## 1. 車載サイバーセキュリティの概要

### なぜ建設機械にもサイバーセキュリティが必要か

建設機械は長らく「閉じたシステム」として運用されてきたため、サイバーセキュリティへの関心が薄い傾向があった。しかし以下の変化により、攻撃対象として無視できない存在になっている。

**接続性の拡大**

| 機能 | 接続先 | リスク |
|------|--------|--------|
| OTA（Over-the-Air）アップデート | クラウドサーバー | 改ざんファームウェアの注入 |
| フリート管理 | 運行管理サーバー | 不正コマンドによる遠隔操作 |
| 遠隔診断 | ディーラー・メーカー | 診断ポートを通じた不正アクセス |
| 人検知・自律制御 | AI処理サーバー | センサーデータの改ざん |

**攻撃の現実性**

- 2020年代以降、農業機械・産業機械を対象としたランサムウェア攻撃が報告されている
- 重機の遠隔操作機能（テレオペレーション）は、攻撃に成功された場合に**人命に直結する**
- 工事現場での不正制御は「建設機械が凶器になる」シナリオを生む

**重要：セキュリティ侵害は安全（Safety）リスクになる**

サイバー攻撃によってブレーキ制御・アーム制御ECUの動作が変化した場合、機能安全（ISO 25119）の前提が崩れる。セキュリティと機能安全は独立した概念ではなく、**セキュリティが侵害されると安全が脅かされる**関係にある。

```
攻撃者の侵入
    │
    ▼
ECUへの不正コマンド送信
    │
    ▼
油圧バルブの誤動作（安全機能の無効化）
    │
    ▼
作業員への危害・機体損壊
```

### ISO/SAE 21434の概要と建設機械への適用

**ISO/SAE 21434とは**

2021年に発行された「道路車両のサイバーセキュリティエンジニアリング」規格。開発ライフサイクル全体（コンセプト〜廃棄）を通じたサイバーセキュリティ管理を義務付ける。

| 章 | 内容 |
|----|------|
| 第5章 | サイバーセキュリティ管理体制 |
| 第6章 | プロジェクト依存サイバーセキュリティ管理 |
| 第9〜15章 | 設計・開発・テスト・運用フェーズの要求 |

**建設機械への適用判断**

ISO/SAE 21434は自動車を対象としているが、建設機械への適用は任意である。ただし以下の理由から**準拠を推奨する**。

- OTA更新機能・遠隔操作機能を搭載した時点で、自動車と同等のリスクが発生する
- 委託先選定時に「ISO/SAE 21434への準拠経験」を要求基準にできる
- 顧客（建設会社・官公庁）からセキュリティ証明を要求されるケースが増加傾向にある

---

### UN/ECE WP.29 R155/R156規制と建設機械の関係

**R155/R156とは**

国連の自動車基準調和世界フォーラム（WP.29）が策定した規制。

| 規制 | 内容 |
|------|------|
| R155 | サイバーセキュリティ管理システム（CSMS）の認証 |
| R156 | ソフトウェアアップデート管理システム（SUMS）の認証 |

**建設機械への直接適用は現時点では不要**

R155/R156は「公道を走行する乗用車・トラック」を対象とした規制であり、建設機械（非公道車両）には現時点で直接適用されない。ただし：

- 将来的な規制拡大の可能性がある（農業機械・建設機械への適用検討が一部国で進行中）
- OTAアップデート機能を搭載する場合、R156相当の管理が事実上必要になる
- 部品・ECUを調達する自動車サプライヤーがR155/R156準拠を要求するケースがある

**推奨対応**：R155/R156の考え方を**参考**として取り込み、特にOTA管理プロセス（R156相当）は整備しておく。

### セキュリティと安全（機能安全）の相互関係

```
┌─────────────────────────────────────────┐
│  機能安全（ISO 25119）                    │
│  「意図しない故障から保護する」           │
│  ・ハードウェア冗長性                     │
│  ・フェイルセーフ動作                     │
│  ・ASIL要求                               │
│                                           │
│  ↑ 前提が崩れる                          │
│                                           │
│  サイバーセキュリティ                     │
│  「意図した攻撃から保護する」             │
│  ・認証・暗号化                           │
│  ・侵入検知                               │
│  ・TARA・CAL要求                          │
└─────────────────────────────────────────┘
```

ISO/SAE 21434は「セキュリティが侵害された場合に、安全側に倒れる設計になっているか」を確認することを求めている（Security-informed Safety Analysis）。

## 2. TARA（脅威分析とリスク評価）の進め方

### TARAとは何か

TARA（Threat Analysis and Risk Assessment）は、「何が攻撃されると困るか」「どれくらい危険か」「どう対策するか」を体系的に整理するプロセスである。

**非専門家向けの説明**

自動車業界のTARAは、建物の防犯設計に例えると分かりやすい。

```
建物の防犯設計            TARAの対応概念
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
守るべき財産を特定         ─→  資産（Asset）の特定
侵入口（窓・ドア）を洗い出す ─→  攻撃対象面（Attack Surface）の特定
どんな泥棒が来るか想定     ─→  脅威（Threat）の分析
被害の大きさを評価         ─→  影響（Impact）の評価
侵入されやすさを評価       ─→  実現可能性（Feasibility）の評価
対策（鍵・警報）を決める   ─→  リスク対策の決定
```

---

### 建設機械の攻撃対象（Attack Surface）の洗い出し

本プロジェクトの構成（Classic AUTOSAR Safety ECU + Linux Gateway）における攻撃対象面を整理する。

#### 物理的な攻撃対象面

| 攻撃対象 | 場所 | リスクの概要 |
|----------|------|-------------|
| OBD/UDSポート（診断コネクタ） | 運転席下部・エンジンルーム | 物理接触により不正診断・制御書き換えが可能 |
| USB・デバッグポート | ECU基板・コンソールパネル | ファームウェア書き換え・シェルアクセス |
| CANバス露出箇所 | ハーネスコネクタ | バスへの接続によりCANメッセージの盗聴・注入 |

#### ネットワーク的な攻撃対象面

| 攻撃対象 | プロトコル | リスクの概要 |
|----------|-----------|-------------|
| CAN/CAN FDバス | CAN | 不正メッセージ注入（制御乗っ取り） |
| J1939バス | J1939 | エンジン・油圧系への不正コマンド |
| 車内Ethernet | TCP/IP、SOME/IP | 中間者攻撃・サービス妨害（DoS） |
| OTA更新経路 | HTTPS/MQTT | 改ざんファームウェア配信 |
| LTE/5G通信 | TCP/IP | クラウドとの通信傍受・偽サーバー誘導 |
| Wi-Fi（整備時接続） | TCP/IP | 認証なし接続による不正アクセス |

---

### TARAの手順

**ステップ1：資産（Asset）の特定**

保護すべき情報・機能を定義する。

| 資産 | 種別 | 保護の観点 |
|------|------|-----------|
| 油圧制御ソフトウェア | 機能資産 | 改ざんによる誤動作防止（Safety） |
| キャリブレーションデータ | データ資産 | 改ざんによる制御精度劣化防止 |
| 診断アクセス認証情報 | データ資産 | 漏洩による不正アクセス防止 |
| 稼働ログ・位置情報 | データ資産 | 漏洩によるプライバシー・競争情報保護 |
| OTA更新パッケージ | 機能資産 | 改ざんによる不正ファームウェア注入防止 |
| CAN通信内容 | データ資産 | 盗聴による制御ロジック解析防止 |

**ステップ2：脅威（Threat）の分析**

STRIDE モデルを使って脅威を体系的に列挙する。

| STRIDEカテゴリ | 意味 | 建設機械での例 |
|----------------|------|--------------|
| Spoofing（なりすまし） | 正規の発信元のふりをする | 正規ECUを装ったCANメッセージ送信 |
| Tampering（改ざん） | データ・ソフトを書き換える | OTAパッケージへの不正コード混入 |
| Repudiation（否認） | 行為の証拠を消す | 攻撃ログの削除 |
| Information Disclosure（情報漏洩） | 機密情報を取り出す | 診断ポートからのカリブレーション値読み出し |
| Denial of Service（サービス妨害） | 正常動作を妨げる | CANバスへのフラッディングで制御不能に |
| Elevation of Privilege（権限昇格） | 本来できない操作をする | 診断セッションを通じた書き込み権限の取得 |

**ステップ3：影響（Impact）の評価**

ISO/SAE 21434では影響を4つの観点で評価する。

| 観点 | 英語 | 評価例（建設機械） |
|------|------|------------------|
| 安全 | Safety | アーム誤動作→作業員への衝突（最大：生命の危機） |
| 財務 | Financial | 機体損壊による修理費・稼働停止損失 |
| 操業 | Operational | 工事現場全体の作業停止 |
| プライバシー | Privacy | 稼働データ・位置情報の漏洩 |

各観点で「重大（Severe）」「重大でない（Major）」「軽微（Moderate）」「無視可（Negligible）」の4段階で評価する。

**ステップ4：実現可能性（Feasibility）の評価**

攻撃がどれくらい現実的かを評価する指標（CVSS・EVITA等を参照）。

| 評価軸 | 内容 |
|--------|------|
| 攻撃経路 | 物理接触が必要か、リモートから可能か |
| 必要な専門知識 | 一般人でも可能か、高度な専門知識が必要か |
| 必要機材 | 特殊機材が必要か、市販品で可能か |
| 攻撃ウィンドウ | 攻撃に要する時間（短時間か長時間か） |

**ステップ5：リスク評価と対策の決定**

影響 × 実現可能性 = リスクレベルとなり、対策方針を決定する。

| リスクレベル | 対策方針 |
|-------------|---------|
| 高（High） | 必須対策（設計変更・実装による対策が必要） |
| 中（Medium） | 推奨対策（コスト・スケジュールと照らし合わせて実施） |
| 低（Low） | 受容または監視継続 |

---

### CAL（Cybersecurity Assurance Level）の判断基準

CALはISO/SAE 21434が定める「セキュリティ対策の厳格さ」のレベルで、CAL 1〜4の4段階がある。

| CAL | 意味 | 適用例（建設機械） |
|-----|------|-----------------|
| CAL 1 | 基本的な対策で十分 | 稼働ログの記録システム（漏洩しても人命への影響なし） |
| CAL 2 | 中程度の対策が必要 | OTA更新機能（改ざんにより機能低下の可能性） |
| CAL 3 | 高度な対策が必要 | 遠隔診断・遠隔操作機能（誤操作による人身事故リスク） |
| CAL 4 | 最高レベルの対策が必要 | 油圧制御ECU（誤動作が直接的に人命に影響） |

CALは前述のTARAの結果（影響レベル）をもとに決定する。CAL 4相当の機能については、設計段階から専門家レビューが必要になる。

### 建設機械での具体的なTARA事例

#### 事例1：遠隔操作ハイジャック

**シナリオ**：攻撃者がLTE/5G通信を通じてフリート管理サーバーを侵害し、遠隔操作コマンドを注入する。

| 項目 | 内容 |
|------|------|
| 攻撃対象 | クラウド〜Gateway ECU間のMQTT通信 |
| 脅威 | Spoofing（偽コマンド注入） |
| 影響：安全 | 重大（Severe）：作業員への重機衝突 |
| 実現可能性 | 中（クラウドサーバーへの侵入が必要） |
| リスク | 高（High） → CAL 3〜4 |
| 対策 | TLS 1.3通信・コマンド認証・人体検知との連動インターロック |

#### 事例2：OTAファームウェア改ざん

**シナリオ**：攻撃者がOTA配信サーバーを侵害し、不正コードを含むファームウェアを配信する。

| 項目 | 内容 |
|------|------|
| 攻撃対象 | OTAサーバー〜Gateway ECU間のHTTPS通信 |
| 脅威 | Tampering（改ざん） |
| 影響：安全 | 重大（機能停止・誤動作の可能性） |
| 実現可能性 | 低〜中（サーバー侵害が必要） |
| リスク | 高（High） → CAL 3 |
| 対策 | コード署名・ハッシュ検証・A/Bパーティション |

#### 事例3：診断ポート悪用

**シナリオ**：整備士を装った攻撃者がOBDポートに接続し、UDSセキュリティアクセスを突破してキャリブレーションデータを書き換える。

| 項目 | 内容 |
|------|------|
| 攻撃対象 | OBD/UDSポート（物理接続） |
| 脅威 | Elevation of Privilege（権限昇格） |
| 影響：安全 | 重大（制御精度の劣化） |
| 実現可能性 | 中（物理接触が必要だが、専用ツールは入手可能） |
| リスク | 高（High） → CAL 3 |
| 対策 | Seed/Key強化・セッション別権限制限・アクセスログ |

## 3. 外部委託における責任分担

### セキュリティ要求を委託先にどう伝えるか

セキュリティ要求は `requirements-spec.md` に記載する機能要求と同様に、明確な「達成基準」とセットで伝える必要がある。曖昧な記述は委託先の解釈に委ねることになり、後工程でのトラブルを生む。

**悪い例（曖昧）**

```
セキュリティ対策を適切に実施すること。
```

**良い例（達成基準つき）**

```
SecOCによるCANメッセージ認証を以下の条件で実装すること。
- 対象メッセージ：油圧制御コマンド（CAN ID: 0x200〜0x2FF）
- MACアルゴリズム：AES-128-CMAC
- Freshness Value長：40ビット以上
- 改ざん検知時の動作：メッセージ破棄・DTC記録・フォールバックモード移行
```

**委託先への提示資料として含めるべき項目**

| 項目 | 内容 |
|------|------|
| TARAの結果 | 攻撃対象・脅威・リスクレベルの一覧（本資料を共有） |
| セキュリティ要求仕様 | CAL別の実装要求（本資料の第4章・第5章を基に作成） |
| テスト要求 | ペネトレーションテスト・ファジングの実施要求（本資料の第6章） |
| 証跡管理要求 | セキュリティ証跡の種類・保管期間・提出形式 |
| インシデント対応要求 | 脆弱性報告窓口・パッチ適用SLA（Service Level Agreement） |

---

### 外部委託先のセキュリティ成熟度の確認方法

`vendor-selection.md` の選定基準にセキュリティ観点を追加する形で、以下を確認する。

**確認項目チェックリスト**

```
□ ISO/SAE 21434に基づく開発実績があるか（納品物・認証書を確認）
□ TUV・BV等の第三者機関によるCSMS認証を取得しているか
□ セキュリティエンジニアが専任で存在するか（兼務ではないか）
□ 過去のプロジェクトで発見された脆弱性の開示プロセスがあるか
□ SBOMの提供実績があるか
□ ペネトレーションテストを外部委託できる体制があるか
```

---

### ソフトウェアSBOM（Software Bill of Materials）の要求

**SBOMとは**

SBOM（エスボム）は、ソフトウェアに含まれるすべてのコンポーネント（OSS・ライブラリ・ツール）の一覧表である。既知の脆弱性（CVE）が報告された際に、**自社製品が影響を受けるかを即座に判断できる**ために必要。

**要求事項として契約に含める内容**

```
委託先は、納品時および更新時にSBOMをSPDX 2.3またはCycloneDX 1.4形式で提供すること。
SBOMには以下の情報を含むこと。
  - コンポーネント名・バージョン・ライセンス
  - 依存関係ツリー
  - 既知の脆弱性（CVE）情報と対応状況
```

**自社側での運用**

受領したSBOMは脆弱性データベース（NVD：National Vulnerability Database）と突合し、影響範囲を定期的に評価する（要確定：評価頻度・担当者）。

---

### 脆弱性開示・パッチ適用プロセスの契約事項

**契約に含めるべき項目**

| 項目 | 内容 | SLA例 |
|------|------|-------|
| 脆弱性報告窓口 | 委託先からの脆弱性報告先（セキュリティ担当者） | - |
| 重大脆弱性報告 | CVSSスコア7.0以上の脆弱性の報告義務 | 発見後48時間以内 |
| パッチ提供SLA | 重大脆弱性のパッチ提供期限 | 報告後30日以内（要確定） |
| サポート期間 | 納品後の脆弱性対応保証期間 | 最低5年（要確定） |
| エスカレーション | パッチ適用が困難な場合のワークアラウンド提供義務 | - |

> 数値はいずれも仮の値。自社の運用体制・機器のライフサイクルに合わせて確定すること。

## 4. Classic AUTOSARにおけるセキュリティ実装

### SecOC（Secure Onboard Communication）の詳細

#### SecOCとは何か

SecOCは、Classic AUTOSARのBSW（Base Software）モジュールの一つで、**CANバス上のメッセージに認証コードを付与する**仕組みである。CANプロトコル自体には認証機能がないため、攻撃者が正規ECUのふりをしてメッセージを送り込める。SecOCはこれを防ぐために設計されている。

```
【SecOCなし】
送信ECU → CANバス → 受信ECU
          ↑
          攻撃者が割り込んで偽メッセージを送れる

【SecOCあり】
送信ECU → [データ + MAC（認証コード）] → CANバス → 受信ECU（MAC検証 OK/NG）
          ↑
          攻撃者はMAC生成に必要な秘密鍵を持っていないので偽造不能
```

#### Message Authentication Code（MAC）の仕組み

MACは「共有鍵 + 送信データ + カウンタ値（Freshness Value）」から計算される認証コードである。

```
MAC = CMAC(AES-128, 秘密鍵, [メッセージデータ | Freshness Value])
```

- **秘密鍵**：送信ECUと受信ECU双方が保持。外部に漏れないようHSMで管理する
- **Freshness Value（鮮度値）**：リプレイ攻撃（過去の正規メッセージを再送する攻撃）を防ぐためのカウンタ。メッセージ送信ごとにインクリメントする

#### SecOC設定パラメータ

| パラメータ名 | 意味 | 推奨値 |
|-------------|------|--------|
| `SecOCFreshnessValueLength` | Freshness Valueのビット長 | 40ビット以上 |
| `SecOCFreshnessValueTxLength` | CANフレームに埋め込むFreshnessのビット長 | 4〜8ビット（下位ビット） |
| `SecOCAuthInfoTxLength` | MACの切り出し長（全MACをCANに乗せるのは困難） | 24〜48ビット |
| `SecOCMacVerifyAttempts` | MAC検証失敗を許容する回数 | 3回（要確定） |
| `SecOCReceptionOverflowStrategy` | 受信バッファあふれ時の動作 | `REJECT`（破棄） |

> Freshness Valueは全ビットをCANフレームに乗せると帯域を圧迫するため、下位数ビットのみ送信し、上位ビットはFVM（Freshness Value Manager）が管理するのが一般的。

#### 鍵管理とHSM（Hardware Security Module）の役割

**HSMとは**

HSMは、ECUの内部チップに内蔵された「鍵専用の金庫」である。CPU（マイコン）とは物理的に分離されたセキュリティドメインで動作し、秘密鍵がソフトウェアから直接読み出せない設計になっている。

```
┌──────────────────────────────────────┐
│ ECUマイコン                           │
│                                      │
│  ┌─────────────┐  ┌───────────────┐  │
│  │ アプリケーション │  │    HSM        │  │
│  │ ソフトウェア  │  │  ・秘密鍵保管  │  │
│  │              │  │  ・MAC計算実行 │  │
│  │  MAC計算要求 ├─►│  ・鍵は外部非公開│ │
│  │  MAC検証要求 │◄─┤  ・鍵更新管理  │  │
│  └─────────────┘  └───────────────┘  │
└──────────────────────────────────────┘
```

**鍵のライフサイクル**

| フェーズ | 処理 | 実施者 |
|----------|------|--------|
| 鍵生成 | HSMの乱数生成器（TRNG）で生成 | 委託先（工場書き込み） |
| 鍵配布 | 工場でのECUプログラミング時に書き込み | 委託先 |
| 鍵更新 | OTAまたは整備ツール経由（要確定：更新頻度・手順） | 自社運用 |
| 鍵廃棄 | 機体廃棄時のHSMリセット | 自社運用 |

> 鍵管理の詳細手順は別途「鍵管理手順書」として作成すること（要確定：担当部門）。

#### SecOC設定のArXML記述例

以下は油圧制御コマンド（CAN ID: 0x201）にSecOCを適用する場合のArXML断片である。実際の値はシステム設計に合わせて確定すること。

### CANバスセキュリティ

#### IDホワイトリスト・レート制限

Gateway ECU（Linux）では、CANバスから受け取るメッセージのフィルタリングを実施する。

**IDホワイトリスト**

許可されたCAN IDのみを通過させ、未知のIDからのメッセージをすべてドロップする。

```c
/* CAN IDホワイトリスト定義例（Gateway ECU側） */
static const uint32_t can_id_whitelist[] = {
    0x200,  /* 油圧コマンド */
    0x201,  /* 走行コマンド */
    0x300,  /* エンジン状態 */
    0x301,  /* 油圧センサー */
    0x700,  /* 診断（UDS） */
};
#define WHITELIST_SIZE (sizeof(can_id_whitelist) / sizeof(can_id_whitelist[0]))
```

**レート制限（Rate Limiting）**

CANバスへのフラッディング攻撃（DoS）を防ぐため、各CAN IDのメッセージ頻度を監視し、閾値を超えた場合にドロップ・アラームを発する。

```
例：CAN ID 0x200（油圧コマンド）の正常送信周期 = 10ms
    受信レート閾値 = 200msg/秒（正常の2倍）
    閾値超過時 → メッセージドロップ + DTC記録 + ログ出力
```

#### CAN IDゲートウェイフィルタリング

CAN/J1939からEthernet（SOME/IP）への変換を行うGateway ECUは、**変換ルールに存在しないCAN IDをEthernetに転送しない**設計とする。これにより、CAN側に侵入した攻撃者がEthernet経由でクラウドに情報を送信する経路を遮断できる。

### 診断ポートのアクセス制御

#### セキュリティアクセス（UDS 0x27）のSeed/Key設計

UDS（Unified Diagnostic Services）のサービス0x27は「セキュリティアクセス」であり、特権操作（フラッシュ書き換え・制御値変更）の前に認証を行う仕組みである。

**Seed/Keyの動作フロー**

```
テスター（診断ツール）                ECU
      │                               │
      │── 0x27 01（Seed要求） ────────►│
      │                               │（乱数Seedを生成）
      │◄─ 0x67 01 [Seed値] ──────────│
      │                               │
      │（KeyをSeed + 秘密鍵から計算） │
      │                               │
      │── 0x27 02 [Key値] ───────────►│
      │                               │（Keyを検証）
      │◄─ 0x67 02（アクセス許可） ───│
      │                               │
```

**Seed/Key設計の注意点**

| 問題 | 対策 |
|------|------|
| Seedが予測可能（固定値・連番） | HSMの乱数生成器（TRNG）を使用する |
| Keyアルゴリズムが公開・解析可能 | アルゴリズムをHSMに格納し、外部から読み出し不能にする |
| ブルートフォース攻撃 | 連続失敗回数制限（3回失敗で30分ロック等・要確定） |
| リプレイ攻撃 | Seedは一度使用したら無効化する（使い捨て） |

#### 診断セッション別の操作制限

UDSは「診断セッション」ごとに操作権限を分離する。建設機械では以下の設計を推奨する。

| セッション | UDS ID | 許可操作 | アクセス者 |
|------------|--------|---------|-----------|
| デフォルトセッション | 0x01 | DTC読み出し・状態確認のみ | 誰でも（認証不要） |
| 拡張診断セッション | 0x03 | キャリブレーション読み出し | セキュリティアクセス後 |
| プログラミングセッション | 0x02 | フラッシュ書き換え | セキュリティアクセス後（強度の高いKey） |

> プログラミングセッションへの移行は、OTA経由ではなく物理接続のみに制限することを推奨する（要確定：運用方針）。

## 5. Linuxシステムのセキュリティ

### OTAアップデートのセキュリティ

#### コード署名によるファームウェア検証

OTAで配信されるファームウェアには**デジタル署名**を付与し、ECU側で署名検証を行ってから適用する。

```
【配信側（クラウド）】
ファームウェア ──→ SHA-256ハッシュ計算
                       │
                       ▼
             RSA-4096またはECDSA-P256で署名
                       │
                       ▼
         [ファームウェア + 署名] をOTAサーバーに配置

【受信側（Gateway ECU）】
[ファームウェア + 署名] を受信
         │
         ▼
公開鍵（HSMに保管）で署名を検証
         │
    OK ──┤── NG（改ざん）→ 適用拒否・アラーム
         ▼
フラッシュへの書き込み
```

**署名用秘密鍵の管理**

- 配信用署名秘密鍵はクラウド側のHSM（または AWS KMS / Azure Key Vault）で管理する（要確定）
- Gateway ECUに書き込まれた公開鍵の更新手順（鍵ローテーション）を事前に設計しておくこと

#### A/Bパーティション構成

OTA適用中の電源断や適用失敗からの回復を保証するため、A/Bパーティション構成を採用する。

```
ストレージ構成：
┌─────────────────────────────────┐
│  Boot Partition（Bootloader）   │ ← 変更不可（書き込み禁止）
├─────────────────────────────────┤
│  パーティション A（現在稼働中） │ ← 変更不可（稼働中）
├─────────────────────────────────┤
│  パーティション B（更新対象）   │ ← OTAで書き込み
├─────────────────────────────────┤
│  Data Partition（設定・ログ）   │ ← 共有
└─────────────────────────────────┘

更新フロー：
1. Bパーティションに新ファームウェアを書き込み
2. 署名検証・整合性チェック
3. Bootloaderに「次回はBから起動」を指示
4. 再起動 → Bで起動
5. 起動成功後、AとBを入れ替え（コミット）
6. 失敗時 → Bootloaderが自動でAに戻す（ロールバック）
```

### クラウド通信のセキュリティ

#### TLS 1.3の適用

Gateway ECU〜クラウド（AWS IoT Core / Azure IoT Hub）間の通信はTLS 1.3を使用する。

| 設定項目 | 推奨値 |
|----------|--------|
| TLSバージョン | 1.3（1.2以下は無効化） |
| 暗号スイート | TLS_AES_256_GCM_SHA384 / TLS_CHACHA20_POLY1305_SHA256 |
| 証明書 | デバイス個別X.509証明書（共有証明書は禁止） |
| 証明書有効期間 | 2年以内（要確定） |

#### デバイス証明書の管理

各機体（Gateway ECU）には個別のX.509証明書を発行し、機体識別と通信認証に使用する。

```
証明書ライフサイクル：
工場 → 証明書発行（自社PKI or AWS ACM PCA） → ECU書き込み
運用中 → 有効期限が90日前になったらOTA経由で更新
機体廃棄 → 証明書の失効申請（CRLまたはOCSP）
```

## 6. セキュリティテストと証跡管理

### ペネトレーションテスト

ペネトレーションテスト（ペンテスト）は、専門の攻撃者視点から実際に侵入を試み、脆弱性を発見するテストである。

**委託先への要求内容**

| 項目 | 内容 |
|------|------|
| 実施タイミング | 量産前（ソフトウェアフリーズ後）・OTA更新後 |
| 実施者 | 第三者セキュリティ機関（委託先とは別会社） |
| 対象スコープ | 全攻撃対象面（第2章のAttack Surface一覧） |
| 報告形式 | 発見した脆弱性・CVSSスコア・再現手順・推奨対策 |
| 対応SLA | 重大脆弱性は報告後30日以内にパッチ提供（要確定） |

**テスト項目例**

```
【物理アクセステスト】
□ OBD/UDSポートへの接続による不正診断コマンド実行
□ CANバスへの直接接続による不正メッセージ注入
□ USBポートへのデバイス接続によるシェルアクセス試行

【ネットワークテスト】
□ MQTTブローカーへの不正接続試行
□ TLS証明書の検証バイパス試行
□ OTAパッケージの改ざんと適用試行

【アプリケーションテスト】
□ UDSセキュリティアクセス（0x27）のブルートフォース
□ DiagnosticSessionControlによる権限昇格試行
□ SecOCのMACバイパス試行
```

## 7. インシデント対応計画

### セキュリティインシデントの検知・対応フロー

```
【検知フェーズ】
ECU異常（DTC）・CANバス異常 ─→ Gateway ECUがログ収集
クラウド側異常検知 ─→ アラート発報
現場作業員からの異常報告 ─→ 整備部門へ連絡

          │
          ▼

【初動対応フェーズ（24時間以内）】
１. 影響範囲の特定（どの機体・どの機能が影響を受けているか）
２. 稼働継続可能かの判断（Safety継続可能か・作業停止が必要か）
３. 証拠の保全（ログの取得・上書きの防止）

          │
          ▼

【分析フェーズ（72時間以内）】
４. 攻撃経路の特定（どこから・何が起きたか）
５. 影響の全体像の把握

          │
          ▼

【回復フェーズ】
６. パッチ適用またはロールバック
７. 再発防止策の検討
８. 社内報告・必要に応じた顧客通知
```

---

### 現場での不正アクセス検知方法

現場の作業員が気づくべき「不審なサイン」を整備マニュアルに含める。

| 不審なサイン | 想定される原因 |
|-------------|-------------|
| 機体の動作が指示と異なる | 遠隔操作の乗っ取り・CANメッセージ注入 |
| 診断ツール未接続なのにDTCが大量発生 | CANバスフラッディング攻撃 |
| OTAアップデートが身に覚えなく完了している | 不正OTAの適用 |
| クラウドダッシュボードに未知の接続元が表示される | 不正なデバイス接続 |
| HMIに見知らぬ画面・メッセージが表示される | マルウェア感染 |

---

### リモートからのソフトウェア無効化・隔離手順

深刻なインシデントが発生した場合、リモートから機体を安全な状態に移行させる手順を事前に設計しておく。

**緊急隔離の手順（要確定：実装設計が必要）**

```
【クラウドコンソールからの操作】
１. フリート管理画面で対象機体を選択
２. 「緊急隔離」コマンドを送信
３. Gateway ECUが受信後：
   - クラウドとの通信を切断（自己隔離）
   - CAN→クラウド転送を停止
   - OTAの受信を拒否
   - フォールバックモードへの移行（最小限の安全動作のみ）

【フォールバックモードの動作】
- 遠隔操作コマンドを一切受け付けない
- 現場オペレーターによるローカル操作のみ許可
- 油圧・走行の最低限の安全制御は継続
```

> 緊急隔離コマンドも認証が必要な設計にすること（攻撃者による悪用を防ぐ）。

### Phase 1（今〜2年）：基本対策の実装

**目標**：最も影響の大きい攻撃対象面をふさぎ、被害を最小化できる基盤を作る。

| 施策 | 担当 | 完了目標 |
|------|------|---------|
| SecOCの要求仕様作成・委託先への伝達 | 自社技術担当 | 要確定 |
| 油圧制御コマンドへのSecOC適用（CAL 4相当） | 委託先 | 要確定 |
| UDS 0x27 Seed/Keyの強化（HSM使用・失敗回数制限） | 委託先 | 要確定 |
| OTAファームウェア署名の実装 | 委託先 | 要確定 |
| A/Bパーティション構成の実装 | 委託先 | 要確定 |
| TLS 1.3によるクラウド通信の実装 | 委託先 | 要確定 |
| CAN IDホワイトリスト・レート制限の実装 | 委託先 | 要確定 |
| Linuxハードニング（不要サービス削除・SSH強化） | 委託先 | 要確定 |
| 委託先へのSBOM提供要求の契約化 | 自社調達担当 | 要確定 |

---

### Phase 2（2〜5年）：TARA実施・ISO/SAE 21434への本格対応

**目標**：体系的なリスク評価と規格準拠を達成し、顧客・規制当局に証明できる体制を作る。

| 施策 | 担当 | 完了目標 |
|------|------|---------|
| TARA（第2章の全機能版）の実施 | 自社 + 外部セキュリティ専門家 | 要確定 |
| ISO/SAE 21434に基づくサイバーセキュリティ計画の策定 | 自社 | 要確定 |
| 委託先のCSMS評価（ISO/SAE 21434準拠確認） | 自社調達 | 要確定 |
| 脆弱性管理プロセスの確立（SBOMの定期チェック） | 自社技術担当 | 要確定 |
| 初回ペネトレーションテストの実施 | 外部セキュリティ機関 | 要確定 |
| インシデント対応計画の整備・訓練 | 自社全体 | 要確定 |
| 証跡管理基盤の整備（クラウドログ集約） | 自社 + 委託先 | 要確定 |

---

### Phase 3（5〜10年）：IDS・セキュリティ運用センターの確立

**目標**：常時監視体制（CSOC）を構築し、新たな脅威に継続的に対応できる組織を作る。

| 施策 | 内容 |
|------|------|
| 車載IDS（Intrusion Detection System）の導入 | CANバスの異常パターン・通信量を常時監視し、自動アラートを発する |
| CSOC（Cyber Security Operation Center）の構築 | 全フリートのセキュリティイベントを一元管理する監視センター（外部委託または内製） |
| 継続的脆弱性管理 | CVEデータベースとSBOMの自動照合・パッチ優先度の自動算出 |
| レッドチーム演習 | 年次でのペネトレーションテスト・インシデント対応訓練の定例化 |
| Adaptive AUTOSAR検討 | Adaptive AUTOSARが提供するセキュリティ機能（IAM・TLS統合）の評価 |
