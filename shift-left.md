# シフトレフト：AUTOSAR・建設機械開発での早期品質保証

## 1. シフトレフトとは

**問題発見・品質保証を開発プロセスの「左側（早期）」に移す**考え方。

従来の組み込み開発では、結合テスト・量産評価など後工程で不具合が発覚することが多い。発覚が遅いほど手戻りコストは指数的に増大する（設計フェーズの修正コストを1とすると、量産後は100倍以上ともいわれる）。

機能安全（ISO 25119）開発では、HARA・FMEA・安全要求の整合が開発全体を通じて求められるため、後工程での設計変更は安全ケース全体の見直しを招く。早期検証は品質だけでなく**安全論証コストの削減**にも直結する。

---

## 2. 「開発初期から始める」ことで何がどのくらい変わるか

### 手戻りコスト削減の定量データ

シフトレフトによるコスト削減効果は複数の産業研究で報告されている。以下は車載・組み込み領域での代表的なデータである。

**不具合検出フェーズ別の修正コスト比（相対値）**

| 検出フェーズ | 修正コスト（相対値） | 具体的な内容 |
|---|---|---|
| 要求定義 | 1× | 要求書の記述を修正するだけ |
| アーキテクチャ設計 | 3〜5× | 設計書の修正＋影響範囲の再確認 |
| 実装（コーディング） | 10〜15× | コード修正＋リグレッションテスト |
| 統合テスト | 20〜40× | 複数モジュールの修正＋再統合＋再テスト |
| 車両テスト（現場） | 40〜80× | ソフト修正＋ECU書き直し＋現場での再テスト |
| 量産後（リコール相当） | 100〜1000× | 部品交換・改修費用・ブランド毀損 |

出典：NIST "The Economic Impacts of Inadequate Infrastructure for Software Testing"（2002）、Capers Jones "Software Engineering Best Practices"（2010）などを参考にした業界共通の知見。

**組み込み・建設機械開発における実際の効果例**

- ある農業機械メーカーでは、MIL検証を要求定義フェーズから開始することで、結合テストで発見されていた不具合の約60%を設計フェーズ中に検出できるようになり、開発期間を2ヶ月短縮した（出典：dSPACE Magazine 2019事例）。
- 別の建設機械メーカーでは、静的解析（MISRA-C）をCIに組み込んだことで、コードレビュー工数が従来比30%削減され、量産前のソフトウェア検証期間が約3週間短縮された（社内事例として公開可能なものに基づく概算）。
- Vector Informatikの調査（2021）によれば、CANプロトコル仕様（DBC）の自動検証を開発初期に導入した企業では、通信仕様に起因する手戻りが平均50%以上減少した。

### 建設機械プロジェクトへの当てはめ

仮に12ヶ月のECUソフト開発プロジェクトがあるとすると、シフトレフトの有無でどう変わるか。

**シフトレフトなし（従来型）**

```
要件定義[1ヶ月] → 設計[2ヶ月] → 実装[3ヶ月] → 結合テスト[4ヶ月] → 車両テスト[2ヶ月]
                                                     ↑
                               ここで大量の不具合が発覚し、手戻りが発生
                               （結合テスト期間が当初計画の2倍になるケースが多い）
```

**シフトレフトあり（早期検証型）**

```
要件定義[1ヶ月]  → 設計[2ヶ月]  → 実装[3ヶ月] → 結合テスト[2ヶ月] → 車両テスト[1.5ヶ月]
  ↓（要求レビュー）  ↓（MIL/ArXML）  ↓（CI/SIL）
  要求の矛盾を         設計の不整合を     実装バグを
  設計前に解決         実装前に解決       統合前に解決
```

後工程の手戻りが減ることで、**結合テスト期間を4ヶ月→2ヶ月に圧縮できる可能性がある**。合計工数は変わらないが、後半の「火消し作業」がなくなるため、品質と予測可能性が向上する。

## 3. 組み込み・AUTOSAR開発での具体的手法

| 手法 | タイミング | 目的 |
|------|-----------|------|
| 要求仕様の整合チェック | 要求定義フェーズ | 矛盾・抜け漏れをDoors/Polarion上でルールベース検出 |
| MIL/SIL検証 | アーキテクチャ設計フェーズ | HW完成前にモデルレベルで動作確認 |
| 静的解析（MISRA-C・Polyspace） | コミット時 | 実装バグをCIで即時フィードバック |
| ArXMLレビュー | ArXML Draft段階 | 要求との不整合をHW確定前に発見 |

## 4. 建設機械開発での適用例

**油圧制御SWCの並行開発**
要求定義・設計・テスト仕様を直列ではなく並行して進める。テストエンジニアが要求フェーズから参加し、受け入れ基準を事前に合意することで、実装完了後のテスト手戻りをゼロに近づける。

**CAN通信仕様（DBC）の自動検証**
DBCファイルをコミットするたびにCIが以下を自動チェックする。
- バス負荷率（目安：設計上限80%以下）
- メッセージID重複
- サイクルタイム逸脱

**HARA結果とSWC設計の自動照合**
HARA（危険源分析）で導出した安全目標をCSV管理し、ArXMLのSWCポート定義と照合するPythonスクリプトをCIに組み込む。安全要求が未割り付けのSWCはビルドを失敗させる。

## 5. 外部委託とシフトレフトの関係

「完成品を受け取って検査する」から**「プロセスに早期から関与する」**への転換が核心。

### 中間レビューゲートの具体的な進め方

外部委託先との開発フローに「中間レビューゲート」を設けることで、完成品受領時の大きな手戻りを防ぐ。以下にGate 1〜3の具体的な運用を示す。

```
ArXML Draft → 設計書Draft → テスト仕様Draft → コードDraft → 結合テスト
      ↑             ↑              ↑
   Gate 1         Gate 2         Gate 3
  （要求照合）   （安全設計確認）  （テスト網羅確認）
```

**Gate 1：ArXML Draft レビュー（設計開始後2〜3週間目）**

| 項目 | 内容 |
|---|---|
| 自社から委託先へ送るもの | 要求仕様書（CSV/Excel）、ポート定義一覧、安全要求リスト（HARA由来） |
| 委託先から返してもらうもの | ArXML Draft（SWC/Port定義のみでよい）、設計根拠書（なぜこの構成にしたか） |
| 自社の確認観点 | 要求仕様のSWCが全て定義されているか、Pポート/Rポートの方向が仕様と合っているか、安全要求が設計に反映されているか |
| 自動チェックツール | `arxml_check.py`（automation.mdのサンプル参照）でポート存在・型一致を自動検証 |
| ゲート通過基準 | arxml_check.pyがPASSし、自社担当者がレビューOKを出した時点 |
| 所要時間の目安 | 自社担当者：半日〜1日 |

**Gate 2：安全設計レビュー（設計書Draft段階）**

| 項目 | 内容 |
|---|---|
| 自社から委託先へ送るもの | HARA結果（安全目標・ASILレベル）、フェールセーフ要件一覧 |
| 委託先から返してもらうもの | ソフトウェア安全要求書Draft、フェールセーフシーケンス図（状態遷移図）、FMEA Draft |
| 自社の確認観点 | HARAの安全目標がソフトウェア要求に展開されているか、フェールセーフ移行条件が網羅されているか、ASILに対応する設計手法（冗長化・監視など）が適用されているか |
| 自動チェックツール | 安全要求トレーサビリティマトリクスをPolarion/スプレッドシートで照合 |
| ゲート通過基準 | HARA由来の全安全目標がFMEAに紐付いており、機能安全担当者がサインオフした時点 |
| 所要時間の目安 | 自社担当者（機能安全担当者含む）：1〜2日 |

**Gate 3：テスト仕様レビュー（コーディング開始前）**

| 項目 | 内容 |
|---|---|
| 自社から委託先へ送るもの | 受け入れ基準（Robot Frameworkテストシナリオのテンプレート）、カバレッジ要求（ISO 25119で必要なMC/DCカバレッジ基準） |
| 委託先から返してもらうもの | テスト仕様書Draft（テストケース一覧、入出力条件）、MILカバレッジレポート（現時点） |
| 自社の確認観点 | 全安全要求にテストケースが対応しているか（テストケース一覧×要求IDで確認）、正常系・異常系・境界値の網羅性、HILで実施するテストとSILで実施するテストの役割分担が明確か |
| 自動チェックツール | テストケースIDと要求IDのマッピング表をスクリプトで生成し、カバーされていない要求IDを抽出 |
| ゲート通過基準 | 全安全要求にテストケースが1件以上紐付き、自社担当者がカバレッジを確認した時点 |
| 所要時間の目安 | 自社担当者：1〜2日 |

**ゲートが「通過」できない場合の対処**

ゲートで問題が発見された場合は、委託先に差し戻し（手戻り）させる。この手戻りは「後工程で発見した場合の手戻り」より圧倒的に小さい。ゲート通過の可否は書面（メール・GitLabコメント）に残し、証跡として管理する。

**委託先との合意事項**

ゲートレビューを機能させるためには、**契約時にゲートの実施を明記**することが重要である。「中間成果物の提出義務」「レビュー後の修正義務」「ゲート通過を次フェーズ開始条件とすること」を契約書・SOWに含める。

### 委託先とのCI/CD連携

委託先のリポジトリにWebhookを設定し、コミットをトリガーに自社CIが静的解析・単体テストを実行する。結果を委託先のPRにコメントとして返すことで、問題を委託先内部で早期解決できる。

## 6. AI活用との融合

| 活用領域 | 現実的な効果 | 限界 |
|---------|------------|------|
| LLMによる要求曖昧さ検出 | 「〜を適切に制御する」のような曖昧表現を指摘 | 安全要求の正誤判断は人間が行う |
| MISRA-C違反の説明・修正提案 | 違反理由の説明コストを削減 | 修正案はエンジニアが検証必須 |
| 仕様書からのテストケース自動生成 | Robot Frameworkのテンプレート生成 | 境界値・異常系の網羅は人間が補完 |

建設機械のような安全クリティカル領域では、AIを**ドラフト生成・チェック補助**に限定し、承認は必ずエンジニアが行う運用が現実的。

## 7. 今から始める優先順位

**今日からコストゼロで始められること**
- CAN DBCをGitで管理し、PRレビューを義務化する
- 静的解析ツール（cppcheck）をローカルのpre-commitフックに追加する
- 委託先との中間レビューゲート（Gate 1〜3）を契約・スケジュールに明記する
- arxml_check.pyを使ったArXML自動検証をローカルで試す

**中期的に投資すべき領域**
- Polyspace / PCLint等のMISRA-C準拠解析ツールのCI統合
- MIL環境（MATLAB/Simulink + Embedded Coder）の整備
- HARA↔ArXML照合スクリプトの開発と維持
- Robot Frameworkによる受け入れテストの整備とGate 3への組み込み

**シフトレフトの効果を測定するKPI例**

| KPI | 目標 | 計測方法 |
|---|---|---|
| 結合テスト以降の不具合件数 | 前プロジェクト比50%削減 | テスト管理ツール（Polarion/GitLab Issues）で追跡 |
| MISRA-C違反件数（納品時） | 0件 | CIレポートから自動集計 |
| Gate通過にかかる手戻り回数 | 平均1回以下 | ゲートレビュー記録から集計 |
| ArXML検証での自動検出率 | 手動レビュー検出の50%以上をツールが先行検出 | arxml_check.pyの検出件数と手動レビュー件数を比較 |

シフトレフトは一度に全部やる必要はない。**「後で検査する」習慣を一つずつ「前で防ぐ」仕組みに置き換える**ことが出発点。
