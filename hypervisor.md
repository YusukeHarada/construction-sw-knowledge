# SoC Hypervisorによる統合アーキテクチャ

**作成日：** 2026年6月16日  
**対象読者：** 技術担当者・アーキテクト  
**資料種別：** 社内検討用（社外秘）

---

## 概要

Classic AUTOSAR（RTOS系）とAdaptive AUTOSAR（Linux系）を単一SoC上のHypervisorで統合するアーキテクチャについて整理する。物理的に別SoCで分離する構成（デュアルSoC）との比較も含める。

---

## 1. Hypervisorとは

Hypervisorとは複数のOSを同一ハードウェア上で安全に分離・共存させるソフトウェア層。車載・組み込み向けにはType 1（ベアメタル型）が主流。

```
┌─────────────────────────────────┐
│  Classic AUTOSAR    │  Linux    │
│  （RTOS制御系）      │  （HMI・  │
│  ASIL-D              │  通信系）  │
│                     │  QM/ASIL-B│
├─────────────────────┴───────────┤
│         Hypervisor（Type 1）     │
│  干渉からの自由（FFI）を保証     │
├─────────────────────────────────┤
│            SoC ハードウェア       │
│  （NXP S32G / Renesas R-Car等）  │
└─────────────────────────────────┘
```

### Type 1 vs Type 2

| 種別 | 動作位置 | 特徴 | 車載採用 |
|---|---|---|---|
| **Type 1（ベアメタル）** | HW上で直接動作 | FFI保証・ASIL認証可・高リアルタイム性 | 主流 |
| **Type 2（ホストOS上）** | ホストOS上で動作 | 性能・安全性で劣る | 車載安全系には不向き |

建設機械プロジェクトにHypervisorを採用する場合、製品選定は開発体制・コスト・認証要件の3軸で判断する必要がある。

| 観点 | QNX Hypervisor for Safety | INTEGRITY-178 tuMP | XEN Project |
|------|--------------------------|-------------------|-------------|
| **主な使用場面** | コックピット・ADAS・ゲートウェイ。安全系とHMI/通信系の同一SoC統合に最も実績が多い。OEMのAUTOSAR案件での採用例が豊富 | 航空・防衛・医療など最高安全等級が必要な用途。DO-178C（航空）認証が強み。自動車でも採用実績あり | クラウドサーバー仮想化が本来の用途。自動車向けでは研究開発・プロトタイプ段階での利用が主。安全認証不要のInfotainment分離程度が現実的 |

| **建設機械への推奨度** | ◎ 将来の統合アーキテクチャ候補として最有力 | △ コスト・人材確保の観点から現実的でない | × 安全認証が必要な建機用途には不適 |

## 3. Classic + Adaptive を統合するメリット・デメリット

### メリット

- 単一SoC上でCAN/LIN制御（Classic）とEthernet/SOME-IP通信・AI処理（Adaptive/Linux）を共存
- SoC数削減によるコスト低減（デュアルSoC比で約20%コスト削減との試算あり）
- 配線・消費電力・基板スペースの削減
- OTA更新をAdaptive側で柔軟に対応できる

### デメリット

- Hypervisor自体のASIL認証・検証コストが高い
- リアルタイム性（レイテンシ・ジッタ）の保証がType 1でも課題になることがある
- ツールチェーン・デバッグが複雑化し、開発期間が伸びる傾向
- SoC故障で全系が停止するリスク

## 4. 機能安全とHypervisorの関係

### ASIL分解（Decomposition）

ASIL-Dの安全要求をASIL-B + ASIL-Bの独立チャネルに分割する手法。Hypervisorがゲスト間の干渉遮断（FFI）を保証することで成立する。

```
ASIL-D 安全要求
    ↓ 分解
ASIL-B チャンネルA（Classic AUTOSAR RTOS）
    ＋
ASIL-B チャンネルB（Adaptive AUTOSAR / Linux）
    ↑
Hypervisorが二者間のFFIを保証
```

実用例：Volvo EX90のボディコントローラがこのASIL混在構成を採用。

### ISO 25119（建設機械）への適用

ISO 25119はISO 26262をベースに農機・建機向けに定義されており、Hypervisorを用いたFFI実証はISO 25119でも適用可能と解釈されつつある。ただし**ISO 25119認証実績を持つHypervisor製品はほぼ存在しない**のが現状で、ISO 26262認証からの転用・論拠付けが必要になる。

## 5. 物理分離（デュアルSoC）との比較

| 観点 | 単一SoC + Hypervisor | デュアルSoC物理分離 |
|---|---|---|
| コスト | 低い（ハード削減） | 高い（SoC・基板追加） |
| FFI証明 | Hypervisor認証で対応 | 物理分離で自明 |
| リアルタイム性 | Hypervisorオーバーヘッドあり | 独立保証しやすい |
| 開発複雑度 | 高い（Hypervisor統合が必要） | 低い（独立して開発可能） |
| 障害影響範囲 | SoC故障で全系停止リスク | 分離によりフェールセーフしやすい |
| デバッグ | 複雑（クロスVM追跡が必要） | 比較的容易 |

### 「物理分離が今は現実的」である根拠

現時点（2026年）において**デュアルSoC物理分離のほうが現実解である**理由を以下に詳述する。単純にコストや安全性だけの問題ではなく、複数の構造的な要因が重なっている。

**（1）ISO 25119認証の論拠構築が困難**

HypervisorのFFI保証は「同一SoC上でゲスト同士が干渉しないこと」をソフトウェア的・論理的に証明する。ISO 26262ではQNX等のASIL-D認証済みHypervisorを用いることでこの証明が成立するが、ISO 25119ではHypervisorを用いた認証事例がほぼない。認証機関（TÜV等）との交渉・論拠構築に数年単位の工数がかかる可能性があり、プロジェクトスケジュールに収まらない。一方、物理的に別SoCに分離した場合、「2つのプロセッサが電気的に独立しているため干渉しない」という証明は直感的かつ審査機関に受け入れられやすい。

**（2）開発チームのHypervisorスキルセットが未成熟**

HypervisorでClassicとLinuxを統合するためには、Hypervisor自体の設定・検証に加え、共有リソース（割り込みコントローラ・DMAコントローラ・メモリ帯域）の競合分析、クロスVM間通信（virtio / shared memory）の設計、そしてHypervisorを含む統合デバッグ環境の整備が必要になる。これらは自動車OEMでも習熟に3〜5年を要するスキルセットであり、国内の建設機械OEMや委託先では経験者が極めて少ない。スキル習得コストを含めると、Hypervisor統合の「ハードウェアコスト削減効果」が開発工数増加によって相殺される。

**（3）デバッグ・故障解析の複雑化**

建設機械は現場でのダウンタイムが大きな損失につながる。単一SoC + Hypervisor構成では、フィールドで発生した不具合の原因がClassic側・Linux側・Hypervisor層のどこにあるかを特定するのが難しく、JTAG等のデバッグツールが複数のゲストOSを横断して動作する必要がある。デュアルSoC構成であれば、問題のあるSoCを切り離してそれぞれ独立に解析できるため、現場サービスエンジニアの対応もシンプルになる。

**（4）SoC単一障害点リスク**

単一SoCが故障した場合、Classic側（油圧制御）とLinux側（HMI・通信）が同時に停止する。建設機械では少なくとも「安全な停止状態への移行」と「基本的な警告表示」が物理的に独立して動作することが求められる場面があり、単一SoCはそのフェールセーフ設計を複雑にする。デュアルSoCであれば、Linux側SoCが故障しても、Classic側が安全停止シーケンスを単独で実行できる。

**（5）実績の非対称性**

John Deere・CNH Industrial・Volvo CEなど建設機械に近い農機・作業機メーカーでのHypervisor統合事例は、2026年時点でまだ研究・先行開発段階が多い。一方、Classic AUTOSAR ECU + LinuxゲートウェイのデュアルSoC構成はZF・ボッシュ・日立Astemoなど主要Tier1で量産実績があり、委託先調達が容易である。「初物」のリスクを避けることが建設機械の長期保守（20〜30年）の観点からも合理的判断となる。

## 6. 建設機械への採用可能性

### 適用が有望なケース

- 自律化（センサー融合・AI処理）とリアルタイム制御を同一筐体に搭載する次世代機
- 電動化によりECU統合のニーズが高まる機種
- スペース・重量・消費電力制約が厳しいコンパクト機

### 現状の課題

- 商用HypervisorのライセンスコストがECU単価に対して高い
- 建設機械OEMのHypervisorツールチェーン整備が未成熟
- ISO 25119認証実績を持つHypervisor製品がほぼ存在しない
- 過酷な電源品質（電圧変動・サージ）への対処がSoC選定で必要

### 農機での先行事例

農機オペレーションターミナル上でHypervisorを用いた安全系・非安全系の分離実行の特許が存在する。建設機械への展開はこれに続く形で進む可能性がある。

---

## 7. 建設機械における推奨判断

| 状況 | 推奨構成 |
|---|---|
| **現時点（2026年）・安全性優先** | デュアルSoC物理分離（Classic AUTOSAR ECU + Linux ECU）。実績・安全性・開発難易度のバランスが最良 |
| **2028年以降・コスト・スペース優先** | 認証済みHypervisor（QNX等）を用いた単一SoC統合を検討。ただしISO 25119認証の論拠構築が前提 |
| **自律化・AI処理が本格化する次世代機** | Hypervisor統合が有望。NXP S32G + QNX Hypervisorの組み合わせが現時点の有力候補 |

**現実的な結論：** 今すぐHypervisor統合に踏み込む必要はない。まずClassic AUTOSAR + Linuxのデュアル構成で土台を作り、Hypervisor統合は次世代機の設計段階で検討するロードマップが妥当。Hypervisor採用を検討する際は、ISO 25119認証の論拠構築・チームのスキル習得・ツールチェーン整備を先行して進め、技術的な準備が整った状態で採用判断を行うことが重要である。
