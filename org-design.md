## 1. コンウェイの法則とソフトウェアアーキテクチャ

### 1.1 コンウェイの法則とは

1967年にメルヴィン・コンウェイが提唱した経験則。

> **「システムを設計する組織は、その組織のコミュニケーション構造を模倣した設計を生み出してしまう」**
>
> — Melvin Conway, 1967

言い換えると、「組織の構造がそのままソフトウェアのアーキテクチャに出る」ということである。エンジニアが意図して設計しても、日常のコミュニケーションや意思決定の経路がモジュール境界や通信仕様に反映されてしまう。

### 1.2 車載ソフトウェアでの具体例

従来の自動車・建設機械メーカーでよく見られる問題：

```
【チーム編成（ECU単位）】
  エンジン制御チーム
  油圧制御チーム
  HMIチーム
  ゲートウェイチーム
       ↓（コンウェイの法則が働く）
【生成されるアーキテクチャ】
  エンジンECU ←→ 油圧ECU ←→ HMI ←→ GW
    ↑             ↑            ↑        ↑
  独自CAN信号   独自CAN信号   独自仕様   未定義インタフェース
```

各チームが自分のECU内部を最適化しようとするが、チーム間の調整コストが高いため通信インタフェースが場当たり的に決まる。結果として：

- 信号の重複定義（油圧チームとエンジンチームが同じ物理量を別の信号名で送る）
- CANの帯域圧迫（調整なく信号を追加し続ける）
- 統合テストで初めて不整合が発覚する

これはコードの品質問題ではなく、**組織設計の問題**である。

### 1.3 逆コンウェイ戦略（Inverse Conway Maneuver）

コンウェイの法則は「組織 → アーキテクチャ」という方向に作用する。逆コンウェイ戦略は、この関係を意図的に利用する。

> **「望ましいアーキテクチャを先に決め、そのアーキテクチャが自然に生まれるような組織設計を行う」**

手順：
1. ターゲットアーキテクチャ（理想のシステム構造）を定義する
2. そのアーキテクチャのモジュール境界に合わせてチームを編成する
3. チーム間のインタフェースを明確にAPIとして定義する
4. 各チームが独立して開発・テストできる状態を目指す

### 1.4 建設機械への適用例

建設機械において目指すアーキテクチャが「Classic AUTOSAR（安全制御）＋ Linux（HMI・通信）のハイブリッド」であれば、組織もそれに合わせて設計する。

```
【ターゲットアーキテクチャ】
  ┌────────────────────────────────────┐
  │  Classic AUTOSAR 側（RTOS）         │
  │  ┌──────────┐  ┌──────────────┐   │
  │  │ 油圧SWC  │  │ エンジン制御  │   │
  │  │  チーム  │  │   SWCチーム  │   │
  │  └────┬─────┘  └──────┬───────┘   │
  │       └─────┬──────────┘           │
  │         ┌───┴────┐                 │
  │         │ BSW/RTE │ ← Platform Team│
  │         └───┬────┘                 │
  └─────────────┼────────────────────-─┘
                │（標準化インタフェース）
  ┌─────────────┼──────────────────────┐
  │  Linux 側   │                      │
  │  ┌──────────┴──┐  ┌─────────────┐  │
  │  │ゲートウェイ  │  │ HMIチーム   │  │
  │  │   チーム    │  │             │  │
  │  └─────────────┘  └─────────────┘  │
  └────────────────────────────────────┘
```

チーム境界とアーキテクチャ境界を一致させることで、チーム間の通信コストをインタフェース定義コスト（仕様書・API契約）に転換できる。

## 2. チームトポロジー

Matthew Skelton と Manuel Pais が著書 "Team Topologies"（2019）で提唱したフレームワーク。「どんなチームをつくるか」と「チームがどう連携するか」を体系化したもの。

### 2.1 4つのチームタイプ

#### Stream-aligned Team（ストリームアラインドチーム）

製品・機能の「価値の流れ（ストリーム）」に沿って編成されたチーム。ユーザーへの価値を継続的に届けることが主目的。

- 車載での例：「油圧制御SWCチーム」「HMIチーム」「ゲートウェイチーム」
- 特徴：エンドツーエンドで機能を届ける責任を持つ
- 理想：他チームへの依存を最小化し、独立してリリースできる

#### Platform Team（プラットフォームチーム）

他チームが使う共通基盤・ツール・サービスを提供するチーム。Stream-aligned Teamの認知負荷を下げることが目的。

- 車載での例：「BSW/RTEチーム（委託先）」「CI/CDインフラチーム」「ArXML管理チーム」
- 特徴：「使いやすいセルフサービス型の基盤」を作ることが使命
- 注意：Platform Teamが官僚的な承認フローを作るとボトルネックになる

#### Enabling Team（イネーブリングチーム）

特定の専門知識を持ち、他チームの能力向上を支援するチーム。「教えて去る」スタイルで能力移転を行う。

- 車載での例：「機能安全チーム（ISO 25119専門）」「サイバーセキュリティチーム」「AUTOSAR移行支援チーム」
- 特徴：長期的な依存を生まないよう、他チームが自立できるよう支援する
- 注意：常設のゲートキーパーになると開発速度が下がる

#### Complicated Subsystem Team（複雑サブシステムチーム）

特定の高度な専門知識が必要で、Stream-aligned Teamでは対応困難な複雑コンポーネントを担当するチーム。

- 車載での例：「通信スタック（SOME/IP、CAN XL）チーム」「暗号化・セキュアブートモジュールチーム」「OTAアップデートエンジンチーム」
- 特徴：数が少なく、高度なドメイン知識が必要
- 注意：存在を正当化できない場合はPlatform Teamに統合する

```
【4つのチームタイプ概念図】

  ┌─────────────────────────────────────────────┐
  │         Stream-aligned Teams                │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
  │  │ 油圧制御  │  │   HMI   │  │ゲートウェイ│  │
  │  │   SWC   │  │ チーム  │  │  チーム   │  │
  │  └─────┬────┘  └────┬─────┘  └─────┬────┘  │
  └────────┼────────────┼───────────────┼───────┘
           │            │               │
  ┌────────┴────────────┴───────────────┴───────┐
  │              Platform Team                  │
  │         （BSW/RTE・CI/CD・共通基盤）          │
  └─────────────────────────────────────────────┘

  ┌─────────────────┐    ┌──────────────────────┐
  │  Enabling Team  │    │ Complicated Subsystem │
  │  （機能安全・   │    │       Team           │
  │  AUTOSAR専門）  │    │  （通信スタック・暗号）  │
  └─────────────────┘    └──────────────────────┘
```

### 2.2 3つのインタラクションモード

チームタイプと同様に重要なのが、チーム間の「連携の仕方」を意識的に選択すること。

#### Collaboration（密な協働）

2チームが一定期間、密に協力して作業する。新技術の探索や初期アーキテクチャ設計フェーズに有効。

- 車載での例：BSW/RTEチームと油圧SWCチームが、新しいSWCのインタフェースを共同設計する期間
- 期間：限定的（数週間〜数ヶ月）。長期化すると責任が曖昧になる
- 注意：終わったら次のモードに移行する

#### X-as-a-Service（サービス提供）

一方のチームが「使いやすいサービス」として機能を提供し、もう一方はそれを消費する。インタフェースが安定していることが前提。

- 車載での例：BSW/RTEチームが「設定済みのBSWパッケージ」を提供し、SWCチームはAPIを呼ぶだけにする
- メリット：依存チームの認知負荷が大幅に下がる
- 前提条件：インタフェース（API仕様・ArXML定義）が安定していること

#### Facilitating（支援）

Enabling Teamが他チームの能力向上を支援するモード。ペアプログラミング、レビュー、ワークショップなどの形で行う。

- 車載での例：機能安全チームが油圧SWCチームのFMEA作業を伴走する
- 目的：最終的に支援チームが不要になること（自立を促す）

### 2.3 コグニティブロード（認知負荷）の考え方

チームトポロジーの重要な概念として「認知負荷」がある。1チームが担当できる複雑さには上限があり、その上限を超えると品質低下・バーンアウトが発生する。

車載開発での認知負荷の源泉：
- AUTOSAR仕様の複雑さ
- 機能安全要求（ASIL分類・FMEA・DFA）
- 複数のECU・通信プロトコルへの対応
- 委託先との調整コスト
- リアルタイム制約とデバッグの難しさ

**対策：1チームの担当領域を絞り、Platform TeamやEnabling Teamに認知負荷を分散する。**

チームサイズの目安：
- 1チームあたり5〜9人（アマゾンの「2ピザルール」に近い考え方）
- 認知負荷が高い領域（AUTOSAR BSW等）は4〜6人でも十分複雑

## 3. 車載ソフトウェア開発における代表的な組織モデル

### 3.1 ドメイン型組織（従来型）

自動車・建設機械業界でもっとも一般的な組織構造。

```
【ドメイン型組織構造】

  開発部門
  ├── パワートレインチーム（エンジン・トランスミッション制御）
  ├── 油圧・シャシーチーム（油圧制御・車体制御）
  ├── ボディ・電装チーム（灯火・空調・ドアロック）
  └── HMI・通信チーム（メータ・無線・CAN通信）
```

**メリット：**
- 専門知識が特定ドメインに蓄積される
- 各チームの担当範囲が明確

**デメリット：**
- ドメイン間の統合が困難（インタフェースの調整に工数がかかる）
- 横断的な関心事（機能安全・セキュリティ）が宙に浮く
- E/E（電気・電子）アーキテクチャの進化（ECU統合・ゾーン型）に対応しにくい
- ソフトウェアプラットフォームを共有する概念が生まれにくい

### 3.2 ゾーン型・中央集中型組織（SDV対応型）

次世代車載アーキテクチャ（中央HPC＋ゾーンECU）への移行に伴い、組織も変化している。

```
【ゾーン型組織構造（移行期）】

  開発部門
  ├── ソフトウェアプラットフォームチーム（OS・BSW・CI/CD）← 強化
  ├── アプリケーションチーム
  │   ├── 安全制御アプリチーム（ASIL-B/D）
  │   └── 非安全アプリチーム（HMI・情報系）
  ├── 機能安全・品質保証チーム（横断）
  └── E/Eアーキテクチャチーム（システム設計）
```

**特徴：**
- ソフトウェアプラットフォームを中心組織が管理する
- アプリケーション（機能SWC）とプラットフォームの分離が明確
- OEMがソフトウェア内製化を進める際に採用される形態

**OEMとTier1の役割変化：**

| 役割 | 従来型 | SDV対応型 |
|------|-------|----------|
| OEM | 要件定義・車両テスト | ソフトウェアアーキテクチャ設計・内製開発強化 |
| Tier1（委託先） | ECU一式開発 | Platform提供・ツール提供 |
| Tier2 | 部品供給 | コンポーネント提供・IP提供 |

### 3.3 スクワッド型（Spotifyモデルの車載適用）

Spotifyが提唱した「スクワッド・トライブ・チャプター・ギルド」モデルを車載開発に適用する動きがある。

```
【スクワッド型概念図】

  トライブ（例：建設機械制御ソフトウェア）
  ├── スクワッド：油圧制御（エンジニア5人、機能安全1人）
  ├── スクワッド：HMI（エンジニア4人、UX1人）
  └── スクワッド：クラウド連携（エンジニア4人）

  チャプター（職能横断）
  ├── 機能安全チャプター（各スクワッドの安全担当者）
  └── テストエンジニアチャプター

  ギルド（自発的な知識コミュニティ）
  ├── AUTOSAR勉強会
  └── セキュリティギルド
```

**採用事例（要確定）：**
- VW CARIAD：大規模なソフトウェア内製組織として設立（当初）
- BMW：ソフトウェア開発の内製強化においてアジャイル組織を試験導入
- Rivian：スタートアップとしてスクワッド型から始動

**車載開発での課題：**
- 機能安全（ISO 26262/ISO 25119）はWプロセスモデルを要求するため、純粋なアジャイルとの整合に工夫が必要
- スクワッドが独立して動ける一方、安全クリティカルな部分は承認プロセスが必須
- 「アジャイルはするが、安全要求はウォーターフォール的に管理する」ハイブリッドアプローチが現実解

## 4. 内製 vs 外部委託の組織設計

### 4.1 判断フレームワーク：何を内製し何を委託するか

「戦略的差別化」と「コモディティ」で分類するアプローチが有効。

```
【内製・委託の判断マトリクス】

                    高 ←── 自社の差別化への貢献度 ──→ 低
                ┌──────────────────┬───────────────────┐
           高   │   内製優先        │  選択的内製        │
                │（競争優位の源泉）  │ または委託+内製管理  │
  知識蓄積の    ├──────────────────┼───────────────────┤
  必要性        │  委託＋内製監督   │  委託優先          │
           低   │（知識は自社に残す）│（コモディティ）     │
                └──────────────────┴───────────────────┘
```

建設機械開発での具体的な分類例：

| 領域 | 分類 | 判断理由 |
|------|------|---------|
| 油圧制御アルゴリズム | 内製 | 建機の核心機能・差別化の源泉 |
| 機械学習による人検知 | 内製または専門委託 | 将来の差別化要因 |
| Classic AUTOSAR BSW | 委託（Platform Team化） | コモディティ・専門性が高すぎる |
| AUTOSAR設定・ArXML管理 | 内製管理・委託実施 | 知識は自社に残す必要あり |
| CI/CDパイプライン基盤 | 内製（またはクラウド活用） | 開発効率に直結 |
| 通信スタック（SOME/IP等） | 委託（Complicated Subsystem） | 高度な専門知識が必要 |
| HMI画面デザイン | 内製 | ユーザー体験の差別化 |

### 4.2 委託先をチームの一部として機能させるインタフェース設計

委託先を「外の人」ではなく「Platform Team」として位置づける。そのためのインタフェース設計が重要。

**明確にすべきインタフェース：**
1. **成果物インタフェース**：何を納品するか（ArXMLファイル、ビルド済みBSWライブラリ等）
2. **コミュニケーションインタフェース**：誰がいつ何を報告するか（週次レポート形式・エスカレーション経路）
3. **技術インタフェース**：APIや設定ファイルの形式（どこまで委託先が決め、どこから自社が決めるか）
4. **品質インタフェース**：受け入れ基準（テストカバレッジ・MISRA準拠・AUTOSAR BSW仕様適合）

**失敗パターン：**
- インタフェースを曖昧にしたまま委託する → 統合フェーズで大量の手戻りが発生
- 委託先に「全部お任せ」にする → 知識が社内に蓄積されない
- 委託先を会議に呼ばない → コンテキスト不足で的外れな成果物が届く

### 4.3 知識の社内蓄積と委託依存のバランス

**「知識の鎖」を断ち切らないための原則：**

1. **委託先の作業に必ず社内エンジニアを1人アサインする（ブリッジエンジニア）**
   - 技術的な詳細を理解し、社内に知識を蓄積する役割
   - 「承認者」ではなく「学習者」として関与する

2. **コアドメインの仕様は社内で書く**
   - 委託先に仕様書を書かせると、仕様の意図が伝わらないまま実装されるリスクがある
   - 「何を作るか」は内製、「どう作るか」は委託するのが原則

3. **委託契約にナレッジトランスファーを明記する**
   - 技術説明会・コードレビュー参加・ドキュメント整備を成果物として定義する

### 4.4 建設機械メーカーとして現実的な内製・委託の境界線

中小規模の建設機械メーカー（30人程度の開発組織）が現実的に実現できる境界：

```
【推奨境界線】

内製（自社エンジニア）
  ├── 要求仕様（機械として何が必要か）
  ├── アーキテクチャ設計（どう分割するか）
  ├── アプリケーション層（油圧アルゴリズム・HMIロジック）
  ├── 統合テスト・検証
  └── 委託先の品質管理・受け入れ

委託（外部パートナー）
  ├── Classic AUTOSAR BSW設定・実装
  ├── AUTOSAR設定ツール操作（Davinci等）
  ├── 通信スタック実装（CAN・Ethernet）
  └── 初期CI/CD構築
```

## 5. DevOps・DevSecOps組織への移行

### 5.1 車載DevOpsの難しさ

Webサービスの世界でDevOpsが普及した背景には「デプロイが数秒でできる」という特性がある。車載ソフトウェアはそのまま適用できない。

**根本的な差異：**

| 観点 | Webサービス | 車載ソフトウェア |
|------|------------|--------------|
| リリースサイクル | 週〜日単位 | 年単位（車両モデルサイクル） |
| デプロイ先 | クラウドサーバー | 実車両（テスト費用・安全確認が必要） |
| 失敗時の影響 | サービス停止 | 人命・安全に関わる可能性 |
| 規制・認証 | 特になし | 機能安全認証（型式認証） |

**それでもDevOpsが必要な理由：**
- ECUが増え、ソフトウェア規模が拡大している
- OTAアップデートにより「デプロイ」の概念が車載にも入ってきた
- 開発中のビルド・テスト自動化によって品質を向上させる余地が大きい

### 5.2 車載DevOpsの現実的な適用範囲

```
【車載DevOpsの適用範囲（現実的）】

  開発フェーズ（DevOpsが有効）
  ├── 自動ビルド（ArXML変更→BSW再生成→コンパイル）
  ├── 静的解析（MISRA違反・AUTOSAR構成チェック）
  ├── SIL（Software-in-the-Loop）テスト自動化
  └── MIL（Model-in-the-Loop）テスト自動化

  検証フェーズ（部分的に適用）
  ├── HIL（Hardware-in-the-Loop）テスト（自動化可能）
  └── 台上試験（一部自動化）

  量産・フィールド（OTA連携）
  └── OTAアップデート配信（差分配信・ロールバック）
```

### 5.3 セキュリティチームの組織上の位置づけ

DevSecOps（開発・運用・セキュリティの統合）を車載開発に適用する際の考え方：

**推奨：セキュリティ担当をEnabling Teamとして配置する**

```
  セキュリティ Enabling Team の役割
  ├── 脅威分析（TARA）の伴走支援
  ├── セキュアコーディングガイドラインの策定・教育
  ├── CI/CDパイプラインへのSAST（静的セキュリティ解析）組み込み
  └── 外部委託先のセキュリティ要件定義支援
```

**避けるべきパターン：**
- セキュリティチームがすべての変更をゲートキープする → 開発速度が著しく低下
- セキュリティを「最後に確認する」後付け工程にする → 手戻りが莫大になる

## 6. 建設機械メーカー（30人規模）への現実的な適用

### 6.1 現状の推定チーム構成と問題

30人・4ドメイン（エンジン/油圧/HMI/電装）体制の典型的な姿：

```
【現状推定構成】

  開発部（30人）
  ├── エンジン制御チーム（7人）
  ├── 油圧制御チーム（8人）
  ├── HMI・表示チーム（6人）
  ├── 電装・CAN通信チーム（5人）
  └── 共通・品質（4人）※機能安全も兼務

  問題：
  - 各チームがECU単位で閉じており、通信仕様の統制者がいない
  - 機能安全担当が4人で全ドメインをカバーしようとして疲弊
  - ソフトウェアプラットフォーム（BSW等）の専任がいない
  - 外部委託先の管理コストが特定の個人に集中している
```

### 6.2 チームトポロジー観点での課題診断

| チームトポロジーの観点 | 現状の問題 |
|---------------------|-----------|
| Stream-aligned Team | ドメイン単位であり「価値の流れ」とずれている（油圧チームは油圧機能の価値を届けられるか？） |
| Platform Team | 不在。BSWやCI/CDは「誰かの担当」になっておらず、属人化 |
| Enabling Team | 機能安全担当が有能でも「4人で全部」は無理。知識移転が追いつかない |
| Complicated Subsystem Team | 通信スタックや暗号処理の専任が不在。問題が起きたとき誰も解決できない |
| インタラクションモード | 全てのチーム間がCollaboration状態（密に連携）になっており、疲弊 |

### 6.3 Classic AUTOSAR外部委託＋Linuxハイブリッドの組織モデル提案

```
【提案組織モデル（30人規模）】

  ┌──────────────────────────────────────────────────┐
  │         Stream-aligned Teams（内製）              │
  │  ┌───────────────┐  ┌──────────┐  ┌──────────┐  │
  │  │ 油圧制御SWCチーム│  │HMIチーム  │  │GWチーム  │  │
  │  │     (7人)      │  │  (5人)   │  │  (4人)  │  │
  │  └───────────────┘  └──────────┘  └──────────┘  │
  └──────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────┐
  │  Platform Team（委託先 + 社内ブリッジ）             │
  │  BSW/RTE・CI/CD・ArXML管理                        │
  │  社内ブリッジエンジニア：2人                        │
  └──────────────────────────────────────────────────┘

  ┌─────────────────────┐  ┌──────────────────────┐
  │   Enabling Team      │  │Complicated Subsystem  │
  │   機能安全（3人）     │  │  Team（委託）         │
  │   各チームを伴走支援  │  │  通信スタック・暗号    │
  └─────────────────────┘  └──────────────────────┘
```

**人員配置の考え方（30人）：**
- Stream-aligned Teams：油圧7人・HMI5人・GW4人 = 16人
- Platform Team（社内ブリッジ）：2人
- Enabling Team（機能安全）：3人
- マネジメント・E/Eアーキテクト：2人
- 委託先への予算配分：残り7人分相当をPlatform/Complicatedサブシステムへ

### 6.4 今すぐできる組織設計の改善3ステップ

**Step 1（即日〜1ヶ月）：「インタフェースオーナー」を指名する**

- 各チーム間の通信仕様（CAN信号定義・APIなど）の責任者を1人決める
- 現状の通信仕様をひとつの文書にまとめ、変更時の承認フローを定める
- コンウェイの法則に対抗する最初の一手

**Step 2（1〜3ヶ月）：機能安全担当を「専任化」する**

- 現在兼務している機能安全担当を部分的にでも専任化する（50%以上）
- 各ドメインチームに「機能安全連絡役」を1人アサインし、Enabling Teamとの窓口にする
- 知識を個人に集中させず「チームの能力」として移転する仕組みを作る

**Step 3（3〜6ヶ月）：委託先との「インタフェース契約」を明文化する**

- 委託先を「Platform Teamの一部」として位置づける
- 成果物・コミュニケーション・品質の3つのインタフェースを文書化する
- 社内ブリッジエンジニアを明示的に任命し、委託先との週次同期に参加させる

## 7. 組織変革のロードマップ

### Phase 1（〜2年）：基盤整備

**目標：「誰が何を担うか」を明確にする**

```
取り組み内容：
├── 機能安全 Enabling Team の設立（専任化）
├── 委託先の Platform Team としての位置づけ確立
│   └── インタフェース契約の明文化
├── インタフェースオーナー制度の導入
│   └── CAN信号・API仕様の一元管理
├── CI/CDパイプラインの整備（ビルド・静的解析の自動化）
└── ブリッジエンジニアの育成（委託先との技術橋渡し役）

成果指標（要確定）：
- 統合フェーズでの手戻り件数の削減
- 機能安全レビューの所要時間短縮
- 仕様変更時の影響範囲把握のリードタイム短縮
```

### Phase 2（2〜5年）：Stream-aligned Team確立

**目標：「機能の価値を独立して届けられるチーム」を作る**

```
取り組み内容：
├── ECU単位からSWC機能単位へのチーム再編
├── HMIチームの独立強化（デザイナー・テスターを含む）
├── ゲートウェイ・クラウド連携の専任チーム化
├── SIL/HILテスト自動化の整備
└── 内製ドメイン（油圧アルゴリズム等）のモジュール化

組織の成熟指標：
- 各チームが他チームの承認なしにインテグレーションテストを実行できる
- 委託先からのBSW更新が自動的にCIに取り込まれる
- 機能安全チームが「ゲートキーパー」から「アドバイザー」に変わる
```

### Phase 3（5〜10年）：SDV対応型組織への移行

**目標：ソフトウェア定義建設機械（SDV）に対応できる組織**

```
取り組み内容：
├── OTAアップデート対応チームの設立
├── クラウド・データプラットフォームチームの拡充
│   （AWS IoT Core / Azure IoT Hub連携）
├── Adaptive AUTOSAR対応の検討・試験導入
├── ML/AI機能（自動化・自律化）のSWCチーム化
└── Hypervisor統合（SoC統合）に向けたアーキテクチャ委員会設立

組織の姿：
- Platform Team：BSW（Classic）+ OS（Linux）+ Hypervisor の統合管理
- Stream-aligned Team：各機能（自律・安全・情報）が独立して開発
- Enabling Team：機能安全・セキュリティ・AI品質保証
- 内製比率：アプリケーション層の80%以上を内製化
```

### ロードマップ全体像

```
現在          2年後            5年後            10年後
  │              │                │                │
  ▼              ▼                ▼                ▼
ECU単位    インタフェース     SWC機能単位       SDV対応組織
4チーム     整備完了          Stream-aligned     フル内製化
属人化      委託先安定化      Team確立          OTA量産対応
機能安全    機能安全          機能安全          自律化対応
```
