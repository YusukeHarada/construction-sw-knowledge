# SDV（Software Defined Vehicle）と建設機械への考察

**作成日：** 2026年6月16日  
**対象読者：** 開発部門・経営層  
**資料種別：** 社内検討用（社外秘）

---

## 1. SDVとは何か

SDV（Software Defined Vehicle）とは、車両の機能・性能・ユーザー体験をハードウェアではなく**ソフトウェアが主導的に定義する**アーキテクチャ概念である。

### 従来の車両との違い

| 観点 | 従来型 | SDV |
|---|---|---|
| 機能追加 | ハードウェア交換が前提 | OTA（無線）ソフトウェア更新で実現 |
| ECU構成 | 機能ごとに独立した分散型ECU | 中央HPC＋ゾーンコントローラに集約 |
| 通信 | CAN中心（低帯域） | Ethernet（高帯域）＋SOME/IP |
| ソフトウェア | ハードウェアに密結合 | ハードウェアから疎結合（HAL整備） |
| ビジネスモデル | 出荷時の機能が固定 | 出荷後も機能追加・課金が可能 |

Teslaが2012年頃から先行実用化し、BMW・VW・Toyotaが追随。現在は業界の主要方向性となっている。

### SDVを実現する技術スタック

```
┌─────────────────────────────────────────┐
│ クラウドサービス・フリート管理・OTA更新       │
├─────────────────────────────────────────┤
│ SOA（サービス指向アーキテクチャ）             │
│ API Gateway / サービスメッシュ               │
├────────────────┬────────────────────────┤
│ Adaptive AUTOSAR│ Linux / Android Auto   │
│ （HPC・自律化）  │ （HMI・コックピット）   │
├────────────────┴────────────────────────┤
│ Classic AUTOSAR（安全クリティカル制御系）     │
├─────────────────────────────────────────┤
│ ハードウェア：HPC＋ゾーンECU＋センサー群      │
│ 通信：Ethernet（100BASE-T1）＋CAN FD       │
└─────────────────────────────────────────┘
```

### E/Eアーキテクチャの変遷

```
分散型（〜2015）→ ドメイン集中型（2015〜2020）→ ゾーン型（2020〜）→ 中央集中型（〜2030）
```

ゾーン型では車両の物理ゾーン（前左・前右・後部等）ごとにゾーンECUを配置し、高速Ethernet骨格で中央HPCと接続する。配線長削減と演算リソース集中が目的。

### AUTOSARとSDVの関係

- **Classic AUTOSAR（CP）**：リアルタイム・安全クリティカルな制御系（パワートレイン・ブレーキ等）に引き続き利用。SDVでも廃止されない
- **Adaptive AUTOSAR（AP）**：Linux上で動作し、SOME/IPによるSOA・動的アプリ展開をサポート。HPC上のSDV機能を担う
- **現実**：CP＋APの混在構成が主流。APのみでASIL-D相当をカバーするのは困難であり、CPとの協調が当面続く

## 2. 建設機械のSDV化動向

### 大手メーカーの動き

| メーカー | 動向 |
|---|---|
| **コマツ** | 2025年9月、Applied Intuitionと戦略的協業を発表。次世代鉱山機械向けSDV＋自律化プラットフォームを共同開発。OTAによるライフサイクル全体での継続改善を軸に据える |
| **日立建機** | デジタルツイン（仮想空間への現場再現）を活用した開発加速。親会社との連携でデジタル化を推進 |
| **Caterpillar** | テレマティクス・AIによる機械制御・電動化・自動機械コントロールを製品化。「データ収集資産としての建機」ビジョン |
| **Volvo CE** | フリート管理・予知保全・電動化に注力。自動運転建機の実証実験を継続 |

コマツとApplied Intuition（自動車向けSDVプラットフォームの有力企業）の協業は、**自動車のSDV技術を建設機械に適用する最初の大規模事例**として業界から注目されている。

### 建設機械固有の障壁

| 障壁 | 内容 |
|---|---|
| **油圧制御の複雑性** | 主機能は油圧回路で実現されており、ソフトウェアで直接定義できる範囲が限られる |
| **J1939との断絶** | 建機の通信標準はJ1939であり、AUTOSAR（Ethernet/CAN FD前提）との直接互換性がなく移植コストが発生 |
| **低ロット・多仕様** | 一機種あたり年間数千台以下が多く、重厚なツールチェーンの費用回収が困難 |
| **過酷環境・長寿命** | 粉塵・振動・極温への対応と、20〜30年の製品寿命を前提とした保守計画が必要 |
| **安全認証の違い** | ISO 13849/ISO 25119（建機）とISO 26262（自動車）は要件が異なる |

### 建設機械でSDVが価値を生む具体的なユースケース

**短期（価値が出やすい）**

- **フリートマネジメント**  
  クラウドに集約した稼働データをもとに、全機材の稼働率・燃費・アイドリング時間をリアルタイムで可視化できる。顧客は稼働パターンの改善や適切な台数配分によって燃料費・リース費用を削減でき、メーカーはデータポータルを通じた継続的な顧客接点と有償サービス（稼働レポート・診断API）の提供が可能になる。

- **予知保全**  
  油温・振動・圧力・エンジン回転数などのセンサーデータをクラウドに常時送信し、機械学習モデルが故障確率の上昇を検知してアラートを発報する。計画外の現場停止を削減することで顧客の損失工期を大幅に減らすとともに、メーカーは部品・サービス収益の予測精度を上げ、サービスエンジニアの派遣を最適化できる。

**中期**

- **遠隔操作**  
  5G・LTE低遅延通信と高精細カメラ映像を組み合わせ、遠隔地のオペレーターが危険現場（崖崩れ・有毒ガス・放射線汚染現場など）や離島・僻地の機械を操縦できる。顧客は作業員の安全リスクを大幅に低減でき、専門オペレーター一人が複数現場を掛け持ち対応することで人件費も削減できる。SDV的には「操作信号の標準化」と「遅延許容制御アルゴリズム」の実装が中心課題となる。

- **半自律化支援**  
  カメラ・LiDARによる人・障害物検知と連動した自動停止、ブレードやバケットの姿勢自動補正、掘削深度・整地精度の自動管理を提供する。オペレーターの熟練度依存を下げて施工精度を均一化でき、特に人手不足が深刻な土木・解体現場でのオペレーター教育コストを削減する価値がある。

**長期**

- **完全自律施工**  
  コマツが鉱山での無人ダンプで実績あり。油圧ショベルの自律掘削が次の主戦場。自律施工が実現すると24時間連続稼働・1オペレーターによる複数台管理が可能となり、機械の時間あたり生産性が根本的に変わる。メーカーにとっては「機械販売」から「施工アウトカム（土量・成形精度）の提供」へのビジネスモデル転換につながる。

## 3. 建設機械に最適なアーキテクチャ考察

### 「AUTOSAR全面採用」が正解でない理由

SDV化にAUTOSARが必須というわけではない。また「Adaptive AUTOSAR全面採用」は現時点では現実的でない。

- Adaptive AUTOSARは自動車業界でも普及途上（高付加価値車・ADAS限定）
- 建設機械向けのAdaptive対応委託先・実績が少ない
- 機能安全（ISO 25119 ASIL相当）のAdaptive上での対応実績が薄い

### 現時点で最も現実的な構成

**Classic AUTOSAR（RTOS系）＋ 通常のLinuxミドルウェア（Linux系）のハイブリッド構成**

```
┌──────────────────────┐    ┌──────────────────────┐
│   Classic AUTOSAR ECU │    │     Linux ECU         │
│                      │    │                      │
│ ・油圧制御SWC         │    │ ・HMI                │
│ ・機能安全（ISO 25119）│◄──►│ ・人検知             │
│ ・J1939ゲートウェイ   │    │ ・クラウド通信        │
│                      │    │ ・フリート管理        │
└──────────────────────┘    └──────────────────────┘
        CAN FD / SOME/IP Gateway
```

| 観点 | ハイブリッド構成 | Adaptive AUTOSAR全面 |
|---|---|---|
| 導入コスト | 低〜中 | 高 |
| 委託先の調達難易度 | 低（実績豊富） | 高（経験者少ない） |
| 成熟度・実績 | 高（John Deere等で実績） | 低（普及途上） |
| ISO 25119対応 | 対応実績あり | 実績が少ない |
| 将来性 | 中（2028年以降に見直し） | 高（標準化の恩恵） |

John Deere・ZF・ボッシュ等がRTOS系制御ECU＋Linuxゲートウェイの構成を採用しており、建設機械での現実解として業界で認知されている。

### AUTOSAR準拠プラットフォームとSDV化の関係

Classic AUTOSAR準拠は**SDV化の土台**になるが、それ自体はSDVではない。

- AUTOSARのポータビリティ（BSW標準化）により、ハードウェア刷新時のソフトウェア移植コストが下がる
- 仕様書・テスト証跡が整備されることで、OTA更新を安全に運用できる前提が生まれる
- ただしRTOS側だけが整備されて「変化できない島」にならないよう、Linux側との通信境界設計が重要

**「ソフトウェア定義」の前提条件：** 仕様書・テストがない内製プラットフォームは「ソフトウェア定義」ではなく「ソフトウェア固定」の状態。OTA更新や機能追加を安全に行うには、機能安全の実効対応が先決。現状の形式的な機能安全対応はここで必ず破綻する。

## 4. 建設機械SDV化のロードマップ

### 今すぐ取り組むべきこと（〜2年：2026〜2027年）

これがないとSDV化はおろかAUTOSAR移行も絵に描いた餅になる。

| アクション | オーナー | 目的 |
|---|---|---|
| 仕様書・テストのない内製プラットフォームの棚卸し | **開発部門マネージャー** が主導。各ECU担当者が現状の設計書・テスト有無を一覧化 | 技術的負債の可視化。何が「ソフトウェア固定」状態かを経営層まで共有する |
| **機能安全の実効対応**（HARA実施・証跡整備） | **機能安全責任者**（社内任命または外部委託）が牽引。開発部門全体が参加 | 形式的な対応から脱し、OTA更新・新機能追加を安全に実施できる基盤を作る |
| Classic AUTOSARへの段階移行PoC（Dcm+Demから開始） | **先行開発チーム**（外部委託パートナーと共同）が実施 | AUTOSAR導入の費用・期間・リスクを実データで経営判断に提供する |
| クラウド連携・フリート管理の初版リリース | **IoT/クラウド担当チーム**が開発。営業・サービス部門が顧客への展開を担当 | 最も早くROIが出るレイヤーで投資回収を開始し、社内のSDV化機運を高める |

### 5年後に目指す姿（2031年）

| 達成目標 | オーナー |
|---|---|
| 最低1機種でClassic AUTOSAR準拠プラットフォームが稼働 | **開発部門長**が成果責任を持ち、量産化判断は経営層が行う |
| 外部委託先と安定した開発・保守体制が確立 | **調達・パートナー管理部門**が委託先との長期契約・SLA管理を担う |
| フリート管理・予知保全が顧客に価値を提供（NPS向上） | **事業部門・サービス部門**が顧客フィードバックを収集し、開発部門にフィードバック |
| OTA更新の仕組みが少なくともLinux系に存在する | **クラウドプラットフォームチーム**がインフラを整備。セキュリティ部門がリリース承認フローを管理 |

### 10年後に目指す姿（2036年）

| 達成目標 | オーナー |
|---|---|
| Adaptive AUTOSARまたは同等のLinux/ROS2基盤で特定作業の半自律化を製品化 | **次世代機開発プロジェクト**（専任チーム）が担当。経営層がロードマップ投資判断を実施 |
| OTAによる機能追加が通常のリリースフローに組み込まれた状態 | **開発部門全体**。リリースマネージャーが安全確認・承認フローを運用 |
| 「データを持つ建設機械メーカー」としての競争優位が確立 | **経営企画・事業開発部門**がデータ収益化戦略・パートナーシップを主導 |

## 5. まとめ：建設機械に今力を入れるべきところ

優先順位をつけると以下の順番になる。

1. **機能安全の実効対応**（現状の形式的対応を脱する）― SDV化の前提
2. **Classic AUTOSAR移行PoC**（RTOS系の技術的負債を解消）― 土台づくり
3. **フリート管理・クラウド連携の強化**（最速でROIが出るSDV要素）― 価値創出
4. **通信標準化**（J1939の外部I/F維持＋内部CAN FD化）― 将来の拡張性
5. **Adaptive AUTOSAR / 自律化**（2028年以降に本格検討）― 将来の競争優位

「AUTOSAR全面採用がゴール」ではなく、**「ソフトウェアで機械の価値を継続的に高められる体制を作ること」がゴール**であり、AUTOSARはそのための手段の一つである。
