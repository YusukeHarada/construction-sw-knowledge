## はじめに

本資料は、建設機械向けプラットフォームソフトウェアをClassic AUTOSAR（RTOS系：油圧制御等の制御系）とLinuxミドルウェア（Linux系：HMI・ゲートウェイ・クラウド通信）のハイブリッド構成へ移行するための費用試算と予算計画をまとめたものである。

記載している金額はすべて**参考値・要確定**である。実際の調達・交渉を通じて自社で確認すること。

> **要確定：** 上記レンジは同規模の自動車・建設機械メーカーの事例を参考にしたものであり、自社のスコープ確定後に精査が必要。


### 「移行しない場合」のリスクコストとの比較

移行に投資しない場合でも、以下のリスクコストが潜在的に存在する。

| リスク項目 | 想定コスト（参考値・要確定） |
|---|---|
| ソフトウェア起因の重大事故による損害賠償・リコール | 数億〜数十億円（1件あたり） |
| 機能安全未対応による主要市場での販売停止 | 販売機会損失：年間数億円規模 |
| 内製コードのレガシー化による保守費増大 | 現状比1.5〜2倍に拡大するリスク |
| 競合他社比のソフトウェア品質格差拡大 | 中長期的な競争力低下（定量化困難） |

**移行投資と移行しない場合のリスクコストを比較すると、5〜7年でリスクコストが投資額を上回る可能性が高い。**

## 2. ツールライセンス費用

### 2-1. AUTOSARコンフィギュレーターツール

AUTOSARコンフィギュレーターツールとは、AUTOSAR（車載ソフトウェアの国際標準規格）に準拠したBSW（基本ソフトウェア：OSや通信スタック等に相当）を設定・生成するためのツールである。

| ツール | 提供元 | 初期費用（参考） | 年間保守費（参考） | 備考 |
|---|---|---|---|---|
| DaVinci Configurator Pro | Vector Informatik（独） | 要確定（数百万円〜） | 初期費の15〜20%程度 | 国内シェア高い |
| EB tresos Studio | Elektrobit（独、Continental子会社） | 要確定（数百万円〜） | 初期費の15〜20%程度 | BSWとセット販売が多い |

**ライセンス形態の種類：**

- **サイトライセンス：** 社内拠点単位で購入。開発者が多い場合はコスト効率が高い
- **フローティング（浮動）ライセンス：** 同時使用数に上限があり、複数開発者で共用する形態
- **ECU単位ロイヤリティ：** 量産するECU（電子制御ユニット）1台あたりに課金される形態。量産台数が多い場合はコストが積み上がるため注意

> **要確定：** ライセンス形態と台数は、量産計画確定後にベンダーと交渉すること。

### 2-2. 静的解析ツール（MISRA-C準拠検査）

MISRA-Cとは、車載組込みソフトウェアに適用されるCコーディング規約であり、機能安全対応には遵守が必要である。

| ツール | 提供元 | 費用感（参考） |
|---|---|---|
| PC-lint Plus | Gimpel Software（米） | 要確定（数十万円/ライセンス） |
| Klocwork | Perforce（米） | 要確定（年間数百万円〜） |
| Polyspace Bug Finder | MathWorks（米） | 要確定（年間数百万円〜） |

### 2-3. 通信テストツール（CANoe / CANalyzer）

CANoeは車内ネットワーク（CAN・LIN・Ethernet等）の開発・テストを行うための業界標準ツール。CANalyzerは解析特化の廉価版に相当する。

| ツール | 用途 | 費用感（参考） |
|---|---|---|
| CANoe（Vector） | CAN・J1939・Ethernetのシミュレーション・テスト | 要確定（数百万円〜/ライセンス） |
| CANalyzer（Vector） | ネットワーク解析・モニタリング | 要確定（百万円台〜） |

> J1939とは建設機械・農業機械で広く使われるCAN通信の上位プロトコル規格である。

### 2-4. HIL機材（ハードウェアインザループ）

HIL（Hardware-in-the-Loop）とは、実ECUに対して車両・環境をシミュレーターで模擬してソフトウェアをテストする手法である。機能安全対応において必須となる。

| 機材 | 提供元 | 費用感（参考） |
|---|---|---|
| SCALEXIO / DS1007等 | dSPACE（独） | 要確定（500万円〜2,000万円/セット） |
| VT System等 | Vector Informatik（独） | 要確定（300万円〜1,500万円/セット） |

> 上記はHILラックの構成規模によって大幅に変動する。まずPoC用の小規模構成から導入し、段階的に拡張することを推奨する。

### 2-5. SIL環境（ソフトウェアインザループ）

SIL（Software-in-the-Loop）とは、実機ハードウェアなしにPC上でソフトウェアをシミュレーション実行してテストする環境である。HILより安価に早期テストが可能。

| ツール | 提供元 | 費用感（参考） |
|---|---|---|
| VEOS | Vector Informatik（独） | 要確定（数百万円〜） |
| VECU-Builder | dSPACE（独） | 要確定（数百万円〜） |

### 2-6. CI/CDインフラ

CI/CD（継続的インテグレーション/継続的デリバリー）とは、ソフトウェアのビルド・テスト・配布を自動化することで開発速度と品質を向上させる仕組みである。

| ツール | 費用感（参考） |
|---|---|
| GitLab EE（オンプレミス） | 要確定（年間数百万円〜、ユーザー数次第） |
| GitHub Enterprise | 要確定（年間数百万円〜） |
| Jenkinsベースの自社構築 | ライセンス費は安価だが構築・運用工数が発生 |

## 5. 認証・監査費用

### 5-1. ISO 25119（農業・建設機械の機能安全規格）

ISO 25119は農業・建設機械に特化した機能安全規格であり、油圧制御等の安全クリティカルな機能に適用される。自動車のISO 26262に相当する。

| 項目 | 費用感（参考） |
|---|---|
| 第三者認証機関（TÜV・SGS等）による審査費用 | 要確定（1,000万〜3,000万円） |
| セーフティケース（安全の論証文書）作成支援 | 要確定（外部コンサル利用の場合：500万〜2,000万円） |
| 社内認証準備工数（設計・ドキュメント整備） | 人件費換算で要確定 |

### 5-2. ISO/SAE 21434（サイバーセキュリティ）

車載システムのサイバーセキュリティに関する国際規格。クラウド連携・OTA機能を持つ場合は対応が必須となる方向に進んでいる。

| 項目 | 費用感（参考） |
|---|---|
| TARA（脅威分析・リスク評価）実施支援 | 要確定（200万〜800万円） |
| 第三者監査費用 | 要確定（500万〜1,500万円） |

### 5-3. ASPICE（自動車ソフトウェアプロセス評価）

ASPICE（Automotive SPICE）とは、車載ソフトウェア開発プロセスの成熟度を評価する国際規格である。Tier1サプライヤーへの委託時には相手先のASPICEレベルの確認が推奨される。

| 項目 | 費用感（参考） |
|---|---|
| ASPICE評価（外部アセッサーによる） | 要確定（200万〜600万円/回） |
| プロセス改善コンサルティング | 要確定（300万〜1,000万円） |

### 5-4. 社内認証担当者の資格取得

## 8. 予算計画の進め方

### 8-1. フェーズ別の予算計上タイミング

| フェーズ | 時期 | 主な予算計上項目 |
|---|---|---|
| 準備・PoC | 〜1年目 | コンサルタント費・PoC委託費・ツール評価費・教育費（初回） |
| 基盤移行 | 2〜3年目 | ツールライセンス（本格導入）・委託費（Phase 2）・HIL機材・採用費 |
| 安全クリティカル移行 | 3〜5年目 | 委託費（Phase 3）・認証費・クラウドインフラ拡張 |
| 量産・保守 | 5年目以降 | 継続委託費・OTAインフラ運用費・クラウド費用 |

### 8-2. CAPEX（資本支出）vs OPEX（運用支出）の分類

| 支出項目 | 分類 | 理由 |
|---|---|---|
| HIL機材購入 | CAPEX | 固定資産として計上 |
| ツールライセンス（永続ライセンス） | CAPEX | 無形固定資産として計上 |
| ツールライセンス（サブスクリプション） | OPEX | 年間費用として計上 |
| 外部委託費（開発） | CAPEX or OPEX | 開発フェーズはCAPEX計上が一般的。会計担当に確認 |
| クラウド費用 | OPEX | 月額費用として計上 |
| 教育・研修費 | OPEX | 費用として計上 |
| 認証費用 | OPEX（またはCAPEX） | 会計担当に確認 |

### 8-3. 経営承認を得るための3軸提示

経営層への説明には以下の3軸を組み合わせることを推奨する。

1. **リスク軸：** 「移行しない場合に顕在化するリスクコスト」の定量化（第7章参照）
2. **コスト軸：** 「投資額とその回収シナリオ」の試算（本資料全体）
3. **競合動向軸：** 「コマツ・日立建機・Caterpillarが既に取り組んでいる事実」の提示

> Caterpillarは2020年代前半に大規模なソフトウェア定義化投資を開始している（要確定・公開情報の範囲）。コマツはKOMTRAXを通じた遠隔管理を既に量産展開済みであり、今後のAutosar化・OTA対応強化が予想される。対応の遅れは、将来のソフトウェア競争力において決定的な格差につながるリスクがある。

### 8-4. 予算申請書チェックリスト

経営承認のための予算申請書に含めるべき項目：

- [ ] 移行の目的と背景（なぜ今か）
- [ ] 移行しない場合のリスクコスト試算

- [ ] フェーズ別の投資金額とマイルストーン

- [ ] CAPEX/OPEX の分類と各年度の予算計上額

## 9. 「コストに見合わない」という反対意見への回答

| 反対意見 | 回答 |
|---|---|
| 「数億円もかけてまで必要か」 | 移行しない場合でも、現状の内製コード保守・品質問題対応・事故リスクに対して毎年コストが発生し続ける。比較すべきは「移行コスト vs 現状継続コスト＋リスクコスト」である。 |
| 「競合他社も同じ問題を抱えているはず」 | コマツ・日立建機は既に外部委託・AUTOSAR化を進めていると見られる（要確定）。先行投資した企業がソフトウェア品質・開発速度で優位に立つ可能性が高い。 |
| 「社内で育てればコストを抑えられるのでは」 | AUTOSARの専門知識はツールベンダーとの協業なしに習得が困難であり、初期段階での内製化は工期・品質リスクが高い。外部委託で基盤を確立した後に内製化比率を高める段階的戦略が現実的である。 |
| 「もう少し様子を見よう」 | 機能安全・サイバーセキュリティへの規制対応は法規制化の動きがあり、対応のタイミングを逃すと認証取得・製品改修のコストが急増する。早期対応ほど費用対効果が高い。 |
| 「ROIが7年では遅い」 | リスクコスト（事故・リコール・販売停止）を加味した場合の実質回収期間は3〜5年に短縮される可能性がある。また、OTA・フリート管理等の新規収益が加わればさらに早期回収が見込まれる。 |
