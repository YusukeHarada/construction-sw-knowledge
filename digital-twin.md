**作成日：** 2026年6月16日  
**対象読者：** 技術担当者・アーキテクト  
**資料種別：** 社内検討用（社外秘）

## 1. デジタルツインとは（建設機械の文脈で）

デジタルツインとは、物理的な機械・現場をソフトウェア上に再現した「仮想モデル」であり、センサーデータとの双方向同期によってリアルタイムに実機の状態を反映する。

### 3つの活用フェーズ

| フェーズ | 目的 | 主な対象 |
|----------|------|----------|
| 開発 | 設計検証・テスト自動化 | ECU・油圧システム・制御ロジック |
| 製造 | 組み立て工程の最適化・品質保証 | 生産ライン・部品トレーサビリティ |
| 運用 | 予知保全・遠隔診断・施工最適化 | フリート全体・現場地形 |

### 業界での実例

- **コマツ**: KOMTRAX／Smart Construction により稼働データをクラウドに集約し、フリート全体の位置・燃料・アラートを可視化
- **Caterpillar**: Cat Digital（旧Minestar）でドローン測量データと機械の施工データを統合し、土工量を自動計算
- **日立建機**: ConSitを通じた遠隔診断サービスで、エラーコードと稼働パターンを組み合わせた故障予測を提供

## 2. 開発フェーズでのデジタルツイン

### MBD（モデルベース開発）との関係

MBDで作成したMATLAB/Simulinkモデルがデジタルツインの「物理モデル層」を構成する。制御ロジックは同一モデルからコード生成（AUTOSAR準拠）されるため、設計変更がシミュレーションと実装に同時反映される。

### SIL/HILテスト環境との連携

- **SIL（Software-in-the-Loop）**: 仮想ECUを用いて制御ソフトウェア単体を検証。実機不要で早期にバグを検出できる
- **HIL（Hardware-in-the-Loop）**: 実ECUに仮想プラントモデルを接続。油圧シミュレーターと組み合わせることでアクチュエーター応答を実機に近い条件で検証可能

### 開発フェーズにおける仮想ECU（SIL環境）の具体的な役割

SIL環境は「実機なし・実チップなし」で制御ソフトウェアを動かすための環境である。建設機械開発では以下の目的で積極的に活用できる。

#### 仮想ECUとは何か

AUTOSARのClassic／Adaptive両方で、ソフトウェアコンポーネント（SWC）やBSWをPCまたはLinuxサーバー上で実行できる形に再ビルドしたものを**仮想ECU（VECU）**と呼ぶ。代表的なツールチェーン：

- **ETAS ISOLAR-EVE**（Classic AUTOSAR向け）：ARXMLから仮想ランタイム環境を生成し、実ECUと同じSWCコードをWindowsまたはLinux上で実行
- **Vector CANoe/AUTOSAR**：仮想ノードとしてSWCを動作させ、CAN/Ethernetシミュレーション環境と接続
- **dSPACE VEOS**：PC上でSILテストを実行し、後でHILへシームレスに移行

#### 仮想ECUとデジタルツインの接続イメージ

```
┌──────────────────────────────────────────────────────┐
│ SIL環境（PC / Linuxサーバー）                         │
│                                                      │
│  ┌─────────────────────┐   ┌────────────────────┐   │
│  │ 仮想ECU（VECU）      │   │ 油圧プラントモデル  │   │
│  │ Classic AUTOSAR SWC │◄──│ Simulink / Modelica│   │
│  │ ・油圧制御ロジック    │──►│ ・シリンダー応答   │   │
│  │ ・Dem（DTC管理）     │   │ ・油温モデル       │   │
│  │ ・Dcm（UDS応答）     │   │ ・ポンプ効率曲線   │   │
│  └──────────┬──────────┘   └────────────────────┘   │
│             │ 仮想CAN / VLAN                         │
│  ┌──────────▼──────────┐                            │
│  │ デジタルツイン       │                            │
│  │ （クラウド接続）     │──► AWS IoT TwinMaker       │
│  │ ・センサーデータ模擬 │    Azure Digital Twins      │
│  └──────────────────────┘                            │
└──────────────────────────────────────────────────────┘
```

#### SIL環境でできること・できないこと

| 検証項目 | SIL | HIL | 実機 |
|---|---|---|---|
| 制御ロジックの正確性 | ○ | ○ | ○ |
| OTA更新フロー（UDS 0x34〜0x37） | ○ | ○ | ○ |
| UDS診断応答（Dcm/Dem） | ○ | ○ | ○ |
| タイミング精度（割り込み・スケジューラ） | △（モデル時間で代替） | ○ | ○ |
| 電圧・温度・EMC影響 | ✗ | △ | ○ |
| AUTOSAR BSWの結合動作 | ○（ISOLAR-EVE使用時） | ○ | ○ |

**SILの最大のメリット**は、実機調達前（設計初期）から制御ソフトウェアの品質を高められることと、CI/CDパイプラインにテストを組み込める点にある。夜間の自動回帰テストをPC上のVECUで回すことで、翌朝にはテスト結果が得られる。

## 3. 運用フェーズでのデジタルツイン

### フリートデータ→クラウド→デジタルツインの流れ

```
実機センサー → 車載ゲートウェイ（4G/5G） → クラウド → デジタルツイン更新 → 可視化・アラート
```

### 予知保全への活用：具体的なアルゴリズム例

運用フェーズのデジタルツインは、センサーデータと物理モデルを照合して「現在の状態と正常状態の乖離」を定量化することが中心になる。以下に代表的な2つの検出手法を示す。

#### アルゴリズム例1：振動異常検知（油圧ポンプ・減速機の異常検出）

**目的**：油圧ポンプや旋回減速機の軸受・歯車の摩耗・損傷を早期検知する。

**入力データ**：
- IMUまたは専用加速度センサーの3軸振動データ（サンプリング：1〜10 kHz）
- エンジン回転数（rpm）
- 油圧ポンプ回転数（rpm）

**アルゴリズムの流れ**：

```
1. 生振動データ取得（高サンプリング）
         │
2. 高速フーリエ変換（FFT）
   → 周波数スペクトルを得る
   → 正常時のベースラインスペクトルと比較
         │
3. 特徴周波数の監視
   軸受損傷周波数 = 回転数 × 軸受係数（BPFO・BPFI・BSF・FTF）
   歯車噛み合い周波数 = 回転数 × 歯数
         │
4. エンベロープ解析（高周波帯の振幅変調を検出）
   → 衝撃的な異常（フレーキング・ピッティング）に有効
         │
5. 異常スコア算出
   スコア = Σ (観測振幅_i - ベースライン振幅_i)² / ベースライン振幅_i
         │
6. 閾値判定
   スコア > 警告閾値 → 注意アラート
   スコア > 危険閾値 → 即時点検推奨
```

**エッジでの実装**：FFTはゲートウェイ（Cortex-A）で処理し、異常スコアのみをクラウドに送信する（生波形は帯域が大きすぎるため常時送信しない）。異常スコアが警告閾値を超えた場合のみ、直近数秒の波形データをバースト送信する。

#### アルゴリズム例2：油温トレンド分析（作動油劣化・クーラー詰まりの検出）

**目的**：作動油の過熱や油圧クーラーの詰まりを予兆段階で検知する。

**入力データ**：
- 作動油温度センサー（油圧タンク・リターン回路）
- 冷却水温度
- 外気温
- 稼働強度（エンジン負荷率・ポンプ流量）

**アルゴリズムの流れ**：

```
1. 稼働条件で正規化した油温モデルを作成
   正規化油温 = 実測油温 - f(外気温, 負荷率, 稼働時間)
   ※ f() はオフライン学習した回帰モデル（線形回帰 or ランダムフォレスト）

2. 正規化後の残差をEWMA（指数加重移動平均）でトレンド化
   EWMA_t = α × 残差_t + (1-α) × EWMA_{t-1}  （α=0.1〜0.3）

3. トレンド検出
   一定期間（例：過去20時間）にわたりEWMAが上昇傾向
   → 線形回帰で傾き slope を算出
   slope > 閾値 → 「クーラー効率低下の疑い」アラート

4. 過去の故障事例との照合
   デジタルツインに蓄積されたフリートデータから
   同様のトレンドを示した後に故障した事例数を参照
   → 「類似パターン機N台中M台が90日以内に故障」として通知

5. 推奨アクション通知
   ・クーラーフィン清掃
   ・作動油サンプリング（粘度・酸価測定）
   ・オイルクーラー交換判断
```

**デジタルツインとの連携**：物理モデル（Modelica/Simscape）が「正常時の油温上昇特性」を与え、実測値との差分を監視する。正常な暖機曲線や負荷応答特性はモデルから算出するため、単純な閾値管理より精度が高い。

### クラウドプラットフォームを用いたデジタルツインの実装方法

#### AWS IoT TwinMakerの使い方

AWS IoT TwinMakerは、センサーデータ・3Dモデル・知識グラフを統合してデジタルツインを構築するマネージドサービスである。

**構成要素と役割**：

| コンポーネント | 役割 | 建設機械での使い方 |
|---|---|---|
| ワークスペース | TwinMakerの管理単位 | 車種ファミリー別に作成（例：油圧ショベル20ton級） |
| エンティティ | 物理資産の仮想表現 | 機体1台＝1エンティティ。コンポーネントで部位を分割（エンジン・油圧・走行） |
| コンポーネント | エンティティの部分定義 | タイムシリーズデータのバインディングをここで定義 |
| データコネクタ | 外部データソースとの接続 | Amazon Timestreamに蓄積したCAN信号をコネクタ経由で参照 |
| シーン | 3D空間上の可視化設定 | glTF形式の機体3Dモデルを読み込み、センサー値をオーバーレイ表示 |
| Grafanaプラグイン | ダッシュボード表示 | Amazon Managed Grafanaと連携し、フリート全体の状態を地図上に表示 |

**実装ステップ（概要）**：
1. Timestreamにテーブルを作成し、MQTT→IoT Core→Timestreamのルールを設定
2. TwinMakerワークスペースを作成し、Timestreamデータコネクタを登録
3. 機体の3DモデルをS3にアップロードし、シーンエディタで配置
4. エンティティにコンポーネントを追加し、油温・油圧・回転数をタイムシリーズにバインド
5. Grafanaでダッシュボードを作成。TwinMakerパネルに3Dビューを埋め込む

**コスト目安**：ワークスペース$0.40/日 + データクエリ件数に応じた従量課金。小規模PoC（機体10台）なら月$100以下で試用可能。

#### Azure Digital Twinsの使い方

Azure Digital Twinsは、オントロジー（知識グラフ）ベースで資産間の関係を定義できることが特徴。

**構成要素と役割**：

| コンポーネント | 役割 | 建設機械での使い方 |
|---|---|---|
| DTDLモデル | 双子の型定義（JSON-LD形式） | 機体・油圧系・エンジン・センサーをそれぞれモデル定義 |
| ツインインスタンス | 実資産に対応する仮想インスタンス | 機体1台＝1インスタンス。子ツイン（部品）を関係で接続 |
| ADT Explorer | ツイン関係のグラフビジュアライザ | 機体→エンジン→シリンダーの親子関係をノード図で確認 |
| IoT Hub連携 | センサーデータの取り込み | IoT Hub→Azure Functions→ADT更新の定型パターン |
| Azure Maps | 地図上の可視化 | 現場地形データと機体位置を重ね合わせ |
| Time Series Insights / ADX | 時系列データの分析 | 油温トレンド分析・故障予測の入力データとして活用 |

**DTDLモデル定義例（油圧ショベルのエンジンコンポーネント）**：
```json
{
  "@context": "dtmi:dtdl:context;2",
  "@id": "dtmi:construction:excavator:Engine;1",
  "@type": "Interface",
  "displayName": "エンジン",
  "contents": [
    { "@type": "Telemetry", "name": "rpm", "schema": "double" },
    { "@type": "Telemetry", "name": "coolantTemp", "schema": "double" },
    { "@type": "Telemetry", "name": "oilPressure", "schema": "double" },
    { "@type": "Property", "name": "modelCode", "schema": "string" },
    { "@type": "Property", "name": "totalHours", "schema": "double" }
  ]
}
```

**IoT Hub → ADT 更新の実装パターン**：
1. IoT Hubにデバイスを登録（1台＝1デバイス）
2. Azure Functions（Event Hub Trigger）でIoTメッセージを受信
3. ADT SDK（C# / Python）でツインのテレメトリ値を更新
4. ADT Explorer またはカスタムWebアプリで可視化

**HoloLens連携（AR表示）**：
Azure Digital TwinsのADT SDKとUnity（MRTK）を組み合わせることで、HoloLensから実機を見るとオーバーレイで油温・稼働時間・アラートが表示されるAR整備支援アプリを構築できる。現場整備士が機体のそばに立ちながらリアルタイムの状態を確認する用途に有効。

### 施工現場のデジタルツイン（地形・機械の統合）

ドローン測量（点群データ）と機械の施工ログを統合することで、設計図面との差分をリアルタイムに確認できる。自律施工・マシンコントロールの精度向上にも寄与する。

## 4. 実装に必要なもの

### センサーデータの種類と収集方法

| データ種別 | 収集方法 |
|-----------|---------|
| エンジン回転数・油温・油圧 | CAN/CANopen → ゲートウェイ |
| IMU（傾斜・振動） | 車載センサー → AUTOSAR SWC |
| 位置情報 | GNSS（RTK） |
| 映像・点群 | カメラ・LiDAR → エッジ処理後に送信 |

### クラウドプラットフォーム選択指針

- **AWS IoT TwinMakerを選ぶ場合**：すでにAWS（S3・Timestream・SageMaker）を使っている、Grafanaでダッシュボードを統一したい、既存のIoT Coreパイプラインを流用したい場合に向く
- **Azure Digital Twinsを選ぶ場合**：資産間の関係（部品構成・現場レイアウト等）をオントロジーで管理したい、HoloLensなどMicrosoftのMR/ARデバイスと連携したい、既存のAzure（Power BI・Teams）エコシステムがある場合に向く

### 物理モデルのソフトウェア化

油圧回路はModelicaまたはSimscapeで記述し、コンテナとしてデプロイ。クラウド上でリアルタイム実機データを境界条件として与えることで、センサーのない内部状態（例：弁の摩耗度）を推定する。

| 時期 | マイルストーン | 具体的な施策 |

AUTOSAR準拠プラットフォームへの移行と並行してデータ収集アーキテクチャを設計することで、開発フェーズのシミュレーション資産を運用フェーズのデジタルツインにそのまま流用できる。これが外部委託移行とSDV化を同時に進める際の最大の効率化ポイントとなる。
