# BIM/CIM連携技術ガイド

対象読者：技術担当者・アーキテクト

---

## 1. BIM/CIMとは

**BIM（Building Information Modeling）** は、建築物の形状・材料・設備・コストなどの情報を統合した3Dデジタルモデルである。単なる3D図面ではなく、属性情報（部材の仕様・施工手順・維持管理情報）を持つデータベースとして機能する。

**CIM（Construction Information Modeling / Management）** は、BIMの概念を土木・インフラ分野へ拡張したものである。道路・橋梁・トンネル・河川・造成工事など、建築物以外の構造物を対象とする。

### 国土交通省によるCIM推進（i-Construction）

国土交通省は2016年より **i-Construction** を推進しており、ICT・BIM/CIMを活用した建設生産システムの効率化を目指している。主要な方針は以下の通りである。

| 項目 | 内容 |
|---|---|
| CIM原則化 | 一定規模以上の直轄工事でBIM/CIM活用を原則化（2023年度〜） |
| 電子納品要件 | 3次元モデルによる成果品納品を段階的に拡大 |
| 出来形管理自動化 | ICT建機による計測・帳票自動生成 |
| 書類削減 | デジタル施工管理による紙書類の大幅削減 |

建設機械との関係では、**施工計画として作成したBIM/CIMモデルを機械制御に直結させる**ことが核心的な概念となる。設計通りの地形を機械が自律的に造成することで、熟練オペレーターへの依存を低減し、施工品質を均一化することが目標である。

## 2. 建設機械とBIM/CIMの連携ユースケース

### ユースケース一覧

| ユースケース | 概要 | 自律化レベル |
|---|---|---|
| 施工ガイダンス（Machine Guidance） | 刃先位置とBIMモデルの差分をHMIに表示。オペレーターが判断・操作 | 人間主体 |
| 施工制御（Machine Control） | BIMモデル通りの地形を半自律的に造成。機械が油圧を自動調整 | 半自律 |
| 進捗管理 | 実際の施工データをBIMモデルに自動フィードバック。クラウドで可視化 | 自動記録 |
| 出来形管理 | 施工後の地形測量データとBIMモデルの差異を自動確認・帳票生成 | 自動検査 |

### 施工ガイダンスの動作概念

```
BIMモデル（設計地形）
        ↓
  差分計算（クラウドまたはLinux ECU）
        ↓
  HMI表示（掘削が必要な深さ・方向を色分け表示）
        ↓
  オペレーターが操作
```

### Machine Controlの動作概念

```
BIMモデル（目標地形）
        ↓
  刃先位置計算（GNSS/RTK + IMU + アーム角度センサ）
        ↓
  目標値との偏差計算（Linux ECU）
        ↓
  油圧弁制御指令（Classic AUTOSAR ECU）
        ↓
  アクチュエータ動作（自動掘削・整地）
```

## 3. 主要なデータフォーマット

| フォーマット | 用途 | 備考 |
|---|---|---|
| IFC（Industry Foundation Classes） | BIMの標準データフォーマット。ISO 16739規格 | 建築・構造物の形状・属性情報を格納 |
| LandXML | 土木向けの設計データ形式。道路・造成の平面・縦断・横断データ | 施工ガイダンス・Machine Controlで広く使用 |
| 3Dポイントクラウド（LAS・PLY） | LiDARによる地形計測データ。出来形管理に使用 | ファイルサイズが大きいため転送・処理に注意 |
| SiteVision / Trimble独自形式 | 施工管理プラットフォームが使用する中間フォーマット（要確定） | ベンダー依存のため標準フォーマットとの変換が必要 |

### 機械側との接続

BIM/CIMデータは機械側（ECU）で直接扱うには適さない。Linux ECUで座標変換・差分計算を行い、Classic AUTOSAR ECUには数値制御指令のみを渡す設計が適切である。ArXMLやDBCとの直接対応はなく、**Linuxゲートウェイ ECUがデータ変換の責務を担う**。

### ECU別役割分担

| ECU | 担当処理 | 採用技術 |
|---|---|---|
| Linux ゲートウェイ ECU | BIMデータ処理・座標変換・差分計算・クラウド通信 | ROS2・Python/C++・MQTT/HTTPS |
| Classic AUTOSAR ECU | 油圧制御ロジック・アクチュエータ管理・フェールセーフ | Classic AUTOSAR BSW・RTE |
| HMI ECU | 施工ガイダンス表示・オペレーターUI | Linux・Qt |
| センサ ECU | GNSS/RTK・IMU・各種センサのデータ収集 | CAN FD接続 |

## 5. 国内規制・標準化動向

### i-Constructionの変遷

```
CALS/EC（1990年代〜）
  ↓ 電子納品・電子商取引の標準化
i-Construction（2016年〜）
  ↓ ICT施工・3次元データ活用・施工管理のデジタル化
BIM/CIM原則化（2023年〜）
  ↓ 直轄工事での3Dモデル活用が標準に
次世代i-Construction（2025年〜）
  ↓ 自動化・省人化・デジタルツイン連携
```

### 主要な規制要件

| 要件 | 内容 | 影響 |
|---|---|---|
| CIM原則化 | 一定規模以上の直轄工事で3Dモデル作成・活用が必須 | 受注者はBIM/CIM対応が必要 |
| 電子納品 | 成果品を3次元データで納品。ICT建機の計測データも含む | 施工データの自動取得・変換が必要 |
| 出来形管理自動化 | ICT建機による3次元計測で帳票を自動生成 | ポイントクラウドとBIMモデルの比較処理が必要 |
| 書類のデジタル化 | 施工管理帳票の電子化推進 | クラウドシステムとの連携が前提 |

## 6. 実装上の課題

### 座標系の統一

建設機械システムでは3つの座標系が存在し、それぞれの変換が必要である。

| 座標系 | 説明 | 変換方法 |
|---|---|---|
| 測地座標系（WGS84・JGD2011） | GNSS/RTKが出力する緯度・経度・標高 | 国土地理院のパラメータ使用 |
| 施工座標系（CAD座標系） | BIM/CIMモデルが使用するローカル座標 | 現場基準点との関係で定義 |
| 機械座標系 | 機体中心を原点とした相対座標 | IMU・アーム角度センサで計算 |

これらの変換はLinux ECU上でリアルタイムに実行する必要がある。

### リアルタイム性要件

| 処理 | 目標更新周期 | 備考 |
|---|---|---|
| 施工ガイダンス表示 | 100ms以内（要確定） | オペレーターが視認できる速度 |
| Machine Control制御指令 | 10〜50ms（要確定） | 油圧応答速度に合わせる |
| 進捗データのクラウド送信 | 数分〜数十分単位 | リアルタイム性は低くてよい |

### オフライン対応

建設現場はLTE/5G電波が届かない環境も多い。Linux ECU内にBIMモデルをキャッシュし、オフライン時でも施工ガイダンス・Machine Controlを継続できる設計が必要である。クラウド復帰時に差分データを同期する仕組みも必要となる。

### BIMモデルの更新管理

設計変更が発生した場合、クラウド上のBIMモデルと機械側のキャッシュの同期タイミングを適切に管理しなければならない。更新前後で機械が異なるモデルで動作すると施工品質に影響するため、バージョン管理と配信制御の仕組みが必要である。

## 7. 自社への示唆・ロードマップ

### フェーズ別展開方針

```
Phase 1（〜2年）        Phase 2（2〜4年）        Phase 3（4〜6年）
────────────────────────────────────────────────────────────────────
施工ガイダンス          Machine Control          クラウドBIM連携
（HMI表示）             （半自律制御）           （進捗自動フィードバック）

・GNSS/RTK搭載          ・Classic AUTOSARへの    ・施工ログの自動収集
・刃先位置計算           制御指令インタフェース   ・出来形管理の自動化
・BIMモデルとの          定義                     ・BIMモデルの自動更新
  差分をHMI表示         ・油圧弁の自動制御         （設計変更の反映）
・LandXML形式の         ・安全監視・フェールセーフ ・ポイントクラウドと
  受信・解析             実装（ISO 25119適用）      BIMモデルの差分比較
```

### AUTOSARとBIM/CIM連携の接点

BIM/CIM連携において、Classic AUTOSAR ECUが直接BIMデータを扱う必要はない。**Linuxゲートウェイ ECUがデータ変換・判断を行い、Classic AUTOSAR ECUには制御指令（目標位置・速度・力）のみを渡す**設計が現実的である。

この役割分担により、AUTOSAR側の設計はBIM/CIMの仕様変更に影響されず、安全機能（ISO 25119対応）を安定して維持できる。

| 役割 | 担当 | 備考 |
|---|---|---|
| BIMデータの解釈・変換 | Linux ECU（非安全系） | 設計変更に柔軟に対応 |
| 安全クリティカルな制御 | Classic AUTOSAR ECU（安全系） | ISO 25119 ASIL相当 |
| 安全監視（オーバーライド） | Classic AUTOSAR ECU | 異常時は自動停止 |

Machine Controlを実装する場合、油圧制御の自動化はISO 25119に基づく機能安全設計が必要となる。Phase 2以降の検討では、`functional-safety.md`・`iso26262-vs-iso25119.md` との整合を確認すること。
