# 検証環境構築ガイド — Classic AUTOSAR 建設機械プラットフォーム

## 1. 全体像：Vモデルと検証レベル

```
要件定義 ─────────────────────────────── 車両テスト
  │                                          │
  アーキ設計 ───────────────────── HIL検証
    │                                    │
    コンポーネント設計 ─────── SIL検証
      │                              │
      モデル設計 ────── MIL検証
               ↓
          実装・統合
```

| レベル | 実施場所 | 主な検証対象 | 担当 |
|--------|----------|--------------|------|
| MIL | PC（Simulink） | 制御アルゴリズムの動作 | 委託先（自社レビュー） |
| SIL | PC（VEOS等） | AUTOSARソフト全体 | 委託先（自社受け入れ） |
| HIL | HILラック | ECU＋実配線 | **自社主体** |
| 車両テスト | 実機 | 統合・機能安全確認 | **自社主体** |

自社1〜2名は MIL/SIL の成果物レビューと、HIL以降の受け入れ検証に集中する。

## 2. MIL（Model-in-the-Loop）

委託先が Simulink でモデルを構築し、自社はレビューと受け入れ基準の合意を担当する。

**建設機械固有の検証ポイント**

- 油圧モデル：ポンプ流量・圧力特性のリファレンスモデルとの誤差 ±2% 以内
- 旋回・走行制御：急操作時のショック抑制（加速度閾値を要件として事前定義）
- アイドルストップ：油温・水温条件との組み合わせ

### 建設機械の油圧制御を例にとったMIL検証の具体例

油圧ショベルのブームリフト制御を例に、MILで何をテストするかを説明する。

**テスト対象：ブームリフト制御モデル（Simulink）**

```
オペレータレバー入力（0〜100%）
        ↓
  制御モデル（Simulink）
        ↓
メインポンプ吐出量指令（L/min）
コントロールバルブ開度指令（%）
```

| テストシナリオ | 入力条件 | 期待する出力 | 合否判定基準 |
|---|---|---|---|
| 正常ブームリフト | レバー50%入力、エンジン回転数1800rpm | バルブ開度50±2%、流量30±1L/min | 定常値が5秒以内に収束 |
| 急操作ショック抑制 | レバーを0→100%に0.1秒でステップ入力 | 圧力スパイクが設定上限（35MPa）以下 | ピーク圧力の超過なし |
| アイドル時の応答 | エンジン回転数800rpm（アイドル）でレバー50% | ポンプ最大吐出量で制限されること | 設計上限流量を超えない |
| フェールセーフ | 圧力センサ断線（フォルト注入） | バルブ指令が0%（全閉）に切り替わる | 300ms以内にフェールセーフ移行 |
| 低温始動 | 油温0℃、粘度高い状態をモデルで模擬 | 操作応答が遅延しても動作すること | 応答時間が要件値以内 |

**MILで検証できること・できないこと**

- 検証できること：制御アルゴリズムの論理的正しさ、過渡応答特性、フェールセーフ遷移の条件
- 検証できないこと：実ECUのCPU処理負荷、割り込みタイミング、実センサのノイズ特性、電気的インタフェース

**成果物（委託先から受領）**

- Simulink モデル一式（.slx）
- テスト結果レポート（Simulink Test またはMAP）
- カバレッジレポート（MC/DC：ISO 25119 の要求）

## 3. SIL（Software-in-the-Loop）

PC 上で AUTOSAR 生成コードを実行し、ECU なしでソフトウェア全体を検証する。

### 建設機械の油圧制御を例にとったSIL検証の具体例

MILではSimulinkモデルを検証したが、SILではそのモデルから生成されたC/C++コード（Embedded Coder等）と、AUTOSARのRTE・BSW構成を合わせた「ソフトウェアとしての動作」を検証する。

**SILで追加して検証できること：**

| 検証項目 | 具体的な内容 |
|---|---|
| CANメッセージのタイミング | 10msサイクルでHydraulicCommandが正しく送信されるか |
| RTE経由のデータ受け渡し | OperatorInput SWCからHydraulicControl SWCへのPport/Rportが正しく配線されているか |
| BSW設定の妥当性 | ComStackの送信スケジューラが仕様通りに動作するか |
| スタックオーバーフロー | タスクの最大スタック使用量がLinuxシミュレータ上で計測できる |
| エラーフレーム受信時の動作 | CANフレームのCRCエラーをSIL上で注入し、エラーハンドリングを確認 |

**主要ツール選択肢**

| ツール | 特徴 | 費用感 |
|--------|------|--------|
| dSPACE VEOS | AUTOSAR対応、Simulink連携◎ | 高（ライセンス制） |
| ETAS ISOLAR-EVE | ISOLAR-A/B と統合 | 中〜高 |
| QEMU + Autosar OS | オープン系、カスタマイズ必要 | ライセンス費ゼロ |

**CI パイプラインへの組み込み（GitLab CI 例）**

```yaml
sil-test:
  stage: test
  script:
    - veos-run --config sil_config.xml --testcases tests/
    - python scripts/parse_results.py --format junit
  artifacts:
    reports:
      junit: results/sil_report.xml
```

委託先の CI に自社担当者がオブザーバーアクセスできる権限を契約時に明記する。

## 4. HIL（Hardware-in-the-Loop）

実 ECU と HIL シミュレータを接続し、実時間で統合テストを行う。

### 建設機械の油圧制御を例にとったHIL検証の具体例

HILではSILでは確認できなかった「実ECUの電気的・時間的な動作」を検証する。

**HILで初めて検証できること：**

| 検証項目 | 具体的な内容 |
|---|---|
| ECUのリアルタイム応答 | レバー入力からバルブ指令CAN送信までのエンドツーエンドのレイテンシ（目標：10ms以内） |
| アナログI/Oの精度 | 圧力センサ模擬電圧（0〜5V）をHILから入力し、ECUのA/D変換値が仕様誤差内か確認 |
| 電源ノイズ耐性 | 12V電源ラインにノイズを重畳し、ECUが誤動作しないことを確認 |
| ウォッチドッグタイムアウト | ソフトウェアが意図的にウォッチドッグをリセットしない状態を作り、ECUリセットが発生することを確認 |
| CANバスエラー注入 | CANコントローラのエラーカウンタをBusOffに誘導し、リカバリシーケンスを検証 |
| 温度ソーク試験 | ECUを-20℃〜85℃の環境で動かしながらHIL通信テストを継続 |

### 「50万円構成」から始める具体的な手順

以下の手順で、予算を最小化しながらISO 25119の最低限の証跡ループを構築できる。

**ステップ1：CANoeの評価版を入手する（0円・1週間）**

1. Vector Informatikのウェブサイト（vector.com）から「CANoe Demo」を申請する。
2. Windows PCにインストールし、委託先から受領したDBCファイル（CANマトリクス）をインポートする。
3. CANoe上でバスシミュレーションが動作することを確認する。

**ステップ2：VN1630Aを接続してECUとの通信を確認する（1〜2週間）**

1. VN1630AをPCにUSBで接続する。Vector Driver Setup（無償）をインストールする。
2. ECU評価ボードのCANコネクタとVN1630AのCAN chをD-Sub 9pinケーブルで接続する。
3. CANoe上でメッセージ受信が確認できれば、最小HIL構成が動作している。

**ステップ3：手動テスト手順書を作成し、CANoe Traceで証跡を取得する（2〜4週間）**

1. Excelで「テストケースID / 入力操作 / 期待するCAN信号 / 合否」の手順書を作成する。
2. テスト実施時にCANoe の「Trace」ウィンドウを開き、ログファイルに保存する（.blf形式）。
3. テスト完了後、ログとExcel手順書をセットでISO 25119の証跡として保管する。

**ステップ4：CANoe Testモジュールで手動テストを自動化する（1〜2ヶ月）**

1. CANoe Test Feature Set（オプション）を使い、CAPLスクリプトまたはPython APIでテストを自動実行できるようにする。
2. Robot Framework + CANoe Python APIを使う場合は、`verification-environment.md`のテスト自動化戦略を参照。

**機能安全検証証跡の生成**

ISO 25119 Part 4 では「テスト仕様 → 実施記録 → 結果」の3点セットが必要。

| 証跡ドキュメント | 生成元 | 管理ツール |
|------------------|--------|------------|
| テスト仕様 | 要件から手動作成 | Polarion / Jama |
| 実施ログ | CANoe Trace / dSPACE自動生成 | バイナリ+PDF保存 |
| 合否判定レポート | スクリプト自動集計 | Polarion へ自動インポート |

## 5. テスト自動化戦略

**Robot Framework ＋ CANoe Python API（ATDD）**

```
Polarion（テスト仕様）
    ↓ エクスポート
Robot Framework（テストスクリプト）
    ↓ CANoe COM API
CANoe（バス刺激・測定）
    ↓ 結果
Polarion（自動インポート）
```

**MISRA-C チェックの自動化**

委託先のビルドパイプラインに組み込み、違反ゼロを Pull Request マージ条件とする。

```bash
# PC-lint Plus / PRQA の例
pclp64 +ffn -u misra_c2012.lnt src/**/*.c > misra_report.txt
```

自社は週次でレポートをレビューし、偏差申請（Deviation）の承認を行う。

## 6. 現実的な構築順序

### 今すぐ始められる最小構成（予算：〜50万円）

1. **CANoe ライセンス取得**（Vector Informatik — 評価版30日 → 購入）
2. **DBC ファイル整備**：委託先から CAN マトリクスを受領し CANoe にインポート
3. **ECU評価ボード**：委託先から提供される評価ボードを CANoe に接続
4. **手動テスト手順書**を Excel で作成し、CANoe Trace で証跡を取得

この段階で ISO 25119 の最低限の証跡ループが回せる。

### フェーズ2（6〜12か月後、予算：〜500万円）

- dSPACE MicroLabBox（DS1202）または SCALEXIO を導入し、I/O 模擬を自動化
- Robot Framework による自動テストスクリプトを整備
- Polarion でテスト管理と要件トレーサビリティを一元化

### フェーズ3（量産前、予算：〜2000万円）

- フルスペック HIL ラック（実ハーネス・アクチュエータ模擬込み）
- SIL→HIL の回帰テストを CI/CD として常時実行
- 機能安全監査に対応した証跡パッケージを自動生成
