# 委託先選定基準

**作成日：** 2026年6月16日  
**対象読者：** 委託先選定担当者（技術担当）  
**資料種別：** 社内運用資料（社外秘）

---

## 概要

AUTOSAR準拠のプラットフォームソフトウェア開発を外部委託する際の、ベンダー選定基準を定義する。選定は「技術力」「規格対応力」「長期パートナーとしての信頼性」の3軸で評価する。

AUTOSAR領域のベンダー選定で最も犯しやすいミスは「できると言っているから大丈夫だろう」という見込み判断である。本資料では「言葉でなく実績と体制で確認する」ことを徹底し、実務で使えるチェックリスト形式で整理する。

**用語補足：**

| 用語 | 読み方 | 意味 |
|---|---|---|
| RFI（Request for Information） | アールエフアイ | 正式発注前に候補ベンダーへ「どんな技術・体制を持っているか」を文書で回答してもらう情報提供依頼。入札の前段として使う |
| ASIL（Automotive Safety Integrity Level） | エイシル | ISO 26262における安全度水準。A〜Dの4段階でDが最高。本プロジェクトでは建設機械向けISO 25119を適用するが、委託先はISO 26262ベースの経験を持つことが多いため、能力確認の指標としてASILを使う |
| ECR（Engineering Change Request） | イーシーアール | 設計変更依頼。設計が固まった後に仕様変更が必要になった際にベンダーへ正式に依頼する手続き |
| SLA（Service Level Agreement） | エスエルエー | 障害対応時間・変更対応期間・稼働率等についての合意事項。契約書に明記することで、納品後のトラブル対応が曖昧にならないようにする |
| UDS（Unified Diagnostic Services） | ユーディーエス | ISO 14229で定められた診断通信の標準仕様。建設機械・車載ECUの診断（故障コード読み出し・ソフト書き換え等）に広く使われる |
| J1939 | ジェイいちきゅうさんきゅう | 建設機械・農業機械・トラック向けのCAN上位通信プロトコル。エンジンECUやトランスミッションECU間の通信に広く使われており、自社機械のCAN配線にも関係する |
| MISRA-C | ミスラシー | 組み込みソフトウェア向けのCコーディング規約。バグになりやすいC言語の記述パターンを禁止するルール集。機能安全開発では準拠が前提とされることが多い |
| HIL（Hardware-in-the-Loop） | エイチアイエル | 実際のECU基板とテスト装置を接続して動作確認するテスト手法。センサーやアクチュエータをシミュレーターで代替するため、実機なしに多くの試験を実施できる |

## 1. 必須要件（Must Have）――これを満たさないベンダーは対象外

以下の項目は、いずれか一つでも満たせないベンダーは選定から除外する。「後から整備する予定」という回答は満たしていないと判断すること。

### 1-1. Classic AUTOSAR対応実績

Classic AUTOSARとは、リアルタイムOS上で動く制御系向けの規格である。油圧制御・ブレーキ制御のような厳密なタイミング制御が必要な機能に適用する。ベンダーがこの規格を「知っている」だけでなく「実際に製品に適用した経験があるか」を確認する。

| 確認項目 | 確認方法 | なぜ必要か |
|---|---|---|
| Classic AUTOSAR BSW・RTE開発実績がある（件数・業種を問う） | 事例資料の提示を求める。匿名可だが業種・規模・担当範囲を明記させる | 実績なしに机上知識だけでは、PoC段階でトラブルが頻発し手戻りコストが膨らむ |
| 使用するツールチェーンを明示できる（DaVinci / EB tresos / ORIENTAIS等） | RFIで記載させ、理由も問う | ツールによって習熟度・サポート体制が異なる。「何でも使えます」は要注意 |
| ArXML形式での成果物納品が可能か | 契約条件として要件に明記し、サンプルを提示してもらう | ArXML納品がなければ将来のベンダー変更・設計継承が困難になる |

**確認のポイント：** 事例を見せてもらう際は「どのモジュールを担当したか」「何人体制か」「どの期間で納品したか」まで聞く。BSWを全部作ったのか、一部モジュールだけなのかで技術の深さが大きく異なる。

### 1-2. Adaptive AUTOSAR / Linuxミドルウェア対応実績

Linux系（HMI・ゲートウェイ・人検知・クラウド通信）の委託も視野に入れるため、以下を別途確認する。Classic AUTOSARと役割が異なり、こちらは「地図処理・画像処理・ネットワーク通信」のような高機能処理に使われる規格である。

| 確認項目 | 確認方法 | なぜ必要か |
|---|---|---|
| Adaptive AUTOSARまたはLinuxベースのミドルウェア開発実績 | 事例資料の提示を求める | Classic専門の会社がAdaptiveを「できます」と言うケースがあるが、規格体系が全く異なるため別スキルとして確認が必要 |
| ara::com / SOME/IP の実装経験 | RFIで確認し、テスト環境の有無も確認 | ara::comはAdaptive AUTOSARの通信基盤。SOME/IPはEthernet上の通信プロトコルで、HMIやゲートウェイに必要になる |

### 1-3. 機能安全対応

機能安全対応とは、規格が求める開発プロセスを踏んで開発し、その証拠を残せることを意味する。「安全な設計をする」ことと「機能安全規格に準拠した開発をする」ことは別物である。

| 確認項目 | 確認方法 | なぜ必要か |
|---|---|---|
| ISO 26262またはISO 25119の開発プロセスを文書化・運用している | 開発プロセス文書（工程フロー・承認手順等）の提示を求める | プロセスがなければ認証審査に耐えられない。「安全を意識して開発しています」は証拠にならない |
| 機能安全対応の開発実績がある | 事例提示（匿名可）を求める。認証取得の有無も確認する | 初めて機能安全対応をするベンダーを採用すると、学習コストを自社が負担する羽目になる |
| ASIL-B以上の対応経験がある | 担当エンジニアのスキル・資格（TÜV SÜD機能安全エンジニア等）を確認する | 建設機械の油圧制御はASIL-B〜C相当の安全レベルが要求される可能性がある。ASIL-Aのみの経験では対応できない場合がある |

### 1-4. セキュリティ対応

コネクテッド機能（クラウド通信・遠隔監視）を持つ製品では、サイバーセキュリティ対応も不可欠になる。

| 確認項目 | 確認方法 | なぜ必要か |
|---|---|---|
| ISO/SAE 21434またはUNECE WP.29対応実績がある | RFIで確認し、具体的な対応内容を記載させる | セキュリティ対応なしに遠隔通信機能を持つ製品を出すことは将来的に法的・社会的リスクになる |
| 脆弱性管理・セキュリティテストのプロセスを持つ | プロセス文書の提示を求め、脆弱性発見から修正・報告までのフローを確認する | 開発後に脆弱性が見つかった際の対応プロセスがないと、修正対応が場当たり的になる |

## 2. 重要要件（Should Have）――選定の差別化に使う評価項目

Must Haveを通過したベンダーを差別化するための評価基準。これらは点数化して比較に使う（セクション4参照）。

### 2-1. 建設機械・重機領域の知識

自動車向けの経験が豊富でも、建設機械固有の環境（振動・電源変動・過酷な温度環境・J1939通信）への対応経験がないと、設計上の考慮漏れが生じることがある。

- **CAN・CAN FD・Ethernetを用いた制御系の開発経験**：建設機械はCANを制御ネットワークとして使う。CAN FDは高速版で、大量のデータを扱う場合に使われる。経験の有無を具体的な実績で確認する。
- **J1939の取り扱い経験**：J1939は建設機械・農業機械・トラック向けのCAN上位プロトコル。エンジンECU・トランスミッションECU等との通信に必要。「CAN経験あり」とJ1939経験は別物であるため必ず区別して確認する。
- **油圧・電動アクチュエータ制御との統合経験**：ソフトウェアが制御するアクチュエータの特性（応答遅れ・非線形性・フェールセーフ時の挙動）を理解しているかどうかで、設計品質が変わる。

### 2-2. 保守運用体制

開発完了が「終わり」ではなく、その後10年以上にわたる保守が続く。長期パートナーとして機能するかを確認する。

- **納品後の長期保守契約が可能か**：保守契約の有無・期間・費用体系を確認する。「都度相談」では見通しが立たない。
- **ECR（設計変更依頼）への対応フローが整備されているか**：変更依頼が来てから対応着手までの手順・期間・費用の算出方法が明確かを確認する。手順がないと変更のたびに交渉から始まることになる。
- **担当エンジニアの継続性が担保されるか**：「担当者が変わったら引き継ぎができなかった」というトラブルは実際に多い。担当者交代ポリシー・引き継ぎ手順を確認する。

### 2-3. 知識移転への協力

自社担当者が「成果物を評価・レビューできる状態」になることが長期的な自社の技術力につながる。ベンダーが協力的かどうかを事前に確認する。

- **自社担当者へのレビュー・技術説明を契約に含められるか**：定期的な技術説明会・設計レビューへの参加を義務化できるかを確認する。「聞けば答える」と「説明する義務がある」では大きく違う。
- **ドキュメントの納品形式・品質水準**：要求仕様書・設計書・テスト仕様書が揃って納品されるか、および自社担当者が読んで理解できる品質かを確認する。ArXMLだけ渡されても読めなければ意味がない。
- **将来的に自社で一部内製化する際の協力姿勢**：長期的に自社でも担える機能を増やしたい場合、ベンダーが「育てることに協力的か」を打診しておく。協力拒否は競合を恐れているサインであり、長期パートナーとしては不安材料になる。

## 3. 選定上の注意点

**会社の規模・知名度で判断しない：** 大手SIerや有名メーカー系列であっても、AUTOSARの実務経験がなければ意味がない。逆に中小規模でも機能安全・AUTOSARに特化した会社は存在する。ブランドではなく実績で判断すること。

**価格の安さのみで判断しない：** 機能安全領域での手戻りコスト（設計のやり直し・認証再取得・納期遅延）は、開発コストの数倍になることがある。安いベンダーを選んで手戻りが生じた場合の総コストは、最初から適切なベンダーを選んだ場合を大きく上回ることが多い。

**「できます」という返答のみで判断しない：** AUTOSARの問い合わせに対して「できます」と答えるベンダーは多い。必ず「どのプロジェクトで・どのモジュールを・何人体制で・どの期間で対応したか」を具体的に確認する。曖昧な回答しか返ってこない場合は実績不足のサインである。

**担当者個人でなく組織の能力を確認する：** 「この人はすごいが会社全体では1人だけ」というケースがある。担当予定エンジニアの人数・バックアップ体制・会社としてのノウハウ継承の仕組みを確認する。

## 4. 評価スコアリング方針

Must Have通過後、以下の重み付けでスコアリングする。スコアリングは定性評価（5段階）×重み係数で合計点を算出し、数値で比較できるようにする。

| 評価軸 | 重み | 評価のポイント |
|---|---|---|
| 技術力（AUTOSAR・機能安全・セキュリティの実績） | 50% | 実績件数・規模・担当範囲の深さ・使用ツールの習熟度 |
| 保守体制・長期パートナーとしての信頼性 | 30% | 保守契約の実績・担当継続性・コミュニケーションの質 |
| コスト | 20% | PoC見積もり・長期保守費用の見通し・変更対応費用の透明性 |

**選定プロセス：** 3社以上にRFIを送付し、Must Have評価と書類審査でスコアリング上位2社まで絞り込む。その2社でPoC（概念実証）を実施し、最終選定はPoC評価を踏まえて行う。1社のみでPoCを行い全体移行を決定しないこと。

## 5. PoC対象機能の推奨選定

### 推奨：診断通信（Dcm + Dem）から始める

最初のPoCとして油圧制御のような安全クリティカル機能は避け、**診断通信（Dcm：Diagnostic Communication Manager＋Dem：Diagnostic Event Manager）**から始めることを推奨する。

DcmはECUに対して「故障コードを読み出す」「ソフトウェアを書き換える」などの診断操作を処理するモジュール。DemはECU内部の故障検出結果を記録・管理するモジュールである。両者は密接に連携して動作する。

**診断通信から始めることを推奨する理由：**
- **合否判定が客観的にできる**：UDS（ISO 14229）という明確な外部仕様があり、「UDSのサービスXに対してY ms以内にレスポンスが返るか」という形で結果を数値判定できる。曖昧さがない。
- **安全機能には干渉しない**：油圧制御と同じECUに載るが、診断通信は制御出力には影響しない。万一PoCで問題が起きても制御系に影響しないため、実機評価リスクが低い。
- **NvM・RTEのSWC連携を実際に検証できる**：NvM（不揮発メモリ管理）やRTEのSWC間通信を含む構成をPoCで実際に動かすことで、本番環境に近いAUTOSARの動作を確認できる。
- **テスト自動化がしやすい**：CANoe等の市販ツールで診断通信のテストを自動化できるため、担当者1〜2名での管理負荷が低い。毎回手作業でテストする必要がない。

**次のステップ（PoC第2フェーズ）：**
Com Stack（PduR＋CanIf）を追加し、通信タイミングと多重化の検証に進む。安全クリティカル機能（油圧制御SWC）は委託先の能力が確認されてから対象にする。無理に安全機能から始めることで、問題が生じた際の影響と対応コストが跳ね上がるリスクを避ける。

### PoCの成功・失敗の判断指標

| 指標 | 合格基準の例 | なぜこの基準か |
|---|---|---|
| タイミング精度 | タスク周期ずれ ±5%以内 | 制御系ではタイミングずれが制御品質に直結する。許容範囲は機能ごとに要求仕様で定める |
| 診断応答 | UDS応答時間 P2/P2*以内 | ISO 14229で定められた応答時間。この基準を超えると診断ツール側でタイムアウト判定される |
| MISRA-C準拠 | Critical違反0件 | MISRA-Cの「Critical」カテゴリは最も危険なコーディングパターン。機能安全コードでは0件が最低ライン |
| ドキュメント品質 | ArXMLと設計書の整合が取れている | 設計書の内容とArXMLの定義が食い違っていると、変更時に何が正しいか判断できなくなる |
| 変更対応能力 | 軽微な変更要求への対応が1週間以内 | PoCの期間中に意図的に軽微な変更依頼を入れ、対応速度とコミュニケーション品質を測る |

**PoCで何を見るか：** PoCの本質は「委託先がAUTOSARツールチェーンを実務レベルで扱えるか」の早期確認にある。機能の完成度より**変更要求への対応速度と設計根拠の説明能力**を重視して評価すること。「なぜそう設計したのか」を説明できないベンダーは、問題が起きた時に対処できない。

## 6. RFI（情報提供依頼）で確認すべき項目

正式発注の前に、候補ベンダーへ以下の内容を文書で提出するよう依頼する。RFIは「お互いの能力と期待値を確認する場」であるため、明確に記載できないベンダーは実力不足と判断してよい。

RFIの送付方法：メールまたは書面でWordまたはPDF形式の質問票を送付し、2週間以内に回答してもらう。口頭での回答は記録が残らないため、必ず文書回答を求めること。

```
RFI質問票

1. Classic AUTOSAR BSW/RTE開発実績
   - 件数・業種・主要モジュール（Os / Com / CanIf / Dcm / Dem / NvM 等）
   - 使用ツールチェーン（DaVinci / EB tresos / ORIENTAIS等）とバージョン
   - 担当エンジニアの人数・体制

2. Adaptive AUTOSARまたはLinuxミドルウェア開発実績
   - 件数・業種・担当範囲
   - ara::com / SOME/IP の実装経験の有無

3. 機能安全対応実績
   - 適用規格（ISO 26262 / ISO 25119）
   - ASILまたはPLレベル
   - 担当した役割（全体 / BSW / SWC）
   - 認証取得実績の有無（取得機関名）

4. セキュリティ対応実績
   - 適用規格（ISO/SAE 21434 / UNECE WP.29）
   - 脅威分析・脆弱性管理のプロセス整備状況

5. 使用するツールチェーンとその理由
   （なぜそのツールを選んでいるか、他ツールとの比較理由）

6. 成果物の納品フォーマット
   - ArXML（スキーマバージョンの明示）
   - 設計書・テスト仕様書のフォーマット例

7. 保守運用の体制と契約形態
   - 保守契約の有無・標準期間・費用体系
   - ECR（設計変更依頼）対応フローとリードタイム

8. 体制図
   - プロジェクト担当者の人数・役割
   - 各担当者のAUTOSAR・機能安全に関するスキル・経験年数

9. 参考見積もり（PoC規模：診断通信Dcm + Dem、対象ECU1台）
   - 工数・費用・納期の目安
```

## 7. PoC段階での最終評価基準

ベンダーを最終決定する前に、PoCで以下を評価する。PoCは単なる技術検証でなく「このベンダーと長期間仕事ができるか」を判断する試験期間として位置づける。

| 評価項目 | 合格基準 | 確認方法 |
|---|---|---|
| 要求仕様の理解度 | 渡した要求に対して的確な質問・確認ができるか。「わかりました」とだけ言って後から大きな問題が出ないか | 質問の質と内容を記録する |
| 中間成果物の品質 | 設計書・ArXMLが標準準拠かつ自社担当者がレビューできる品質か | 中間レビュー会議を2〜3回設ける |
| コミュニケーション | 問題発生時に早期に報告・エスカレーションできるか。隠蔽や先送りがないか | 週次報告の質・問題報告のタイムラグを記録する |
| スケジュール管理 | マイルストーンの達成精度。遅延する場合に事前に報告できるか | ガントチャートと実績を比較する |
| 成果物の受け入れ検証対応 | 自社側のレビューコメントに誠実・迅速に対応できるか。「仕様通り」と押し返してこないか | レビュー指摘への対応時間と品質を記録する |

**PoCで合格基準を満たさなかった場合：** 次点ベンダーにPoC実施を依頼するか、要件定義の再検討を行う。「せっかくPoCまでやったから」という理由で合格基準に満たないベンダーに全体移行を決定しないこと。PoC段階での妥協が本番の大きなリスクになる。

## 8. 契約時に確認・明記すべき条件の骨子

選定完了後、以下を正式契約書に反映させること。法務・調達部門と協議の上で最終化すること。

| 契約条件 | 具体的な内容 | なぜ明記が必要か |
|---|---|---|
| 著作権・ソースコードの権利帰属 | 成果物（ソースコード・設計書・ArXML）の著作権は自社に帰属することを明記 | 明記がないとベンダーが権利を主張し、後からコピーや利用に制約が生じる場合がある |
| ArXML・設計ドキュメントの納品義務 | ArXMLおよびすべての設計ドキュメントを納品対象として明示 | 「ブラックボックス」での納品を防ぐ。将来のベンダー変更に必須 |
| 引き継ぎ義務の明記 | 契約終了時にベンダー変更を可能にするための引き継ぎ期間・範囲を定める | 引き継ぎ義務の規定がないと、ベンダー変更時に多大なコストと時間を要する |
| 機能安全・セキュリティ上の瑕疵担保 | ベンダー側の設計ミスに起因する機能安全・セキュリティ上の問題に対する責任範囲を定める | 事故発生時の責任分担が曖昧だと、全責任を自社が負うリスクがある |
| SLA（保守対応条件）の明記 | 障害対応時間・変更対応リードタイム・稼働率等の合意事項を数値で明記 | 「できる限り対応する」では不十分。数値基準があって初めて契約上の根拠になる |
