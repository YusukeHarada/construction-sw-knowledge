## はじめに

OTA と SBOM は、SDV（Software Defined Vehicle）化を進める上で避けて通れない二つの技術・管理概念である。どちらも「ソフトウェアをどう安全に届け、管理するか」というテーマに直結する。本資料では、両概念を知らない読者でも理解できるよう基本から説明し、建設機械への適用を具体的に考える。

---

## 1. OTA（Over-The-Air）とは

### 1.1 基本概念

**OTA とは**、無線通信（携帯網・Wi-Fi・Bluetooth 等）を経由してソフトウェア・ファームウェア・設定・地図データ等を更新する技術の総称である。スマートフォンの「システムアップデート」が身近な例にあたる。

**なぜ今 OTA が重要か**

従来の組み込みシステムはリリース後に変更することを想定していなかった。しかし SDV 化が進む現代では、次の三つのニーズがソフトウェア更新を常態化させている。

| ニーズ | 内容 |
|---|---|
| セキュリティパッチ | 脆弱性が発見されたら即座に修正を配布する必要がある |
| 機能追加・改善 | 販売後も機能を追加してユーザー体験を向上させる |
| キャリブレーション変更 | 制御パラメータを現場条件に合わせて最適化する |

**OTA で更新できるものの種類**

- **Firmware OTA（FOTA）**：MCU のファームウェア（フラッシュ書き込み）
- **Software OTA（SOTA）**：Linux 上のアプリケーション・ミドルウェア
- **Calibration OTA**：制御パラメータ（PID ゲイン・センサ補正値等）
- **Map OTA**：ナビ・自律走行用の地図データ

**OTA と「ソフトウェアリコール」の関係**

ソフトウェア不具合が原因のリコールは、従来ディーラー入庫による書き換えが必要だった。OTA が整備されていれば、条件を満たした車両に対してソフトウェアリコール相当の修正を非接触で配布できる。コスト削減・対応速度の向上という観点から自動車業界での普及が進んでいる。

### 1.2 車載 OTA の仕組み

#### アーキテクチャ全体像

```
[OTA クラウドサーバー]
        │  HTTPS / MQTT
        ▼
[テレマティクス GW / Central GW]
        │  CAN / Ethernet / SOME/IP
        ▼
[ドメインコントローラー / ECU 群]
```

クラウドサーバーが更新パッケージを配信し、車載の Central GW（またはテレマティクス GW）が受信・検証を行い、各 ECU へ転送する。建設機械では Linux ゲートウェイがこの役割を担うことが現実的である。

#### FOTA と SOTA の違い

| 項目 | FOTA（Firmware OTA） | SOTA（Software OTA） |
|---|---|---|
| 対象 | MCU ファームウェア | Linux アプリ・ミドルウェア |
| 更新単位 | フラッシュ書き込み（バイナリ全体） | パッケージ単位（rpm / deb / コンテナ） |
| 停止要件 | 原則エンジン停止・電源安定が必要 | オンライン更新も可能なケースがある |
| 難易度 | 高い（書き込み失敗で文鎮化リスク） | 低い（OSレベルのロールバック機構が使える） |

#### A/B パーティション（デュアルバンク）とロールバック

フラッシュメモリを A/B の二面に分割し、稼働中は A 面を使いながら B 面に新バージョンを書き込む手法。書き込み完了後に切り替え、起動に失敗した場合は自動的に A 面に戻る（ロールバック）。これにより更新失敗による文鎮化リスクを大幅に低減できる。

```
[A 面：稼働中 v1.0]  ←── 現在ブート中
[B 面：更新中 v1.1]  ←── バックグラウンドで書き込み
         ↓ 書き込み完了・検証OK
[A 面：旧 v1.0]      ←── ロールバック用に保持
[B 面：稼働中 v1.1]  ←── ブート切り替え
```

#### 差分更新（Delta Update）

ファームウェア全体を転送する代わりに、旧バージョンとの差分のみを転送する手法。通信量・転送時間を大幅に削減できるため、セルラー回線での OTA では必須となりつつある。代表的なアルゴリズムに bsdiff・VCDIFF・Zephyr の FOTA ライブラリ等がある。

#### コード署名と検証

更新パッケージが正規の配信元から改ざんなく届いていることを確認するために、公開鍵暗号方式による署名を用いる。

```
[OTA サーバー側]
  更新パッケージ → 秘密鍵で署名 → 署名付きパッケージを配信

[ECU 側]
  パッケージ受信 → 公開鍵で署名検証 → 検証 OK なら書き込み
                                        検証 NG なら破棄
```

証明書は階層構造（証明書チェーン）で管理し、ルート認証局（Root CA）→ 中間 CA → デバイス証明書という構造で信頼の連鎖を確立する。

#### 安全状態管理

機械が動作中に ECU のファームウェアを書き換えると、制御の喪失や機器の損傷につながる。OTA 実装では書き込み実行条件を厳密に定義する必要がある。

- エンジン停止・駐車ブレーキ確認後にのみ書き込みを開始する
- 書き込み中は外部電源またはバッテリー電圧の閾値管理を行う
- 書き込み途中での電源断を検知した場合はロールバックを実行する

### 1.3 UNECE WP.29 R156（OTA 規制）

**R156 とは**、国連の自動車規制フォーラム（WP.29）が策定した OTA 向け規制である。2022年7月以降に型式認定を受ける新型乗用車・商用車に対して義務化されている（日本・EU・韓国等が採択）。

**R156 が要求する主な事項**

| 要求事項 | 内容 |
|---|---|
| SUMS（ソフトウェア更新管理システム） | 更新のプロセス全体を文書化・管理するシステムの整備 |
| バージョン管理 | 全 ECU のソフトウェアバージョンを追跡できること |
| ロールバック | 更新失敗時に以前の状態に戻せること |
| セキュリティ | 不正な更新を防ぐコード署名・認証の実装 |
| 変更影響評価 | 更新が型式認定条件に影響しないかの確認 |

**建設機械への適用**

現状、建設機械は R156 の直接対象外である。しかし以下の理由から、将来的な対応を検討する価値がある。

- 建設機械の型式承認規制が強化された場合に同様の要求が波及する可能性がある
- OTA を実装する上でのベストプラクティスとして R156 要求を参考にできる
- 欧米向け輸出機については、現地規制の動向を継続的に確認する必要がある（要確定）

**OTA 実装前に整備すべきプロセス**

技術的な OTA 基盤を構築する前に、以下のプロセス・体制が必要である。

1. SUMS（ソフトウェア更新管理システム）の設計
2. リリース承認フロー（誰が何を承認するかの明確化）
3. 変更管理プロセス（バージョン番号・変更履歴の一元管理）
4. ロールアウト計画（対象機体の選定・段階的展開・緊急停止手順）

### 1.4 OTA プラットフォームの現状

代表的な OTA プラットフォームを整理する。

#### AWS IoT Greengrass / FreeRTOS OTA

AWS が提供するエッジコンピューティングフレームワーク。Linux 搭載デバイスと MCU（FreeRTOS）の両方に対応している。AWS IoT Core との統合により、デバイス管理・証明書管理・OTA ジョブ管理を一元的に扱える。クラウド連携を AWS で統一している場合に適する。

#### Azure IoT Hub / Device Update for IoT Hub

Microsoft が提供する IoT デバイス管理プラットフォーム。Device Update for IoT Hub は apt パッケージや画像（イメージ）ベースの更新に対応しており、差分更新（Delta Update）もサポートする。Azure エコシステムを採用している場合に適する。

#### Eclipse Kanto / COVESA

OSSのECU向けOTAフレームワーク。Eclipse Foundation 傘下のプロジェクトであり、COVESA（旧 GENIVI）の車載ソフトウェア標準に沿った設計になっている。特定クラウドへの依存を避けたい場合や、自社でフルスタックを制御したい場合に選択肢となる。

#### 商用 OTA プラットフォーム

- **Harman Ignite OTA**：Samsung Harman が提供する自動車 OEM 向け商用ソリューション。FOTA・SOTA 両対応。
- **Airbiquity Choreo**：OEM 採用実績が多い商用 OTA プラットフォーム。差分更新・証明書管理・キャンペーン管理機能を持つ。

#### 建設機械での現実的な選択肢

クラウド連携を既に AWS または Azure で構築している場合は、同一エコシステムの OTA サービスを選ぶのが導入コストを抑えられる。Linux ゲートウェイ向けの SOTA から始め、段階的に FOTA へ拡張するアプローチが現実的である。

---

### 1.5 機能安全と OTA の関係

OTA によってソフトウェアを更新しても、**更新後のソフトウェアは更新前と同じ機能安全要求（ISO 25119 相当）を満たし続けなければならない**。このため以下のプロセスが必要となる。

- OTA 配布前に SIL（Software-in-the-Loop）テストおよび HIL（Hardware-in-the-Loop）テストを通過させる
- テスト環境と同一のバイナリを配布する（ビルド再現性の確保）
- OTA キャンペーン管理（対象機体の絞り込み・段階的ロールアウト・問題発生時の緊急停止）

## 2. SBOM（Software Bill of Materials）とは

### 2.1 基本概念

**SBOM とは**、ソフトウェアを構成するコンポーネント（ライブラリ・フレームワーク・OSS 等）を一覧化した「部品表」である。製造業における BOM（Bill of Materials）のソフトウェア版と考えるとわかりやすい。

製品の部品表があれば「どの部品にリコールが発生したか」をすぐに特定できるように、SBOM があれば「脆弱性が発見された OSS を使っている製品はどれか」を即座に特定できる。

**なぜ SBOM が重要か**

現代のソフトウェアは膨大な数の OSS に依存しており、その依存関係は複雑化している。OSSに脆弱性が発見された場合、SBOM なしでは「自社製品がそのOSSを使っているかどうか」を確認するだけで多大な工数がかかる。

**Log4Shell 事件（2021年）**

Apache Log4j という広く使われている Java ライブラリに深刻な脆弱性が発見された事件。世界中の多くの企業が「自社製品に Log4j が含まれているか」の調査に追われた。SBOM を整備していた企業は数時間で確認できたが、整備していなかった企業は数週間かかった例もある。この事件が SBOM 需要を大幅に加速させた。

---

### 2.2 SBOM に含まれる情報

標準的な SBOM には以下の情報が含まれる。

| 情報 | 説明 |
|---|---|
| コンポーネント名 | ライブラリ・フレームワーク名 |
| バージョン | 使用している具体的なバージョン番号 |
| ライセンス | MIT・GPL・Apache 2.0 等のライセンス種別 |
| 依存関係 | コンポーネント間の依存ツリー |
| ハッシュ値 | ダウンロードしたバイナリの整合性確認用 |
| サプライヤー情報 | 開発元・入手経路 |
| CVE 照合情報 | 既知の脆弱性（CVE）との照合結果 |

---

### 2.3 SBOM のフォーマット標準

主要な三つの標準が存在する。

#### SPDX（Software Package Data Exchange）

Linux Foundation が策定。ISO/IEC 5962 として国際標準化されている。ライセンスコンプライアンス管理を主目的として設計されており、OSS ライセンス管理に長い実績を持つ。

#### CycloneDX

OWASP（Open Web Application Security Project）が策定。セキュリティユースケース（脆弱性管理・サプライチェーンリスク管理）に特化した設計になっている。

#### SWID（Software Identification Tags）

ISO/IEC 19770-2 として標準化されている。IT 資産管理（ITAM）の文脈で使われることが多い。

**どれを使うべきか**

車載・組み込み業界では **SPDX と CycloneDX が主流**である。委託先に SBOM を要求する場合は、どちらか一方に統一して指定することを推奨する。脆弱性管理を重視するなら CycloneDX、ライセンスコンプライアンスを重視するなら SPDX が向いている。

---

### 2.4 規制・規格による SBOM 要求

#### UNECE WP.29 R155（サイバーセキュリティ）

R155 はサイバーセキュリティ管理システム（CSMS）を要求する規制であり、SBOM という言葉は使っていないが、「ソフトウェアサプライチェーンのリスク管理」としてコンポーネント管理が間接的に求められる。

#### ISO/SAE 21434

車両サイバーセキュリティの国際規格。ソフトウェアサプライチェーンのリスク識別・評価を要求しており、SBOM はその実践手段として位置づけられる。

#### 米国大統領令 EO 14028（2021年）

バイデン政権が発令したサイバーセキュリティ強化に関する大統領令。連邦政府に納入するソフトウェアに SBOM の提供を義務付けた。民間企業へも波及する契機となっている。

#### EU サイバーレジリエンス法（CRA）

2024年に成立した EU 規制。IoT を含むデジタル製品に対してサイバーセキュリティ要件とともに SBOM の整備を義務付ける予定である（段階的施行）。

**建設機械への影響**

建設機械に対する SBOM の直接的な法的義務は現時点では限定的である。ただし、EU 向け輸出機については CRA の適用可能性を確認する必要がある（要確定）。また、顧客（ゼネコン・官公庁等）からのサプライチェーンセキュリティ要求が強まる動きもあり、事前に対応を進めておくことが望ましい。

---

### 2.5 SBOM の生成ツール

#### Syft（Anchore）

OSS の SBOM 生成ツール。コンテナイメージ・ファイルシステム・Linux パッケージを解析して SPDX または CycloneDX 形式の SBOM を出力する。CI/CD パイプラインへの組み込みが容易であり、まず試してみるツールとして最適。

```bash
# 例：Linux ルートファイルシステムから SBOM を生成
syft dir:/path/to/rootfs -o spdx-json > sbom.spdx.json
```

#### Black Duck（Synopsys）

商用の OSS 管理プラットフォーム。バイナリスキャン・ライセンス管理・脆弱性照合を統合している。ソースコードが入手できないサードパーティコンポーネントのバイナリ解析が可能な点が強み。

#### FOSSA

ライセンスコンプライアンス管理に強い商用サービス。多言語・多パッケージマネージャーに対応しており、ライセンス違反リスクの自動検出ができる。

#### Dependency-Track（OWASP）

SBOM を継続的に監視するためのプラットフォーム（OSS）。生成した SBOM をインポートし、新しい CVE が公開された際に自動的にアラートを出す。SBOM の「生成」だけでなく「継続監視」に使う。

#### Reuse（FSFE）

ソースコードファイル単位でライセンス情報を管理するツール（OSS）。各ソースファイルに SPDX 識別子を付与する規約に従って管理することで、正確な SBOM 生成の基盤となる。

---

### 2.6 車載・組み込みでの SBOM 課題

**バイナリのみで提供されるコンポーネント**

Classic AUTOSAR の商用 BSW（MCAL・COM スタック等）はバイナリ形式で提供されることが多く、ベンダーが SBOM を提供しない場合がある。Syft のようなバイナリスキャンツールでも内部 OSS の全量を検出できるとは限らない。委託契約時に SBOM 提供を条件として明記することが重要である。

**組み込みツールチェーンの SBOM**

コンパイラ（GCC・IAR 等）・リンカ・RTOS カーネルも SBOM の対象となりうる。ツールチェーンベンダーへの SBOM 提供要求が必要な場合がある（要確定）。

**委託先からの SBOM 受領**

外部委託でソフトウェアを開発する場合、納品物に SBOM を含めることを契約条件として明記しておかないと、後から要求しても対応してもらえないケースがある。SOW（作業範囲定義書）および検収条件に SBOM のフォーマット・粒度・更新タイミングを明記することを推奨する。

## 3. OTA と SBOM の関係

OTA と SBOM は独立した概念に見えるが、実際には強く依存し合っている。

**OTA 更新のたびに SBOM が変わる**

OTA でソフトウェアを更新すると、使用するライブラリのバージョンが変わる。SBOM はソフトウェアのバージョンに紐づくため、OTA 更新のたびに SBOM を更新・バージョン管理する必要がある。

**脆弱性発見から OTA パッチ適用までのフロー**

```
① 新しい CVE が公開される
        ↓
② SBOM を照合し、影響を受けるコンポーネント・対象機体を特定
        ↓
③ 修正版ソフトウェアをビルド・テスト（SIL/HIL）
        ↓
④ OTA キャンペーンを作成し、対象機体に段階的に配布
        ↓
⑤ SBOM を更新（修正後のバージョンに差し替え）
```

「**SBOM がなければ OTA での脆弱性管理はできない**」。OTA の仕組みを整備しても、どの機体にどのバージョンのソフトウェアが入っているかを追跡できなければ、脆弱性の影響範囲を特定できない。両者はセットで整備する必要がある。

## 4. 建設機械への適用ロードマップ

### Phase 1：今〜2年

**目標**：SBOM 生成の開始と Linux GW 限定の OTA 基盤構築

- 既存の Linux ゲートウェイシステムに Syft を導入し、SBOM 生成を試行する
- Dependency-Track を構築し、生成した SBOM の継続的脆弱性監視を始める
- AWS Greengrass または Azure IoT Hub の評価環境を構築し、Linux GW への SOTA を試験する
- 委託先との新規契約に SBOM 提供条件を追加する
- OTA キャンペーン管理・承認フローのプロセスを設計する

### Phase 2：2〜5年

**目標**：Classic AUTOSAR 側の SBOM 整備と ECU 全体への OTA 拡張

- BSW ベンダーに対して SBOM 提供を要求し、RTOS・Classic AUTOSAR スタックの SBOM を整備する
- A/B パーティション対応 ECU の設計を新規搭載 ECU から開始する
- FOTA 対応 ECU への段階的な OTA 拡張（まずはゲートウェイ配下の非安全クリティカル ECU から）
- OTA 後の機能安全検証プロセスを確立する（SIL/HIL 通過を必須とするリリースゲートの設定）
- SUMS（ソフトウェア更新管理システム）の正式運用開始

### Phase 3：5〜10年

**目標**：自動脆弱性検知から OTA パッチ適用までのパイプライン自動化

- CVE 公開 → SBOM 照合 → 影響機体特定 → アラート通知の自動化
- テスト自動化（SIL）と OTA 配布パイプラインの連携
- 全 ECU の OTA 対応完了・Hypervisor 統合後の OTA 管理への発展（2028年以降の次世代機）

## 5. 今すぐできること

### SBOM で今日から始めるステップ

1. **Syft をインストールし、手元の Linux システムで SBOM を生成してみる**
   - 既存の組み込み Linux ルートファイルシステムに対して実行するだけで、使用 OSS の一覧が得られる
   - 量の多さと把握できていない OSS の数に気づくことが第一歩

2. **生成した SBOM を Dependency-Track にインポートし、既知 CVE との照合を確認する**
   - 現在のシステムにどれほどの既知脆弱性が存在するかを可視化する

3. **次回の委託案件から SBOM 提供を SOW に明記する**

### OTA で今日から始めるステップ

1. **AWS Greengrass または Azure IoT Hub の無料評価環境を構築し、Linux GW への OTA を試験する**
   - 実際に動かすことで、通信設計・証明書管理・リリースフローの課題が具体化する

2. **OTA 適用時の安全状態管理条件を定義する**
   - 「どの状態のときに書き込みを許可するか」を設計ドキュメントに落とし込む

3. **OTA キャンペーン管理プロセスの草案を作成する**
   - 誰がリリースを承認し、どのように段階ロールアウトするかを決める

### 委託先への要求事項に追加すべき事項

| 要求事項 | 詳細 |
|---|---|
| SBOM 提供 | 納品時に SPDX または CycloneDX 形式の SBOM を提出すること |
| SBOM 更新 | ソフトウェア更新のたびに SBOM を更新すること |
| OTA 対応設計 | A/B パーティション対応・コード署名対応を設計に含めること |
| 脆弱性対応期間 | 納品後 X 年間はセキュリティパッチを提供すること（要確定） |
| ツールチェーン SBOM | 使用コンパイラ・RTOS 等のバージョン情報も開示すること |
