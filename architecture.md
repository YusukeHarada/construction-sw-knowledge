# アーキテクチャ概要

## 1. E/Eアーキテクチャの変遷と建設機械への適用

### アーキテクチャの進化

| 世代 | 構成 | 特徴 | 建設機械への適用 |
|------|------|------|-----------------|
| 分散型 | 機能ごとに専用ECU | ECU数が多く、配線複雑 | 現在の主流（既存機種） |
| ドメイン集中型 | パワートレイン／ボディ等ドメインごとにメインECU | ECU数削減、Ethernetバックボーン | 本プロジェクトの目標段階 |
| ゾーン型 | 車体ゾーンごとにゾーンECU + 中央コンピュータ | 完全SDV化 | 将来（5〜10年後） |

建設機械は自動車より開発サイクルが長く、部品調達制約も大きい。まずドメイン集中型を外部委託で確立し、段階的にSDV化を進める。

## 2. 推奨アーキテクチャ構成

### 現在：Classic AUTOSAR + Linux Gateway（物理分離）

以下の図は、建設機械における推奨構成の全体像を示す。各ノード間のプロトコルとデータの流れ方向を明示している。

**ノード間プロトコル・データフロー詳細**

| 区間 | プロトコル | 速度 | 主なデータ内容 | 方向 |
|------|-----------|------|--------------|------|
| Safety ECU ↔ Gateway ECU | CAN FD | 最大2Mbps | 油圧指令値・バルブ開度・走行速度指令 / センサーフィードバック | 双方向 |
| J1939 ECU群 ↔ Gateway ECU | J1939（CAN 2.0B） | 250kbps | エンジンRPM・油温・油圧・DTC | 双方向 |
| Gateway ECU ↔ クラウド | TLS 1.3 over LTE/5G | 〜数十Mbps | 稼働ログ・センサー集約値・OTAパッケージ | 双方向 |
| Gateway ECU ↔ HMI | CAN FD または Ethernet | 2Mbps〜 | 警告表示・操作受付・映像ストリーム | 双方向 |

### 将来：SoC Hypervisor統合

Classic AUTOSARとLinuxを単一SoC上のHypervisorで分離実行。物理ECU数を削減しつつ、安全ドメインのアイソレーションを維持する（詳細は `hypervisor.md` 参照）。

## 3. SOME/IPゲートウェイアーキテクチャ

SOME/IP（Scalable service-Oriented MiddlewarE over IP）はAdaptive AUTOSARやLinux系ミドルウェアがEthernet上でサービスを呼び出す際に用いるプロトコル。CAN FDで通信するClassic AUTOSAR側との橋渡しが「ゲートウェイ変換」の主な役割となる。

```
Classic AUTOSAR側（Safety ECU）         Linux側（Gateway / AP）
────────────────────────────────────────────────────────────────
  CAN FD フレーム                          SOME/IP メッセージ
  ┌──────────────────────────┐             ┌─────────────────────────────┐
  │ ID(11/29bit) | DLC | Data│             │ Header: ServiceID | MethodID│
  │ 例）0x200 | 8B | [油圧値]│             │ Payload: [油圧値(float32)]  │
  └──────────────────────────┘             └─────────────────────────────┘
           │ CAN FD                                    ▲
           ▼ 受信                                      │ vsomeip / CommonAPI
  ┌────────────────────────────────────────────────────┐
  │                  Gateway ECU                        │
  │                                                     │
  │  ┌──────────────────┐       ┌────────────────────┐ │
  │  │  AUTOSAR COM     │       │  Signal Router     │ │
  │  │  （受信・解析）   │──────►│  （マッピング・変換）│ │
  │  │                  │       │                    │ │
  │  │ ・PDU受信        │       │ CAN シグナル名      │ │
  │  │ ・シグナル抽出   │       │   → SOME/IP サービス│ │
  │  │ ・スケーリング   │       │     メソッドへ変換  │ │
  │  └──────────────────┘       └────────────────────┘ │
  └────────────────────────────────────────────────────┘
           │
           ▼ Ethernet（SOME/IP）
  Linux側サービスが「HydraulicPressureService」として油圧値を購読・利用
```

**ゲートウェイ設計方針**

ゲートウェイECUはシグナル変換のみを担う薄い層とし、ビジネスロジックを持たない設計とする。これにより変換レイヤのテストが容易になる。具体的には以下のルールを守る。

- シグナルのスケーリング変換（例：CAN生値 → 物理値 MPa）はゲートウェイで行ってよい
- 「このセンサー値が閾値を超えたら何かをする」というロジックはゲートウェイに実装しない。あくまでLinux側のサービスかSafety ECUが判断する
- 変換マッピング定義はコード上にハードコードせず、設定ファイル（ARXML / JSONマップ）で外部化し、変更容易性を確保する

## 4. セキュリティアーキテクチャ

### 多層防御の考え方

| 層 | 技術 | 役割 |
|----|------|------|
| メッセージ認証 | SecOC（AUTOSAR） | ECU間CAN FDフレームのMAC検証 |
| 鍵管理・暗号 | HSM（オンチップ） | 秘密鍵の安全な保持、AES/SHA処理 |
| ネットワーク分離 | VLAN + ゾーンFirewall | 安全系／情報系の通信ドメイン隔離 |
| 外部接続 | TLS 1.3 + 証明書 | クラウド・診断通信の暗号化 |

### SecOC（Secure Onboard Communication）の具体的な保護対象

SecOCはAUTOSARが定義するメッセージ認証のメカニズムで、CAN FDフレームに**MAC（Message Authentication Code）** を付加することで「送信元のなりすまし」と「メッセージの改ざん」を防ぐ。以下に建設機械固有の保護対象例を示す。

| 保護対象メッセージ | なぜ守る必要があるか | 攻撃が成功した場合のリスク |
|---|---|---|
| **油圧バルブ開度指令**（Safety ECU → バルブコントローラ） | 不正な開度指令でバケット・アームが意図せず動作する | 周囲作業員への接触事故、機体転倒 |
| **走行制御指令**（Safety ECU → 走行モータ） | 停止指令の改ざんで機体が止まらなくなる | 人身事故・衝突 |
| **非常停止コマンド**（遠隔操作端末 → Safety ECU） | 攻撃者が非常停止を無効化・偽の停止指令を送信する | 遠隔操作不能・ランサムウェア的妨害 |
| **OTA更新の認証トークン**（クラウド → Gateway ECU） | 不正なファームウェアを書き込まれる | 機能破壊・バックドア埋込み |
| **診断セッション認可**（Dcm → Safety ECU） | 不正なECU書き換え・パラメータ変更 | 安全設定の無効化 |

SecOCはCAN FD上の全安全関連メッセージに適用する。HSMは安全ECUに内蔵し、鍵がSWに露出しない構成とする。特に**CANバスは物理的に接続できれば誰でもフレームを送信できる**ため、MACによる送信元検証は不可欠である。

## 5. 建設機械固有の考慮事項

### 油圧制御系の安全アーキテクチャ

自動車のE-gas（電子スロットル）と同様に、油圧バルブ制御でも「指令経路」と「監視経路」を独立させる。

```
操作指令 ──► [主制御CPU: ASIL-C]──► 油圧バルブ
                     │
                     ▼ 監視（クロスチェック）
             [監視CPU: ASIL-D] ──► 緊急遮断弁
```

監視CPUは主制御CPUの出力を独立して検証し、異常時は油圧系統を安全状態（バルブ閉）へフェールセーフさせる。

### J1939とAUTOSARの共存

既存の油圧・エンジンECUはJ1939ベースで通信している。移行期間中は以下の方針で共存させる。

- J1939 ↔ CAN FD変換はゲートウェイECUで吸収
- AUTOSARのPDUマッピングでJ1939 PGNをシグナルとして取り込む
- 新規ECUのみClassic AUTOSARで実装し、既存ECUは原則改修しない

これにより既存機種への影響を最小化しながら、新制御機能をAUTOSAR上に段階的に移行できる。

**J1939 PGNマッピング例**

| J1939 PGN | 内容 | AUTOSARシグナル名（PDU） | 変換処理 |
|---|---|---|---|
| PGN 61444 (EEC1) | エンジンRPM | Engine_SpeedSignal | 0.125 rpm/bit スケーリング |
| PGN 65263 (EFL/P1) | エンジン油温 | Engine_OilTemp | −40°Cオフセット変換 |
| PGN 65257 (LFC) | 燃料消費量 | Fuel_Consumption | 累積値のリセット管理が必要 |

移行初期はゲートウェイのマッピングテーブルをYAML/ARXMLで管理し、J1939側ECUが将来AUTOSAR化された際にゲートウェイ設定のみ変更する構成とする。
