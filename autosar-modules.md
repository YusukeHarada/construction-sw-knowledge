# AUTOSAR BSWモジュール解説 — Ethernet通信系を中心に

対象読者：建設機械メーカーの技術担当者

---

## 1. BSWモジュール全体像

### AUTOSARの層構造とは何か

AUTOSARはソフトウェアを「役割ごとの層」に分けて整理している。上位の層ほどアプリケーションに近く、下位の層ほどハードウェアに近い。この層構造の最大のメリットは「下位の層を交換してもアプリが動き続ける」点にある。たとえばマイコンをルネサスからNXPに変えても、上位のBSW・SWCはそのまま使える。

```
【層の全体像】

  ┌──────────────────────────────────────────────┐
  │  SWC（Application Software Component）      │
  │  開発者が機能ロジックを書く唯一の領域        │
  └──────────────────┬───────────────────────────┘
                     │
  ┌──────────────────┴───────────────────────────┐
  │  RTE（Runtime Environment）                 │
  │  SWCとBSWの間の"配線"を自動生成する         │
  └──────────────────┬───────────────────────────┘
                     │
  ┌──────────────────┴───────────────────────────┐
  │  BSW（Basic Software）                      │
  │  通信・OS・診断・メモリ管理などのミドルウェア │
  │  （本ドキュメントの主題）                    │
  └──────────────────┬───────────────────────────┘
                     │
  ┌──────────────────┴───────────────────────────┐
  │  MCAL（Microcontroller Abstraction Layer）  │
  │  マイコンのペリフェラル（CAN/SPI/PWM等）    │
  │  をAUTOSAR標準APIで抽象化する               │
  └──────────────────┬───────────────────────────┘
                     │
              【ハードウェア（マイコン）】
```

### 通信関連レイヤー構造（CAN系 / Ethernet系 対比）

```
┌─────────────────────────────────────────────────────────────┐
│                   SWC（ソフトウェアコンポーネント）              │
│  例）油圧制御SWC、人検知受信SWC                               │
└───────────────────────────┬─────────────────────────────────┘
                            │ RTE（ランタイム環境）
                            │ ← SWCはRteのAPIだけを呼ぶ。下位層を意識しない
┌───────────────────────────┴─────────────────────────────────┐
│                        COM                                  │
│              （シグナル／PDUの送受信管理）                      │
│  ← CAN/Ethernetを問わず、SWCからは同じIFで扱える             │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────┴─────────────────────────────────┐
│                       PDUR                                  │
│               （PDUのルーティング振り分け）                     │
│  ← 「このPDUはCAN1へ」「こっちはUDPへ」を自動で振り分ける     │
└──────────────┬──────────────────────────┬───────────────────┘
               │ CAN系                    │ Ethernet系
┌──────────────┴────────┐   ┌─────────────┴─────────────────┐
│        CANIF          │   │           SoAD                │
│   （CANドライバIF）    │   │   （ソケット操作アダプタ）      │
└──────────────┬────────┘   └──────┬──────────────┬─────────┘
               │                  │ SOME/IP       │ その他
               │           ┌──────┴──────┐        │
               │           │  SOME/IP    │        │
               │           │ Transformer │        │
               │           └──────┬──────┘        │
               │                  └──────┬─────────┘
┌──────────────┴────────┐   ┌─────────────┴─────────────────┐
│      CAN Driver       │   │           TcpIp               │
│  （CAN FD対応を含む）  │   │  （UDP/TCP プロトコルスタック） │
└───────────────────────┘   └─────────────┬─────────────────┘
                                          │
                              ┌───────────┴───────────┐
                              │   EthIf / Eth Driver   │
                              │  （Ethernetドライバ）   │
                              └───────────────────────┘

  ※ SecOC（セキュリティモジュール）は COM と IF の間に横断的に適用
  ※ SD（Service Discovery）は SoAD と並列で動作し、サービス起動/終了を管理
```

## 2. 主要モジュールの役割

### COM（Communication）

SWCとBSW間でシグナル・PDUを受け渡す通信サービス層。送信タイミング（周期・イベント）の管理も担う。SWCはRteのAPIを経由してCOMのシグナルバッファに値を書き込み・読み出しするだけでよく、「どのバスに送るか」「どのCANフレームに収めるか」は意識しなくてよい。

**このモジュールがないと何が困るか：** SWCがCANフレームのビット位置を直接操作するコードを書かなければならなくなる。プロトコルが変わるたびにSWCを修正する必要が生じ、アプリロジックと通信処理が混在して保守不可能なコードになる。

**主な設定項目：**
- シグナルの送信モード（周期送信 / イベント送信 / 周期+イベント）
- 受信タイムアウト監視（Deadline Monitoring）：指定時間内にデータが届かない場合にエラーを通知
- フィルタリング：値が変化したときだけ送信するなど

### PDUR（PDU Router）

受信PDUを適切な上位モジュール（COM、SoAD等）へルーティングする中継ハブ。CAN/Eth混在構成でも単一の窓口となる。

**このモジュールがないと何が困るか：** CANとEthernetが混在する構成で、各モジュールが直接互いを呼び出す必要が生じ、モジュール間の依存が複雑に絡み合う。ゲートウェイ処理（CANで受けたPDUをEthernetへ転送する）もPDURのルーティング設定だけで実現できるが、PDURがなければゲートウェイのためだけに個別のコードを書かなければならない。

**主な設定項目：**
- ルーティングテーブル：「送信元モジュールのPDU ID → 送信先モジュールのPDU ID」のマッピング
- ゲートウェイ設定：CAN→Eth、Eth→CANの転送経路

### CANIF（CAN Interface）

CANドライバとPDURの間に位置し、CANフレームをPDU単位に変換する。複数CANチャンネル（CAN1・CAN2等）の多重化も行い、PDURからは「CANのどのチャンネルか」を意識せずに扱える。

**このモジュールがないと何が困るか：** PDURが直接CANドライバのハードウェア固有APIを呼ぶ必要が生じ、マイコンを変えるとPDURの修正が必要になる。CANIFがあることでCANドライバ（ハードウェア固有）とPDUR（ハードウェア非依存）を切り離せる。

**主な設定項目：**
- CAN ID ↔ PDU IDのマッピング（どのCAN IDフレームをどのPDUとして上位に渡すか）
- フィルタリング：不要なCAN IDを受信しない設定
- 対応CANチャンネル（物理コントローラ番号）

### SoAD（Socket Adaptor）

UDP/TCPソケットとPDURを橋渡しするアダプタ。SOME/IPなどのペイロードをPDUとして上位に渡す。「ソケット」というUNIX系の概念をAUTOSARのPDU体系に翻訳する役割を担っている。

**このモジュールがないと何が困るか：** アプリケーションやCOMが直接BSD socketのsend/recv関数を呼ぶ必要が生じる。LinuxアプリをAUTOSARに移植した際に一番苦労するのがこの変換部分であり、SoADがこれを抽象化することでPDURからは「ネットワークの種類」が隠蔽される。

**主な設定項目：**
- SocketConnectionGroup：UDPかTCPか、リモートIPアドレス・ポート、ローカルポート
- PDUHeaderEnabled：SOME/IPヘッダを自動で付与するかどうか

### SOME/IP Transformer

PDUをSOME/IPフォーマットへシリアライズ／デシリアライズする変換器。SoADと組み合わせて使用。CANのシグナルは「バイト列の固定位置に数値がある」という単純な構造だが、SOME/IPのペイロードはC++の構造体をバイト列に変換（シリアライズ）する必要がある。この変換をBSW側で担うのがSOME/IP Transformerである。

**このモジュールがないと何が困るか：** 各SWCが独自にシリアライズコードを書く必要が生じる。エンディアン変換・アライメント・配列の長さプリフィックスなどのルールが統一されず、送信側と受信側で解釈が食い違うバグが発生しやすくなる。

**主な設定項目：**
- データ型のシリアライズルール（エンディアン・アライメント）
- ヘッダ付与の有無（Service ID・Method IDの埋め込み）

### SD（Service Discovery）

SOME/IP Service Discovery。サービスの公開（Offer）・検索（Find）・購読（Subscribe）をネットワーク上で実行する。Ethernet上では機器のIPアドレスやポート番号が起動時まで確定しないため、SDが「今このサービスはここにいます」を自動通知することで、各ECUがハードコードなしにサービスを見つけられる。

**このモジュールがないと何が困るか：** 全ECUのIPアドレス・ポート・サービス状態を設計時に固定し、ハードコードしなければならなくなる。ECUを追加・削除するたびに全ノードのコードを修正する必要が生じ、スケーラビリティがなくなる。

**SDの動作パラメータ（設定が必要な主要値）：**
- `InitialDelayMin/Max`：起動後最初のOfferを送るまでの待機時間（複数ECUが同時起動する際の衝突防止）
- `CyclicOfferDelay`：Offerを定期的に繰り返す間隔。消費側はこの値の2〜3倍をタイムアウトとして設定する
- `TTL`：Offer/Subscribeの有効期限（秒）。切れると再ネゴシエーションが必要

### TcpIp

AutosarスタックのTCP/IPプロトコル実装。UDPマルチキャスト（SOME/IP-SD）やTCP接続管理を担う。LinuxであればOS内蔵のTCP/IPスタックが使えるが、Classic AUTOSARが動く組み込みマイコンではOSにTCP/IPが含まれないため、BSW内でTcpIpモジュールとして提供される。

**このモジュールがないと何が困るか：** マイコン上でEthernetを使う場合、独自にTCP/IPスタックを実装またはサードパーティ製OSSを組み込む必要がある。AUTOSARのTcpIpはSoADと標準インターフェースで接続されており、TcpIpを交換してもSoAD以上の層に影響が出ない。

### EthIf / Eth Driver

EthIfはEthernetドライバを抽象化するインターフェース層、Eth Driverはハードウェア依存の送受信処理を行う。CANIFとCAN Driverの関係と同じ構造。マイコンメーカーがEth Driverを提供し、AUTOSARのEthIfがその上に被さる。

**このモジュールがないと何が困るか：** EthernetコントローラのICが変わると、TcpIp以上のすべての層に影響が出る。EthIf/Eth Driverの境界でハードウェア依存部分を隔離することで、IC変更のインパクトをEth Driverのみに限定できる。

### SecOC（Secure Onboard Communication）

COMとIF層の間でメッセージ認証コード（MAC）を付与・検証する。CANとEthernet両系統に適用可能。MACは「このメッセージは正規の送信者から来た改ざんされていないメッセージだ」を受信側が確認するための短い署名のようなもの。機能安全（ISO 26262）だけでなくサイバーセキュリティ（ISO 21434）への対応でも必須になりつつある。

**このモジュールがないと何が困るか：** 攻撃者が不正なCANフレームや偽SOME/IPイベントを送りつけて油圧バルブを誤動作させるサイバー攻撃を防げない。建設機械でもISO 21434準拠が求められるケースが増えており、後付けでSecOCを追加するのは困難なため設計初期から組み込むこと。

## 3. SOME/IPとは何か

### プロトコル概要

SOME/IP（Scalable service-Oriented MiddlewarE over IP）はEthernet上でサービス指向通信を実現するAUTOSAR標準プロトコル。三つの通信パターンを持つ。

- **メソッド呼び出し**：クライアントがサーバーの処理を要求し、結果を受け取る（RPC相当）。例：HMIからエンジン始動コマンドを送りACKを受け取る
- **イベント通知**：サーバーが変化を購読者へ配信（Pub/Sub相当）。例：カメラECUが人検知イベントを他のECUへ一斉配信する
- **フィールド**：サービスが保持する値をGet/Setまたはイベントで共有する。例：車速センサECUが現在の車速フィールドを持ち、他ECUがGetで取得する

### CAN通信（シグナルベース）との違い

| 観点 | CAN | SOME/IP |
|---|---|---|
| 通信モデル | 全ノードにブロードキャスト | サービス単位のアドレッシング |
| データ定義 | シグナル（ビット位置固定） | シリアライズされたデータ構造 |
| サービス発見 | 不要（ID固定） | SDで動的発見 |
| 帯域 | 最大8 MB/s（CAN FD） | 100 Mbps〜 |
| 用途 | リアルタイム制御・低レイテンシ | 大容量データ・カメラ映像・クラウド連携 |
| プロジェクト内の担当 | 油圧制御・車体制御・診断 | 人検知・HMI・クラウドテレメトリ |

**CANとEthernetを共存させる理由：**  
CANは低レイテンシ・高信頼性・電気的堅牢性が優れており、油圧バルブのリアルタイム制御には今も最適な選択肢である。一方でカメラ映像の転送・クラウド連携・HMIとの大容量通信にはEthernetの帯域が必要。この二種のプロトコルを1つのECU内でAUTOSARが統合管理するのが本プロジェクトの構成である。

### Classic AUTOSARでの実装経路

```
SWC → RTE → COM → PDUR → SoAD → SOME/IP Transformer → TcpIp → EthIf → Eth Driver
                    ↑
                    SD（サービス発見は並列動作。SoADにソケットオープンを指示する）
```

SoADがUDPソケットを管理し、SOME/IP TransformerがPDUをSOME/IPヘッダ付きバイト列に変換する。

### Adaptive AUTOSARでの実装

`ara::com` APIがSOME/IPバインディングを内包しており、Classic BSW相当の機能をミドルウェアが直接提供する。アプリケーションはサービスを「プロキシ（クライアント側）」「スケルトン（サーバー側）」として扱うだけでよい。  
ただしAdaptive AUTOSARは規格・ツールがまだ成熟しておらず、Classic AUTOSARと比べて実績・対応ベンダーが限られる（詳細はtoolchain.mdを参照）。

## 4. CAN通信とEthernet通信の経路比較

### CAN信号がSWCに届くまで（詳細版）

```
【CANフレーム受信の流れ】

CAN線（物理信号）
    │
    ▼
┌─────────────────────────────────────────┐
│  CAN Driver（MCAL）                     │
│  ・CANコントローラのHWレジスタを直接操作 │
│  ・受信したフレームをCAN IF へ渡す       │
└──────────────────┬──────────────────────┘
                   │ (CAN ID + DLC + データ8バイト)
                   ▼
┌─────────────────────────────────────────┐
│  CANIF                                  │
│  ・CAN IDをPDU IDに変換                 │
│    例）CAN ID 0x100 → PDU "HydCmd_PDU" │
│  ・複数CANチャンネルを1つのIFに集約     │
└──────────────────┬──────────────────────┘
                   │ (PDU ID + データ)
                   ▼
┌─────────────────────────────────────────┐
│  PDUR                                   │
│  ・ルーティングテーブルを参照           │
│    「HydCmd_PDUはCOMへ転送」            │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│  COM                                    │
│  ・PDUのバイト列からシグナルを抽出      │
│    例）Byte0-1 → BoomLiftCmd (uint16)  │
│  ・係数・オフセットを適用して物理値に変換│
│  ・シグナルバッファに格納               │
│  ・タイムアウト監視（Deadline Monitoring)│
└──────────────────┬──────────────────────┘
                   │ (シグナル値)
                   ▼
┌─────────────────────────────────────────┐
│  RTE                                    │
│  ・SWCからの Rte_Read_XXX() 呼び出しに  │
│    シグナルバッファの値を返す           │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│  SWC（油圧制御SWC等）                   │
│  ・Rte_Read_BoomLiftCmd(&val) を呼ぶだけ│
│  ・CANやビット配置は一切意識しない      │
└─────────────────────────────────────────┘
```

### SOME/IPサービスがSWCに届くまで（詳細版）

```
【SOME/IP イベント受信の流れ（人検知イベントの例）】

Ethernetケーブル（物理信号）
    │
    ▼
┌─────────────────────────────────────────┐
│  Eth Driver（MCAL）                     │
│  ・Ethernetコントローラを直接操作       │
│  ・MACフレームをEthIFへ渡す            │
└──────────────────┬──────────────────────┘
                   │ (MACフレーム)
                   ▼
┌─────────────────────────────────────────┐
│  EthIf                                  │
│  ・VLANタグの処理、フレームの振り分け   │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│  TcpIp                                  │
│  ・IP/UDPヘッダを解釈してペイロードを取得│
│  ・マルチキャストグループへの参加管理   │
└──────────────────┬──────────────────────┘
                   │ (UDP ペイロード)
           ┌───────┴──────────────┐
           │ SD受信 (Port:30490)  │ データ受信 (Port:50100)
           ▼                      ▼
┌────────────────┐    ┌─────────────────────────────────────┐
│  SD            │    │  SoAD                               │
│ OfferServiceを │    │  ・ソケットとPDU IDを対応付け       │
│ 受け取り、     │    │  ・受信ペイロードをPDUとしてPDURへ  │
│ SoADにソケット │    └──────────────────┬──────────────────┘
│ オープンを指示 │                       │
└────────────────┘                       │ (PDU + SOME/IPペイロード)
                                         ▼
                          ┌─────────────────────────────────────┐
                          │  SOME/IP Transformer                │
                          │  ・SOME/IPヘッダを解析             │
                          │    Service ID / Event ID の確認    │
                          │  ・ペイロードをデシリアライズ       │
                          │    DetectionStatus・Confidence等   │
                          └──────────────────┬──────────────────┘
                                             │ (PDU)
                                             ▼
                                    ┌─────────────────┐
                                    │  PDUR           │
                                    └──────┬──────────┘
                                           │
                                           ▼
                                    ┌─────────────────┐
                                    │  COM            │
                                    │  シグナルバッファ│
                                    └──────┬──────────┘
                                           │
                                           ▼
                                    ┌─────────────────┐
                                    │  RTE → SWC      │
                                    └─────────────────┘
```

**2つの経路を比較すると：**  
CAN経路の方が層が少ないため低レイテンシ。SOME/IP経路はDeserialize処理（Transformer）とSD処理が追加されるが、その代わり大容量データ・複雑なデータ構造・サービス指向の動的接続が可能になる。

## 5. 建設機械での適用イメージ

### 構成例

| SWC | 通信種別 | 経路上の主要モジュール | 備考 |
|---|---|---|---|
| 油圧制御SWC | CANシグナル（バルブ開度・圧力値） | CAN Driver → CANIF → PDUR → COM → RTE | CAN FD 10ms周期 |
| 人検知サービス消費SWC | SOME/IPイベント（カメラ検知結果） | Eth Driver → EthIf → TcpIp → SoAD → Transformer → PDUR → COM → RTE | UDP Multicast |
| J1939ゲートウェイSWC | J1939 PGN（エンジン回転数） | CAN Driver（CAN2） → CANIF → PDUR → COM → RTE | エンジンECUから受信してCAN1のシグナルへ変換 |
| 診断SWC | UDS on CAN（DiagBus） | CAN Driver（CAN3） → CANIF → CanTp → DCM | 診断ルーティングはCOM経由しない |

### データフローの全体像（ECU内部）

```
                  ┌──────────────────────────────────────────────────────┐
                  │                  VehicleCtrlECU の内部               │
                  │                                                      │
  CAN1 ──────────►│  CANIF ──► PDUR ──► COM ──► RTE ──► 油圧制御SWC    │
  (制御バス)       │                                  ◄──                │
                  │                                       └►PDUR►CANIF──►│──► CAN1
                  │                                                      │
  CAN2 ──────────►│  CANIF(J1939) ──► PDUR ──► COM ──► RTE ──► GW-SWC │
  (J1939)         │                                                      │
                  │                                                      │
  Eth ───────────►│  EthIf ──► TcpIp ──► SoAD ──► Transformer          │
  (SOME/IP)       │               ↑                     │               │
                  │              SD                     ▼               │
                  │                             PDUR ──► COM ──► RTE   │
                  │                                            ──► 人検知SWC │
                  └──────────────────────────────────────────────────────┘
```

油圧バルブの制御値はCAN FDで10 ms周期送信（低レイテンシ・高確実性）。カメラECUが発行する「人検知イベント」はSOME/IP Pub/SubでEthernet経由受信し、人検知SWCが警報やブーム動作制限に利用する。J1939ゲートウェイSWCはエンジンECUからのPGN（エンジン回転数・トルク）をAUTOSARシグナルに変換して内部バスへ橋渡しする。SecOCは両系統のメッセージに適用し、改ざん検知を行う。

## 6. モジュール一覧早見表

| モジュール | 略称の意味 | 位置（層） | 担当すること | ないと困ること |
|---|---|---|---|---|
| COM | Communication | BSW サービス | シグナル送受信・タイムアウト監視 | SWCがビット操作をしなければならなくなる |
| PDUR | PDU Router | BSW サービス | PDUルーティング | CAN/Eth間のゲートウェイが個別実装になる |
| CANIF | CAN Interface | BSW ECAL | CAN ID↔PDU変換・多重化 | PDURがHW固有CANドライバを直接呼ぶ |
| SoAD | Socket Adaptor | BSW サービス | ソケット↔PDU変換 | アプリがsocket APIを直接呼ぶ |
| SOME/IP Transformer | - | BSW変換 | シリアライズ・デシリアライズ | 各SWCが独自変換コードを書く |
| SD | Service Discovery | BSW サービス | Offer/Find/Subscribe | 全IPアドレス・ポートをハードコード |
| TcpIp | - | BSW サービス | TCP/UDPプロトコル実装 | マイコンにTCP/IPスタックがない |
| EthIf | Ethernet Interface | BSW ECAL | Ethドライバ抽象化 | Eth IC変更で上位層全修正 |
| SecOC | Secure OC | BSW サービス | MAC付与・検証 | 不正フレーム注入を検知できない |
