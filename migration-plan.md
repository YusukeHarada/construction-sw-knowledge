## 1. 移行の基本戦略

### 1.1 「一括移行」vs「段階移行」

| 観点 | 一括移行 | 段階移行 |
|------|----------|----------|
| 移行期間 | 2〜3年で完結 | 4〜6年かかる |
| 現行保守コスト | 移行完了まで二重化なし | 並行運用期間は二重コスト |
| リスク集中度 | 高い（一度に全機能が切り替わる） | 低い（小さな単位で検証できる） |
| 失敗時の影響 | 全機能が影響を受ける | 当該機能のみ切り戻し可能 |
| 仕様書・テストがない場合 | 非現実的（全量を同時に仕様化できない） | 現実的（1機能ずつ仕様化できる） |
| 委託先への知識移転 | 一度に大量の暗黙知を伝える必要がある | 小さな単位で徐々に移転できる |

**結論：仕様書・テストが存在しない現状では一括移行は不可能。段階移行一択とする。**

### 1.2 Strangler Fig Patternが現実的な理由

Strangler Fig Pattern（絞め殺しイチジク方式）は、既存システムを即座に廃止せず、新システムが機能を少しずつ「包み込む」ように置き換えていく移行パターンである。

本プロジェクトに適用する根拠は以下の通り。

1. **仕様化コストの分散**  
   現行コードの仕様化は1機能あたり数週間かかる。一括では30人×数ヶ月の仕様化工数が同時に発生し、開発が止まる。段階移行では優先度順に1機能ずつ対応できる。

2. **委託先の習熟を活用できる**  
   最初のフェーズ（Dcm/Dem）で委託先が自社システムの癖・通信仕様・開発環境に習熟する。後半フェーズの品質向上につながる。

3. **現行製品の保守継続**  
   既存の製品ラインは移行完了まで現行システムで動作し続ける必要がある。段階移行なら「移行済み機能」と「未移行機能」を並行運用できる。

4. **安全クリティカル機能の段階的検証**  
   油圧制御等の安全クリティカル機能を最後に移行することで、前フェーズのノウハウを蓄積した上で検証できる。

### 1.3 新旧並行運用期間の考え方

移行期間中は「新システム（AUTOSAR準拠）」と「旧システム（内製）」が一定期間共存する。この並行運用期間について以下の原則を定める。

```
[並行運用の構成イメージ]

 ECU
 ┌─────────────────────────────────┐
 │  新: AUTOSARスタック（移行済み機能）  │
 │  旧: 内製コード（未移行機能）        │
 │                                 │
 │  ← 同一ECU上での段階移行の場合 →  │
 └─────────────────────────────────┘

または

 ECU A（旧）         ECU B（新）
 内製コード  ←通信→  AUTOSARスタック
 （全機能）          （移行済み機能）
```

**並行運用のルール**

- 同一機能を新旧両方で同時に「本番動作」させない（どちらかがプライマリ、もう一方は検証用）
- 本番移行の切り替えは機能単位で行い、切り替え日を明確に記録する
- 旧システムへの機能追加は原則禁止（移行先の整合性が取れなくなるため）
- 並行運用期間の上限を機能ごとに設定する（ダラダラと二重保守が続かないよう管理する）

### 1.4 移行判断の3原則

移行の可否・順序・方式を判断するとき、常に以下の3原則を優先順位順に適用する。

**原則1：安全性（最優先）**  
ISO 25119に基づく機能安全要求を満たしているか。安全クリティカル機能の移行は、Safety Analysisが完了するまで開始しない。

**原則2：実現可能性**  
委託先の技術力・自社の知識移転能力・スケジュールを踏まえて実現可能か。「できる」と「やる」を混同しない。技術的に可能でも、体制が整っていなければ移行しない。

**原則3：ROI（費用対効果）**  
移行コスト（仕様化・委託費・検証費）に見合う効果（保守性向上・規格準拠・技術負債削減）が得られるか。ROIが見込めない機能は移行の対象外とする判断も行う。

## 2. 現状の内製コードの棚卸し方法

### 2.1 棚卸しシートテンプレート

全機能を以下のシートで一元管理する。各項目の記入担当は「現行機能の担当者」とする。

| 項目 | 記入内容 | 備考 |
|------|----------|------|
| ECU名 | 対象ECUの識別名 | 例：エンジンECU、油圧ECU |
| 機能名 | 機能の名称 | 例：診断通信、J1939受信処理 |
| コード行数（概算） | ソースコードの行数 | ヘッダ込みの概算でよい |
| 使用言語 | C / C++ / アセンブリ 等 | |
| 仕様書の有無 | ○ / △（不完全）/ × | △の場合は何が欠けているか記載 |
| テストの有無 | ○ / △（一部）/ × | △の場合はカバレッジ概算を記載 |
| 担当者（現行） | 現在の機能オーナー | 退職・異動リスクも把握する |
| 暗黙知の有無 | 高 / 中 / 低 | 後述のヒアリングで判定 |
| 外部依存 | ハード依存・他機能依存・社外ライブラリ等 | |
| 複雑度 | 高 / 中 / 低 | 状態機械の数・割り込み処理・タイミング制約等を根拠に判定 |
| 安全影響度 | ASIL D / C / B / A / QM | ISO 25119のASIL相当で記載 |
| 移行候補フェーズ | Phase 1 / 2 / 3 / 対象外 | 第3節の優先度評価後に記入 |
| 備考 | 特記事項 | |

**棚卸しシートの運用方針**

- スプレッドシートで管理し、全ドメイン担当者が編集可能にする
- 最初の2週間で「ECU名・機能名・担当者」だけ埋める（全量把握が先）
- その後4週間でコード行数・仕様書有無・テスト有無を埋める
- 複雑度・安全影響度・暗黙知はヒアリング完了後に記入する

### 2.2 技術的負債の分類

棚卸しで把握した技術的負債を以下の4種類に分類し、移行計画への影響度を整理する。

**カテゴリA：ドキュメント不足**  
- 仕様書なし・コメントなし・設計図なし
- 移行前に仕様化が必要。委託先への提供物が増える
- 対策：担当者ヒアリング + リバースエンジニアリングで最低限の仕様書を作成

**カテゴリB：テスト不足**  
- 単体テストなし・統合テストの手順が属人的
- 移行後の機能同等性検証が困難になる
- 対策：移行前に「現行動作のブラックボックステスト」を記録する（入力→期待出力を実機で確認・記録）

**カテゴリC：アーキテクチャ問題**  
- 機能間の依存関係が複雑・グローバル変数の多用・割り込み駆動の状態機械
- AUTOSARのSWCモデルへのマッピングが困難
- 対策：移行前にリファクタリングを行うか、AUTOSAR側の設計で吸収するか委託先と協議

**カテゴリD：規格非準拠**  
- MISRA-C非準拠・セキュリティコーディング規約違反
- 委託先への提供コードをそのままベースラインにできない
- 対策：MISRA-C対応は委託先スコープに含める。提供時に「参考コード」として位置づける

### 2.3 暗黙知の洗い出し方（担当者ヒアリング手順）

担当者が退職・異動した場合に機能が「ブラックボックス」になるリスクを防ぐため、以下の手順でヒアリングを実施する。

**ヒアリング前の準備（技術リード）**

1. コードを事前に読み、「なぜこの実装にしたのか分からない箇所」をリストアップする
2. 過去のバグ票・変更履歴から「繰り返し修正が入っている箇所」を特定する
3. ヒアリング所要時間は1機能あたり1〜2時間を確保する

**ヒアリング項目**

| # | 質問 | 引き出したい情報 |
|---|------|----------------|
| 1 | この機能が「正しく動いている」状態を、どう確認しますか？ | 非公式なテスト手順・確認方法 |
| 2 | 過去に一番ハマったバグは何ですか？原因と対策は？ | ハードウェア依存・タイミング問題等の落とし穴 |
| 3 | このコードを変更するとき、必ず注意することは？ | 暗黙の制約・副作用 |
| 4 | 仕様書に書かれていないが「こう動かなければならない」ことは？ | 暗黙の要件 |
| 5 | 他の機能・ECUとの「この順番でないとダメ」な初期化・通信はある？ | 起動シーケンスの依存関係 |
| 6 | ハードウェアの個体差・ロット差で挙動が変わる箇所はある？ | ハード依存の回避策 |
| 7 | 引き継ぐとしたら、まず何を説明しますか？ | 担当者が重要と考えるポイント |

**ヒアリング後の処理**

- 録音（同意を得た上で）またはメモを即日文書化する
- 「暗黙知シート」としてWikiに保存し、委託先への提供物に含める
- ヒアリング結果を担当者に確認してもらい、認識ずれを修正する

### 2.4 棚卸し完了の判断基準

以下の全項目を満たした時点で棚卸し完了とする。

- [ ] 全ECUの機能リストが作成され、機能の数が確定している
- [ ] 全機能について棚卸しシートの必須項目（担当者・仕様書有無・テスト有無・安全影響度）が埋まっている
- [ ] 担当者が在籍している全機能のヒアリングが完了している
- [ ] 技術的負債がA〜Dに分類され、移行への影響度が整理されている
- [ ] コード行数の集計が完了し、規模感が把握できている
- [ ] 移行対象外（AUTOSAR化しない機能）の判断が終わっている

## 3. 移行優先度の決め方

### 3.1 優先度評価マトリクス

各機能を以下の3軸で評価し、優先度スコア（最大15点）を算出する。

**軸1：安全影響度（1〜5点、逆スコア）**  
安全影響度が高い機能ほど「後回し」にすることが重要なため、スコアを逆転させる。

| 安全影響度 | スコア |
|-----------|--------|
| ASIL D（最高安全要求） | 1点 |
| ASIL C | 2点 |
| ASIL B | 3点 |
| ASIL A | 4点 |
| QM（安全要求なし） | 5点 |

**軸2：移行容易性（1〜5点）**

| 条件 | スコア |
|------|--------|
| 仕様書あり・テストあり・暗黙知少・依存少 | 5点 |
| 仕様書△・テスト△・暗黙知中程度 | 3点 |
| 仕様書なし・テストなし・暗黙知高・複雑な依存 | 1点 |

**軸3：ビジネス価値（1〜5点）**

| 条件 | スコア |
|------|--------|
| AUTOSAR化で顧客価値・規格対応・保守性に大きく寄与 | 5点 |
| 中程度の価値 | 3点 |
| ほぼ現行のまま動けばよい・差別化要素なし | 1点 |

**スコア判定**

| 合計スコア | 移行フェーズ |
|-----------|-------------|
| 12〜15点 | Phase 1（PoC）候補 |
| 8〜11点 | Phase 2（基盤移行）候補 |
| 4〜7点 | Phase 3（安全クリティカル）候補 |
| 3点以下 | 移行対象外または最終フェーズ |

### 3.2 「最初に移行すべき機能」の選定基準

**PoC推奨：Dcm（診断通信）＋ Dem（診断イベント管理）**

最初のPoCとしてDcm＋Demを推奨する理由は以下の通り。

1. **安全影響度が低い（QM〜ASIL A相当）**  
   診断機能は車両の通常動作とは独立しており、診断機能が停止しても走行・作業には即座に影響しない。移行中の問題が現場での安全事故に直結しにくい。

2. **AUTOSARの標準化度が高い**  
   DcmはUDS（ISO 14229）に準拠した標準モジュールであり、委託先の習熟度にかかわらず実績あるArXML設定で対応できる。

3. **機能境界が明確**  
   「OBD2診断ポートから来るUDSリクエストに応答する」という機能境界が分かりやすく、他機能への依存が限定的。

4. **委託先の力量を低リスクで評価できる**  
   PoCで委託先のArXMLコンフィグ品質・テスト体制・コミュニケーション能力を評価できる。問題があっても影響が最小限。

5. **既存の診断ツールで回帰テストができる**  
   現行の診断テスタ（CANalyzer/CANoe等）をそのまま使って新旧比較ができる。

### 3.3 なぜ安全クリティカル機能を最後にすべきか

油圧制御・走行制御等のASIL C/D機能を最初に移行しない理由を明確にしておく。

- **HARA（Hazard Analysis and Risk Assessment）が未実施**  
  安全クリティカル機能の移行にはHARAから始まるV字プロセスが必要。PoCより先に着手すると移行中のシステムが安全要求不明の状態になる。

- **委託先がシステムの癖を理解していない状態で触らせるのは危険**  
  Phase 1〜2で委託先が自社の通信仕様・タイミング要求・ハード依存を把握してから安全機能に着手する。

- **証跡が残せない**  
  ISO 25119の機能安全証跡は開発プロセス全体にわたる。最初からプロセスを守って進めることが必要で、「後から証跡を作る」は原則として認められない。

### 3.4 移行を後回しにすべき機能

以下の特性を持つ機能は、Phase 3以降または移行対象外として扱う。

| 特性 | 理由 |
|------|------|
| 独自プロトコル（AUTOSARに対応モジュールがない） | SWC内でソフト的に実装する必要があり設計工数が大きい |
| 複雑な状態機械（状態数が20以上・割り込みが絡む） | AUTOSAR SWCのRunnable設計との整合に時間がかかる |
| ハードウェアに強く依存した制御（タイマ・DMA直叩き） | MCU Abstraction Layer設計が複雑になる |
| 移行後の検証手段がない機能 | Go/No-Goの判断ができないため移行リスクが高い |

## 4. 機能別移行ロードマップ

### 4.1 Phase 1：PoC（0〜6ヶ月）

**目標：** Dcm＋Demの移行により、委託先の技術力を評価しAUTOSAR開発基盤を確立する。

---

#### 診断通信（UDS/Dcm/Dem）の移行手順

**Step 1：既存診断コードの仕様化（Month 1〜2）**

1. 現行コードから対応しているUDSサービスIDを全量リストアップする  
   - 最低限：0x10（Session Control）, 0x11（ECU Reset）, 0x14（Clear DTC）, 0x19（Read DTC）, 0x22（Read Data by ID）, 0x27（Security Access）, 0x2E（Write Data by ID）

2. 各サービスIDについて以下を仕様化する  
   - 受け入れるサブファンクション・Data Identifier
   - 応答データのフォーマット（バイト列・エンディアン）
   - セッション種別（Default / Extended / Programming）ごとの動作差異
   - タイムアウト値（P2、P2*）の設定値

3. DTCリスト（Demで管理するフォルト）を全量抽出する  
   - DTC番号・フォルト名・検出条件・回復条件・保存条件

4. 仕様書レビューを実施し、委託先への提供物として確定する

**Step 2：DcmモジュールのArXMLコンフィグ作成（Month 2〜4）**

委託先が行うメイン作業。自社技術リードが以下を確認する。

- Session Control設定（DefaultSession/ExtendedSession/ProgrammingSession）
- Security Access のシードアルゴリズム（セキュリティ上、実装詳細は自社で行うか慎重に検討）
- Dcm_Config のDID（Data Identifier）マッピング
- DemとのConnector定義（Dcm → Dem の DTC読み出しI/F）
- ComM, BswM との連携設定

**Step 3：既存テスト機器との互換性確認（Month 4〜6）**

1. 現行システムで使用している診断テスタ（CANalyzer / CANoe / サービスツール等）で新システムを接続する  
2. 全UDSサービスIDの応答を旧システムと比較する（差分が出た場合は仕様の解釈ずれを疑う）  
3. 診断テストケースを文書化し、以降のフェーズの検証テンプレートとして流用する

**Phase 1 完了基準**

- [ ] DcmとDemがターゲットECU上で動作している
- [ ] 全UDSサービスIDの応答が旧システムと一致している
- [ ] DTCの発生・保存・消去が正常に動作している
- [ ] 委託先のArXML品質レビューが完了している
- [ ] 委託先の力量評価が完了し、Phase 2継続の判断が行われている

---

### 4.2 Phase 2：基盤移行（6〜24ヶ月）

**目標：** 通信系・OS/タスク管理を移行し、安全クリティカル機能以外の基盤を確立する。

---

#### 通信系（CAN送受信・J1939）の移行

**Step 1：既存DBCファイルの整理・正規化（Month 6〜9）**

建設機械のCANネットワークは長年の改訂でDBCファイルが断片化していることが多い。移行前に整理する。

チェックリスト：
- [ ] 全CANバスのDBCファイルが最新版に統一されている
- [ ] 未使用・廃止されたメッセージIDが削除されている
- [ ] 同一メッセージIDが複数のDBCで定義されている矛盾がない
- [ ] J1939 PGN（Parameter Group Number）とSPN（Suspect Parameter Number）が正確に記載されている
- [ ] Cycle Time（送信周期）・Timeout（受信タイムアウト）が全メッセージに設定されている
- [ ] DBCファイルのバージョン管理がGit等で行われている

**Step 2：COM/PDUR/CANIF設定への変換手順**

DBCファイルからAUTOSARのArXMLコンフィグへの変換方針を定める。

```
[変換の流れ]

DBCファイル
  ↓（ツール変換 または 手動）
System Description (ARXML)
  ↓
CANIF    ── CANフレーム単位の送受信制御
  ↓
PDUR     ── PDU（Protocol Data Unit）のルーティング
  ↓
COM      ── シグナル単位のアクセス（アプリケーション層）
  ↓
SWC（アプリケーション）
```

変換時の注意点：
- マルチプレクスメッセージ（Mux）は COM側の設定が複雑になる。事前に件数を把握する
- エンディアン変換（Intel / Motorola）の設定ミスは通信化けの原因になる。必ず実機で確認する
- 送信タイミング（Cyclic / Event / Mixed）はCOM設定で制御する。現行のタイミング仕様を事前に記録する

**Step 3：J1939アドレス管理の移行**

J1939ではECUがネットワーク上でアドレスを主張（Address Claiming）するプロセスが定義されている。

- 現行コードのAddress Claimingロジックを把握する（独自実装 or RTOS提供ライブラリ）
- AUTOSARでのJ1939対応はBSWに標準モジュールがある（J1939Tp, J1939Rm等）
- J1939優先度・PGN送信設定はArXMLで設定する
- 既存のJ1939アドレスを変更しないよう注意する（他ECU・農業機械等との互換性）

**Phase 2 通信系 完了基準**

- [ ] 全CANバスで送受信が正常動作している
- [ ] CANoe / CANalyzerでのキャプチャ比較で旧システムとのフレーム差分がない
- [ ] J1939 Address Claimingが正常に完了している
- [ ] タイムアウト検出が旧システムと同等に動作している

---

#### OS・タスク管理の移行

**Step 1：既存RTOSタスク構成のSWCマッピング**

現行RTOSのタスク構成をAUTOSAR SWCのRunnable（実行単位）にマッピングする。

棚卸しシートに以下を追記する：

| タスク名 | 実行周期 | 優先度 | 担当機能 | 対応SWC候補 | 対応Runnable |
|---------|---------|--------|---------|------------|-------------|
| Task_10ms | 10ms | 高 | 油圧制御 | HydraulicCtrl_SWC | Run_HydCtrl_10ms |
| Task_100ms | 100ms | 中 | J1939受信処理 | CanReceive_SWC | Run_J1939Recv_100ms |
| ... | | | | | |

**Step 2：タイミング・優先度の引き継ぎ方法**

AUTOSARではOSの設定（OSEK OS）でタスク・アラームを定義する。

- 現行RTOSのタスク周期をそのままOSEK OSのAlarm Cycleに設定することを基本とする
- タスク優先度の逆転（Priority Inversion）が現行システムで意図的に設計されていないか確認する
- OSコンフィグ（Os_Cfg.c/h）の生成はArXMLから自動生成する。手動編集しない
- タイミング移行後はタイミングアナライザ（要確定）でCPU負荷・Deadlineミスがないか検証する

**Phase 2 OS移行 完了基準**

- [ ] 全Runnableが設計通りの周期で実行されている
- [ ] CPU負荷が（要確定）%以下に収まっている
- [ ] スタックオーバーフローが検出されていない
- [ ] Deadline Missed が発生していない

---

### 4.3 Phase 3：安全クリティカル移行（24〜60ヶ月）

**目標：** 油圧制御等のASIL C/D機能をISO 25119に準拠した形で移行する。

---

#### 油圧制御SWCの移行

**Step 1：HARA実施 → 安全要求定義 → SWC設計の順序**

```
[V字プロセスの左辺（設計フェーズ）]

機能安全計画の策定
  ↓
HARA（Hazard Analysis and Risk Assessment）
  ├── ハザード同定（油圧暴走・予期せぬ動作等）
  ├── 重篤度（S）・暴露頻度（E）・制御可能性（C）の評価
  └── ASIL判定（D/C/B/A/QM）
  ↓
FSC（Functional Safety Concept）策定
  ├── Safety Goal定義
  └── FSR（Functional Safety Requirement）定義
  ↓
TSC（Technical Safety Concept）策定
  ├── TSR（Technical Safety Requirement）定義
  └── SWC・HWへの安全要求の割り付け
  ↓
SWC詳細設計
  ├── AUTOSAR SWCとしてのインターフェース定義
  ├── MISRA-C準拠の実装
  └── 安全機構（モニタリング・フェールセーフ）の設計
```

**Step 2：既存制御ロジックのMISRA-C対応方法**

現行コードをそのままAUTOSAR SWCに移植することはせず、以下のアプローチを取る。

1. 現行コードは「動作仕様の参考」として位置づける（委託先への提供は参考情報）
2. MISRA-C:2012準拠で制御ロジックを再実装する（再実装であるため証跡が残りやすい）
3. 静的解析ツール（要確定）でMISRA-C違反を検出・修正する
4. コードレビューは自社技術リードと委託先の両方で実施する

MISRA-C対応の工数は、制御ロジックの複雑度によって大きく異なる。Phase 2完了時点で委託先に見積もりを取ること。

**Step 3：HILを使った機能同等性検証**

```
[HIL（Hardware-in-the-Loop）検証構成]

 テストPC（CANoe等）
   ↓ CAN/Ethernet
 HILシミュレータ
   ├── 油圧センサ信号模擬
   ├── アクチュエータ応答模擬
   └── 異常注入（センサ断線・値異常等）
   ↓ アナログ/デジタルI/O
 ターゲットECU（新AUTOSARスタック）
```

検証項目：
- [ ] 通常動作（各種操作パターン）で旧システムと同等の制御出力が得られる
- [ ] 異常検知（センサ故障等）でフェールセーフ動作が正しく動作する
- [ ] 安全機構（モニタリング機能）が正しく機能する
- [ ] 応答時間（レイテンシ）が旧システムと同等以上

**Phase 3 完了基準**

- [ ] HARA・FSC・TSCの文書が完成し内部承認されている
- [ ] SWCがMISRA-C:2012に準拠している（静的解析ツールで確認）
- [ ] HIL検証で全テストケースがパスしている
- [ ] 安全証跡一式（HARA・設計・テスト記録）が揃っている
- [ ] 実機フィールドテストが完了している

### 5.3 知識移転セッションの進め方

知識移転は「一度説明して終わり」ではなく、フェーズをまたいで継続的に行う。

**キックオフセッション（Phase 1開始前）**

- 参加者：自社全ドメインリード + 委託先技術責任者
- 内容：システム全体像・通信構成・主要ECUの役割・開発環境の説明
- 所要時間：2日間（1日目：座学、2日目：実機確認）
- 成果物：委託先が作成した理解確認資料（自社がレビューする）

**機能別セッション（各フェーズ開始前）**

- 参加者：対象機能の自社担当者 + 委託先担当者
- 内容：担当機能の詳細仕様説明・ハードウェア依存の説明・過去のバグ事例
- 所要時間：1機能あたり半日〜1日
- 成果物：Q&A記録（後から参照できるよう文書化）

**定例技術レビュー（フェーズ期間中）**

- 頻度：隔週（設計フェーズ）、週次（実装・検証フェーズ）
- 内容：ArXML設計レビュー・テスト結果確認・課題共有
- 形式：オンライン可（画面共有でArXML/コードをレビュー）

## 6. リスク管理

### 6.1 主要リスクと対策

**リスク1：暗黙知の喪失**

| 項目 | 内容 |
|------|------|
| 発生確率 | 高（現時点でドキュメントが存在しない） |
| 影響度 | 高（移行後に原因不明の動作異常が発生する） |
| 対策 | 第2節のヒアリング手順を確実に実施する。担当者が退職するより前に実施する。移行完了まで担当者が在籍できるよう人事と調整する。 |
| 検知方法 | 棚卸しシートの「暗黙知の有無：高」の機能数を毎月モニタリングする |

**リスク2：機能の欠落・デグレード**

| 項目 | 内容 |
|------|------|
| 発生確率 | 中（仕様化漏れがある場合に発生する） |
| 影響度 | 高（現場での予期せぬ動作につながる） |
| 対策 | ブラックボックステスト（入力→出力記録）を移行前に全機能実施する。移行後に同一テストを実行し差分を確認する。 |
| 検知方法 | 各フェーズ完了前の機能同等性検証テストで検知する |

**リスク3：スケジュール遅延**

| 項目 | 内容 |
|------|------|
| 発生確率 | 高（現状の規模と体制を考えると遅延は想定内） |
| 影響度 | 中（フェーズが延びる。ただし安全優先のため遅延は許容する） |
| 対策 | フェーズの完了基準を明確にし、「完了してから次フェーズへ」を徹底する。スコープを削減する判断を早期に行う。 |
| 検知方法 | 月次のマイルストーン達成率を確認する |

**リスク4：委託先の技術力不足**

| 項目 | 内容 |
|------|------|
| 発生確率 | 中（Phase 1のPoCで見極められなかった場合） |
| 影響度 | 高（Phase 2以降で発覚すると切替コストが大きい） |
| 対策 | Phase 1のPoC完了時点で委託先の力量を正式評価する。評価基準を事前に定め、基準未達の場合はPhase 2開始前に委託先変更または追加支援を判断する。 |
| 評価基準 | ArXML品質・テストカバレッジ・コミュニケーション応答速度・課題解決能力 |

**リスク5：移行期間中の現行製品保守**

| 項目 | 内容 |
|------|------|
| 発生確率 | 高（移行期間中も現行製品の不具合対応は発生する） |
| 影響度 | 中（移行作業リソースを現行保守に取られる） |
| 対策 | 移行担当チームと現行保守担当チームを分離する。移行担当が現行保守に引き込まれるルールを作らない。移行フェーズ中の現行コード変更は最小限にし、変更した場合は移行先への反映漏れがないよう管理する。 |

**リスク6：機能安全証跡の欠如**

| 項目 | 内容 |
|------|------|
| 発生確率 | 高（現行コードに証跡がないため、移行プロセスで証跡を新規作成しないと証明できない） |
| 影響度 | 高（Phase 3の安全クリティカル機能移行がISOに準拠できない） |
| 対策 | Phase 1〜2から証跡管理プロセスを習慣化しておく。委託先への発注書・受入書・レビュー記録を全て保存する。Phase 3前に機能安全コンサルタント（要確定）のレビューを受ける。 |

### 6.2 移行中止・ロールバックの判断基準

以下の条件に該当する場合、移行中止またはロールバックを検討する。

**即時ロールバック（当該機能を旧システムに戻す）**

- 移行後の実機で安全上の問題（人身・機械損傷リスク）が確認された
- 移行後の通信異常により他ECUに波及影響が出た
- 機能同等性検証で重大な差分が解消できない

**フェーズ一時停止**

- 委託先の技術力不足が明確になり、PhaseのGo/No-Go基準を連続2回達成できない
- 自社の知識移転担当者の退職等により、移行に必要な情報が失われた
- 機能安全上の問題が発見され、設計の見直しが必要

**プロジェクト全体の中断判断**

- 委託先が倒産・プロジェクト撤退した
- 採用するAUTOSARスタックが廃盤になった（要確定）
- 規制・規格の変更により移行計画全体の見直しが必要

中断・ロールバック判断はプロジェクトオーナー（開発部門長）が行う。技術リードは判断材料を提供する役割であり、単独で判断しない。

## 7. 移行の成功基準と検証方法

### 7.1 各フェーズの「移行完了」の定義

**Phase 1 完了定義**

- Dcm＋Demが量産相当のECU上で動作している
- 指定した全UDSサービスIDで旧システムと機能同等性が確認されている
- 委託先の力量評価が完了し、Phase 2継続可否の判断が記録されている

**Phase 2 完了定義**

- 全CANバスの通信が移行済みECU上で正常動作している
- J1939通信が旧システムと同等に動作している
- 全Runnableが設計通りの周期で動作し、CPU負荷が設計上限以内に収まっている
- 旧システムとのCANフレーム比較テストが全ケースパスしている

**Phase 3 完了定義**

- HARA・FSC・TSCの文書が完成し内部承認されている
- 移行対象の全安全クリティカル機能のSWCがMISRA-C準拠で実装されている
- HIL検証で全テストケースがパスしている
- 実機フィールドテストで安全動作が確認されている
- 安全証跡一式が揃っており、第三者（要確定）のレビューを受けている

### 7.2 新旧システムの機能同等性検証手順

各フェーズで以下の手順を実施する。

```
[機能同等性検証フロー]

1. テストケースの準備（移行前）
   ├── 実機で現行システムの入力→出力を記録する
   ├── 境界値・異常値のテストケースを追加する
   └── テストケースを文書化する（以降のリグレッションテストに使用）

2. 新システムでの検証（移行後）
   ├── 同一テストケースを新システムで実行する
   └── 出力を記録する

3. 差分分析
   ├── 差分がある場合：仕様の解釈ずれか実装バグかを切り分ける
   ├── 仕様の解釈ずれ：仕様書を修正し、委託先に再実装を依頼する
   └── 実装バグ：委託先が修正し、テストを再実行する

4. 合格判定
   └── 全テストケースで差分なし または 差分が意図的な仕様変更として承認されている
```

### 7.3 委託先からの納品物チェックリスト

各フェーズの納品物を以下で確認する。

**設計フェーズ納品物**

- [ ] ArXMLファイル一式（バージョン管理されている）
- [ ] SWC設計書（インターフェース定義・Runnable設計）
- [ ] トレーサビリティマトリクス（要求 ↔ 設計 ↔ テスト）
- [ ] 使用ツール・ライセンス一覧

**実装フェーズ納品物**

- [ ] ソースコード（Gitリポジトリ、コミット履歴付き）
- [ ] ビルド手順書（自社環境で再現可能なレベル）
- [ ] 静的解析結果（MISRA-C違反ゼロまたは偏差申請書付き）
- [ ] コードレビュー記録

**検証フェーズ納品物**

- [ ] テスト仕様書
- [ ] テスト結果報告書（全件の合否記録）
- [ ] 不具合管理表（発見・修正・再テストの記録）
- [ ] カバレッジ測定結果（Phase 3ではMC/DC必須）

### 7.4 本番移行の判断基準（Go/No-Go）

本番移行（旧システムから新システムへの切り替え）は以下の全項目がGOであることを確認してから実施する。

| 判断項目 | Go条件 | 判断者 |
|---------|--------|--------|
| 機能同等性 | 全テストケースが合格している | 技術リード |
| 安全性（Phase 3） | 安全証跡が揃い内部承認されている | 安全担当者 |
| 納品物 | チェックリストが全件クリア | 技術リード |
| 委託先サポート | 本番移行後の不具合対応体制が合意されている | プロジェクトマネージャー |
| 切り戻し手順 | ロールバック手順が文書化され、動作確認済み | 技術リード |
| 現場確認 | フィールドテストで問題がないことを現場担当者が確認している | 現場担当者 |
| 経営承認（Phase 3） | 開発部門長の承認が得られている | 部門長 |

いずれか1項目でもNo-Goの場合は、問題が解消するまで本番移行を延期する。「スケジュールが迫っているから」という理由でGo/No-Goの判断を妥協しない。
