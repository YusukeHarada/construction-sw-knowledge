# J1939 機能安全通信・アタッチメント対応

J1939プロトコル上での機能安全通信の設計指針と、建設機械アタッチメントごとのJ1939対応をまとめる。プロトコル基礎は `j1939-detail.md`、機能安全全般は `functional-safety.md` を参照。本資料はその両者の橋渡しを目的とする。

---

## 1. 機能安全規格とJ1939の関係

### 1.1 J1939は機能安全通信規格ではない

J1939（SAE J1939シリーズ）はアプリケーション層〜ネットワーク層のプロトコル仕様であり、**機能安全のための通信保護機能は規定していない**。

| 観点 | J1939の状況 |
|---|---|
| CRCによるビットエラー検出 | CAN物理層（ISO 11898-1）のCRC15で対応。ただし単一フレームのみ |
| メッセージ欠落検知 | 規定なし。受信側がタイムアウト監視を独自実装する |
| メッセージ順序保証 | 規定なし |
| メッセージ偽装・リプレイ攻撃防止 | 規定なし（セキュリティはSecOCで別途対応） |
| E2E通信保護 | 規定なし。AUTOSAR E2E Profileで上乗せする |

ISO 25119（建設機械・農業機械向け機能安全）がJ1939通信に安全要求（AgPL c/d/e相当）を課す場合、**アプリケーション層で追加の保護機構を実装する**必要がある。

### 1.2 ISO 25119とJ1939通信の接点

ISO 25119 Part 5（通信系機能安全）は、安全関連の制御コマンドをCAN経由で送受信する際に以下を要求する（要確定・規格本文で確認すること）：

| 要求項目 | 内容 |
|---|---|
| 伝送エラー検出 | ビット反転・バーストエラーの検出（CAN CRCで部分的に対応） |
| メッセージ遅延/消失検知 | 受信タイムアウト監視が必須。タイムアウト時間はFTTI（Fault Tolerant Time Interval）内に設定 |
| 意図しないメッセージ繰り返し | Alive Counter（シーケンス番号）による古いメッセージの排除 |
| メッセージ誤挿入検知 | メッセージ認証（高AgPL要求時はSecOCまたはMAC） |
| アドレス競合時の安全処置 | アドレスクレーム失敗時にコントローラが安全状態へ遷移すること |

AgPL要求レベルに応じた通信保護の目安：

| AgPL | 主な要求 | J1939上の実装方法 |
|---|---|---|
| a / b | タイムアウト監視 | 受信タイムアウト検知 + 安全状態遷移 |
| c | ＋Alive Counter | SPNに1バイトのシーケンスカウンタを埋め込む |
| d | ＋CRC保護 | AUTOSAR E2E Profile 1/2相当をデータフィールドに追加 |
| e | ＋メッセージ認証 | SecOC（MAC付与）または冗長通信 |

## 2. J1939上の通信安全設計

### 2.1 タイムアウト監視（受信タイムアウト）

最も基本的かつ重要な保護手段。安全関連PGNが規定周期内に受信されない場合、安全状態へ遷移する。

```
設計手順：
① HARAでFTTI（故障許容時間）を決定する
   例：アーム緊急降下防止 → FTTI = 200ms

② 安全関連PGNの送信周期を決定する
   例：作業機制御PGN → 50ms周期送信

③ タイムアウト閾値を設定する
   例：タイムアウト = 3周期分 = 150ms（FTTI 200ms 以内に収まるよう設定）

④ タイムアウト検知時の安全状態遷移を設計する
   例：作業機指令を停止し、現在位置でホールド → ブザー鳴動 → DM1送信
```

タイムアウト値の根拠をSafetyに記載し、HARAのFTTI導出と対応付けること。

### 2.2 Alive Counter（シーケンスカウンタ）

同一メッセージの繰り返し受信（リプレイ）や古いメッセージの誤検知を防ぐ。J1939の8バイトデータフィールドに1バイト（0〜255のカウントアップ）を埋め込む。

```
データフィールドへの埋め込み例（作業機制御PGN）：

Byte 0: Command Type（操作種別）
Byte 1: Arm Target Speed（アーム目標速度）
Byte 2: Bucket Target Speed（バケット目標速度）
Byte 3: Boom Target Speed（ブーム目標速度）
Byte 4: Alive Counter（0〜255、毎フレームインクリメント）← 追加
Byte 5: CRC8（Byte 0〜4 のCRC）← 追加
Byte 6-7: 予備
```

受信側は、Alive Counterが前回値+1でない場合にエラーカウントを増やし、閾値超えで安全状態へ遷移する設計にする。

**注意**：Alive CounterとCRCをデータフィールドに追加すると、実際の制御データに使えるバイト数が減る。8バイトの制約上、複数PGNに分割するか、信号の解像度を落とす設計判断が必要。

### 2.3 AUTOSAR E2E Profileの適用

AUTOSARのE2EトランスフォーマーをJ1939信号に適用することで、ソフトウェアが自動でCRC・カウンタを付加・検証できる。

| E2E Profile | オーバーヘッド | 適用可否 |
|---|---|---|
| Profile 1 | 2バイト（CRC8 + Counter4bit）| 8バイト制約内に収まれば適用可能 |
| Profile 2 | 3バイト（CRC8 + Counter8bit）| 制約厳しい |
| Profile 4 | 1バイト（CRC4 + Counter4bit）| オーバーヘッド最小。AgPL c/d向け |
| Profile 11 | 1バイト（CRC4 + Counter4bit）| Profile 4の改良版 |

**建設機械での現実的な選択**：J1939の8バイト制限とAUTOSAR E2Eの相性は良くない。制御データが多い場合は、CAN FD移行（64バイト）またはプロプライエタリなAlive Counter実装を検討する。詳細は `autosar-modules.md` のE2Eセクションを参照。

### 2.4 SecOCのJ1939適用

SecOC（Secure Onboard Communication）はMACをフレームに付与し、メッセージ改ざん・なりすましを防ぐ。J1939適用時の課題：

| 課題 | 内容 |
|---|---|
| データ長制限 | 8バイト中にMAC（最小4バイト相当）を収めると制御データが圧迫される |
| J1939TP との相性 | 8バイト超えのTPメッセージにはSecOCを適用しやすい |
| 建設機械での優先度 | 外部ネットワーク接続なし・物理的な閉域ネットワークでは必須ではない場合が多い |

**推奨**：同一車内のJ1939制御バスはSecOCより**タイムアウト監視＋Alive Counter**を優先する。外部接続（クラウド・V2X）が生じるゲートウェイECUにSecOCを適用する構成が現実的。詳細は `secoc.md` を参照。

## 3. Safety-Relevant PGN の設計指針

### 3.1 安全重要度による分類

J1939上を流れるPGNを機能安全の観点から分類する。

| 分類 | 代表的なPGN例 | AgPL目安 | 通信保護要求 |
|---|---|---|---|
| 緊急停止・安全停止 | 独自Proprietary PGN（E-Stop指令） | d/e | タイムアウト＋Alive Counter＋CRC |
| 作業機制御（動作指令） | 独自Proprietary PGN（アーム・ブーム速度指令） | c/d | タイムアウト＋Alive Counter |
| 転倒防止・安定制御 | 独自Proprietary PGN（傾斜角・荷重制限） | c/d | タイムアウト＋Alive Counter |
| エンジン・走行制御 | EEC1（PGN 61444）・CCVS（PGN 65265） | b/c | タイムアウト監視 |
| 故障通知 | DM1（PGN 65226） | b | タイムアウト監視（1s周期監視） |
| モニタリング（非安全） | ET1（冷却水温）・EFL（油量）等 | a | 任意 |

### 3.2 緊急停止PGNの設計

緊急停止（E-Stop）は最も高いAgPL要求が課される通信の一つ。

```
設計例：E-Stop PGN（Proprietary B、ブロードキャスト）

送信元：HMI / オペレータパネル
送信先：全ECU（ブロードキャスト）
送信周期：10ms（通常）/ E-Stop発動時は即時連続送信

データフィールド（8バイト）：
  Byte 0: E-Stop Command（0x00=通常, 0x01=E-Stop, 0xFF=Invalid）
  Byte 1: E-Stop Source（誰がE-Stopを要求したか：オペレータ/外部/ソフト）
  Byte 2: Alive Counter（0〜255）
  Byte 3: CRC8（Byte 0〜2）
  Byte 4-7: 予備（将来拡張用）

受信側ECUの応答：
  - タイムアウト（30ms超）→ E-Stopと同等の安全状態へ遷移
  - Alive Counter異常 → エラーカウント。3回連続で安全状態へ遷移
  - CRC不一致 → 即時安全状態へ遷移
```

### 3.3 作業機制御PGNのタイムアウト設計例

```
HARA結果（例）：
  Hazard: アーム自重降下による人身事故
  AgPL: d
  FTTI: 200ms（ISO 25119 Part 2の事例から設定。要確定）

通信設計：
  送信周期: 50ms
  タイムアウト閾値: 150ms（3周期）← FTTI 200ms 以内
  タイムアウト時動作: アーム速度指令=0にフォールバック、HMIへ警告表示
```

## 4. アタッチメントのJ1939対応

建設機械のアタッチメントがJ1939バスに接続される場合、以下の規格・仕組みが関係する。

### 4.1 関連規格の全体像

```
標準PGN（汎用）
  SAE J1939-71: エンジン・変速機・油圧系の一般的なPGN/SPN
  SAE J1939-75: 土工機械（Earthmoving Machines）向け拡張PGN

アタッチメント接続プロトコル
  ISO 11783 (ISOBUS): 農業機械向けだが建設機械アタッチメントにも流用
  SAE J1939-81: アドレスクレームによる動的接続

電動アタッチメント
  ISO 25119関連: BMS・インバータ通信への適用
```

### 4.2 J1939-75（土工機械向け拡張）

J1939-75（Application Layer for Earthmoving Machines）は、掘削機・ブルドーザ・クレーン等に固有のPGNを追加定義している規格。

J1939-71の標準PGNで対応できない建設機械固有の物理量（作業機圧力・角度・荷重等）が対象。

**J1939-75で定義される信号の例（要確定・規格本文で確認）**：

| 信号カテゴリ | 信号例 | 用途 |
|---|---|---|
| 作業機角度 | ブーム角度・アーム角度・バケット角度 | 干渉防止・自動平坦化 |
| 油圧圧力 | シリンダ圧力・ポンプ吐出圧 | 過負荷検知・安全制御 |
| 作業機荷重 | バケット積載量・吊り荷重量 | 転倒防止・ペイロード管理 |
| 車体傾斜 | 前後・左右傾斜角 | 転倒防止・傾斜補正 |
| 走行系 | クローラ速度・旋回速度 | 走行安全制御 |

**注意**：J1939-75は有償SAE規格。実際に使用するSPN番号はSAE規格原典で確認すること。

### 4.3 アタッチメント別の対応状況

#### バケット（Bucket）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | バケット角度・積載量のSPNが定義されている（要確定） |
| プロプライエタリ拡張 | 刃先磨耗センサ・石質判定等は各社独自 |
| 機能安全関連 | バケット角度による「人への接触エリア」制限はAgPL c/d相当になりえる |
| アドレスクレーム | 電動バケット（センサ付き）はJ1939ノードとして起動時にアドレスクレーム |

#### ブレーカ（Hydraulic Breaker）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | ブレーカ固有のSPNは標準定義が少ない。各社プロプライエタリが主流（要確定） |
| 代表的な信号 | 打撃回数・打撃エネルギー・作動油温度・ブランク打ち検知 |
| 機能安全関連 | ブランク打ち（空打ち）検知は油圧系保護に重要。タイムアウト監視が有効 |
| 注意点 | ブレーカ自体がJ1939ノードを持つケースは少ない。油圧回路の状態をショベル側ECUが把握する構成が多い |

#### グラップル（Grapple）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | グラップル専用のSPNは標準定義が限定的（要確定） |
| 代表的な信号 | グリップ開閉角度・把持力・作業姿勢 |
| 機能安全関連 | 把持解放による落下物事故はAgPL dを要求しうる。把持確認信号の監視が重要 |
| アドレスクレーム | センサ内蔵スマートグラップルはJ1939ノードを持つ製品が増えている |

#### クイックカプラ（Quick Coupler）

| 項目 | 内容 |
|---|---|
| 特徴 | アタッチメントを油圧で着脱する装置。取り付け確認が機能安全上最重要 |
| J1939対応 | クイックカプラ確認信号はプロプライエタリPGNまたはデジタルI/Oが多い |
| 機能安全関連 | 取り付け不完全でのアタッチメント脱落はAgPL eに相当する可能性がある。二重確認（機械ロック＋電気確認）が必要 |
| 通信設計 | 「取り付け確認OK」フラグをAlive Counter付きで常時送信する設計が安全 |

#### ISOBUS（ISO 11783）対応アタッチメント

農業機械向けに発展したISOBUS（J1939をベースとした拡張規格）は、建設機械のアタッチメントにも一部転用されている。

| 特徴 | 内容 |
|---|---|
| ベース規格 | ISO 11783（J1939を基盤として上位層を拡張） |
| 接続方式 | トラクタ（ベース機械）にアタッチメントが接続し、ISOBUS上で制御データをやりとり |
| 機能 | アタッチメント自己記述（どの信号を持つかを機械が宣言）・ECU管理・制御コマンド |
| 建設機械への転用 | アタッチメント交換頻度が高い機械（マルチカプラ搭載機等）でISO 11783ベースの通信を採用するケースがある |
| 課題 | ISO 11783はAUTOSAR標準モジュールの外。対応には個別実装が必要 |

## 5. アドレスクレームと機能安全の関係

アドレスクレームは機能安全設計において見落としやすいリスクポイント。

### 5.1 アドレスクレーム失敗時のリスク

```
シナリオ：
  バケット交換後、新しいアタッチメントコントローラが起動する
  → アドレスクレームで他のECUと競合
  → アドレス取得に失敗（SA=0xFE で通信断念）
  → アタッチメントコントローラがJ1939バスから孤立
  → ベース機械側がアタッチメントからの信号受信不能
  → タイムアウト検知 → 安全状態遷移（作業機停止）
```

**設計要点**：
- アドレスクレーム失敗は「通信断」と同等のフェイルとして扱う。
- ベース機械側のタイムアウト監視が機能することが安全バックアップになる。
- AUTOSARのJ1939Nmはアドレスクレームの成否を上位アプリに通知する。この通知を機能安全ロジックに組み込む。

### 5.2 アドレスクレームの起動シーケンスと安全状態

```
推奨起動シーケンス（安全側設計）：

システム起動
  │
  ├── 全ECUがアドレスクレーム開始
  │
  ├── 250ms 待機
  │
  ├── アドレスクレーム成功 → 通常通信へ移行
  │
  └── アドレスクレーム失敗 → 【安全状態】作業機全停止を維持
                              オペレータにアラーム表示
                              エンジン起動は可（走行のみ許可等の段階的制限も可）
```

## 6. AUTOSARでの実装ポイント

### 6.1 J1939Nm と機能安全の連携

```
J1939Nm → CanSM（通信状態管理）→ DEM（故障管理）→ FiM（機能禁止管理）

J1939Nm がアドレスクレーム失敗を検知
  → CanSM に通信バス障害として報告
  → DEM が故障イベントを記録（DTC生成）
  → FiM が「作業機制御」機能を禁止フラグON
  → アプリが機能禁止を確認して安全状態へ遷移
```

### 6.2 タイムアウト監視の設定（ArXML）

ComConfigにおいて、安全関連I-PDUには必ずタイムアウト監視（Deadline Monitoring）を設定する。

```
設定項目（ArXML）：
<I-PDU>
  <NAME>WorkImplement_Control_Rx</NAME>
  <I-PDU-TIMING-SPECIFICATIONS>
    <CYCLIC-TIMING>
      <TIME-PERIOD>0.05</TIME-PERIOD>  <!-- 50ms周期 -->
    </CYCLIC-TIMING>
    <EVENT-CONTROLLED-TIMING>
      <NUMBER-OF-REPETITIONS>3</NUMBER-OF-REPETITIONS>
    </EVENT-CONTROLLED-TIMING>
  </I-PDU-TIMING-SPECIFICATIONS>
  <!-- タイムアウト = 150ms（3周期分）はPDUM側で設定 -->
</I-PDU>
```

タイムアウト検知時の処理はDEM経由でFiMに連携し、アプリがフォールバック動作を実装する。

### 6.3 委託先への要件定義チェックリスト

- [ ] 安全関連PGN一覧を作成し、各PGNにAgPL・タイムアウト値・Alive Counter要否を明記する
- [ ] アドレスクレーム失敗時の安全処置を要件に記載する（FiM連携か、アプリ直接ハンドリングか）
- [ ] J1939-75を参照するSPNのSAE規格購入・提供方法を合意する
- [ ] アタッチメントコントローラのNAMEとデフォルトSAを機械ごとに割り当て管理する担当を決める
- [ ] E2E / Alive Counter の実装責任（委託先か自社か）を明確にする
- [ ] ISO 25119 Part 5 の適用範囲（どのPGNが安全関連通信か）を正式にHARA結果から導出する
