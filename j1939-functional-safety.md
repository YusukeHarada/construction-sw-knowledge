# J1939 機能安全通信・アタッチメント対応

J1939プロトコル上での機能安全通信の設計指針と、建設機械アタッチメントごとのJ1939対応をまとめる。プロトコル基礎は `j1939-detail.md`、機能安全全般は `functional-safety.md` を参照。本資料はその両者の橋渡しを目的とする。

---

## 1. 機能安全規格とJ1939の関係

### 1.1 J1939は機能安全通信規格ではない

J1939（SAE J1939シリーズ）はアプリケーション層〜ネットワーク層のプロトコル仕様であり、**機能安全のための通信保護機能は規定していない**。

| 観点 | J1939の状況 |
|---|---|
| CRCによるビットエラー検出 | CAN物理層（ISO 11898-1）のCRC15で対応。ただし単一フレームのみ |
| メッセージ欠落検知 | 規定なし。受信側がタイムアウト監視を独自実装する |
| メッセージ順序保証 | 規定なし |
| メッセージ偽装・リプレイ攻撃防止 | 規定なし（セキュリティはSecOCで別途対応） |
| E2E通信保護 | 規定なし。AUTOSAR E2E Profileで上乗せする |

ISO 25119（建設機械・農業機械向け機能安全）がJ1939通信に安全要求（AgPL c/d/e相当）を課す場合、**アプリケーション層で追加の保護機構を実装する**必要がある。この保護機構を標準化した規格が **SAE J1939-76**（セクション7参照）である。

### 1.2 ISO 25119とJ1939通信の接点

ISO 25119 Part 5（通信系機能安全）は、安全関連の制御コマンドをCAN経由で送受信する際に以下を要求する（要確定・規格本文で確認すること）：

| 要求項目 | 内容 |
|---|---|
| 伝送エラー検出 | ビット反転・バーストエラーの検出（CAN CRCで部分的に対応） |
| メッセージ遅延/消失検知 | 受信タイムアウト監視が必須。タイムアウト時間はFTTI（Fault Tolerant Time Interval）内に設定 |
| 意図しないメッセージ繰り返し | Alive Counter（シーケンス番号）による古いメッセージの排除 |
| メッセージ誤挿入検知 | メッセージ認証（高AgPL要求時はSecOCまたはMAC） |
| アドレス競合時の安全処置 | アドレスクレーム失敗時にコントローラが安全状態へ遷移すること |

AgPL要求レベルに応じた通信保護の目安：

| AgPL | 主な要求 | J1939上の実装方法 |
|---|---|---|
| a / b | タイムアウト監視 | 受信タイムアウト検知 + 安全状態遷移 |
| c | ＋Alive Counter | SPNに1バイトのシーケンスカウンタを埋め込む |
| d | ＋CRC保護 | AUTOSAR E2E Profile 1/2相当をデータフィールドに追加 |
| e | ＋メッセージ認証 | SecOC（MAC付与）または冗長通信 |

## 2. J1939上の通信安全設計

### 2.1 タイムアウト監視（受信タイムアウト）

最も基本的かつ重要な保護手段。安全関連PGNが規定周期内に受信されない場合、安全状態へ遷移する。

```
設計手順：
① HARAでFTTI（故障許容時間）を決定する
   例：アーム緊急降下防止 → FTTI = 200ms

② 安全関連PGNの送信周期を決定する
   例：作業機制御PGN → 50ms周期送信

③ タイムアウト閾値を設定する
   例：タイムアウト = 3周期分 = 150ms（FTTI 200ms 以内に収まるよう設定）

④ タイムアウト検知時の安全状態遷移を設計する
   例：作業機指令を停止し、現在位置でホールド → ブザー鳴動 → DM1送信
```

タイムアウト値の根拠をSafetyに記載し、HARAのFTTI導出と対応付けること。

### 2.2 Alive Counter（シーケンスカウンタ）

同一メッセージの繰り返し受信（リプレイ）や古いメッセージの誤検知を防ぐ。J1939の8バイトデータフィールドに1バイト（0〜255のカウントアップ）を埋め込む。

```
データフィールドへの埋め込み例（作業機制御PGN）：

Byte 0: Command Type（操作種別）
Byte 1: Arm Target Speed（アーム目標速度）
Byte 2: Bucket Target Speed（バケット目標速度）
Byte 3: Boom Target Speed（ブーム目標速度）
Byte 4: Alive Counter（0〜255、毎フレームインクリメント）← 追加
Byte 5: CRC8（Byte 0〜4 のCRC）← 追加
Byte 6-7: 予備
```

受信側は、Alive Counterが前回値+1でない場合にエラーカウントを増やし、閾値超えで安全状態へ遷移する設計にする。

**注意**：Alive CounterとCRCをデータフィールドに追加すると、実際の制御データに使えるバイト数が減る。8バイトの制約上、複数PGNに分割するか、信号の解像度を落とす設計判断が必要。

### 2.3 AUTOSAR E2E Profileの適用

AUTOSARのE2EトランスフォーマーをJ1939信号に適用することで、ソフトウェアが自動でCRC・カウンタを付加・検証できる。

| E2E Profile | オーバーヘッド | 適用可否 |
|---|---|---|
| Profile 1 | 2バイト（CRC8 + Counter4bit）| 8バイト制約内に収まれば適用可能 |
| Profile 2 | 3バイト（CRC8 + Counter8bit）| 制約厳しい |
| Profile 4 | 1バイト（CRC4 + Counter4bit）| オーバーヘッド最小。AgPL c/d向け |
| Profile 11 | 1バイト（CRC4 + Counter4bit）| Profile 4の改良版 |

**建設機械での現実的な選択**：J1939の8バイト制限とAUTOSAR E2Eの相性は良くない。制御データが多い場合は、CAN FD移行（64バイト）またはプロプライエタリなAlive Counter実装を検討する。詳細は `autosar-modules.md` のE2Eセクションを参照。

### 2.4 SecOCのJ1939適用

SecOC（Secure Onboard Communication）はMACをフレームに付与し、メッセージ改ざん・なりすましを防ぐ。J1939適用時の課題：

| 課題 | 内容 |
|---|---|
| データ長制限 | 8バイト中にMAC（最小4バイト相当）を収めると制御データが圧迫される |
| J1939TP との相性 | 8バイト超えのTPメッセージにはSecOCを適用しやすい |
| 建設機械での優先度 | 外部ネットワーク接続なし・物理的な閉域ネットワークでは必須ではない場合が多い |

**推奨**：同一車内のJ1939制御バスはSecOCより**タイムアウト監視＋Alive Counter**を優先する。外部接続（クラウド・V2X）が生じるゲートウェイECUにSecOCを適用する構成が現実的。詳細は `secoc.md` を参照。

## 3. Safety-Relevant PGN の設計指針

### 3.1 安全重要度による分類

J1939上を流れるPGNを機能安全の観点から分類する。

| 分類 | 代表的なPGN例 | AgPL目安 | 通信保護要求 |
|---|---|---|---|
| 緊急停止・安全停止 | 独自Proprietary PGN（E-Stop指令） | d/e | タイムアウト＋Alive Counter＋CRC |
| 作業機制御（動作指令） | 独自Proprietary PGN（アーム・ブーム速度指令） | c/d | タイムアウト＋Alive Counter |
| 転倒防止・安定制御 | 独自Proprietary PGN（傾斜角・荷重制限） | c/d | タイムアウト＋Alive Counter |
| エンジン・走行制御 | EEC1（PGN 61444）・CCVS（PGN 65265） | b/c | タイムアウト監視 |
| 故障通知 | DM1（PGN 65226） | b | タイムアウト監視（1s周期監視） |
| モニタリング（非安全） | ET1（冷却水温）・EFL（油量）等 | a | 任意 |

### 3.2 緊急停止PGNの設計

緊急停止（E-Stop）は最も高いAgPL要求が課される通信の一つ。

```
設計例：E-Stop PGN（Proprietary B、ブロードキャスト）

送信元：HMI / オペレータパネル
送信先：全ECU（ブロードキャスト）
送信周期：10ms（通常）/ E-Stop発動時は即時連続送信

データフィールド（8バイト）：
  Byte 0: E-Stop Command（0x00=通常, 0x01=E-Stop, 0xFF=Invalid）
  Byte 1: E-Stop Source（誰がE-Stopを要求したか：オペレータ/外部/ソフト）
  Byte 2: Alive Counter（0〜255）
  Byte 3: CRC8（Byte 0〜2）
  Byte 4-7: 予備（将来拡張用）

受信側ECUの応答：
  - タイムアウト（30ms超）→ E-Stopと同等の安全状態へ遷移
  - Alive Counter異常 → エラーカウント。3回連続で安全状態へ遷移
  - CRC不一致 → 即時安全状態へ遷移
```

### 3.3 作業機制御PGNのタイムアウト設計例

```
HARA結果（例）：
  Hazard: アーム自重降下による人身事故
  AgPL: d
  FTTI: 200ms（ISO 25119 Part 2の事例から設定。要確定）

通信設計：
  送信周期: 50ms
  タイムアウト閾値: 150ms（3周期）← FTTI 200ms 以内
  タイムアウト時動作: アーム速度指令=0にフォールバック、HMIへ警告表示
```

## 4. アタッチメントのJ1939対応

建設機械のアタッチメントがJ1939バスに接続される場合、以下の規格・仕組みが関係する。

### 4.1 関連規格の全体像

```
標準PGN（汎用）
  SAE J1939-71: エンジン・変速機・油圧系の一般的なPGN/SPN
  SAE J1939-75: 土工機械（Earthmoving Machines）向け拡張PGN

アタッチメント接続プロトコル
  ISO 11783 (ISOBUS): 農業機械向けだが建設機械アタッチメントにも流用
  SAE J1939-81: アドレスクレームによる動的接続

電動アタッチメント
  ISO 25119関連: BMS・インバータ通信への適用
```

### 4.2 J1939-75（土工機械向け拡張）

J1939-75（Application Layer for Earthmoving Machines）は、掘削機・ブルドーザ・クレーン等に固有のPGNを追加定義している規格。

J1939-71の標準PGNで対応できない建設機械固有の物理量（作業機圧力・角度・荷重等）が対象。

**J1939-75で定義される信号の例（要確定・規格本文で確認）**：

| 信号カテゴリ | 信号例 | 用途 |
|---|---|---|
| 作業機角度 | ブーム角度・アーム角度・バケット角度 | 干渉防止・自動平坦化 |
| 油圧圧力 | シリンダ圧力・ポンプ吐出圧 | 過負荷検知・安全制御 |
| 作業機荷重 | バケット積載量・吊り荷重量 | 転倒防止・ペイロード管理 |
| 車体傾斜 | 前後・左右傾斜角 | 転倒防止・傾斜補正 |
| 走行系 | クローラ速度・旋回速度 | 走行安全制御 |

**注意**：J1939-75は有償SAE規格。実際に使用するSPN番号はSAE規格原典で確認すること。

### 4.3 アタッチメント別の対応状況

#### バケット（Bucket）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | バケット角度・積載量のSPNが定義されている（要確定） |
| プロプライエタリ拡張 | 刃先磨耗センサ・石質判定等は各社独自 |
| 機能安全関連 | バケット角度による「人への接触エリア」制限はAgPL c/d相当になりえる |
| アドレスクレーム | 電動バケット（センサ付き）はJ1939ノードとして起動時にアドレスクレーム |

#### ブレーカ（Hydraulic Breaker）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | ブレーカ固有のSPNは標準定義が少ない。各社プロプライエタリが主流（要確定） |
| 代表的な信号 | 打撃回数・打撃エネルギー・作動油温度・ブランク打ち検知 |
| 機能安全関連 | ブランク打ち（空打ち）検知は油圧系保護に重要。タイムアウト監視が有効 |
| 注意点 | ブレーカ自体がJ1939ノードを持つケースは少ない。油圧回路の状態をショベル側ECUが把握する構成が多い |

#### グラップル（Grapple）

| 項目 | 内容 |
|---|---|
| J1939-75対応 | グラップル専用のSPNは標準定義が限定的（要確定） |
| 代表的な信号 | グリップ開閉角度・把持力・作業姿勢 |
| 機能安全関連 | 把持解放による落下物事故はAgPL dを要求しうる。把持確認信号の監視が重要 |
| アドレスクレーム | センサ内蔵スマートグラップルはJ1939ノードを持つ製品が増えている |

#### クイックカプラ（Quick Coupler）

| 項目 | 内容 |
|---|---|
| 特徴 | アタッチメントを油圧で着脱する装置。取り付け確認が機能安全上最重要 |
| J1939対応 | クイックカプラ確認信号はプロプライエタリPGNまたはデジタルI/Oが多い |
| 機能安全関連 | 取り付け不完全でのアタッチメント脱落はAgPL eに相当する可能性がある。二重確認（機械ロック＋電気確認）が必要 |
| 通信設計 | 「取り付け確認OK」フラグをAlive Counter付きで常時送信する設計が安全 |

#### ISOBUS（ISO 11783）対応アタッチメント

農業機械向けに発展したISOBUS（J1939をベースとした拡張規格）は、建設機械のアタッチメントにも一部転用されている。

| 特徴 | 内容 |
|---|---|
| ベース規格 | ISO 11783（J1939を基盤として上位層を拡張） |
| 接続方式 | トラクタ（ベース機械）にアタッチメントが接続し、ISOBUS上で制御データをやりとり |
| 機能 | アタッチメント自己記述（どの信号を持つかを機械が宣言）・ECU管理・制御コマンド |
| 建設機械への転用 | アタッチメント交換頻度が高い機械（マルチカプラ搭載機等）でISO 11783ベースの通信を採用するケースがある |
| 課題 | ISO 11783はAUTOSAR標準モジュールの外。対応には個別実装が必要 |

### 4.5 アタッチメント別通信設定の管理（J1939-76適用時の課題）

アタッチメントごとにPGN定義・J1939-76 SPDUプロファイル・タイムアウト値が異なる場合、ベース機械側ECUはそれらを**全種類分保持する**必要がある。これがClassic AUTOSARとの組み合わせで特有の問題を生む。

#### なぜ全種類分の設定が必要か

```
アタッチメントA（バケット）：
  PGN: 0xFF10（独自）, SCRC8, timeout=150ms, AgPL d

アタッチメントB（ブレーカ）：
  PGN: 0xFF20（独自）, SCRC4, timeout=300ms, AgPL c

アタッチメントC（グラップル）：
  PGN: 0xFF30（独自）, SCRC8, timeout=100ms, AgPL d

→ ベース機械ECUは上記3種類すべての設定を保持しないと
  どのアタッチメントが接続されても対応できない
```

#### Classic AUTOSARの制約

Classic AUTOSARのCOM設定・E2E設定はArXML（ECUコンフィギュレーション）に**静的**に記述される。アタッチメントが交換されるたびに動的に通信設定を切り替える標準的な仕組みはない。

| 課題 | 内容 |
|---|---|
| ArXML静的定義 | サポートする全アタッチメント分のI-PDU・E2E設定をArXMLに記述しておく必要がある |
| 実行時切り替え | アタッチメント接続時にどの設定セットを「有効」にするかは、アプリ層またはBSW拡張で対応する |
| 設定爆発 | サポートアタッチメント種類×安全プロファイルの組み合わせが増えると設定管理が複雑になる |
| ECUメモリ | 未使用のアタッチメント設定もECUフラッシュに常駐する。メモリ容量の計画が必要 |

#### 対処アプローチ

**アプローチ1：静的ライブラリ方式（現実的な第一選択）**

サポートするアタッチメント種類をあらかじめ決め、全設定をArXMLに書き込む。アタッチメント接続時はNAMEのFunction Codeを読み取り、対応する設定セットを有効化する。

```
ECU内部の論理構成：

アドレスクレーム完了
  → アタッチメントのNAMEを取得
  → Function Code（NAMEの8bit フィールド）を確認
  → 対応する設定セット（PGN・SPDUプロファイル・timeout）を有効化
  → 有効化された設定でSPDU検証・タイムアウト監視を開始
  → 未知のFunction Code → 接続拒否 or 非安全モードで起動
```

- **利点**：シンプル。ArXMLで管理。認証証跡が明確。
- **欠点**：新規アタッチメント追加のたびにECUソフト更新（OTA）が必要。

**アプローチ2：設定ダウンロード方式**

新規アタッチメント対応時にOTA（または診断チャネル経由）でECUの設定データだけを更新する。コード変更なしで対応アタッチメントを追加できる。

- **利点**：フィールドでの柔軟な拡張が可能。
- **欠点**：設定データの検証・機能安全証跡の管理が複雑になる。OTA/SBOMの管理と連動が必要（`ota-sbom.md` 参照）。

**アプローチ3：ISOBUS（ISO 11783）自己記述方式**

ISO 11783ベースのアタッチメントは接続時に自分の通信仕様を宣言する（アタッチメント自己記述）。ベース機械はこの宣言を受け取り動的に設定を構築する。

- **利点**：未知のアタッチメントにも原理上対応可能。
- **欠点**：J1939-76のSPDU設定（SCRCプロファイル等）は静的合意が前提のため、動的自己記述とは相性が良くない。ISO 11783はAUTOSAR標準外で個別実装が必要。機能安全認証の証跡も複雑になる。

#### 設定管理の実務的指針

```
推奨方針：アプローチ1（静的ライブラリ）を基本とし、OTAで拡張

サポートアタッチメント種類：開発着手前に確定する
  ├── 標準アタッチメント（バケット・ブレーカ・グラップル等）→ ArXMLに初期収録
  └── 将来アタッチメント → OTAによる設定追加で対応

設定セットの単位：
  アタッチメント1種類 = {
    PGN番号リスト（送受信）,
    各PGNのSPDUプロファイル（SCRC4/8/16とSC検証方法）,
    タイムアウト値（PGN毎のFTTI由来）,
    アタッチメントのNAME Function Code（識別キー）,
    AgPL要求（HARA由来）
  }
```

**重要**：設定セット自体も機能安全の管理対象。追加・変更時はHARAの再評価とSafety Caseの更新が必要。安易に設定ファイルをフィールドで書き換えられる設計にしないこと。

## 5. アドレスクレームと機能安全の関係

アドレスクレームは機能安全設計において見落としやすいリスクポイント。

### 5.1 アドレスクレーム失敗時のリスク

```
シナリオ：
  バケット交換後、新しいアタッチメントコントローラが起動する
  → アドレスクレームで他のECUと競合
  → アドレス取得に失敗（SA=0xFE で通信断念）
  → アタッチメントコントローラがJ1939バスから孤立
  → ベース機械側がアタッチメントからの信号受信不能
  → タイムアウト検知 → 安全状態遷移（作業機停止）
```

**設計要点**：
- アドレスクレーム失敗は「通信断」と同等のフェイルとして扱う。
- ベース機械側のタイムアウト監視が機能することが安全バックアップになる。
- AUTOSARのJ1939Nmはアドレスクレームの成否を上位アプリに通知する。この通知を機能安全ロジックに組み込む。

### 5.2 アドレスクレームの起動シーケンスと安全状態

```
推奨起動シーケンス（安全側設計）：

システム起動
  │
  ├── 全ECUがアドレスクレーム開始
  │
  ├── 250ms 待機
  │
  ├── アドレスクレーム成功 → 通常通信へ移行
  │
  └── アドレスクレーム失敗 → 【安全状態】作業機全停止を維持
                              オペレータにアラーム表示
                              エンジン起動は可（走行のみ許可等の段階的制限も可）
```

## 6. AUTOSARでの実装ポイント

### 6.1 J1939Nm と機能安全の連携

```
J1939Nm → CanSM（通信状態管理）→ DEM（故障管理）→ FiM（機能禁止管理）

J1939Nm がアドレスクレーム失敗を検知
  → CanSM に通信バス障害として報告
  → DEM が故障イベントを記録（DTC生成）
  → FiM が「作業機制御」機能を禁止フラグON
  → アプリが機能禁止を確認して安全状態へ遷移
```

### 6.2 タイムアウト監視の設定（ArXML）

ComConfigにおいて、安全関連I-PDUには必ずタイムアウト監視（Deadline Monitoring）を設定する。

```
設定項目（ArXML）：
<I-PDU>
  <NAME>WorkImplement_Control_Rx</NAME>
  <I-PDU-TIMING-SPECIFICATIONS>
    <CYCLIC-TIMING>
      <TIME-PERIOD>0.05</TIME-PERIOD>  <!-- 50ms周期 -->
    </CYCLIC-TIMING>
    <EVENT-CONTROLLED-TIMING>
      <NUMBER-OF-REPETITIONS>3</NUMBER-OF-REPETITIONS>
    </EVENT-CONTROLLED-TIMING>
  </I-PDU-TIMING-SPECIFICATIONS>
  <!-- タイムアウト = 150ms（3周期分）はPDUM側で設定 -->
</I-PDU>
```

タイムアウト検知時の処理はDEM経由でFiMに連携し、アプリがフォールバック動作を実装する。

## 7. J1939-76（Application Layer - Functional Safety）

### 7.1 J1939-76とは

SAE J1939-76は**J1939上の機能安全通信を標準化した規格**である。セクション2で説明したAlive Counter＋CRCによる手動実装を、SPDU（Safety PDU）として体系化・標準化したものに相当する。

| 項目 | 内容 |
|---|---|
| 規格名 | SAE J1939-76 Application Layer - Functional Safety（要確定・発行年確認） |
| 目的 | J1939メッセージに安全保護データ（Safety Counter＋Safety CRC）を付加し、機能安全規格が求めるエラー検出要件を満たす |
| 対応規格 | ISO 25119 Part 5・IEC 61784-3（安全関連通信のエラー分類に対応） |
| 位置付け | J1939-71（アプリ層）の上位層として、安全関連PGNに対して適用する |

**セクション2との関係**：本資料で説明した「SPNにAlive CounterとCRC8を手動で埋め込む手法」は、J1939-76が標準化しているアプローチと同等の考え方である。認証（ISO 25119）を取得する場合、独自実装よりもJ1939-76準拠の実装のほうが**認証機関に対する説明・証跡が容易**になる。

### 7.2 SPDU（Safety PDU）の構造

J1939-76はSPDU（Safety PDU）という概念を定義する。通常のJ1939 PDUのデータフィールドにSafety Counter（SC）とSafety CRC（SCRC）を追加したものがSPDU。

```
通常のJ1939メッセージ（8バイト）：
┌─────────────────────────────────────────────────┐
│          Application Data（最大8バイト）          │
└─────────────────────────────────────────────────┘

SPDU（J1939-76適用後、例：SCRC8の場合）：
┌──────────────────────────┬────┬────────┐
│ Application Data（6バイト）│ SC │ SCRC8  │
│                          │1byte│  1byte │
└──────────────────────────┴────┴────────┘
```

| フィールド | サイズ | 役割 | 対応するエラー種別 |
|---|---|---|---|
| SC（Safety Counter） | 1バイト（0〜255循環） | 送信毎にインクリメント | 繰り返し・消失・順序誤り |
| SCRC（Safety CRC） | 4/8/16/32bit（要確定） | SC＋データ＋SAに対するCRC | データ破損・誤挿入・アドレス誤り |

### 7.3 SCRCのバリアントと安全レベル（要確定）

SPDUはSCRCのビット長に応じて複数のバリアントが存在し、それぞれ異なる安全要求レベルに対応する（具体的なバリアント番号・ビット長はSAE規格本文で確認すること）。

| SCRC長 | 目安のAgPL | 特徴 |
|---|---|---|
| 4bit（SCRC4） | b/c | オーバーヘッド最小（SC1バイト＋SCRC0.5バイト相当）。帯域制約が厳しい場合 |
| 8bit（SCRC8） | c/d | SC1バイト＋SCRC1バイト計2バイトのオーバーヘッド。建設機械の多くのユースケースに対応 |
| 16bit（SCRC16） | d/e | 計3バイトのオーバーヘッド。高安全要求（緊急停止・クイックカプラ確認等） |
| 32bit（SCRC32） | e | 計5バイトのオーバーヘッド。8バイトデータ制約のため制御データが3バイトしか残らない |

### 7.4 対応できるエラー種別

IEC 61784-3（機能安全通信のエラー分類）が定義するエラー種別に対して、J1939-76 SPDUがどう対応するかを示す。

| エラー種別 | 内容 | SPDUでの対応 |
|---|---|---|
| Corruption（破損） | データの一部がビット反転等で変化 | SCRCで検出 |
| Unintended Repetition（繰り返し） | 同じメッセージが重複して届く | SCで検出（カウンタが進まない）|
| Incorrect Sequence（順序誤り） | 複数メッセージの順序が入れ替わる | SCで検出 |
| Loss（消失） | メッセージが届かない | タイムアウト監視（SPDU外・受信側で実装）|
| Unacceptable Delay（遅延超過） | 規定時間内に届かない | タイムアウト監視（SPDU外） |
| Insertion（誤挿入） | 別の送信元から偽のメッセージが挿入 | SCRCの計算にSAを含めることで検出 |
| Masquerading（なりすまし） | 別のECUが自分のふりをして送信 | SCRCにSAを含めることで部分的に対応 |
| Addressing（アドレス誤り） | 宛先を間違えたメッセージを受信 | 受信側フィルタ＋SCRCで対応 |

**タイムアウト監視はSPDUの外**：Loss（消失）とDelay（遅延超過）は受信側のタイムアウト監視で対応する。セクション2のタイムアウト設計はJ1939-76 SPDUと組み合わせて使うものである。

### 7.5 AUTOSAR E2E Profileとの比較

AUTOSARのE2E TransformerとJ1939-76 SPDUは役割が重複する部分がある。

| 比較項目 | AUTOSAR E2E Profile | J1939-76 SPDU |
|---|---|---|
| 標準化主体 | AUTOSAR Consortium | SAE International |
| 主な適用領域 | 自動車（ASIL準拠） | 農業・建設機械（AgPL準拠） |
| J1939との親和性 | 中（J1939の構造を意識していない） | 高（J1939のフレーム・SAを前提に設計） |
| 認証実績 | ISO 26262・IEC 61508の豊富な実績 | ISO 25119向けとして設計 |
| AUTOSARモジュール | E2E Transformer（標準モジュールあり） | 専用モジュールなし（要確定・BSWベンダ確認）|
| 建設機械での選択 | 自動車系委託先との連携・AUTOSAR標準準拠を優先する場合 | J1939規格との整合・ISO 25119認証証跡を重視する場合 |

**現時点での推奨方針**：
- AUTOSAR Classic BSWを委託先が構築する場合 → **E2E TransformerをJ1939安全関連I-PDUに適用**する方が委託先のスキルと合致しやすい。
- J1939-76準拠が顧客仕様・認証機関から要求される場合 → BSWベンダ（EB・Vector等）の対応状況を確認した上で選択する（要確定）。

### 7.6 建設機械開発での実務的な影響

J1939-76を知っておくべき理由：

1. **認証審査での説明力**：ISO 25119認証取得時、審査員に「通信保護をどう実現したか」を説明する際、「J1939-76に準拠した」と言えれば根拠が明確になる。
2. **委託先との仕様合意**：委託先（特にEUの農業機械系ベンダ）はJ1939-76を知っている場合が多い。仕様書に「安全関連PGNにJ1939-76 SPDU準拠の保護を適用すること」と明記できる。
3. **ProprietaryなSPDU実装との整合**：自社のカスタムAlive Counter＋CRCがJ1939-76と互換性を持つかを確認しておく。完全互換でなくても、J1939-76のエラー種別対応表を参照してカバレッジを評価する資料として活用できる。

## 8. J1939-77（CAN FD Functional Safety Assurance Data）

### 8.1 J1939-77とは

SAE J1939-77（発行：2025年11月、`j1939-77_202511`）は**CAN FD上での機能安全通信を定義した規格**である。J1939-76がClassical CAN（8バイト制限）向けであるのに対し、J1939-77はJ1939-22（CAN FD Data Link Layer）を前提に設計されている。

```
J1939の機能安全通信規格の対応関係：

Classical CAN（J1939-21ベース）  →  J1939-76（SPDU方式、8バイト制限内に収める）
CAN FD（J1939-22ベース）        →  J1939-77（Assurance Trailer方式、大ペイロード対応）
```

| 項目 | J1939-76 | J1939-77 |
|---|---|---|
| 対応CAN | Classical CAN（250/500kbps） | CAN FD（J1939-22、データフェーズ最大2Mbps） |
| 保護データの配置 | データフィールド内に埋め込む（SPDU） | メッセージ末尾に追加する（Assurance Trailer） |
| CRC強度 | 4/8/16bit（SCRCバリアント） | 32bit または 64bit |
| シーケンス番号 | 8bit（SC、0〜255循環） | 32bit（より長い周期で繰り返し検出） |
| ペイロード | 8バイト制限内 | 最大65,526バイト（Profile 3、FD-TP使用） |
| 接続認証 | SAに依存（リンクローカル） | DataID（アドレス非依存のシステム固有認証） |

### 8.2 J1939-22との関係

J1939-77はJ1939-22が定義する以下の2つのフレーム形式の「予約済み安全領域」を利用する。

- **Multi-PG（マルチPGフレーム）**：J1939-22が導入したコンテナフレーム。1つのCAN FDフレームに複数のPGNを格納できる。このフレーム内にAssurance Trailerを挿入する領域が予約されている。
- **FD-TP（FD Transport Protocol）**：J1939-22のCAN FD向けTPプロトコル。Classical CAN TPより最大転送サイズが大幅に拡大。FD-TPセッション内にもAssurance Trailerを組み込む設計。

J1939-22がCAN FDの「土台」、J1939-77がその土台の上に乗る「機能安全層」という関係。

### 8.3 Assurance Trailer の構造

J1939-77のAssurance Trailerはメッセージデータの末尾に付加される。Classical CANのSPDU（データ内埋め込み）と異なり、ペイロードデータと分離されているため、データ設計への影響が少ない。

```
J1939-77 メッセージ構成（概念）：

┌──────────────────────────────────┬────────────────────────────────┐
│  Application Data（任意長）      │  Assurance Trailer             │
│  通常の制御・状態データ          │  Sequence Number（32bit）      │
│                                  │  CRC（32bit または 64bit）     │
│                                  │  Length（CRC計算対象バイト数） │
│                                  │  DataID（Profile 2/3のみ）     │
└──────────────────────────────────┴────────────────────────────────┘
```

| フィールド | サイズ | 役割 |
|---|---|---|
| Sequence Number | 32bit | 送信毎にインクリメント。繰り返し・順序誤りを検出 |
| CRC | 32bit（Profile 1/2）または 64bit（Profile 3） | データ＋Sequence Numberに対するCRC。破損・誤挿入を検出 |
| Length | 可変 | CRCの計算対象となったデータのバイト数 |
| DataID | Profile 2/3のみ | システム固有の接続認証ID。SAに依存しない認証を提供 |

### 8.4 3つのプロファイル

| プロファイル | ペイロード | CRC | DataID | 主な特徴 |
|---|---|---|---|---|
| Profile 1 | 固定 8バイト | 32bit | なし | オーバーヘッド最小。Classical CANからの移行に近い使い方 |
| Profile 2 | 可変 0〜19バイト | 32bit | あり | DataIDによるルーティング対応。ゾーンアーキ対応機器向け |
| Profile 3 | 最大 65,526バイト | 64bit | あり | 大ペイロード＋最高強度のCRC。将来の高安全要求用途向け |

**DataIDとは**：J1939-76のSPDUではCRC計算にSource Address（SA）を含めることで送信元を識別するが、アドレスクレームによってSAが変わった場合に認証が破れるリスクがある。J1939-77のDataIDはSAに依存しないシステム固有の64bit識別子で、この問題を解決する。

### 8.5 建設機械開発への影響

**現在（Classical CANが主流）**

現行の建設機械J1939バスにはJ1939-77は適用できない。J1939-77はCAN FD（J1939-22）が前提であり、現在のClassical CAN環境では無関係。

**CAN FD移行時（将来）**

`can-fd-tsn.md` で整理しているCAN FD移行を実施する際、J1939-22ベースに切り替えると同時にJ1939-77による機能安全通信が選択肢に入る。

```
移行のタイミングと機能安全通信規格の対応：

現在（Classical CAN）
  → J1939-76（SPDU）を適用 ← 今すぐ使える

CAN FD移行後（J1939-22採用時）
  → J1939-77（Assurance Trailer）が選択肢
  → J1939-76との後方互換性の設計も要検討
```

**J1939-77が特に注目される点**：
1. **32bit Sequence Number**：J1939-76の8bit SCに比べ、長周期の通信やバースト通信でも繰り返し検出精度が高い
2. **64bit CRC（Profile 3）**：大容量データを1セッションで安全に転送可能（OTA更新・デジタルツイン連携等の将来用途）
3. **DataIDによるアドレス非依存認証**：アタッチメント交換・アドレスクレーム変動がある建設機械環境に適する

**注意**：J1939-77は2025年11月発行の非常に新しい規格。現時点でAUTOSARの標準モジュール対応状況・BSWベンダ（EB・Vector等）の実装提供状況は要確認。
