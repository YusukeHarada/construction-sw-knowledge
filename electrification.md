# 建設機械の電動化とソフトウェアへの影響

## 1. 建設機械電動化の現状

### 電動化の3パターン

| パターン | 概要 | 主な用途 |
|---|---|---|
| フル電動 | エンジンなし・バッテリー＋電動モータのみ | ミニショベル・フォークリフト・小型ホイールローダー |
| ハイブリッド | エンジン＋電動モータ・回生エネルギー活用 | 中型ショベル・クレーン |
| 電動油圧 | 電動モータで油圧ポンプを駆動 | 既存機械への段階的電動化 |

フル電動は主に小型・中型機械から普及が進んでいる。大型機械（大型油圧ショベル・大型ブルドーザー等）はエネルギー密度の制約からハイブリッドまたは水素燃料電池との組み合わせが現実的な解となる見込みである。

### 主要メーカーの動向

- **コマツ**：PC30E-5（電動ミニショベル）を市場投入。20トン以上クラスでは水素・電動の両方を開発中
- **日立建機**：ZX55U-6EB（電動ミニショベル）を欧州市場向けに展開
- **ボルボ CE**：ECR25 Electric・L25 Electricなどをラインナップ。2030年代に中型機電動化を目標
- **カタピラー・ジョンディア**：大型機向けのバッテリー電動・水素エンジンを並行開発

### 電動化加速の背景

- EU排出ガス規制（Stage V）の段階的強化と都市部での非排ガス作業要件
- 建設現場のカーボンニュートラル目標（ゼネコン各社のScope 3削減）
- 夜間・屋内作業での低騒音・排ガスゼロの需要（トンネル工事・ビル解体等）
- バッテリーコスト低下（2030年目安：要確定）による経済性向上

## 2. 電動化によって生まれる新しいソフトウェア要件

### BMS（Battery Management System）

BMSは電動建機において最も重要なソフトウェアコンポーネントである。

```
主要機能：
  ├── SOC推定（State of Charge：残量管理）
  ├── SOH推定（State of Health：劣化度管理）
  ├── セル電圧・温度の個別監視
  ├── セル均等化（パッシブ／アクティブ均等化）
  ├── 過充電・過放電・過電流保護
  └── 絶縁抵抗監視（高電圧系の地絡検出）
```

### モータ制御（インバータ制御）

- **FOC（Field Oriented Control：磁界方向制御）**：高効率・高応答のトルク制御。従来のエンジン制御より高頻度の制御ループ（要確定：1kHz以上）が必要
- **インバータ保護**：IGBT／SiC素子の過熱・短絡保護

### エネルギー回生

建設機械特有の回生機会：
- アーム・ブーム降下時の位置エネルギー回生
- スイング制動時の慣性エネルギー回生
- 走行制動時の回生ブレーキ

回生した電力をバッテリーに戻すか補機電源に転用するかの制御判断がソフトウェアに求められる。

### 電力分配制御

複数の電動アクチュエータ（走行・旋回・作業機・補機）への電力優先制御：

```
バッテリー（400V系）
    │
    ├── 走行モータ（インバータ）
    ├── 旋回モータ（インバータ）
    ├── 油圧ポンプモータ（インバータ）
    └── DC/DCコンバータ → 補機（24V系：空調・照明・制御ECU）
```

SOCが低下した際にどのアクチュエータを優先するか（安全要件と作業効率のトレードオフ）をソフトウェアで定義する必要がある。

### 熱マネジメント

- バッテリーパックの温度を適正範囲（要確定：15〜45℃程度）に保つ冷却制御
- パワーエレクトロニクス（インバータ・DC/DC）の冷却制御
- 低温環境でのバッテリー加熱（プレコンディショニング）

## 3. 電気アーキテクチャの変化

### 電圧体系の移行

```
現状（エンジン機）        電動化後
─────────────────────────────────────────────────
12V / 24V 単一系    →    高電圧系（400〜800V）＋
                         低電圧系（24V）の分離構成
```

高電圧系と低電圧系は物理的・電気的に分離し、DC/DCコンバータを介して接続する。高電圧系には絶縁監視装置（IMD：Insulation Monitoring Device）の常時監視が必要となる。

### 電気アーキテクチャ概略図

```
┌─────────────────────────────────────────────────┐
│              高電圧系（400〜800V）               │
│                                                  │
│  ┌──────────┐    ┌──────────────────────────┐   │
│  │ バッテリー│────│      PDU（電力分配ユニット）│   │
│  │  パック  │    └──────┬──────┬──────┬──────┘   │
│  └──────────┘         │      │      │          │
│                   走行INV 旋回INV HYD-INV        │
│                       │      │      │          │
│                   走行MOT 旋回MOT 油圧ポンプMOT  │
└─────────────────────────────────────────────────┘
              │
         DC/DCコンバータ
              │
┌─────────────────────────────────────────────────┐
│              低電圧系（24V）                     │
│                                                  │
│   BMS-ECU  ゲートウェイ  HMI  各制御ECU  補機    │
└─────────────────────────────────────────────────┘
```

### ネットワーク要件の変化

BMSは各セルの電圧・温度データを高頻度（要確定：10〜100ms周期）で送信する。従来のJ1939/CAN（250kbps）ではデータ量が増大するため、BMS専用のCAN FDバスまたはEthernet化が現実的な選択肢となる。これは既存のネットワーク設計を見直す契機となる。

### 機能安全要件の拡張

電動化により以下の新たなハザードへの対応が必要となる：

| ハザード | 関連規格 | 対応概要 |
|---|---|---|
| 感電（高電圧系の地絡） | IEC 60364・ISO 6469 | 絶縁監視・インターロック |
| 熱暴走（バッテリー） | IEC 62133・UN/ECE R100 | 温度監視・過充電保護・ベント設計 |
| 過充電・過放電 | ISO 25119 AgPL | BMSソフトウェアの安全設計 |
| 電磁妨害（EMI） | CISPR・ISO 11452 | インバータノイズ対策 |

ISO 25119（農業機械・建設機械向け機能安全規格）に加えて、高電圧系固有の規格（IEC 60730等）への対応が追加される。

## 4. AUTOSARへの移行と電動化の関係

### 電動化は「ゼロから設計し直すチャンス」

電動建機のBMSソフトウェアやモータ制御ソフトウェアは、既存のエンジン制御コードとは全く異なる新規開発となる。これは**レガシーコードを引き継がない状態でAUTOSAR設計を適用できる絶好の機会**である。

既存内製コードの技術的負債（仕様書なし・テストなし）を電動化部分には持ち込まない方針を最初から決めることができる。

### Classic AUTOSARによるBMS統合

```
BMS SWC
├── BMSCore（SOC/SOH演算・セル監視）
├── BMSThermal（温度管理・冷却制御）
└── BMSProtection（過充電・過放電・過電流保護）

利用するAUTOSARモジュール：
  COM  ── BMSデータのCAN FD送信
  Dem  ── バッテリー異常のDTC管理
  NvM  ── SOH履歴・累積充放電回数の不揮発保存
  Wdg  ── BMS-ECUのウォッチドッグ管理
```

### 将来のAdaptive AUTOSAR：エネルギー管理SOAサービス化

2030年以降の集中型アーキテクチャでは、エネルギー管理をSOAサービスとして提供する構成が想定される：

```
ara::com サービス例：
  EnergyManagement/GetSOC          → 現在のSOC取得
  EnergyManagement/SetChargingMode → 充電モード設定
  PowerAllocation/RequestPower     → 電力要求
```

## 5. クラウド・OTAとの連携

### バッテリー状態データのクラウド送信

BMSのリアルタイムデータ（SOC・SOH・温度・充放電電流）をLinuxゲートウェイ経由でAWS IoT Core／Azure IoT Hubに送信する。フリート全体のバッテリー状態を一元管理し、劣化傾向を早期検知する。

### 充電スケジュール最適化

- 複数台の電動建機が並列充電する際の電力ピーク抑制（デマンドコントロール）
- 夜間の低電力単価帯への充電シフト
- クラウドからの充電開始・停止指示

### BMSパラメータのOTA更新

SOC推定アルゴリズムの改善（実運用データに基づくキャリブレーション）やセル均等化パラメータをOTAで更新する。BMSパラメータの更新はバッテリー保護に直結するため、WP.29 R156準拠のセキュアOTA手順が必須となる。

### 電動建機の予知保全

- セルの劣化傾向（容量低下・内部抵抗増加）からバッテリー交換時期を予測
- インバータの温度異常傾向からパワーエレクトロニクス障害を事前検知

## 6. 建設機械固有の電動化課題

| 課題 | 内容 | ソフトウェアへの影響 |
|---|---|---|
| 電力変動が大きい | 掘削時（最大電力）←→アイドル（最小電力）が頻繁に切り替わる | BMSのSOC推定精度・電力分配制御の応答性 |
| 稼働時間が長い | 1日8〜12時間の連続稼働 | バッテリー熱管理・急速充電対応・バッテリー交換式設計 |
| 過酷環境 | 高温（砂漠・夏季）・低温（寒冷地）・振動・粉塵 | 温度範囲を拡大した熱マネジメント制御・防振設計 |
| 充電インフラ未整備 | 建設現場に電源がない、または電力容量が小さい | 現場発電機対応の充電制御・バッテリー交換方式の検討 |
| 大型機の電動化限界 | 20t以上クラスはバッテリーエネルギー密度が不足 | 水素燃料電池・ハイブリッドとのソフトウェア共通化設計 |

## 7. 自社への示唆・移行ロードマップ

### 電動化対応ソフトウェアをAUTOSARで設計する意義

電動化部分を新規AUTOSAR設計で立ち上げることで、以下を実現できる：

1. 仕様書・ArXMLベースの設計資産を最初から確立できる
2. 委託先と共通のAUTOSAR言語で仕様管理できる
3. 機能安全（ISO 25119 AgPL）の証跡をAUTOSARワークフローで生成できる
4. 既存エンジン制御コードの技術的負債を電動化部分に持ち込まない

### 既存コードとBMSコードの共存期間

```
移行期間のECU構成イメージ：

エンジン制御ECU（既存内製コード）── エンジン機に搭載継続
BMS・モータ制御ECU（新規AUTOSAR設計）── 電動機専用
ゲートウェイECU（Linux）── 両方の診断データを集約・クラウド送信
```

エンジン機と電動機が混在するフリートに対して、共通のゲートウェイ・クラウド基盤で両方を管理する構成とする。

### 2026〜2035年フェーズ計画

| フェーズ | 期間 | 内容 |
|---|---|---|
| Phase 1：基盤整備 | 2026〜2028 | Classic AUTOSAR移行（エンジン機）・BMSソフトウェア要件定義・電動化技術調査 |
| Phase 2：電動化初期対応 | 2028〜2030 | 電動ミニショベル向けBMS・モータ制御SWCをAUTOSARで開発・SIL検証環境整備 |
| Phase 3：電動化展開 | 2030〜2033 | 中型機電動化対応・エネルギー管理クラウド連携・OTAによるBMSパラメータ更新 |
| Phase 4：大型機・次世代 | 2033〜2035 | ハイブリッド・水素燃料電池対応・Adaptive AUTOSARエネルギー管理SOA化 |

## まとめ

電動化は建設機械ソフトウェア開発にとって大きな技術的挑戦であると同時に、**レガシーコードを引き継がずにAUTOSAR準拠設計を適用できる稀少な機会**でもある。BMS・モータ制御・電力分配制御という新規ソフトウェアドメインを、最初からAUTOSAR SWC設計・機能安全証跡管理の枠組みで構築することで、将来の電動化展開とソフトウェア品質向上を同時に達成することを目指す。
