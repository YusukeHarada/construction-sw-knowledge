# AUTOSAR移行・SDV化 ロードマップ

建設機械（RTOS系 + Linux系ハイブリッド構成）における、内製プラットフォームからAUTOSAR準拠・外部委託モデルへの移行計画。

---

## 1. 全体フェーズ概要

| フェーズ | 期間 | テーマ |
|----------|------|--------|
| Phase 1 | 今〜2年 | 土台整備：証跡・プロセス・PoC |
| Phase 2 | 2〜5年 | 移行完了：本格稼働・コネクテッド化 |
| Phase 3 | 5〜10年 | SDV化：半自律・次世代アーキテクチャ |

---

## 2. フェーズ別 詳細と技術スタック

### Phase 1：土台整備（今〜2年）

**やること**

| タスク | 時期（目安） | 担当 | 使用ツール・手段 | 完了の判断基準 |
|--------|-------------|------|----------------|---------------|
| ISO 25119に基づくHARAの実施と安全要求の文書化 | 1〜6ヶ月目 | 機能安全マネージャー | HARA分析テンプレート・AIドラフト支援 | HARA文書が内部レビューで承認され、安全目標が確定していること |
| 既存コードの機能安全証跡の整備（後付け記録） | 1〜12ヶ月目 | システムアーキテクト＋担当エンジニア | コードレビュー記録・静的解析ログ整理 | 主要制御モジュールに「変更履歴・レビュー記録・テスト根拠」が残っていること |
| Classic AUTOSAR PoCをRTOS系（油圧制御）の一部で実施 | 3〜12ヶ月目 | アーキテクト＋委託先 | VectorまたはDENSOTen系BSW評価版 | 油圧バルブ制御の1機能がAUTOSAR BSW上で動作し、既存仕様を満たすこと |
| CI/CDパイプライン構築（ビルド・静的解析・単体テスト自動化） | 2〜9ヶ月目 | インフラ担当 | Jenkins or GitLab CI + MISRA解析ツール | マージ時に自動でビルド・MISRA解析が走り、結果がSlack通知されること |
| 外部委託先の選定・契約（要求仕様・受け入れ基準の定義） | 1〜6ヶ月目 | システムアーキテクト＋調達 | RFI/RFP・技術評価 | ASPICE CL2以上・ISO 25119対応の委託先と基本合意書が締結されていること |

**判断ポイント**：Phase 1 から Phase 2 への移行判断は「PoCで実現した機能が量産要件を満たせるか」の技術評価と、「委託先が要求仕様通りに成果物を納入できているか」のプロセス評価を両方クリアすることを条件とする。

**技術選定ポイント**

| 領域 | 採用技術 | 選定理由 |
|------|----------|---------|
| RTOS系BSW | Classic AUTOSAR（VectorまたはDENSOTen系ベンダー） | 建設機械・農業機械での採用実績があるベンダーを優先 |
| Linux系MW | Yocto + 独自ミドルウェア（この段階では標準化範囲外） | Phase 2でAdaptive AUTOSAR移行を見据えた構成にしておく |
| 静的解析 | MISRA-C対応ツール（PC-lint Plus / Polyspace） | 認証審査で実績のあるツールを選ぶ |
| CI/CD | Jenkins or GitLab CI + SIL環境 | 既存インフラとの統合コストを考慮 |
| クラウド連携 | LinuxゲートウェイからAWS IoT Core / Azure IoT Hub（PoC） | この段階ではPoC範囲に留め、Phase 2で本格化 |

### Phase 2：移行完了（2〜5年）

**やること**

| タスク | 時期（目安） | 担当 | 使用ツール・手段 | 完了の判断基準 |
|--------|-------------|------|----------------|---------------|
| Classic AUTOSAR本格稼働（油圧制御系の主要ECU） | 25〜36ヶ月目 | 委託先＋受け入れテスト担当 | HILテスト環境 | 量産機体での累計稼働が所定時間を超え、重大バグゼロが確認されること |
| OTA基盤の構築（Linuxゲートウェイ経由でFirmware OTA） | 25〜42ヶ月目 | クラウド/OTA担当 | Mender or SWUpdate | フィールド機体50台以上への更新が自動ロールバックなしで完了できること |
| フリート管理システムの本番稼働（稼働時間・エラーログ収集） | 30〜48ヶ月目 | クラウド/OTA担当 | AWS IoT Fleet Management | 全稼働機体のリアルタイム状態をダッシュボードで確認できること |
| Linux系のセキュリティ強化（SELinux / SecureBoot / 証明書管理） | 24〜42ヶ月目 | クラウド/OTA担当＋外部セキュリティ専門家 | HSM・SELinuxポリシー設計ツール | ペネトレーションテストで重大脆弱性がゼロであること |
| サプライヤへのASIL要求を要件書に明文化 | 24〜30ヶ月目 | 機能安全マネージャー＋調達 | 標準ASIL要求テンプレート | 主要ECUサプライヤ全社との契約書にASIL要件が明記されていること |

**判断ポイント**：Phase 2 から Phase 3 への移行は「OTA・フリートデータの収集基盤が安定稼働しているか」を前提条件とする。フィールドフィードバックループが機能していないまま自律機能開発を始めると、現場課題が設計に反映されないリスクがある。

**技術選定ポイント**

| 領域 | 採用技術 | 選定理由 |
|------|----------|---------|
| OTA | Mender or SWUpdate（Linux系） / AUTOSAR UCM（RTOS系将来） | オフライン環境・低帯域環境でも動作するものを優先 |
| フリート管理 | AWS IoT Fleet Management / Azure IoT Hub Device Twins | 既存クラウド契約との統合コストで選択 |
| セキュリティ | HSM活用・SecOC（AUTOSAR）・TLS 1.3（Linux） | 建設機械は屋外作業で長期稼働。証明書の有効期限管理も設計に含める |
| テスト | HILテスト自動化・カバレッジ計測 | Phase 1で構築したCI/CDと連携させる |

### Phase 3：SDV化（5〜10年）

**やること**

| タスク | 時期（目安） | 担当 | 使用ツール・手段 | 完了の判断基準 |
|--------|-------------|------|----------------|---------------|
| 人検知・周辺認識の精度向上と半自律機能の段階的搭載 | 60〜84ヶ月目 | AI/センサ担当＋機能安全マネージャー | LiDAR・カメラ統合・ADAS評価環境 | 施工現場での人検知精度（再現率）が社内安全基準を満たすこと |
| Hypervisor統合の検討・PoC（次世代機プラットフォーム） | 72〜96ヶ月目 | システムアーキテクト | Type-1 Hypervisor（QNX Hypervisor等） | RTOS系とLinux系の分離実行を1つのSoCで実現し、レイテンシ要件を満たすこと |
| Adaptive AUTOSARの適用検討（高性能SoC搭載ECU向け） | 72〜108ヶ月目 | アーキテクト＋委託先 | Adaptive AUTOSAR準拠スタック評価 | POCとして1機能のサービス指向通信（SOME/IP）が動作すること |
| OTA・フリートデータを活用したフィールドフィードバックループの確立 | 60〜84ヶ月目 | クラウド/OTA担当＋データエンジニア | フリートデータ分析基盤・MLパイプライン | フィールド異常データから問題箇所を特定し、次回OTAに反映するサイクルが回っていること |

**判断ポイント**：Phase 3 は技術の成熟度に大きく依存する。Hypervisor統合・Adaptive AUTOSARは2026年時点でも建設機械向けの実績が少ない。PoCは次世代機プラットフォームで先行実施し、現行機への適用は成功確認後とする。

## 3. 必要な人材・スキルセット

### 自社に置くべき人材（要求定義・受け入れ検証）

| 役割 | 人数目安 | 求めるスキル | 採用・育成の方法 |
|------|----------|-------------|----------------|
| 機能安全マネージャー | 1〜2名 | ISO 25119 / IEC 61508の理解、HARA・FMEAの実施経験 | 自動車・農機業界からの中途採用を優先。社内育成の場合は外部トレーニング（TÜV・Exida等）＋OJTで最低2年 |
| システムアーキテクト | 1〜2名 | RTOS/Linux両系統の俯瞰的設計力、外部委託管理 | 自動車OEM・ティア1経験者の中途採用。または社内のシニアエンジニアにAUTOSARトレーニング（Vector等主催）を受けさせる |
| 受け入れテスト担当 | 2〜3名 | テスト仕様策定・HIL/SIL環境の操作・合否判定 | 社内ハードウェアエンジニアの転換育成が可能。HIL環境ベンダー（dSPACE等）の研修を活用 |
| クラウド/OTA担当 | 1〜2名 | AWS / Azure IoT、デバイス管理、セキュリティ基礎 | クラウドサービス経験者を中途採用。AWS・Azureの認定資格取得を奨励 |

**採用・育成の補足方針**：

- **中途採用の優先領域**：機能安全マネージャーとシステムアーキテクトは社内育成に2〜3年かかる。Phase 1 開始に合わせて即戦力を採用することが最も効果的。求人媒体では自動車・農業機械・産業機械分野のAUTOSAR経験者を対象にする。
- **社内育成の領域**：受け入れテスト担当・クラウド/OTA担当は、既存の組み込みエンジニアや制御エンジニアをトレーニングで転換育成できる。外部認定研修（1〜2週間）＋社内OJTの組み合わせが現実的。
- **大学・研究機関との連携**：AUTOSAR・機能安全を研究テーマとする研究室との共同研究・インターンシップ受け入れは、中長期の採用パイプラインになりうる。

### 外部委託先に求めるスキル

- Classic AUTOSAR BSW構築・MCAL開発実績（建設機械・農業機械での実績があればなお良い）
- MISRA-C準拠・ASPICE CL2以上のプロセス保有
- ISO 26262（自動車）または ISO 25119（農業機械）対応経験
- HILテスト環境の構築・運用能力

### 将来的に内製化を検討する領域

- アプリケーション層の制御ロジック（機械知識との融合が必要）
- フリートデータ分析・機械学習パイプライン
- OTAキャンペーン管理・ロールアウト制御

## 4. マインドセットの転換

### 「作る組織」から「定義し検証する組織」へ

これまでの内製開発では「動けばよい」が基準だった。外部委託・標準準拠への移行後は、**何を作らせるかを定義する力**と**できあがりを正しく検証する力**が自社の核心スキルになる。コードを書く量は減っても、要求仕様の質と受け入れ基準の厳密さが品質を決める。

この転換を怠ると、委託先は「言われたものを作るだけ」となり、本来自社が持つべき機械知識・現場知識が要求仕様に反映されない。定義する側の力が弱いと、外部委託は単なるコスト要因に終わる。

### 技術負債と向き合う姿勢

「仕様書なし・テストなし・証跡なし」の現状は負債だが、一括解消は不可能。Phase 1では**新規開発分から正しいプロセスを適用**しつつ、既存コードは影響範囲を限定しながら段階的に置き換える。完璧を求めて動きを止めるより、記録を残しながら前進することを優先する。

### 外部標準に乗ることの意味

AUTOSARやISO 25119は制約ではなく**共通言語**である。標準に準拠することで、委託先・顧客・認証機関との対話コストが下がり、人材採用・サプライヤ交渉でも有利に働く。自社固有の「独自仕様」を減らすほど、長期的な維持コストは下がる。

### ケーススタディ：他社はどうマインドセットを転換したか

**農業機械メーカー A社（欧州）**：完全内製エンジン制御システムを持っていたが、2010年代後半にAUTOSARへの移行を決断。初期は「自社の機械はどこも同じにできない」という現場抵抗が強かった。転機になったのは、委託先から「AUTOSARに乗らないと長期サポートを続けられない」という通告を受けたこと。外部からの圧力を契機に、アーキテクト2名とセーフティマネージャー1名の採用を実行し、「自社はドメイン知識を持つ、実装はサプライヤが担う」という役割分担を社内に徹底した。移行完了まで4年かかったが、その後の機能追加・変更コストは大幅に下がったと報告されている。

**産業機械メーカー B社（国内）**：「仕様書がなくても現場で調整すれば動く」文化が長年続いていた。AUTOSAR移行を機に、要求仕様レビューを月次の経営会議アジェンダに組み込んだ。管理職が仕様の曖昧さを指摘するレビューを繰り返すことで、「書かれていないことは実装しなくていい」という意識がエンジニアに浸透。最初の1年は工数増と感じられたが、手戻り件数が2年目から明確に減少した。

**重機メーカー C社（北米）**：SDV化を見据えてソフトウェア部門を分社化し、ハードウェア部門から切り離した組織構造の転換を行った。分社後は採用・評価制度・ツール選定をソフトウェア基準で設計でき、「ハードウェア開発の延長としてのソフトウェア」から脱却。ただし分社化は社内調整コストを要するため、本ロードマップでは分社化を前提とせず、まず役割・スキルの明確化から始めることを推奨する。

**教訓の共通点**：いずれのケースでも、マインドセット転換の起点は「外部からの圧力か危機感の共有」だった。自発的な変化を待つより、フェーズ移行の判断基準を経営レベルの共通言語にすることが、組織の転換を加速させる。
