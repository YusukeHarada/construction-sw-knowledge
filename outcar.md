# Out-car技術概念 — 建設機械SDVアーキテクチャにおける車外通信システム

---

## 1. Out-carとは何か

**Out-car**とは、車両・機械の**外部**との通信・連携システムの総称である。対義語は**In-car**（コックピット・車内ネットワーク・乗員UI）。

| 区分 | 対象範囲 |
|---|---|
| Out-car | クラウド通信、OTA配信、フリート管理、V2X、遠隔操作、現場インフラ連携 |
| In-car | コックピット、HMI、車内ネットワーク（CAN/LIN/Ethernet）、HVAC、ECU間通信 |

建設機械におけるOut-car構成：

```
                    [クラウド（AWS / Azure）]
                      ↑ MQTT / HTTPS / gRPC
            [Linux ゲートウェイECU]
           /          |           \
      [LTE/5G]    [V2X通信]    [現場インフラ]
      クラウド      機械間・        BIM/CIM・
      テレマティクス 現場インフラ    測量データ
```

## 2. V2X概念

**V2X（Vehicle to Everything）** は機械と外部エンティティとの通信の総称。建設機械では「Machine to X（M2X）」と呼ぶ場合もある。

### V2Xカテゴリ

| カテゴリ | 意味 | 建設機械での活用例 |
|---|---|---|
| V2V | 機械間通信 | 複数機協調作業・衝突回避 |
| V2I | インフラ通信 | 現場信号機・ゲート・センサ連携 |
| V2N | クラウド通信 | フリート管理・テレマティクス |
| V2P | 歩行者（作業員）検知 | 人検知・接近警告・自動停止 |

### 通信規格比較

| 規格 | 方式 | 特徴 | 建設機械での位置づけ |
|---|---|---|---|
| DSRC（ITS-G5 / WAVE） | 専用無線（5.9 GHz帯） | 低遅延・インフラ要構築 | 閉鎖現場向け（要確定） |
| C-V2X（LTE-V2X） | セルラー | 既存LTEインフラ活用 | 現実的な第一選択 |
| C-V2X（NR-V2X / 5G） | 5G | 超低遅延・高信頼 | 2028年以降の遠隔操作向け |

## 3. クラウド連携

### 接続構成

```
[建設機械]
  └─ Classic AUTOSAR ECU（制御系）
      └─ CAN FD
          └─ [Linux ゲートウェイECU]
              └─ LTE / 5G モデム
                  └─ TLS 1.3
                      ├─ AWS IoT Core（要確定）
                      └─ Azure IoT Hub（要確定）
```

### プロトコル選択

| プロトコル | 特徴 | 用途 |
|---|---|---|
| MQTT | 軽量・Pub/Sub・低帯域 | テレマティクス定期送信 |
| HTTPS | 汎用・REST API | ファームウェア取得・設定変更 |
| gRPC | 低遅延・バイナリ | 遠隔操作コマンド（要確定） |

### 収集データ例

- 稼働時間・エンジン回転数・燃料消費量
- 油圧・温度センサ値（異常予兆検知用）
- 故障コード（J1939 DM1 / UDS DTC）
- GPS位置情報・作業量（掘削量・積載量）

### クラウドデータパイプライン

```
IoT Core / IoT Hub
  → データレイク（S3 / Azure Data Lake・要確定）
      → ストリーム処理（Kinesis / Event Hub・要確定）
          → BI分析（QuickSight / Power BI・要確定）
          → AI/ML異常検知（SageMaker / Azure ML・要確定）
              → 保全アラート通知
```

## 4. フリート管理

### 複数機統合管理システム

```
[フリート管理ダッシュボード]
  ├─ リアルタイム状態可視化
  │    ├─ 位置情報（GIS地図上にプロット）
  │    ├─ 稼働状態（作業中・待機・停止・故障）
  │    ├─ 燃料残量・稼働時間
  │    └─ アクティブDTC一覧
  ├─ 予兆保全
  │    ├─ センサ時系列データ→異常検知モデル
  │    └─ 保全アラート→整備部門通知
  └─ 作業実績レポート
       └─ 日次・月次の稼働率・生産性集計
```

### 業界先行事例（概要レベル）

- コマツ：KOMTRAX（稼働管理）→ Smart Construction（施工DX）として発展
- キャタピラー：Cat Connect / VisionLink によるフリート遠隔管理
- これらを参考に自社要件を整理し、クラウドプラットフォームと管理システムを選定（要確定）

## 5. 遠隔操作・遠隔監視

### ユースケース

| シナリオ | 内容 |
|---|---|
| 危険現場作業 | 崩落リスク・汚染土壌現場での遠隔操作 |
| 夜間施工 | オペレータを現場に配置せず遠隔で継続施工 |
| 熟練者不足対応 | 1人の熟練者が複数機械を遠隔で監視・補助操作 |

### 通信要件

| 項目 | 要件 | 備考 |
|---|---|---|
| 往復遅延（RTT） | 100 ms以下（要確定） | 操作感・安全性に直結 |
| 映像品質 | HD 30fps以上（要確定） | カメラ複数台（全周） |
| 冗長性 | LTE＋サブ回線（要確定） | 通信断時の自動停止 |

### セキュリティ

- 相互認証（クライアント証明書）による操作端末認証
- なりすまし防止：コマンドへのタイムスタンプ＋署名
- 操作権限の二重確認（ロック解除→操作許可）

## 6. OTAアップデートのOut-car側

### ファームウェア配信フロー

```
[クラウド OTA管理サーバ]
  └─ 配信対象選定（機種・バージョン・ロールアウト順）
      └─ HTTPS / MQTT でゲートウェイに通知・転送
          └─ [Linux ゲートウェイECU]
              ├─ 署名検証（公開鍵証明書）
              ├─ ローカルストレージに展開
              └─ UDS（0x34/0x36/0x37）で制御ECUに書込
```

### デプロイメント管理

- ターゲット管理：機種・シリアル番号・現行バージョンの一元管理
- 段階ロールアウト：少数機先行→問題なければ全台展開
- ロールバック：書込失敗時の旧バージョン自動復帰

### WP.29 R156対応（概要）

国連規則 UNECE R156はOTAソフトウェア更新管理に関する規制。建設機械への直接適用は現状義務ではないが、将来の規制動向および委託先（自動車業界出身）との共通言語として把握しておく：

- ソフトウェア更新管理システム（SUMS）の整備
- 更新の識別・追跡・記録保持
- サイバーセキュリティ管理と連携（R155）

## 7. 現場インフラ連携

### 測量データ連携

| データ種別 | 形式 | 活用 |
|---|---|---|
| 3D点群 | LAS / E57 | 設計面との差分計算・出来形管理 |
| BIM/CIM | IFC / LandXML | 施工計画・干渉チェック |
| 電子小黒板・施工管理 | 独自フォーマット / クラウドAPI | 出来形記録・官庁提出書類自動生成 |

### ICT施工（マシンコントロール）との連携

```
[測量会社 / 設計事務所]
  └─ 3D設計データ（LandXML等）
      └─ [現場クラウドサーバ / 施工管理システム]
          └─ LTE/Wi-Fi
              └─ [建設機械 Linux GW]
                  └─ 設計面データをHMIへ配信
                      └─ [コックピット HMI]
                           施工ガイダンス表示（残土量・目標深さ）
```

## 8. セキュリティ（Out-car特有のリスク）

Out-car通信は外部ネットワークに直接接続するため**最大の攻撃面（Attack Surface）**となる。

### 主要リスクと対策

| リスク | 対策 |
|---|---|
| 通信傍受・改ざん | TLS 1.3による暗号化・証明書ピンニング |
| なりすまし（OTAサーバ偽装） | OTAパッケージへの署名検証（PKI） |
| 遠隔操作コマンド偽装 | HMAC/署名付きコマンド＋タイムスタンプ検証 |
| APIへの不正アクセス | APIゲートウェイ（WAF）＋OAuth 2.0認証 |
| 証明書管理 | HSM（Hardware Security Module）での秘密鍵保護（要確定） |

### 多層防御の考え方

```
[外部ネットワーク]
  └─ APIゲートウェイ / WAF
      └─ TLS終端 / 証明書検証
          └─ [Linux ゲートウェイECU]
              ├─ ファイアウォール（iptables / nftables）
              ├─ 受信データ検証（スキーマ・値域チェック）
              └─ [車内ネットワーク（信頼ゾーン）]
```

ISO/SAE 21434に基づくTARA（脅威分析・リスクアセスメント）でOut-car通信を最高リスク区分として評価し、対策を体系化することを推奨する。

## 9. Out-car移行ロードマップ

```
現状（〜2025年）
  孤立した機械・手動報告・テレマティクスなし
    ↓
Phase 1（2025〜2027年）
  LTE接続確立・稼働データクラウド収集開始
  Linux GW構築・MQTT/AWS IoT Core連携
  基本フリート管理（位置・稼働状態可視化）
    ↓
Phase 2（2027〜2029年）
  フリート管理高度化（予兆保全・アラート通知）
  OTA配信基盤整備（SUMS準拠・署名検証）
  現場インフラ連携（BIM/CIM・施工管理システム）
    ↓
Phase 3（2029年〜）
  V2X導入（V2P人検知・V2V機械間協調）
  遠隔操作対応（5G・低遅延通信・全周カメラ）
  自律協調施工（複数機自動連携・AI施工計画）
```
