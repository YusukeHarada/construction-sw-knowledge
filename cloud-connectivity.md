# クラウド連携アーキテクチャ

**作成日：** 2026年6月16日  
**対象読者：** 技術担当者・アーキテクト  
**資料種別：** 社内検討用（社外秘）

---

## 概要

建設機械のECU群をクラウドに接続し、フリート管理・予知保全・OTA更新・遠隔監視を実現するためのアーキテクチャを整理する。

---

## 1. 推奨アーキテクチャ：Linuxゲートウェイ経由構成

Classic AUTOSAR ECU（RTOS系）をクラウドに直接接続するのではなく、**Linuxゲートウェイが通信を一元管理する構成**を推奨する。

```
┌─────────────────────────────────────────────────────┐
│ Classic AUTOSAR ECU群                               │
│ （油圧制御・CAN通信）                                │
└────────────┬────────────────────────────────────────┘
             │ CAN / J1939
┌────────────▼────────────────────────────────────────┐
│ Linux Gateway（Cortex-A / x86）                     │
│                                                     │
│  ・AWS Greengrass / Azure IoT Edge                 │
│  ・ローカルMLモデル（予知保全・エッジ推論）          │
│  ・証明書管理 / TLS終端                             │
│  ・OTAエージェント                                  │
│  ・CAN → MQTT 変換                                  │
└────────────┬────────────────────────────────────────┘
             │ MQTT over TLS（4G/5G/衛星）
┌────────────▼────────────────────────────────────────┐
│ クラウドプラットフォーム                              │
│  AWS IoT Core / Azure IoT Hub                      │
│  ・OTA Jobs・フリート管理・デジタルツイン            │
│  ・S3・SageMaker / Azure ML（機械学習）             │
└─────────────────────────────────────────────────────┘
```

### なぜ直接接続ではなくゲートウェイ経由か

| 観点 | RTOS ECU直接接続 | Linuxゲートウェイ経由 |
|---|---|---|
| TLSハンドシェイク | MCUでのコストが大きい（〜10秒/接続） | Linuxで高速処理 |
| 証明書ストレージ | MCUでの確保が困難 | Linux側で一元管理 |
| CAN→Ethernet変換 | 自前実装が必要 | Greengrass等で標準対応 |
| OTA管理 | 複雑 | Greengrassが一元管理 |
| セキュリティ | 機能が限定的 | DMZ構成・ファイアウォールを実装可能 |

## 2. Amazon FreeRTOS（FreeRTOS）とClassic AUTOSARの関係

### 通常のFreeRTOSとの違い

**FreeRTOS（OSS）**：リアルタイムカーネルのみ。MITライセンス。  
**Amazon FreeRTOS（旧a:FreeRTOS、現FreeRTOS統合版）**：FreeRTOS + AWS接続ライブラリ群。

追加されるAWSライブラリ：
- **coreMQTT**：MQTT/TLS実装
- **AWS OTA Library**：OTAエージェント（A/Bパーティション対応）
- **Device Shadow**：デバイス状態のクラウド同期
- **AWS IoT Jobs**：リモートジョブ実行

### FreeRTOSとClassic AUTOSARは共存できるか

**結論：同一ECU上での共存は原則不可。役割分担による併用が現実的。**

Classic AUTOSARはそれ自体がOSAL（OS抽象化層）を含む完結したソフトウェアスタックであり、独自のOSIF（OS Interface）を通じてOS（例：OSEK準拠OS）を抽象化している。FreeRTOSをClassic AUTOSARのOS実装として差し込むことは技術的には可能だが、AUTOSARのOSコンフォーマンス要件（タスクスケジューリング・アラーム等）とFreeRTOSの動作モデルの乖離により、認証取得が困難になる。

現実的な使い分けは以下の通り：

| コンポーネント | OS | 役割 |
|---|---|---|
| Classic AUTOSAR ECU（既存） | OSEK準拠OS（例：ETAS RTA-OS） | 油圧制御・安全機能・J1939通信 |
| Linuxゲートウェイ | Linux | MQTT通信・OTA管理・エッジML |
| 新規追加の低コストMCU | FreeRTOS + Amazon FreeRTOS | センサー収集・補助機能・AWS直接接続 |

**将来的に新規ECUや低コストMCU（STM32・ESP32等）を追加する場合の選択肢**として、Amazon FreeRTOSは有力候補になる。その場合もClassic AUTOSARとは別バスに接続し、ゲートウェイを介してデータを統合する設計を採ること。

## 3. クラウドプラットフォーム比較

### 機能比較

| 観点 | AWS IoT Core | Azure IoT Hub |
|---|---|---|
| OTA | AWS IoT Jobs + OTA Library | Azure Device Update（ADU） |
| デバイス管理 | IoT Device Management | IoT Hub Device Provisioning Service（DPS） |
| エッジ処理 | AWS Greengrass | Azure IoT Edge |
| 機械学習連携 | SageMaker | Azure Machine Learning |
| デジタルツイン | AWS IoT TwinMaker | Azure Digital Twins |
| タイムシリーズDB | Amazon Timestream | Azure Time Series Insights / ADX |
| 建設機械での実績 | コマツ（KOMTRAX・Landlog） | Caterpillar（Cat Connect） |

### 料金感（2026年時点の目安）

料金は変動するため公式ページで確認すること。ここでは規模感の把握を目的として参考値を示す。

**AWS IoT Core**
- メッセージング：$1.00 / 100万メッセージ（最初の10億）
- デバイス接続：$0.042 / デバイス・月（常時接続）
- Rules Engine（データルーティング）：$0.15 / 100万ルール実行
- 試算例：1,000台の建設機械が1分1回テレメトリを送信 → 約43万メッセージ/日 → 月間約$13（メッセージ費のみ）

**Azure IoT Hub**
- 無料プランあり：デバイス数8,000台・日8,000メッセージまで（開発用）
- S1プラン：$10/月・ユニット、1ユニットで400,000メッセージ/日
- 試算例：1,000台が1分1回送信 → 日144万メッセージ → S1×4ユニット = $40/月程度
- OTA（ADU）は IoT Hub の上位レイヤで別途 Device Update アカウントが必要

**コスト最適化のポイント**
- エッジ側で信号フィルタリングを行いクラウドへの送信頻度を下げる（エンジン稼働中のみ送信、停車中はバッファリング等）
- 差分OTAを利用し大容量ファームウェアの転送コストを削減
- Tiered Messagingの適用（AWSは最初の100億メッセージが安い）

### 日本語サポート・日本リージョン

| 項目 | AWS | Azure |
|---|---|---|
| 国内リージョン | 東京・大阪（両方IoT Core対応） | 東日本・西日本（両方IoT Hub対応） |
| 日本語技術ドキュメント | ほぼ全サービスに存在（一部機械翻訳） | ほぼ全サービスに存在（品質高め） |
| 日本語サポートプラン | ビジネス・エンタープライズプランで日本語対応 | Standard以上で日本語対応 |
| 日本のパートナー | APN（AWS Partner Network）に多数 | SI大手（NTT・富士通・NEC等）が対応 |
| 公的認証（ISMAP等） | IoT Core含む主要サービスが対応済み | IoT Hub含む主要サービスが対応済み |

建設機械の国内展開では**東京リージョンを使用し、データ主権の観点から国内完結構成**を検討すること。海外展開では現地リージョン（北米・欧州）を追加する。

### 建設機械業界の実例

**コマツ KOMTRAX（2001年〜）**
- 世界最大規模の建設機械テレマティクス
- GPS＋携帯/衛星通信で稼働状況・燃料・アラートを収集
- AWSと連携し、建設現場OS「Landlog」に発展
- 機械学習によるエンジンオイル劣化予測等の予知保全を実装

**Cat Connect（Caterpillar）**
- Azure IoT Hub経由でデータ集約
- Azure Digital Twinsを用いた現場全体シミュレーション

## 4. OTA更新のアーキテクチャ要件

機能安全との関係でOTAには以下が必須。

### 必須要件

| 要件 | 内容 | 実現方法 |
|---|---|---|
| **二重バンク（A/Bパーティション）** | 更新失敗時にロールバック可能 | AWS OTA Library / SWUpdate |
| **コード署名検証** | 更新イメージにECDSA署名を付与・検証 | AWS IoT Jobs + Code Signing |
| **安全状態遷移** | OTA実行中はエンジン停止・安全停止状態へ移行 | 機能安全設計で規定 |
| **差分OTA** | 帯域制約（3G/衛星）での転送量削減 | BSDiff / AWS IoT Core対応 |

### 安全状態遷移：エンジン稼働中にOTAが来た場合のシーケンス

エンジン稼働中は即座にOTAを適用せず、**安全停止後にのみ書き込みを実行する**。以下にシーケンスを示す。

```
クラウド             Linuxゲートウェイ              Classic AUTOSAR ECU
   │                      │                               │
   │ OTAジョブ通知         │                               │
   │─────────────────────>│                               │
   │                      │ 車両状態確認要求               │
   │                      │──────────────────────────────>│
   │                      │                               │
   │                      │ 応答：エンジン稼働中           │
   │                      │<──────────────────────────────│
   │                      │                               │
   │                      │ ダウンロード開始               │
   │ ファームウェア転送     │（ダウンロードのみ・書込保留）  │
   │─────────────────────>│                               │
   │                      │                               │
   │                      │ ダウンロード完了               │
   │                      │ 署名検証OK                    │
   │ ジョブ進捗更新        │                               │
   │<─────────────────────│                               │
   │                      │                               │
   │                      │  ─ 待機 ─（エンジン停止まで）  │
   │                      │                               │
   │                      │ エンジン停止通知               │
   │                      │<──────────────────────────────│
   │                      │                               │
   │                      │ OTA適用可能確認               │
   │                      │──────────────────────────────>│
   │                      │ 確認応答OK                    │
   │                      │<──────────────────────────────│
   │                      │                               │
   │                      │ 書込開始指示（UDS 0x34〜0x37）│
   │                      │──────────────────────────────>│
   │                      │ 書込完了・CRC確認OK            │
   │                      │<──────────────────────────────│
   │                      │                               │
   │                      │ ECUリセット指示               │
   │                      │──────────────────────────────>│
   │                      │                               │
   │                      │ 再起動・新バージョン起動確認    │
   │                      │<──────────────────────────────│
   │ ジョブ完了報告        │                               │
   │<─────────────────────│                               │
```

**重要な状態遷移ルール：**
1. エンジン稼働中（キースイッチON・作業中）はダウンロードのみ実施。書込禁止
2. ダウンロード完了後はゲートウェイ内バッファに保持し、エンジン停止イベントを待つ
3. エンジン停止後も、一定時間（例：30秒）経過後に車両静止を確認してから書込開始
4. 書込中に電源断が発生した場合はBバンクに切り替えてロールバック
5. ロールバック3回失敗した場合はサービスモードに移行し、整備士による手動対処を要求

**重要：** エンジン稼働中のOTA適用は禁止。「いつOTAを適用するか」のポリシーを機能安全要件として定義すること。

## 5. セキュリティ（ISO/SAE 21434対応）

### デバイス認証

- **X.509クライアント証明書によるmTLS認証**が標準
- 製造時にHSM（Hardware Security Module）またはSecure Element（例：ATECC608A）へ秘密鍵を焼き込む
- AWS Fleet Provisioningによるスケーラブルな証明書配布が可能

### 通信暗号化

- TLS 1.2以上（TLS 1.3推奨）を使用
- MCUリソース制約がある場合：ゲートウェイ内でTLS-PSKを終端し、外向きのみ証明書ベースにする構成も有効

### 証明書ライフサイクル管理

建設機械は**10年超の運用寿命**があるため、証明書の自動更新（Rotation）の仕組みが必須。AWS IoT Device Defenderで異常通信パターン（予期しないIP・過剰メッセージ）を検知する。

### TARAでの位置付け

ISO/SAE 21434が求めるTARA（Threat Analysis and Risk Assessment）では、クラウド接続は「外部インターフェース」として高リスク評価される。

```
必須対策：
  ・ゲートウェイのDMZ構成（インターネット側 ↔ CAN側の分離）
  ・ファイアウォールルールの明示化
  ・不審な通信パターンの監視・アラート
  ・ECUへの直接書き込み禁止（ゲートウェイからCAN制御系への一方向制限）
```

## 6. エッジコンピューティング（ゲートウェイ上のローカル処理）

クラウドに全データを送るのではなく、ゲートウェイ上でエッジ処理することで帯域・レイテンシを削減できる。

| 処理 | エッジ処理 | クラウド処理 |
|---|---|---|
| 異常検知（即時） | ゲートウェイ上のローカルMLモデル | 精度の高い後処理 |
| CAN信号フィルタリング | 必要なデータのみ送信 | 集積データの分析 |
| 人検知アラート | リアルタイム処理（低レイテンシ必須） | ログ蓄積・統計 |
| 予知保全 | 簡易モデルでローカル判定 | 精密モデルで詳細分析 |

**AWS Greengrass**や**Azure IoT Edge**がこのエッジ処理を担うコンポーネントとして機能する。

| 4G/LTE | 中 | 中 | 市街地・郊外の通常施工 | 最も一般的 |
