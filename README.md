# autosar-plan

建設機械向けプラットフォームソフトウェアのAUTOSAR準拠・外部委託移行に関する検討資料。

---

## 背景

建設機械の制御系・通信系ソフトウェアを内製で開発してきたが、以下の課題がある。

- 機能安全（ISO 25119）・セキュリティ規格への対応が必要だが、現状の内製体制では対応不可能
- 仕様書・設計書・テストが存在しない状態で4〜5年の開発が蓄積されている
- AUTOSARを自社で習得・導入するリソースがない

これらを踏まえ、**AUTOSAR準拠のプラットフォームを外部委託で構築する**方針への移行を検討している。

## ドキュメント一覧

### 1. [社内提案資料](proposal.md)
上司・幹部への提案用。「なぜ現状維持では駄目か」「なぜAUTOSAR外部委託か」を経営リスクの観点から説明する。

### 2. [委託先選定基準](vendor-selection.md)
AUTOSAR対応ベンダーを選定する際の必須要件・評価基準・RFI項目・契約条件を定義する。

### 3. [要求仕様の作り方](requirements-spec.md)
外部委託先に渡すための要求仕様の構成・書き方・フォーマットを定義する。AUTOSARが決めることと自社が決めることの切り分けを明確にする。

### 4. [通信仕様の書き方](communication-spec.md)
CAN・EthernetのメッセージリストをAUTOSARツールチェーンへのインプットとして定義する方法。DBCファイル形式・SOME/IPサービス定義・J1939固有事項を含む。

### 5. [ツールチェーン](toolchain.md)
通信仕様・要求仕様からAUTOSARのBSWコンフィグ・コード生成までのツールチェーン全体像。Vector・EB tresos・ETASの比較と、自社担当者が把握すべき最低限の知識を整理する。

## 移行方針の概要

```
現状                          移行後
─────────────────────────────────────────────
内製BSW・RTE（仕様書なし）  →  Classic AUTOSAR BSW・RTE（外部委託）
内製Linuxミドルウェア        →  Adaptive AUTOSAR / Linuxミドルウェア（外部委託）
独自通信定義                 →  DBC / ArXML による標準化された通信定義
自社がコードを書く           →  自社は要求定義・受け入れ検証に専念
```

## アクションプラン

1. **PoC（3〜6ヶ月）** — 一機能を限定してAUTOSAR準拠ベンダーに委託し実現性を検証
2. **移行計画策定** — PoCの結果を踏まえてベンダー選定・移行優先順位を決定
3. **段階的移行** — 安全クリティカルな制御系（RTOS系）から順次AUTOSAR準拠へ移行
