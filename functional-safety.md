## 1. 機能安全の概要

### 1.1 規格の全体像

機能安全（Functional Safety）とは、「システムが期待どおりに動かないとき（故障・誤作動）に人が傷つかないよう、ソフトウェアや電気システムで安全を担保する」ための設計・開発・検証の体系的なアプローチである。建設機械ソフトウェアに関係する主要規格を以下に整理する。

| 規格 | 適用分野 | 安全水準単位 | 特徴 |
|------|---------|------------|------|
| **ISO 26262** | 乗用車・トラック | ASIL A〜D | 自動車業界の事実上標準。要求事項が詳細で実績豊富 |
| **ISO 25119** | 農業・建設機械 | AgPL a〜e | ISO 26262をベースに農機・建機向けに適応。ガイダンス的な部分も多い |
| **ISO 13849** | 機械（全般） | PLa〜PLe | 機械安全の観点から制御システムを評価。電気回路・油圧回路も対象 |
| **IEC 62061** | 機械（全般） | SIL 1〜3 | 電気・電子系に特化。PLと並行適用される場合もある |

### 1.2 ISO 25119とISO 26262の違い

ISO 26262は乗用車を前提に策定されたため、そのまま建設機械へ適用すると過剰要求になる箇所がある。ISO 25119はその差異を吸収するために設計されている。

**主な相違点：**

| 観点 | ISO 26262 | ISO 25119 |
|------|-----------|-----------|
| 適用機械 | 量産乗用車・商用車 | 農業機械・建設機械（低量産品を含む） |
| 安全水準 | ASIL A〜D（4段階） | AgPL a〜e（5段階、ISO 13849のPLに整合） |
| ソフトウェアプロセス | 極めて詳細（Part 6） | ISO 26262 Part 6を参照しつつ緩和可能な項目あり |
| ハードウェア要求 | FMEDA・SPF/LFM等の定量指標 | 同様だが一部簡略化された評価方法も認める |
| 量産規模 | 数万〜数百万台 | 数台〜数千台（低量産品への配慮あり） |
| 文書量 | 非常に多い | ISO 26262より少ないが増加傾向にある |

**実務上の判断：** 外部委託先がISO 26262ベースの開発プロセスを持つ場合、ISO 25119への適用マッピングを委託先に確認する。ISO 26262準拠の証跡の多くはISO 25119でも有効である（要確定）。

### 1.3 ISO 13849との関係と使い分け

ISO 13849は「機械安全」の規格であり、電気・電子・油圧を含む制御システム全体を対象とする。一方、ISO 25119はソフトウェアとE/E（電気電子）システムの機能安全に重点を置く。

建設機械では以下のように使い分けることが多い。

```
システム全体の安全設計
├── 機械・油圧回路の安全 → ISO 13849（PLe等）
│   └── 緊急停止スイッチ、インターロック等
└── E/Eシステムの安全 → ISO 25119 / ISO 26262
    ├── ECUソフトウェア（Classic AUTOSAR領域）
    └── Linuxゲートウェイ（人検知・HMI）
```

ISO 13849で要求されるPLは、そのままISO 25119のAgPLに対応させることができる（PLe = AgPL e）。HARAの結果がAgPL dを要求する場合、電気系はISO 25119でAgPL dを達成し、機械系はISO 13849でPL dを達成することで、システム全体としての安全水準を満たす構成が一般的である。

### 1.4 「形式的対応」と「実効対応」の違い

**形式的対応（Compliance by Documentation）** とは、規格が要求する文書や証跡を揃えることで対応済みとする状態を指す。書類審査は通過できるが、実際のシステムに安全が組み込まれているかは別問題である。

**実効対応（Real Functional Safety）** とは、設計・実装・検証の各段階で安全要求が正しく反映・追跡・検証されている状態を指す。

| 観点 | 形式的対応 | 実効対応 |
|------|-----------|---------|
| HARA | 「エラーなし」の前提でリスクを低評価 | 最悪ケースを想定した保守的な評価 |
| 安全要求 | 曖昧な文章で記載（「適切に制御する」等） | 測定可能・検証可能な要求として記載 |
| トレーサビリティ | ドキュメント間のリンクが形式的に存在 | 要求〜設計〜実装〜テストが実際に追跡可能 |
| テスト | 証跡として存在するが合否基準が曖昧 | 安全要求に基づく明確な合否基準 |
| 変更管理 | 変更時にHARAや安全要求を更新しない | 変更影響分析を必ず実施し証跡を更新 |

> **注意：** 認証機関（TÜV等）の審査官は形式的な書類だけでなく、設計とドキュメントの整合性・開発者への口頭確認も行う。形式的対応だけでは審査を通過できないケースが増えている。

### 1.5 建設機械に求められる安全水準の具体例

| 機能 | 典型的なAgPL目標 | 根拠 |
|------|----------------|------|
| 油圧アームの動作制御 | AgPL d | 誤作動時に作業員が死亡する可能性あり。高頻度操作 |
| 走行駆動系の制御 | AgPL d | 暴走・急発進による重大事故リスク |
| 緊急停止（全軸） | AgPL e | 最後の砦。最高水準を要求 |
| 旋回ロック制御 | AgPL c〜d | 不意旋回による人身事故リスク |
| 人検知システム（カメラ） | AgPL b〜c | 補助機能として位置づけるか、主要安全機能とするかで変わる（要HARA） |
| エンジン回転数制御 | AgPL c | 暴走・過回転リスクあり |
| HMI表示（油圧状態） | AgPL a〜b | 情報提供のみで直接制御しない場合 |

※ 上記は参考値。実際の値はHARAで決定すること（要確定）。

## 2. HARA（ハザード分析とリスク評価）の進め方

### 2.1 HARAとは

HARA（Hazard Analysis and Risk Assessment：ハザード分析とリスク評価）は、「どんな故障がどんな状況で起きると、誰がどれだけ危険にさらされるか」を体系的に洗い出し、それぞれのリスクを数値化するプロセスである。

HARAのアウトプットは**安全目標（Safety Goal）** と**AgPL（達成すべき安全水準）**であり、後続の全開発作業（設計・実装・テスト）の大本になる。HARAが不正確だと、以降の作業がすべて的外れになる。

```
HARA実施
  ↓
安全目標（例：「エンジン停止後にアームが動作してはならない」）
  ↓
AgPL（例：AgPL d）
  ↓
機能安全要求（FSRS）
  ↓
技術安全要求（TSRS）→ ソフトウェア安全要求（SSRS）
  ↓
設計・実装・テスト
```

### 2.2 ハザード分析の手順（ステップバイステップ）

**Step 1：アイテム定義（境界の確定）**

分析対象システム（アイテム）の境界を明確にする。境界が曖昧だとハザードの抜け漏れが生じる。

```
例：油圧制御ECU
  境界内：ECUソフトウェア、マイコン、CAN通信I/F、センサー入力処理
  境界外：油圧バルブ本体（機械系）、オペレーターの操作意図
  インターフェース：操作レバー入力、圧力センサー入力、CANバス
```

**Step 2：使用状況（Operational Situation）の列挙**

システムがどのような状況で使われるかを洗い出す。同じ故障でも状況によってリスクが大きく変わる。

```
例：油圧アーム制御の使用状況
  OS-01：掘削中（作業員が近傍にいる可能性あり）
  OS-02：走行中（周囲に歩行者がいる可能性あり）
  OS-03：メンテナンス中（整備士が機械直下に入る場合あり）
  OS-04：アイドリング停車中
  OS-05：輸送中（機械搭載状態）
```

**Step 3：ハザードの同定**

各使用状況に対して、アイテムの誤作動（故障・誤動作）が引き起こしうるハザードを列挙する。「〜が故障したとき、〜が起きる」という形式で記載する。

```
例：
  H-01：アーム降下指令の誤出力 → 意図しないアーム降下 → 作業員を押しつぶす
  H-02：走行停止指令の不達 → 制動不能 → 障害物・人への衝突
  H-03：エンジン停止後のアーム保持失敗 → 自重降下 → 周囲の人・物への衝突
  H-04：操作量の誤算出（過大値） → 急激な動作 → 転倒・周囲への危害
```

**Step 4：リスクパラメータの評価**

各ハザードについて、S・E・Cの3つのパラメータを評価する。

**S（Severity：深刻度）** ─ 最悪ケースで起きうる被害の大きさ

| レベル | 定義 | 例 |
|--------|------|----|
| S0 | 傷害なし | 財産のみへの軽微な影響 |
| S1 | 軽傷 | 処置不要〜軽い医療処置で回復 |
| S2 | 重傷 | 回復可能な重大傷害 |
| S3 | 生命の危機・死亡 | 回復不能な重大傷害、死亡 |

**E（Exposure：暴露頻度）** ─ そのハザードが発生しうる状況に置かれる頻度

| レベル | 定義 | 例 |
|--------|------|----|
| E1 | ごくまれ | 年に1回未満 |
| E2 | まれ | 月1回程度 |
| E3 | 時々 | 週1回程度 |
| E4 | 頻繁 | ほぼ毎日 |
| E5 | 非常に頻繁 | 常時・作業中ほぼ常に |

**C（Controllability：制御可能性）** ─ ハザードが発生したとき、人が回避できる可能性

| レベル | 定義 | 例 |
|--------|------|----|
| C0 | 通常は制御可能 | ほぼ誰でも回避可能 |
| C1 | 容易に制御可能 | 多くの人が回避可能 |
| C2 | 通常は制御可能 | 50%程度が回避可能 |
| C3 | 制御困難 | 回避できる人はまれ |
| C4 | 制御不能 | ほぼ回避不可能 |

**Step 5：AgPLの決定**

S・E・Cの組み合わせからAgPLを決定する。ISO 25119 Part 1のリスクグラフまたは以下の対応表を参照する。

```
AgPL決定の簡易ガイド（要ISO 25119 Part 1で確認）：
  S3 × E4/E5 × C3 → AgPL e
  S3 × E3/E4 × C2/C3 → AgPL d
  S3 × E2 × C1/C2 → AgPL c
  S2 × E3/E4 × C2/C3 → AgPL c
  S2 × E1/E2 × C1 → AgPL b
  S1以下（最悪でも軽傷） → AgPL a または 安全要求なし
```

> **注意：** 上記は概念的な対応であり、実際の決定にはISO 25119 Part 1のリスクグラフを使用すること（要確定）。

### 2.3 建設機械でのHARA事例

#### 事例1：油圧アーム降下制御

| 項目 | 内容 |
|------|------|
| ハザード | アーム降下指令の誤出力により、意図しないアーム降下が発生する |
| 使用状況 | 掘削作業中（オペレーターの視野外に補助作業員がいる） |
| S | S3（作業員を直撃すれば死亡） |
| E | E4（掘削中は頻繁に操作する） |
| C | C3（アーム降下速度が速ければ回避不能） |
| → AgPL | AgPL d |
| 安全目標 | 「エンジン稼働中、アームは操作レバーの指令がない限り降下を開始してはならない」 |

#### 事例2：走行駆動制御

| 項目 | 内容 |
|------|------|
| ハザード | 走行停止指令が不達となり、制動不能状態が発生する |
| 使用状況 | 傾斜地での走行中（下方に作業員がいる） |
| S | S3（重機が作業員に激突すれば死亡） |
| E | E3（傾斜地での走行は日常的） |
| C | C3（重機の慣性が大きく回避困難） |
| → AgPL | AgPL d |
| 安全目標 | 「走行停止要求から500ms以内に駆動力をゼロにしなければならない（傾斜地含む）」 |

#### 事例3：人検知システム（カメラ）

| 項目 | 内容 |
|------|------|
| ハザード | 人検知の誤検知・未検知により、人への衝突を回避できない |
| 使用状況 | 旋回動作中（後方に作業員がいる） |
| S | S3（旋回ブームが作業員に当たれば死亡） |
| E | E3〜E4（旋回は頻繁な操作） |
| C | C2〜C3（オペレーターが後方を確認しながら操作できる場合はC2） |
| → AgPL | AgPL c〜d（人検知をどの安全アーキテクチャに位置づけるかで変わる） |
| 安全目標 | 「人検知システムの警告から2秒以内にオペレーターが認識可能な警報を出力しなければならない」 |

> **補足：** 人検知システムをHMI補助機能（AgPL b）として位置づけるか、旋回制御の安全機能（AgPL c〜d）として組み込むかによって、求められる開発負荷が大きく異なる。この判断はHARAの段階で確定させること。

### 2.4 HARAのアウトプットテンプレート

**安全目標記載テンプレート：**

```
[安全目標ID]: SG-001
[対応ハザード]: H-01（意図しないアーム降下）
[AgPL]: AgPL d
[安全目標]: エンジン稼働中、操作レバーの降下指令がない場合に、
            アームは自重降下を含む下方向の動作を開始してはならない。
[FTTI（故障許容時間間隔）]: 200ms（要確定）
[Safe State（安全状態）]: アーム保持（現在位置で停止）
[除外事項]: 意図的なメンテナンスモード（専用キー操作で有効化）

[機能安全要求への展開]
  FSRS-001: アーム降下駆動信号は、操作レバー入力が検出された場合のみ
            ECUから出力すること
  FSRS-002: センサー異常検出から200ms以内に安全状態へ遷移すること
  FSRS-003: アーム位置センサーの二重化により単一故障で安全機能を喪失
            しないこと
```

## 3. Safety Plan（機能安全計画）の作成

### 3.1 Safety Planに記載すべき項目

Safety Planは「機能安全をどのようにして達成するか」のマスタープランである。プロジェクト開始時に作成し、フェーズごとに更新する。

| セクション | 記載内容 |
|-----------|---------|
| 1. スコープ | 対象アイテム・適用規格（ISO 25119）・期間 |
| 2. 組織・役割 | 機能安全マネージャー・開発チーム・品質・委託先の役割と責任 |
| 3. 機能安全ライフサイクル | 各フェーズ（概念〜廃棄）の計画、担当者、スケジュール |
| 4. 証跡・成果物一覧 | 各フェーズで作成・受領する証跡の一覧と管理場所 |
| 5. ツール管理計画 | 使用ツールのTC（Tool Confidence）クラスと資格化計画 |
| 6. 委託管理計画 | 委託先との契約・レビュー・証跡受け取り方針 |
| 7. 変更管理計画 | 変更要求発生時の影響分析・再評価手順 |
| 8. コンピテンシー計画 | 機能安全に関する担当者のトレーニング計画 |
| 9. リスク管理 | プロジェクトリスクの識別と対策 |
| 10. 内部監査計画 | フェーズゲートレビューの計画 |

### 3.2 外部委託を前提とした責任分担

自社が全てを内製しないケースでは、責任分担の明確化が特に重要である。暗黙の分担は必ずトラブルを招く。

| 活動 | 自社（発注者） | 委託先（受注者） |
|------|--------------|----------------|
| アイテム定義 | 主担当 | 支援 |
| **HARA** | **主担当**（自社が行うべき） | 支援・レビュー |
| **安全目標の確定** | **主担当**（最終決裁は自社） | 入力提供 |
| 機能安全要求（FSRS） | 主担当・承認 | 作成・レビュー |
| 技術安全要求（TSRS） | 承認 | 主担当 |
| ソフトウェア安全要求（SSRS） | レビュー・承認 | 主担当 |
| ソフトウェア設計・実装 | レビュー | 主担当 |
| ソフトウェアテスト | 受入確認 | 主担当 |
| ハードウェア設計 | 承認 | 主担当 |
| 統合テスト（HW+SW） | 参加・確認 | 主担当 |
| Safety Case（全体安全論証） | **最終責任者** | 入力・支援 |
| 認証機関対応 | **主窓口** | 技術支援 |

> **重要：** HARAと安全目標は「何を作るか」を定義する最上位の安全活動であり、外部委託しても最終責任は自社にある。委託先が作成した場合も、自社技術者が内容を理解し承認できる状態にしておく必要がある。

### 3.3 機能安全マネージャーの役割と要件

**役割：**
- Safety Planの作成・維持・更新
- 機能安全ライフサイクル全体の監督
- 証跡の完全性確認
- 認証機関とのコミュニケーション窓口
- 開発チームへの機能安全要求の展開確認

**要件（ISO 25119が求める能力）：**

```
必須知識：
  ✓ ISO 25119（または ISO 26262）の要求事項の理解
  ✓ HARAの実施経験（または指導できる知識）
  ✓ FMEA/FTAの基礎知識
  ✓ ソフトウェア安全開発プロセスの理解
  ✓ 証跡管理・トレーサビリティの知識

望ましい経験：
  ✓ 機能安全に関するプロジェクト参画経験（3年以上）
  ✓ 外部認証機関との交渉経験
  ✓ 委託先管理経験
```

**社内任命 vs 外部委託の判断：**

| 判断軸 | 社内任命が望ましい | 外部委託が有効 |
|--------|-------------------|--------------|
| HARAの内容理解 | 自社製品を熟知した人材が適切 | 専門的なHARAファシリテートは外部が効率的 |
| Safety Case最終承認 | 自社責任者が必須 | - |
| 認証機関対応 | 自社窓口が必要（外部補佐は可） | コンサルタント活用可 |
| 継続的な証跡管理 | 自社が継続担当すべき | - |

> **推奨：** 少なくとも1名の社内機能安全マネージャーを任命し、外部コンサルタントを副担当として活用する形態が現実的である。完全な外部委託は避けること（自社にノウハウが蓄積されない）。

### 3.4 機能安全ライフサイクルと証跡の対応表

| フェーズ | 主な活動 | 必要な証跡（成果物） |
|---------|---------|-----------------|
| **概念フェーズ** | アイテム定義、HARA | アイテム定義書、HARA報告書、安全目標一覧 |
| **システム開発** | 機能安全要求定義、アーキテクチャ設計 | FSRS、TSRS、安全アーキテクチャ設計書、FMEA |
| **ソフトウェア開発** | SW安全要求、設計、実装、単体テスト | SSRS、SWアーキテクチャ設計書、詳細設計書、単体テスト報告書 |
| **統合・検証** | HW/SW統合、システムテスト | 統合テスト計画書・報告書、安全機能検証報告書 |
| **妥当性確認** | 実機での安全機能確認 | 妥当性確認計画書・報告書 |
| **Safety Case** | 安全論証の統合 | Safety Case文書（GSN/CAE形式） |
| **量産・運用** | 変更管理、不具合対応 | 変更影響分析書、不具合報告書 |
| **廃棄** | 廃棄時の安全考慮 | 廃棄計画書 |

## 4. 証跡管理の実務

### 4.1 必要な証跡の種類と管理方法

証跡（Evidence）は機能安全対応を「証明」するための文書・データ・ログの総称である。単に文書が存在するだけでなく、内容が正確で互いに整合していることが求められる。

**証跡の分類：**

| カテゴリ | 主な証跡 | 管理場所の例 |
|---------|---------|------------|
| プロセス証跡 | Safety Plan、コンピテンシー記録、レビュー議事録 | 社内SharePoint / Git |
| 分析証跡 | HARA報告書、FMEA、FTA | 専用ツール or 管理ファイルサーバー |
| 要求証跡 | FSRS、TSRS、SSRS、要求追跡マトリクス（RTM） | 要求管理ツール（IBM DOORS等）または表計算 |
| 設計証跡 | 安全アーキテクチャ設計書、詳細設計書 | ドキュメント管理システム |
| テスト証跡 | テスト計画書、テスト報告書、カバレッジ報告書 | テスト管理ツール |
| ツール証跡 | ツール資格化記録（TC）、ツール検証報告書 | プロジェクト管理システム |

**管理の基本原則：**

```
1. バージョン管理：全ての証跡にバージョン・改訂履歴を記載
2. 承認記録：作成者・レビュア・承認者の署名（電子署名可）と日付
3. 変更追跡：変更時に何をなぜ変更したかを記録
4. アクセス制御：改ざん防止のための権限管理
5. バックアップ：プロジェクト終了後も一定期間保管（要確定：10年程度）
```

### 4.2 「証跡があれば良い」という落とし穴

多くのプロジェクトで観察される誤解と、その実態を整理する。

| 落とし穴 | 具体的な問題 | 対策 |
|---------|------------|------|
| **形式的なHARA** | リスクが低くなるように意図的に（または無意識に）パラメータを設定している | HARAはワーストケース思考で実施。第三者レビューを必ず行う |
| **コピペ要求** | 前プロジェクトのFSRSをほぼ流用しているが、現システムに合っているか未確認 | 要求をHARAの安全目標と突合せ、根拠を確認する |
| **形骸化したテスト** | テスト報告書はあるが、テストケースが安全要求と結びついていない | テスト計画書にSSRS/TSRSのIDを明記し追跡可能にする |
| **更新されない証跡** | 仕様変更後も古いHARAやFSRSが「最新」として管理されている | 変更管理プロセスに証跡更新チェックを組み込む |
| **審査員への過信** | 認証機関が審査したから安全だ、という誤解 | 認証は「プロセスが規格に従っているか」の確認であり、システムの安全性を保証するものではない |

### 4.3 外部委託先から受け取る証跡のチェックリスト

委託先から証跡を受領する際のチェック項目。形式的な受け取りではなく、内容の妥当性確認まで行うこと。

**共通チェック：**
```
□ ドキュメントにバージョン・日付・承認者が記載されているか
□ 適用規格（ISO 25119 / ISO 26262）のどのパートに対応するか明記されているか
□ 前回版からの変更履歴が記載されているか
□ 自社で承認したSSRS/FSRSと内容が整合しているか
```

**HARA・安全要求の受け取り時：**
```
□ 全ての安全目標について、S/E/CパラメータとAgPLの根拠が記述されているか
□ 安全目標から機能安全要求（FSRS）への展開が論理的に追えるか
□ 見落とされているハザードや使用状況はないか
□ 安全状態（Safe State）とFTTIが全ての安全目標に記載されているか
```

**設計書の受け取り時：**
```
□ 安全アーキテクチャ図にASIL/AgPL分解が明記されているか
□ E2E保護・Watchdog等の安全機構が設計に含まれているか
□ FMEA/FTAで単一故障（Single Point Failure）が評価されているか
□ ソフトウェア設計がSSRSの全項目をカバーしているか（RTMで確認）
```

**テスト報告書の受け取り時：**
```
□ テストケースがSSRS/TSRSのどの要求を検証しているかが明記されているか
□ 全安全要求に対するテストカバレッジが100%（または免除理由が記載）であるか
□ MC/DC（修正条件/決定カバレッジ）等の構造カバレッジ要求を満たしているか
□ 故障注入テスト（Fault Injection Test）の結果が含まれているか
□ 全テストが合格しているか（NOGOテストの場合は回帰テスト記録があるか）
```

### 4.4 ToolChain資格（TC1〜TC3）の説明と判断基準

開発ツール（コンパイラ、コードジェネレーター、テストツール等）の品質が低ければ、それを使って作ったソフトウェアも信頼できない。ISO 25119（および参照するISO 26262 Part 8）では、ツールの信頼性を「TCクラス」で評価する。

**TCクラスの概要：**

| TCクラス | 意味 | 必要な対応 |
|---------|------|-----------|
| **TC1** | ツールの誤出力がソフトウェアに影響しない、または検出可能 | 特別な資格化不要。使用理由を記録するのみ |
| **TC2** | ツールの誤出力が影響する可能性があるが、その影響を他の手段で検証可能 | ツールの検証計画を作成し、既存の証跡（ユーザー事例等）で対応可能 |
| **TC3** | ツールの誤出力が直接安全に影響し、他の手段での検出が困難 | 完全なツール資格化が必要（ツールベンダーの資格証明または自社検証） |

**建設機械プロジェクトでの判断例：**

| ツール | 典型的なTCクラス | 判断理由 |
|--------|--------------|---------|
| Classic AUTOSARコードジェネレーター（RTE生成） | TC3（場合によりTC2） | 生成コードがSWCに直接影響し、誤生成を単体テストで完全には検出できない |
| コンパイラ（gcc, greenhills等） | TC2〜TC3 | コンパイラバグはテストで大半は検出可能（TC2）だが、最適化オプション依存の問題はTC3相当の注意が必要 |
| 静的解析ツール（MISRA checker等） | TC1〜TC2 | コードを変更しないため影響は低い（TC1）だが、誤検出・見逃しはある（TC2） |
| テスト自動化フレームワーク | TC2 | テスト結果に影響するが、テスト自体の妥当性確認で対応可能 |
| モデリングツール（MATLAB/Simulink等） | TC3 | コード生成を含む場合はTC3。モデルのみの場合はTC2 |

> **実務上のポイント：** ツールベンダーがISO 26262/ISO 25119対応の「ToolQualification Kit」を提供しているかを確認する。Vector、dSPACE等の主要ベンダーは資格化証明書を提供している場合が多い（要確定）。

### 4.5 認証機関（TÜV等）との交渉・審査の進め方

**主要な認証機関（Functional Safety Assessment機関）：**

| 機関 | 特徴 |
|------|------|
| TÜV SÜD / TÜV Rheinland | ドイツ系。自動車・機械分野で豊富な実績。日本にも拠点あり |
| SGS / Intertek | グローバル展開。多分野対応 |
| Bureau Veritas | 欧州系。建設機械分野の実績あり |
| 日本品質保証機構（JQA） | 国内機関。ISO認証取得後の機能安全審査も対応（要確定） |

**審査プロセスの一般的な流れ：**

```
Step 1：審査依頼・スコープ合意（プロジェクト初期）
  └─ 適用規格・対象システム・審査の範囲を合意する
  └─ 審査費用・スケジュールの見積もりを取得

Step 2：フェーズゲートレビュー（概念・システム・SW開発フェーズ）
  └─ 各フェーズ終了時に証跡を提出し審査官がレビュー
  └─ Observation（軽微な指摘）とNon-Conformance（不適合）の対応

Step 3：最終審査（Safety Case審査）
  └─ Safety Case全体の審査
  └─ 開発チームへの口頭確認（インタビュー）
  └─ 実機での安全機能デモンストレーション

Step 4：認定書の発行
  └─ 合格後、Functional Safety Assessment Report と認定書を発行
  └─ 認定書の有効期間と更新条件を確認

Step 5：量産後のフォローアップ
  └─ 変更発生時の再審査（変更の規模による）
  └─ 定期的なサーベイランス審査
```

**交渉・準備のポイント：**

```
□ 審査機関は早期（フェーズ1開始前）から関与させる（最終審査だけ依頼は最悪の手順）
□ 審査機関との窓口は自社の機能安全マネージャーが担当する
□ 不適合（Non-Conformance）は必ず書面で受け取り、対応記録を残す
□ 審査機関の指摘は要求事項の解釈として参考にする（最終判断は自社）
□ 審査費用は開発フェーズの早期から予算計上する（後から確保は困難）
```

## 5. Classic AUTOSARにおける機能安全実装

### 5.1 ASIL分解とSWCへの安全要求割り付け

Classic AUTOSARはソフトウェアをSWC（Software Component）の集合として構成する。機能安全の観点では、HARAで決定したAgPL/ASILの要求をどのSWCに割り付けるかを明確にする必要がある。

**ASIL分解（ASIL Decomposition）の概念：**

1つのASIL D要求を満たすために、ASIL D対応のシングルチャネルで実装する代わりに、独立したASIL B + ASIL Bの2チャネルに分解して実現する手法がある。

```
ASIL D要求（単一実装）
  → 開発コストが非常に高い

ASIL D = ASIL B(D) + ASIL B(D) に分解
  → 各チャネルはASIL B要求で開発
  → 独立性（Independence）の証明が必要
  → 合計開発コストは下がる場合が多い

※ AgPL dも同様の分解が可能（AgPL d = AgPL b + AgPL b）
```

**SWCへの安全要求マッピング例：**

```
安全目標：「アームは操作指令がない限り降下してはならない」（AgPL d）

↓ ASIL分解

チャネルA（AgPL b相当）：ArmControlSWC
  - 操作レバー入力を処理し、降下指令を生成
  - 異常検出時に降下禁止フラグを設定

チャネルB（AgPL b相当）：SafetyMonitorSWC
  - ArmControlSWCの出力を独立監視
  - 不正な降下指令を検出した場合に安全状態（降下禁止）に遷移

独立性の確認：
  - 両SWCは異なる開発者/チームが開発
  - 共通原因故障（CCF）の分析を実施
  - 共有メモリ・共有リソースの最小化
```

### 5.2 Runnable・ポートへの安全要求マッピング方法

Classic AUTOSARでは、SWCの処理はRunnable（実行単位）として定義される。安全要求はRunnable単位まで落とし込む。

```
SSRS-001: アーム降下指令は、操作レバー入力信号が有効値の場合のみ生成すること

↓ マッピング

SWC: ArmControlSWC
  Runnable: rbl_ArmControl_10ms（10msサイクル実行）
    Port: LeverInput（R-Port, AUTOSAR Interface: If_LeverInput）
      → この入力値が有効範囲内かをRunnable内で検証
    Port: ArmDriveCmd（P-Port, AUTOSAR Interface: If_ArmCmd）
      → 有効入力がある場合のみ降下指令値を出力
      → 無効入力・入力なしの場合はゼロ値（降下禁止）を出力
    
  ArXML設定（概念）：
    <REQUIRED-DATA-ELEMENT-REFS>
      <REQUIRED-DATA-ELEMENT-REF>LeverInput/LeverValue</REQUIRED-DATA-ELEMENT-REF>
    </REQUIRED-DATA-ELEMENT-REFS>
    <!-- 入力信号のタイムアウト監視もここで設定 -->
```

### 5.3 SafetyManager SWCの役割と設計パターン

SafetyManager SWCは、複数の安全機能を横断的に監視・制御するSWCである。Classic AUTOSARプロジェクトでは1つのSWCとして設計することが多い。

**SafetyManagerの主な役割：**

```
SafetyManagerSWC（AgPL d相当）
  ┌─────────────────────────────────────────┐
  │ 1. システム状態監視                        │
  │    - 各SWCのAlive監視（デッドロック検出）    │
  │    - E2Eエラーカウンタの監視               │
  │    - Watchdog token管理                  │
  │                                          │
  │ 2. 安全状態遷移管理                        │
  │    - Normal → SafeState の遷移制御         │
  │    - SafeState での安全出力（全軸停止等）    │
  │    - SafeState からの復帰条件管理           │
  │                                          │
  │ 3. 故障ログ記録                            │
  │    - NvM（不揮発メモリ）への故障記録         │
  │    - DTC（診断コード）の設定               │
  │    - DTCはDEM（Diagnostic Event Manager） │
  │      経由で管理する                        │
  └─────────────────────────────────────────┘
```

**状態遷移設計パターン：**

```
[Normal State]
  ↓ 安全機能異常検出（E2Eエラー / Alive監視失敗 / センサー異常）
[SafeState Requested]
  ↓ 全アクチュエーター停止指令送信（FTTI以内）
[SafeState Active]
  ↓ リセット条件成立（オペレーター操作 or 自動回復）
[Normal State]（または再起動）
```

### 5.4 E2E（End-to-End）保護プロファイルの設定

E2E保護は、CAN通信やRTE内部通信における**データ破損・遅延・消失・繰り返し**を検出するための仕組みである。CRC（巡回冗長検査）とカウンタを組み合わせて実装する。

**E2Eプロファイルの選択：**

| プロファイル | CRC長 | カウンタ | 適用場面 |
|------------|-------|---------|---------|
| Profile 1 | 8bit | 4bit | 低帯域CAN通信（8バイトフレーム） |
| Profile 2 | 16bit | 4bit | 中程度のデータ長 |
| Profile 5 | 32bit | 8bit | 高信頼性が必要な長いデータ |
| Profile 7 | 64bit | 8bit | 高精度・大容量データ（Ethernet含む） |

**Classic AUTOSARでのE2E設定（ArXML概念）：**

```xml
<!-- E2E保護の設定例（ArXMLの概念、実際の記法はツールで生成） -->
<E2E-PROFILE-COMPATIBILITY-PROPS>
  <E2E-PROFILE-CONFIGURATION>
    <PROFILE-NAME>PROFILE_02</PROFILE-NAME>
    <DATA-LENGTH>8</DATA-LENGTH>   <!-- 8バイトCAN DLC -->
    <MAX-DELTA-COUNTER-INIT>1</MAX-DELTA-COUNTER-INIT>
    <MAX-NO-NEW-OR-REPEATED-DATA>3</MAX-NO-NEW-OR-REPEATED-DATA>
    <SYNC-COUNTER-INIT>15</SYNC-COUNTER-INIT>
  </E2E-PROFILE-CONFIGURATION>
</E2E-PROFILE-COMPATIBILITY-PROPS>
```

**安全要求へのE2E要件記載例：**

```
SSRS-E2E-001: ArmDriveCmdポートの通信データはE2E Profile 2で保護すること
SSRS-E2E-002: E2EエラーカウントがMaxErrorCount（=3、要確定）を超えた場合、
              SafetyManagerSWCは安全状態遷移を開始すること
SSRS-E2E-003: E2E保護はRTEのTransformerレイヤーで適用し、
              SWCのアプリケーションコードはE2Eを意識しないこと
```

### 5.5 Watchdog・安全状態遷移の設計

Watchdogは「ソフトウェアが正常動作していること」を外部のハードウェア（またはSWモジュール）に定期的に通知する仕組みである。

**Classic AUTOSARのWatchdog Manager（WdgM）を使った設計：**

```
Application Layer
  SafetyManagerSWC
    └─ rbl_SafetyMgr_10ms でWdgM_CheckpointReached()を呼ぶ
       └─ Checkpoint: WDGM_CHECKPOINT_SAFETY_MGR_MAIN

WdgM（Watchdog Manager）
  └─ 全Runnableの"Alive Supervision"を実施
     └─ rbl_ArmControl_10ms が10msごとに実行されているか確認
     └─ rbl_SafetyMgr_10ms が実行されているか確認
  └─ 異常検出時にWdgM_GlobalStatusをWDGM_GLOBAL_STATUS_STOPPEDに設定

WdgIf（Watchdog Interface）→ Wdg（Watchdog Driver）
  └─ 外部Watchdog ICへのtrigger信号をWdgM異常時に停止
     └─ 外部Watchdog ICがECUをリセット

→ リセット後: ECU再起動 → 安全状態への遷移
```

## 6. 「形式的対応」から「実効対応」への移行ステップ

### 6.1 現状の形式的対応の典型的な問題パターン4つ

建設機械の内製ソフトウェア開発で観察されることが多い形式的対応のパターンを示す。

**パターン1：HARAのリスク過小評価**

```
問題：「アーム誤作動が発生しても、熟練オペレーターなら対処できる」という
      根拠なき前提でC（制御可能性）を低く評価し、AgPLを1〜2段階下げている。

症状：安全目標が「AgPL b」と結論づけられているが、実際の故障ケースを
      検討すると明らかにAgPL d相当のハザードが含まれている。
```

**パターン2：安全要求のトレーサビリティ欠落**

```
問題：FSRSは存在するが、ソフトウェアのどのモジュール・関数がその要求を
      実装しているか追跡できない。テストとの紐付けも存在しない。

症状：「安全要求が変更になったが、何を修正すればよいかわからない」という
      状況が発生する。また、テストを追加しても何をカバーしているかが不明。
```

**パターン3：テストの有名無実化**

```
問題：テスト報告書は存在するが、テストケースが「動作確認」レベルであり、
      故障時の挙動（フォールトインジェクション）を全く検証していない。

症状：単体テストは全て「合格」だが、実機でセンサー断線が発生したときに
      安全状態に遷移するかどうかが未検証。
```

**パターン4：変更時の証跡未更新**

```
問題：ソフトウェアが変更された際に、HARAや安全要求書が更新されない。
      ドキュメントと実装が乖離している状態で「最新版」として管理されている。

症状：委託先やレビュアーに「このソフトの仕様書はどれですか」と聞くと
      複数のバージョンが提示され、どれが正しいかわからない。
```

### 6.2 実効対応に向けた優先順位付けの方法

全てを一度に改善しようとすると破綻する。以下の基準で優先順位を付ける。

| 優先度 | 対象 | 基準 |
|--------|------|------|
| **最優先** | HARAの再評価 | 全ての安全対応の根拠となるため、ここが誤っていると後続が全て無駄になる |
| **高** | 安全目標〜SSRSのトレーサビリティ確立 | 証跡の骨格。存在しないと審査で即アウト |
| **中** | 故障注入テスト（フォールトインジェクション） | 実効対応の証明として最も説得力がある |
| **低** | 全ドキュメントの書式統一 | 内容が正しければ書式は後で整える |
| **最後** | 全項目の完全カバレッジ | パイロット機能で実績を作ってから拡大する |

### 6.3 フェーズ別機能安全対応ロードマップ

| フェーズ | 期間（目安） | 主な活動 | マイルストーン |
|---------|------------|---------|-------------|
| **Phase 0：現状把握** | 1〜2ヶ月 | 現状のHARA・安全要求・テストのギャップ分析。機能安全マネージャーの任命 | ギャップ分析報告書の完成 |
| **Phase 1：基盤整備** | 3〜6ヶ月 | Safety Plan作成。HARAの再実施（主要機能から）。要求管理ツール導入 | Safety Plan承認。主要機能のHARA完了 |
| **Phase 2：パイロット実装** | 6〜12ヶ月 | パイロット機能（1〜2機能）で実効対応の完全実施。Classic AUTOSARへの実装 | パイロット機能のSafety Case完成。第三者レビュー受審 |
| **Phase 3：展開** | 12〜24ヶ月 | パイロットで確立した手順を残り機能に展開。認証機関による最終審査 | 全機能のSafety Case完成。認証取得 |

### 6.4 小さく始めるためのパイロット機能の選び方

パイロット機能は「小さいが代表的」なものを選ぶ。以下の基準で選定する。

```
パイロット機能選定の基準：
  ✓ AgPL c〜d相当の機能（高すぎず、やりがいがある）
  ✓ 境界が明確で他機能との依存が少ない
  ✓ テスト環境が整備されている（またはテスト容易性が高い）
  ✓ 開発担当者が機能安全に前向きであること
  ✓ 委託先も同意しており、証跡作成に協力的

推奨パイロット候補（建設機械）：
  案A：エンジン緊急停止機能（AgPL c）
       → 単純だが安全上重要。Classic AUTOSARでの実装例として最適
  案B：走行速度制限機能（AgPL c〜d）
       → 速度センサー入力〜駆動出力の全体フローをカバーできる
```

## 7. よくある失敗パターンと対策

### 失敗パターン1：HARAを委託先に丸投げする

**状況：** 「機能安全は委託先が詳しいから、HARAも委託先にやってもらおう」と判断し、自社はレビューするだけにした。

**問題：** 委託先は自社製品・使用状況・現場リスクを知らない。保守的に評価すると開発コストが上がるため、意図せずリスクを低く見積もるインセンティブが働く。結果として、重要なハザードが見落とされる。

**対策：**
```
・HARAは自社の熟練オペレーター・現場担当者・技術者が主体となって実施する
・委託先はHARAのファシリテート（進行役）として活用する
・現場担当者へのヒアリングをHARAセッションに組み込む
```

---

### 失敗パターン2：「Classic AUTOSAR導入=機能安全対応」と誤解する

**状況：** Classic AUTOSARを採用したことで、BSW（基本ソフトウェア）のモジュールが機能安全対応済みになると期待する。

**問題：** Classic AUTOSARのBSWモジュールはAgPL対応品である場合があるが、それはBSWが機能安全要求を満たして実装されているということであり、アプリケーション（SWC）の安全要求は別途定義・実装する必要がある。ツールが安全機能を「自動的に」実装してくれるわけではない。

**対策：**
```
・BSW製品のAgPL適合範囲を正確に把握する（AUTOSAR BSP製品のSafety Manualを読む）
・アプリケーション層（SWC）の安全要求はSSRSとして別途定義・実装する
・委託先にSWCの機能安全実装責任を明確に割り当てる
```

---

### 失敗パターン3：機能安全担当を1人に集中させる

**状況：** 機能安全マネージャーを1人任命し、その人が全ての機能安全活動を抱え込む体制にした。

**問題：** その人が退職・異動すると、機能安全の知見と証跡の解釈が失われる。また、1人では客観的な評価が難しく、重要なハザードや要求の抜け漏れを内部で指摘できなくなる。

**対策：**
```
・最低でも機能安全マネージャー＋バックアップ1名の2人体制とする
・機能安全活動の内容をチームで共有する定期レビューを設ける（月1回以上）
・証跡はドキュメント管理システムで全員がアクセスできる状態にする
・機能安全トレーニングは複数名が受講する
```

---

### 失敗パターン4：変更時にHARAを更新しない

**状況：** ソフトウェアや機能が変更になったが、「HARAは済んでいる」として再評価を省略した。

**問題：** 変更によって新たなハザードが生まれることがある。例えば、センサーの種類変更により検出遅延が増加し、FTTIを超過するケースや、機能追加によりSWCのインターフェースが変わりE2E保護の範囲が変わるケースがある。変更後に重大インシデントが発生した場合、変更影響分析を実施していないことが法的責任の根拠になりうる。

**対策：**
```
・変更管理プロセスのステップ1に「機能安全への影響確認」を組み込む
・影響確認の結果（影響あり/なし）を変更管理記録に残す
・「影響あり」の場合はHARA・安全要求・テストの更新をトリガーする
・変更影響分析のテンプレートを用意し、担当者が機能安全マネージャーでなくても実施できるようにする
```

---

### 失敗パターン5：認証取得をゴールにしてしまう

**状況：** 機能安全対応の目的が「TÜVの認定書を取ること」になり、認定取得後は証跡のメンテナンスや変更時の再評価を行わなくなった。

**問題：** 認定書は取得時点のシステムに対して発行される。変更後のシステムへの適用可否は別途評価が必要である。また、認定書があってもフィールドでインシデントが発生すれば、製造者責任を問われる可能性がある。

**対策：**
```
・認証取得は「機能安全活動の適切な実施を第三者が確認した」という意味に留める
・量産後も変更管理・証跡更新を継続する体制を維持する
・年1回程度、機能安全マネージャーが証跡の有効性を確認するレビューを実施する
・インシデント発生時の機能安全観点での調査手順を事前に定めておく
```

## 付録A：用語集

| 用語 | 説明 |
|------|------|
| HARA | Hazard Analysis and Risk Assessment。ハザード分析とリスク評価 |
| AgPL | Agriculture Performance Level。ISO 25119における安全水準（a〜e） |
| ASIL | Automotive Safety Integrity Level。ISO 26262における安全水準（A〜D） |
| FSRS | Functional Safety Requirements Specification。機能安全要求仕様書 |
| TSRS | Technical Safety Requirements Specification。技術安全要求仕様書 |
| SSRS | Software Safety Requirements Specification。ソフトウェア安全要求仕様書 |
| RTM | Requirements Traceability Matrix。要求追跡マトリクス |
| FTTI | Fault Tolerant Time Interval。故障が発生してから安全状態に遷移するまでの許容時間 |
| Safe State | 安全状態。故障検出時にシステムが移行する安全な動作状態 |
| CCF | Common Cause Failure。共通原因故障。独立した2チャネルが同一の原因で同時故障すること |
| E2E保護 | End-to-End保護。通信データの破損・遅延・消失をCRC＋カウンタで検出する仕組み |
| Safety Case | 安全論証。システムが安全であることを論理的に示す文書（GSN/CAE形式で記述することが多い） |
| TC（Tool Confidence） | ツールの信頼性クラス（TC1〜TC3）。TC3が最も高い対応が必要 |
| WdgM | Watchdog Manager。Classic AUTOSARのBSWモジュール。Runnableの生存監視を担当 |
| DEM | Diagnostic Event Manager。Classic AUTOSARのBSWモジュール。診断イベント管理 |
| Classic AUTOSAR | AUTOSAR規格のうちRTOS上のECU向けプラットフォーム。本資料の主対象 |
| Adaptive AUTOSAR | AUTOSAR規格のうちLinux等のOS上で動作する次世代プラットフォーム。本プロジェクトでは将来検討 |
| ArXML | AUTOSAR規格で定義されたXMLベースの記述フォーマット |
| J1939 | SAE J1939プロトコル。建設・農業機械向けのCAN上位プロトコル |
| SOME/IP | Scalable service-Oriented MiddlewarE over IP。Adaptive AUTOSARで使われるEthernet上の通信プロトコル |
