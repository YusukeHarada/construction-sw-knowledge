# AUTOSARを前提とした要求仕様の作り方

**作成日：** 2026年6月16日  
**対象読者：** 自社技術担当者（AUTOSAR知識不要で使えることを想定）  
**資料種別：** 社内運用資料（社外秘）

---

## 概要

外部委託先にAUTOSAR準拠のプラットフォームソフトウェアを依頼するために必要な要求仕様の作り方を定義する。

要求仕様書とは「自社が委託先に何を作ってほしいかを文書で伝えるもの」であり、委託開発においては設計・テスト・費用・品質判断の全てがこの文書に基づいて行われる。「だいたいこういうもの」という口頭説明では、後から「言った言わない」「解釈が違った」という問題が必ず起きる。

本資料の基本的な考え方は「AUTOSARが決めていること」と「自社が決めなければならないこと」を分離することにある。全てを一から自社で仕様化しようとする必要はなく、AUTOSARが標準で定めている部分は「標準に準拠すること」と書くだけでよい。

---

## 用語一覧（読み始める前に確認する）

| 用語 | 読み方 | 意味 |
|---|---|---|
| SWC（Software Component） | エスダブリューシー | ソフトウェアを機能単位で分割した部品。「油圧制御SWC」「人検知SWC」のように、機能ごとに独立した部品として設計する。AUTOSAR設計の基本単位 |
| BSW（Basic Software） | ビーエスダブリュー | OS・ドライバ・通信スタックなど、ハードウェアを抽象化するソフトウェア層。委託先が実装するが、何が必要かは自社が要求仕様で指定する |
| RTE（Runtime Environment） | アールティーイー | SWC同士のデータのやり取りを中継する層。RTEのコードはツールが自動生成するため自社で実装する必要はないが、何をやり取りするかは自社が定義する |
| Runnable | ランナブル | SWC内で定義される処理の単位。「10msごとに油圧制御計算を実行する」という処理の塊がRunnable。呼び出し条件（周期・イベント）とセットで定義する |
| ポート（Port） | ポート | SWC同士がデータをやり取りするための出入口。データを出す側を「Provide Port」、受け取る側を「Require Port」と呼ぶ |
| ArXML | アーエックスエムエル | AUTOSARが定めた標準の設計データ形式（XML）。委託先がツールで生成する設計データがこの形式で出力される。自社はこれをレビューして「要求通りか」を確認する |
| HARA | ハラ | Hazard Analysis and Risk Assessment（ハザード分析・リスク評価）の略。機能ごとに「何が危険か」「どの程度危険か」を分析して安全レベルを決める作業 |
| ASIL | エイシル | Automotive Safety Integrity Level（自動車安全度水準）。A〜Dの4段階でDが最も厳しい。建設機械には直接適用されないが、委託先との対話で基準として使われる |
| PL（Performance Level） | ピーエル | ISO 13849・ISO 25119で使う安全水準の表現。a〜eの5段階でeが最も厳しい。建設機械の機能安全はこの指標で語ることが多い |
| NvM（Non-volatile Memory） | エヌブイエム | 電源オフ後もデータを保持する不揮発メモリの管理モジュール。故障コード・設定値の保存に使う |
| ECU（Electronic Control Unit） | イーシーユー | 制御用コンピューター基板。油圧制御ECU・HMI用ECUのように機能別に搭載される |
| HIL（Hardware-in-the-Loop） | エイチアイエル | 実際のECU基板にテスト装置をつなぎ、センサーやアクチュエータをシミュレーターで代替してテストする手法 |
| FMEA | エフエムイーエー | Failure Mode and Effects Analysis（故障モード影響解析）。各部品・機能の故障パターンを列挙し、それが全体にどう影響するかを分析する文書 |
| J1939 | ジェイいちきゅうさんきゅう | 建設機械・農業機械・トラック向けCAN上位プロトコル。エンジンECU・トランスミッションECUとの通信に使われる |
| MISRA-C | ミスラシー | 組み込みソフトウェア向けCコーディング規約。バグになりやすい記述を禁止するルール集。機能安全対応では準拠が必須とされる |
| UDS（Unified Diagnostic Services） | ユーディーエス | ISO 14229で定めた診断通信の標準仕様。故障コード読み出し・ソフト書き換え等の操作を行うプロトコル |
| トレーサビリティ | トレーサビリティ | 要求仕様・設計・テストの各段階が対応しているかを追跡できること。「この要求はこの設計で実現され、このテストで確認した」という対応関係が明確な状態 |

## 1. 基本的な考え方

### AUTOSARが決めていること（自社が実装・決定しなくてよいもの）

以下の項目は、AUTOSARの規格そのものに仕様が定められている。自社の要求仕様書には「AUTOSARの規格に従うこと」と書くだけでよく、詳細を自社が決める必要はない。

- BSW各モジュールのAPI（Application Programming Interface：機能部品の呼び出し方）とインターフェース仕様
- RTE（Runtime Environment）の動作仕様：SWC間のデータ受け渡しルール
- OS（AUTOSAR OS）のスケジューリング仕様：タスクの優先度管理・実行順序の規則
- 通信スタック（COM・PDUR・CANIF等）のレイヤー構造：CAN信号がどう処理されるかの基本構造

**補足：** RTEのCコードは委託先がツール（Vector DaVinci等）を用いて自動生成する。ただし、その入力となるArXML（ポート定義・Runnable定義）の内容は自社が承認する必要がある。「自動生成だからチェック不要」ではなく、「自動生成の入力定義が要求通りかを確認する」のが自社の役割である。

### 自社が決めなければならないこと（要求仕様として記述するもの）

AUTOSARが決めていない「何を作るか」の部分が自社の責任範囲になる。以下を要求仕様書に記述する。

| 決めること | 具体例（建設機械） | なぜ自社が決めるのか |
|---|---|---|
| どのSWCが必要か | 油圧制御SWC・人検知SWC・ゲートウェイSWC | 自社の機械が何をするかは自社にしか判断できないため |
| 各SWCのポート定義 | ValveCommandポート（型：uint16、単位：%） | データの型・意味・単位を定義しなければ、委託先が間違った型で実装するリスクがある |
| Runnableの周期・起動条件 | 油圧制御は10ms周期、人検知は100ms周期 | 制御周期は性能と安全に直結するため、自社で機能要件として決める |
| ECUマッピング | 油圧制御SWCはRTOS系ECUに配置 | どのECUに何を載せるかはシステム設計の決定事項 |
| 機能安全レベルの割り当て | 油圧制御SWCはISO 25119 PLd相当 | HARAの結果に基づく安全レベルは自社の責任で定める |

## 2. 機能安全レベルの考え方

**重要：** 本資料では実装例としてASILという表記を使用しているが、これはISO 26262（自動車向け）由来の概念である。建設機械向けにはISO 25119（農業・建設機械向け機能安全）またはISO 13849（機械安全）の**PL（Performance Level）**が適用される。

要求仕様作成時は、適用規格とレベルを以下の形で明記すること。

```
例：
[FS-HYD-001] ISO 25119 PLd相当の安全機能を満たすこと
[FS-HYD-001] ISO 26262 ASIL-Bを満たすこと（自動車向け規格を意図的に採用する場合）
```

どちらの規格を使うかは、認証機関・顧客要件・委託先の対応能力を踏まえて決める。迷う場合は委託先または安全コンサルタントに相談する。

### ASILレベルの割り当て方法

ASILやPLは、担当者が「これは危ないから高い安全レベルにしよう」と感覚で決めるものではなく、HARA（ハザード分析・リスク評価）という手順に従って客観的に決定する。

HARAの手順は概略以下のとおりである。

1. **ハザードを洗い出す**：「何が起きたら人に危害が及ぶか」を機能ごとに列挙する。例：「油圧制御が誤動作してアームが急降下する」「人検知が失敗して作業区域に人がいるのに動作を継続する」
2. **リスクを評価する**：各ハザードについて「深刻度（Severity）」「暴露確率（Exposure）」「制御可能性（Controllability）」の3軸でスコアを付ける
3. **安全レベルを決定する**：スコアの組み合わせに基づいてASIL（またはPL）を決定する

HARAの実施は高度な専門知識を要するため、委託先に支援を依頼するか、別途安全コンサルタントに依頼することを強く推奨する。自社で機能一覧（「この機能が誤動作するとどうなるか」のリスト）を準備しておくと、HARA支援の費用を抑えられる。

### 建設機械の機能安全レベルの目安

以下はあくまで参考目安であり、HARAを実施して確定すること。

| 機能 | 誤動作の影響 | 目安の安全レベル |
|---|---|---|
| 油圧アーム制御 | アームが急降下・急旋回し人に当たる | PLd（ASIL-B〜C相当） |
| 走行制御 | 意図しない走行・急加速 | PLd（ASIL-B〜C相当） |
| 人検知・作業区域監視 | 人を検知できず機械が動作を継続 | PLc〜PLd |
| 診断通信 | 故障コードの読み誤り（制御には非直結） | PL不要〜PLa |
| HMI表示 | 表示が不正確（操作ミスを誘発） | PLb〜PLc |

## 3. 要求仕様の構成

要求仕様書は「システム全体を対象にした仕様書（System Requirements Specification）」と「SWCごとの仕様書（SWC Requirements Specification）」の2層で構成する。最初に全体仕様書を作り、その後 SWCごとに詳細化するという順序で進める。

### 3-1. システム要求仕様（System Requirements Specification）

委託先への入力として最初に渡す最上位ドキュメント。「何のシステムを・どういう構成で・何のために作るか」を全体像として定義する。

```
システム要求仕様書のテンプレート

1. 目的・スコープ
   - 対象システムの概要（例：建設機械の制御システム AUTOSAR移行プロジェクト Phase 1）
   - 本仕様書が対象とする領域（RTOS系 / Linux系 / 両方）
   - 使用するAUTOSARスキーマバージョン（例：Classic R4.4 / Adaptive R22-11）
     ※委託先ツールチェーンとの整合を取るため、契約時に確定させること。
       バージョンが異なるとArXMLの互換性がなくなり後で大きな問題になる

2. システム構成
   - ECU構成図（何台のECUがあり、どのSWCが動くか）
     例：
       ・RTOS系ECU（Renesas RH850搭載）：油圧制御SWC・ブレーキ制御SWC・診断SWC
       ・Linux系ECU（Arm Cortex-A搭載）：HMI SWC・人検知SWC・クラウド通信SWC
   - ネットワーク構成図（CAN・CAN FD・Ethernetの接続関係・バス速度）
     例：
       ・作業機制御CAN（500kbps）：油圧制御ECU ↔ 作業機コントローラ
       ・車体CAN（250kbps J1939）：エンジンECU ↔ 油圧制御ECU ↔ ゲートウェイECU
       ・Ethernet（100BASE-T1）：ゲートウェイECU ↔ Linux系ECU

3. 機能要求
   ※機能ごとにIDを付けて列挙する。1つの要求には1つの意味だけを書く（AND・ORで複数を混ぜない）

   [FR-001] 油圧制御SWCは10msサイクルで制御値を出力すること
   [FR-002] 人検知SWCは検知結果をCAN経由でRTOS系ECUに通知すること
   [FR-003] ゲートウェイSWCはJ1939バスとCANの間でメッセージ変換を行うこと
   [FR-004] 診断SWCはUDS（ISO 14229）に準拠した診断サービスを提供すること
   [FR-005] HMI SWCはオペレータの入力を受け取り、作業モードの選択を油圧制御SWCに通知すること

4. 非機能要求
   [NFR-001] 油圧制御SWCはISO 25119 PLd相当の安全機能を満たすこと
   [NFR-002] 起動時間はエンジンスタートから制御準備完了まで3秒以内であること
             ※建設機械は始動直後に作業機操作が必要なケースがあるため
   [NFR-003] 外気温 -30℃〜+70℃の動作環境を満たすこと（建設現場の屋外環境を想定）
   [NFR-004] 電源電圧が18Vを下回った場合、制御出力を安全値に移行しECUを保護すること
             ※建設機械はエンジン始動時・大型負荷投入時にバッテリー電圧が大きく落ち込む。
               この要件がないと電圧低下時に誤動作が発生するリスクがある
   [NFR-005] セキュリティ要件：Ethernetインターフェース経由の不正コマンドを拒否すること

5. インターフェース要求
   - SWC間のポート定義（詳細はSWC要求仕様書参照）
   - 外部（センサー・アクチュエータ）とのI/O定義
     例：
       ・アームポジションセンサー（アナログ 0〜5V → IoHwAb経由でSWCに入力）
       ・油圧バルブ指令（PWM出力 → IoHwAb経由でアクチュエータに出力）
   - 外部サプライヤ製ECU（エンジンECU等）とのJ1939インターフェース
     ゲートウェイSWCがJ1939信号をAUTOSARシグナルに変換し、内部SWCに提供すること
     ※エンジンECUのJ1939 PGN定義（エンジン回転数・エンジン負荷率等）は別紙通信仕様書を参照
   - 通信メッセージ一覧（別紙：通信仕様書を参照）

6. 制約条件
   - 使用するマイコン・SoC：RTOS系 Renesas RH850G4KH / Linux系 （型番を記入）
   - 使用するAUTOSARベースソフト（指定がある場合はここに記載。なければ「委託先提案とする」）
   - 対応が必要な規格：ISO 25119（機能安全）・ISO/SAE 21434（セキュリティ）
   - 使用言語：C（Classic AUTOSAR SWC） / C++（Adaptive AUTOSAR SWC）
   - コーディング規約：MISRA-C:2012（Classic）/ MISRA-C++:2008（Adaptive）
```

### 3-2. SWC要求仕様（SWC Requirements Specification）

SWC（Software Component）ごとに作成する。システム要求仕様書で列挙したSWCの数だけ作成する。最初は全SWCを一度に作ろうとせず、PoCの対象SWCから始めるとよい。

```
SWC要求仕様書のテンプレート（記入例：油圧制御SWC）

SWC名：HydraulicControlSWC

1. 概要
   油圧バルブへの制御値を算出し出力するSWC。
   オペレータのジョイスティック入力・システム状態を受け取り、
   アームの動作指令をPWM指令値として油圧バルブに出力する。

2. ポート定義
   ※データを「受け取る」のがRequireポート、「出す」のがProvideポート

   | ポート名         | 種別    | データ型    | 接続先              | 内容・単位                          |
   |-----------------|---------|------------|---------------------|------------------------------------|
   | OperatorInput   | Require | uint16      | OperatorInputSWC   | ジョイスティック入力値（0〜1000、1=0.1%） |
   | ValveCommand    | Provide | uint16      | IOHWAB（アクチュエータ）| バルブ開度指令（0〜1000、1=0.1%）    |
   | SystemStatus    | Require | uint8       | SystemManagerSWC   | 0:正常、1:警告、2:フォルト           |
   | SafetyRequest   | Require | boolean     | SafetyManagerSWC   | TRUE:フェールセーフ動作を要求        |

   ※データ型はApplicationDataType（物理量：単位・レンジ）と
     ImplementationDataType（ECU内の表現型）を分けて委託先に定義させること。
     例：ValveCommandの物理量は 0〜100%、実装型は uint16（係数0.1、オフセット0）
     この分離がないと、「値をそのまま使ったら10倍になった」という設計ミスが起きる

3. Runnable定義
   ※RunnableはSWC内の処理の塊。いつ・どのくらいの頻度で動くかを必ず定義する

   | Runnable名              | 起動条件       | 周期   | 安全レベル | 説明                             |
   |------------------------|--------------|-------|-----------|----------------------------------|
   | HydraulicControl_10ms  | TimingEvent  | 10ms  | PLd相当   | 通常制御計算（入力→バルブ指令算出）|
   | SafetyMonitor_5ms      | TimingEvent  | 5ms   | PLd相当   | 出力値の範囲チェック・フェールセーフ判定|

   ※10ms周期の根拠：油圧バルブの応答周波数を考慮したもの。
     周期を変える場合は油圧系の機構担当者に確認すること

4. 機能要求
   [FR-HYD-001] OperatorInputポートの値（0〜1000）を比例制御アルゴリズムで変換し、
                ValveCommandポートへ出力すること
   [FR-HYD-002] SystemStatusがFault（値=2）またはSafetyRequestがTRUEの場合、
                ValveCommandをフェールセーフ値（0）に固定すること
   [FR-HYD-003] OperatorInputが無効値（範囲外）の場合、ValveCommandを0に固定し、
                SystemManagerSWCへ異常を通知すること

5. 機能安全要求
   [FS-HYD-001] ISO 25119 PLd相当の安全機能を満たすこと
   [FS-HYD-002] ValveCommandが1000（100.0%）を超えた場合、即座に0に戻し、
                フォルトを記録してSystemManagerSWCに通知すること
   [FS-HYD-003] SafetyMonitor_5msが3サイクル以上連続で応答しない場合、
                フェールセーフ動作に移行すること

   ※フェールセーフ時の目標状態（ValveCommand=0）は油圧アーム保持バルブの
     有無によって「安全」かどうかが変わる。
     「バルブ指令0でアームが重力で下降しないか」を機構担当者と必ず確認すること。
     確認が取れたら「ValveCommand=0でアームが安全側（保持）になることを機構設計書[文書番号]で確認済み」
     と注釈に記入する。

6. J1939固有事項（J1939バスに接続するSWCの場合のみ記載）
   - J1939アドレスクレーム処理の責任コンポーネントを明記すること
     （GatewaySWCが担当するか、各SWCが個別に担当するかを決める）
   - 外部エンジンECU等とのJ1939通信はGatewaySWCを経由させること
     （HydraulicControlSWCがJ1939を直接参照する設計は避ける）
```

## 4. 要求の書き方のルール

### 良い要求の4条件

| 条件 | 悪い例 | 良い例 | なぜ良い例の方がよいか |
|---|---|---|---|
| 具体的（数値・条件が明確） | 「速く動作すること」 | 「10msサイクルで制御値を出力すること」 | 「速く」は人によって解釈が異なる。委託先が「100msで十分速い」と判断する可能性がある |
| 検証可能（合否を数値で判断できる） | 「安全であること」 | 「ISO 25119 PLdを満たすこと」 | 「安全か」は主観的で判断できない。規格番号と水準があれば第三者が検証できる |
| 単一の意味（1要求1条件） | 「入力を処理して出力し、異常時は停止すること」 | 「OperatorInputポートの値を受け取りValveCommandポートへ出力すること」と「異常検知時はValveCommandを0に固定すること」を別々の要求として記載する | 1文に複数条件を書くと、テスト時に「どちらの条件でパスか」が曖昧になる |
| 実装を規定しない（HOWでなくWHATを書く） | 「PID制御を使うこと」 | 「オペレータ入力に対してXX ms以内にバルブ指令値を更新すること」 | 実装方法（PIDかどうか）は委託先の判断。自社が決めるのは「何を達成するか」 |

### 要求IDの付け方

要求にはIDを付けて管理する。IDがあることで、「この設計はどの要求を実現しているか」「このテストはどの要求を確認しているか」の対応関係（トレーサビリティ）を追跡できるようになる。

```
IDの形式：[カテゴリ-コンポーネント略称-連番]

[FR-XXX-NNN]   機能要求（何をするか）
[NFR-XXX-NNN]  非機能要求（どのくらいの品質・性能で）
[FS-XXX-NNN]   機能安全要求（安全機能としての要件）
[SEC-XXX-NNN]  セキュリティ要求（セキュリティとしての要件）
[IF-XXX-NNN]   インターフェース要求（他との接続仕様）

XXX：対象コンポーネントの略称
  HYD  = 油圧制御（Hydraulic）
  BRK  = ブレーキ制御（Brake）
  HMI  = HMI表示
  GW   = ゲートウェイ（Gateway）
  DET  = 人検知（Detection）
  SYS  = システム全体

NNN：連番（001〜。欠番が出ても詰めない。削除した要求の番号は再利用しない）

記入例：
  [FR-HYD-001]  油圧制御の機能要求 第1番
  [FS-HYD-002]  油圧制御の機能安全要求 第2番
  [IF-GW-003]   ゲートウェイのインターフェース要求 第3番
```

## 5. AUTOSARのコンフィグとして定義するもの

要求仕様とは別に、委託先が作成するArXMLに含まれる定義がある。自社はこれを**レビューできる状態**を目指す。全てを完全に読み解く必要はなく、「自分が書いた要求が正しく反映されているか」を確認できれば十分である。

| 定義項目 | ArXML上の要素名 | 自社が確認すべき点 | 確認の仕方 |
|---|---|---|---|
| SWCの構造 | ApplicationSoftwareComponentType | ポート名・型が要求仕様と一致しているか | SWC要求仕様書のポート定義表と照合する |
| ポートインターフェース | PortInterface | データ型・方向（Provide/Require）が正しいか | 「ValveCommandはProvide、uint16」のように1行ずつ照合 |
| Runnableの周期 | TimingEvent | 周期が要求通りか（例：10msが10msになっているか） | Runnable定義表と照合する |
| ECUへの配置 | SwcToEcuMapping / EcuInstance | 正しいECUに配置されているか | ECU構成図と照合する |
| 通信マッピング | SystemSignalMapping | CAN/EthのメッセージとSWCのポートが正しく対応しているか | 通信仕様書と照合する |

**ArXMLのレビュー方法：** ArXMLはXML形式のテキストファイルだが、数千〜数万行になることがある。全てをテキストで読む必要はなく、以下の方法で確認する。

- **Vector DaVinci Developer等のビューア**を使うと、ArXMLの内容をGUIで表示・ナビゲートできる。委託先に「DaVinciビューアで確認できる形で中間成果物を共有してほしい」と依頼する。
- **テキストエディタ（VSCode・Notepad++）で要求IDを検索**する。ArXMLのコメントや要素名に要求IDが含まれているかを確認することで、どの要求が設計に反映されているかを追跡できる。
- **差分確認**には、バージョン管理ツール（Git等）または専用の差分ツールを使い、前回版からの変更箇所に絞って確認する。

**AUTOSARスキーマバージョンの注意：** Classic R4.2/R4.3/R4.4・Adaptive R22-11等でArXMLのスキーマ（書式）が異なる。契約時にバージョンを固定し、成果物のバージョンを統一しないと、異なるバージョンのArXMLを組み合わせた際にツールでエラーが発生する。

## 6. 要求仕様作成のステップ（最初から始める場合の手順）

以下のStep 1から順番に進める。各Stepを完了させてから次に進む必要はなく、委託先と並行しながら進めることもできる。

```
Step 1：システム全体の機能一覧を作る（箇条書きで十分）
  ・「この機械は何ができるか」を機能単位でリストアップする
  ・例：「アーム上下・旋回制御」「走行制御」「エンジン始動制御」
        「人検知時の動作停止」「故障コード表示」「遠隔監視通信」
  ・完璧でなくてよい。後で追加・修正しながら進める
  ↓
Step 2：機能ごとにSWCの候補を洗い出す
  ・1機能＝1SWCが基本だが、密接に関連する機能はまとめることもある
  ・「自社が作る機能」と「AUTOSARのBSWが提供する機能」を分ける
  ・例：CAN通信の制御はBSW（CanIf/Com）が担当 → SWCとしては作らない
  ↓
Step 3：SWC間のデータのやり取りをインターフェース図で描く
  ・A3の紙や白板に「どのSWCがどのSWCに何を渡すか」を矢印で描く
  ・この図があることで、ポートの抜け漏れを早期に発見できる
  ・例：OperatorInputSWC → [ジョイスティック値] → HydraulicControlSWC
              HydraulicControlSWC → [バルブ指令] → IoHwAb（ハードウェア）
  ↓
Step 4：各SWCのポート定義・Runnable定義を表形式で記述する
  ・セクション3-2のテンプレートに従い、SWCごとに表を埋める
  ・ポート名・データ型・接続先・単位を明記する
  ・周期（何msごとに動くか）はこの段階で確定させる
  ↓
Step 5：機能安全レベル（HARA結果に基づく）を機能ごとに割り当てる
  ・HARAが未実施の場合は委託先または安全コンサルタントに支援を依頼する
  ・自社では「この機能が誤動作するとどうなるか」のリストを準備しておくと作業が進めやすい
  ・セクション2の目安表を参考にしながら、専門家の意見を求める
  ↓
Step 6：トレーサビリティマトリクスを作成する
  ・要求ID → 設計（ArXMLの要素名または設計書の章番号）→ テスト項目番号 の対応表を作る
  ・Excelで十分。最初は要求IDとテスト項目の対応だけでもよい
  ・例：
    | 要求ID      | 設計要素（ArXML）           | テスト項目     |
    |[FR-HYD-001] | HydraulicControl_10ms      | TC-HYD-001    |
    |[FS-HYD-001] | SafetyMonitor_5ms          | TC-HYD-010    |
  ・変更影響分析・認証審査の際に必須となる
  ↓
Step 7：委託先にレビューしてもらい、抜け漏れ・曖昧さを指摘してもらう
  ・要求仕様書ドラフトをPDF＋Word形式で送付する
  ・「2週間以内に質問/指摘/提案の3列で回答してほしい」と依頼する
  ・委託先からの質問が多い箇所は仕様が曖昧なサイン → 明確化する
  ・「この部分はAUTOSAR規格で決まっているので要求不要」という指摘も受け入れる
```

## 7. 受け入れ基準の定め方

委託先から成果物を受け取る際の判断基準を事前に定めておく。「完成しました」という言葉ではなく、「確認項目全てをパスしたか」で受け入れを判断する。HIL環境が整備されていない場合は、SILテストまたはPCベースシミュレーションによる代替確認方法を委託先と事前に合意しておくこと。

```
受け入れ基準テンプレート（1SWCごとに作成する）

機能：[機能名。例：油圧制御SWC]
要求ID：[FR-HYD-001], [FS-HYD-001], ...（関連する要求IDを全て列挙）

確認方法：
  □ HIL（Hardware-in-the-Loop）テストで動作確認
    ・実際のECU基板にテスト装置をつなぎ、センサー入力をシミュレートして動作確認する
    ※HIL環境未整備の場合：SILテスト（PCシミュレーション）を代替として使用。
      この場合は委託先がテスト結果のレポートを提出すること

  □ Vector CANoeまたはCANalyzerでCAN信号を確認
    ・制御指令が正しいタイミング・値でCAN上に出力されているかをオシロスコープ的に確認
    ・CANoeのテストモジュールで自動テストを実施することを推奨

  □ 設計書とArXMLの整合性確認
    ・要求仕様書のポート定義表 ↔ ArXMLの内容が一致しているかをレビュー会議で確認

  □ 機能安全ドキュメントの納品確認
    ・HARA（ハザード分析・リスク評価）が完了し文書化されているか
    ・FMEA（故障モード影響解析）が作成されているか
    ・安全計画書（Safety Plan）が納品されているか

合格基準：
  - 全確認項目をパスすること（条件付きパスは不可。未解決は持ち越さない）
  - 未解決の不具合がゼロであること
  - 全ての確認項目について結果が記録された「受け入れテストレポート」が提出されていること

納品物チェックリスト（受け取る前に必ず確認する）：
  □ ArXML一式（スキーマバージョンが契約と一致しているか確認）
  □ ツールチェーンプロジェクトファイル（DaVinci等。環境を移行する際に必要）
  □ テスト仕様書（何をどうテストしたかの手順書）
  □ テストレポート（テスト実施結果・不具合対応履歴）
  □ 安全計画書（Safety Plan：機能安全対応のどのステップを実施したかの記録）
  □ トレーサビリティマトリクス（要求ID → 設計 → テスト項目の対応表）
  □ FMEA（故障モード影響解析文書）
  □ ビルド手順書（ソースコードからバイナリを再現できるかの確認に必要）
```

## 付録：よくある間違いと対策

| よくある間違い | 結果 | 対策 |
|---|---|---|
| 「細かいことは委託先に任せる」と言って要求を定義しない | 委託先が独自判断で実装し、後から「思っていたものと違う」が続出する | 最低限ポート定義・Runnable周期・安全レベルは自社で決めてから委託する |
| フェールセーフ動作を「安全値に固定すること」とだけ書く | 「安全値が何か」が機能によって異なるため、委託先が勝手に値を決める | 「ValveCommand=0」「走行モーター出力=0かつブレーキON」のように数値と状態を具体的に書く |
| 要求IDを付けずに書く | 設計・テストとの対応関係が追えなくなり、認証審査・変更管理で困る | 最初からID付きで書く習慣を徹底する |
| 要求仕様書を一度作ったら更新しない | 設計が変わっても要求仕様書が古いままになり、何が正しいか誰もわからなくなる | 変更があるたびに要求仕様書をバージョン管理ツールで更新し、変更履歴を残す |
| 受け入れ基準を事前に決めない | 「できました」と言われたときに何を確認すればよいか分からなくなる | セクション7のテンプレートを使い、委託前に受け入れ基準を合意しておく |
