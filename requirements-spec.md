# AUTOSARを前提とした要求仕様の作り方

**作成日：** 2026年6月16日  
**対象読者：** 自社技術担当者（AUTOSAR知識不要で使えることを想定）  
**資料種別：** 社内運用資料（社外秘）

---

## 概要

外部委託先にAUTOSAR準拠のプラットフォームソフトウェアを依頼するために必要な要求仕様の作り方を定義する。「AUTOSAR側で決まっていること」と「自社が決めなければならないこと」を分離することが基本的な考え方である。

---

## 1. 基本的な考え方

### AUTOSARが決めていること（自社が実装・決定しなくてよいもの）

- BSW各モジュールのAPI・インターフェース仕様
- RTE（Runtime Environment）の動作仕様
- OS（AUTOSAR OS）のスケジューリング仕様
- 通信スタック（COM・PDUR・CANIF等）のレイヤー構造

**補足：** RTEのCコードは委託先がツール（Vector DaVinci等）を用いて自動生成する。ただし、その入力となるArXML（ポート定義・Runnable定義）の内容は自社が要求仕様として承認する必要がある。

### 自社が決めなければならないこと（要求仕様として記述するもの）

- 何のSWC（Software Component：ソフトウェアの機能単位）が必要か
- 各SWCが持つポート（インターフェース）の定義
- どのRunnable（処理単位）を何Hzで動かすか
- どのECUに何を配置するか（ECUマッピング）
- 機能安全レベルの割り当て

---

## 2. 機能安全レベルの考え方

**重要：** 本資料では実装例としてASILという表記を使用しているが、これはISO 26262（自動車向け）由来の概念である。建設機械向けにはISO 25119（農業・建設機械向け機能安全）またはISO 13849（機械安全）のPL（Performance Level）が適用される。

要求仕様作成時は、適用規格とレベルを以下の形で明記すること。

```
例：
[FS-HYD-001] ISO 25119 PLd相当の安全機能を満たすこと
[FS-HYD-001] ISO 26262 ASIL-Bを満たすこと（自動車向け規格を意図的に採用する場合）
```

**ASILレベルの割り当て方法：** ASILはHARA（Hazard Analysis and Risk Assessment：ハザード分析・リスク評価）の結果から導出する。HARAは各機能のハザード（危険事象）を洗い出し、深刻度・暴露確率・制御可能性から安全レベルを決める作業である。HARAの実施は委託先に支援を依頼するか、別途安全コンサルタントに依頼することを推奨する。

---

## 3. 要求仕様の構成

### 3-1. システム要求仕様（System Requirements Specification）

委託先への入力として最初に渡す最上位ドキュメント。

```
1. 目的・スコープ
   - 対象システムの概要
   - 本仕様書が対象とする領域（RTOS系 / Linux系）
   - 使用するAUTOSARスキーマバージョン（例：Classic R4.4 / Adaptive R22-11）
     ※委託先ツールチェーンとの整合を取るため、契約時に確定させること

2. システム構成
   - ECU構成図（何台のECUがあり、何が動くか）
   - ネットワーク構成図（CAN・Ethernet等の接続関係）

3. 機能要求
   - [FR-001] 油圧制御SWCは10msサイクルで制御値を出力すること
   - [FR-002] 人検知SWCは検知結果をCAN経由でRTOS系ECUに通知すること
   - （機能ごとにIDを付けて列挙する）

4. 非機能要求
   - [NFR-001] 油圧制御SWCはISO 25119 PLd相当の安全機能を満たすこと
   - [NFR-002] 起動時間はXX秒以内であること
   - [NFR-003] セキュリティ要件（認証・暗号化等）
   - [NFR-004] 電源電圧がXX Vを下回った場合、制御出力を安全値に移行しECUを保護すること
     ※建設機械は電源環境が過酷（バッテリー電圧変動・負荷投入）なため必須

5. インターフェース要求
   - SWC間のポート定義
   - 外部（センサー・アクチュエータ）とのI/O定義
   - 外部サプライヤ製ECU（エンジンECU等）とのJ1939インターフェースは
     ゲートウェイSWCで変換し、AUTOSARシグナルとして内部に取り込むこと
   - 通信メッセージ一覧（別紙：通信仕様書を参照）

6. 制約条件
   - 使用するマイコン・SoC
   - 使用するAUTOSARベースソフト（指定がある場合）
   - 対応が必要な規格（ISO 25119等）
```

### 3-2. SWC要求仕様（SWC Requirements Specification）

SWC（Software Component）ごとに作成する。

```
SWC名：HydraulicControlSWC

1. 概要
   - 油圧バルブへの制御値を算出し出力するSWC

2. ポート定義
   | ポート名         | 種別    | データ型    | 接続先              |
   |-----------------|---------|------------|---------------------|
   | OperatorInput   | Require | uint16      | OperatorInputSWC   |
   | ValveCommand    | Provide | uint16      | IOHWAB（アクチュエータ）|
   | SystemStatus    | Require | uint8       | SystemManagerSWC   |

   ※データ型はApplicationDataType（物理量：単位・レンジ）と
     ImplementationDataType（ECU内の表現型）を分けて委託先に定義させること。
     例：ValveCommand の物理量は -100〜100%、実装型は uint16（係数0.1、オフセット-100）

3. Runnable定義
   | Runnable名              | 起動条件       | 周期   | 安全レベル |
   |------------------------|--------------|-------|-----------|
   | HydraulicControl_10ms  | TimingEvent  | 10ms  | PLd相当   |

4. 機能要求
   - [FR-HYD-001] オペレータ入力値をXXアルゴリズムで変換してバルブ指令値を出力すること
   - [FR-HYD-002] システムステータスがFaultの場合はバルブ指令値をフェールセーフ値に固定すること

5. 機能安全要求
   - [FS-HYD-001] ISO 25119 PLd相当の安全機能を満たすこと
   - [FS-HYD-002] バルブ指令値の範囲外出力を検出し、フェールセーフ動作に移行すること
     ※フェールセーフ時の目標状態（Hold/Retract/0等）はアクチュエータごとに別紙安全概念書で定義すること。
       「0固定」が安全側になるかどうかは機構設計者と必ず確認すること
       （例：油圧保持バルブがない場合、0指令でアームが重力で下降する可能性がある）

6. J1939固有事項（J1939バスに接続するSWCの場合）
   - J1939アドレスクレーム処理の責任コンポーネントを明記すること
   - 外部エンジンECU等とのJ1939通信はゲートウェイSWCを経由させること
```

## 4. 要求の書き方のルール

### 良い要求の条件

| 条件 | 悪い例 | 良い例 |
|---|---|---|
| 具体的 | 「速く動作すること」 | 「10msサイクルで制御値を出力すること」 |
| 検証可能 | 「安全であること」 | 「ISO 25119 PLdを満たすこと」 |
| 単一の意味 | 「入力を処理して出力すること」 | 「OperatorInputポートの値を受け取り、ValveCommandポートへ出力すること」 |
| 実装を規定しない | 「PID制御を使うこと」 | 「オペレータ入力に対してXX ms以内にバルブ指令値を更新すること」 |

### IDの付け方

```
[FR-XXX-NNN]   機能要求
[NFR-XXX-NNN]  非機能要求
[FS-XXX-NNN]   機能安全要求
[SEC-XXX-NNN]  セキュリティ要求
[IF-XXX-NNN]   インターフェース要求

XXX：対象コンポーネントの略称（HYD=油圧、HMI、GW=ゲートウェイ等）
NNN：連番（001〜）
```

## 5. AUTOSARのコンフィグとして定義するもの

要求仕様とは別に、委託先が作成するArXMLに含まれる定義として以下がある。自社はこれを**レビューできる状態**を目指す。

| 定義項目 | ArXML上の要素 | 自社が確認すべき点 |
|---|---|---|
| SWCの構造 | ApplicationSoftwareComponentType | ポート名・型が要求と一致しているか |
| ポートインターフェース | PortInterface | データ型・方向が正しいか |
| Runnableの周期 | TimingEvent | 周期が要求通りか |
| ECUへの配置 | SwcToEcuMapping / EcuInstance | 正しいECUに配置されているか |
| 通信マッピング | SystemSignalMapping | CAN/Ethのメッセージと正しく対応しているか |

**ArXMLのレビュー方法：** ArXMLはXML形式だが数千〜数万行になることがある。Vector DaVinci等のビューア、またはNotepad++のXPath検索を用いて差分・構造を確認する。テキストエディタで目視する場合は要求IDで検索して関連箇所に絞ること。

**AUTOSARスキーマバージョン：** Classic R4.2/R4.3/R4.4・Adaptive R22-11等でArXMLのスキーマが異なる。契約時にバージョンを固定し、成果物のバージョンを統一すること。

## 6. 要求仕様作成のステップ

```
Step 1：システム全体の機能一覧を作る（箇条書きで十分）
  ↓
Step 2：機能ごとにSWCの候補を洗い出す
  ↓
Step 3：SWC間のデータのやり取りをインターフェース図で描く
  ↓
Step 4：各SWCのポート定義・Runnable定義を表形式で記述する
  ↓
Step 5：機能安全レベル（HARA結果に基づく）を機能ごとに割り当てる
         → HARAが未実施の場合は委託先または安全コンサルタントに支援を依頼
  ↓
Step 6：トレーサビリティマトリクスを作成する
         → 要求ID → 設計（ArXML） → テスト項目の対応を表にする
         → 変更影響分析・認証審査の際に必須
  ↓
Step 7：委託先にレビューしてもらい、抜け漏れ・曖昧さを指摘してもらう
         → 要求仕様ドラフトをPDF+Word形式で送付し、
           2週間以内に「質問/指摘/提案」の3列で回答してもらうよう依頼する
```

## 7. 受け入れ基準の定め方

委託先から成果物を受け取る際の判断基準を事前に定めておく。HIL環境が整備されていない場合は、SILテストまたはPCベースシミュレーションによる代替確認方法を委託先と合意しておくこと。

```
受け入れ基準テンプレート

機能：[機能名]
要求ID：[FR-XXX-NNN]

確認方法：
  □ HIL（Hardware-in-the-Loop）テストで動作確認
    ※HIL環境未整備の場合：SILテスト/PCシミュレーションに代替
  □ ベクターCANoeまたはCANalyzerでCAN信号を確認
  □ 設計書とArXMLの整合性確認
  □ 機能安全ドキュメント（HARA・FMEA・安全計画）の納品確認

合格基準：
  - 全確認項目をパスすること
  - 未解決の不具合がゼロであること

納品物チェックリスト：
  □ ArXML一式（スキーマバージョン確認）
  □ ツールチェーンプロジェクトファイル
  □ テスト仕様書・テストレポート
  □ 安全計画書（Safety Plan）
  □ トレーサビリティマトリクス（要求→設計→テスト）
  □ FMEA（故障モード影響解析）
```
