# 通信仕様（CAN・Ethernet メッセージリスト）の書き方

**作成日：** 2026年6月16日  
**対象読者：** 自社技術担当者・委託先  
**資料種別：** 社内運用資料（社外秘）

---

## 概要

AUTOSARでは通信仕様はArXMLとして定義されるが、その入力として人間が読める形式の通信仕様書が必要である。本ドキュメントでは、AUTOSARのツールチェーンへのインプットとなる通信仕様の書き方を定義する。

**なぜ通信仕様書がツールチェーンのインプットになるか：**  
AUTOSARのBSWコンフィグツール（DaVinci Configurator Pro等）は、DBCファイルやArXMLを読み込んで通信設定（CANIF・COM・PDUR等）を自動構成する。逆に言えば、通信仕様書のフォーマットや記載内容が不完全だと、ツールへのインポートが失敗したり、委託先が独自に解釈して実装するリスクがある。通信仕様書の品質がそのままコードの品質に直結する。

## 通信プロトコル別フォーマット対応表

| プロトコル | 仕様フォーマット（マスター） | テストツール | 備考 |
|---|---|---|---|
| CAN / CAN FD | **DBC**（デファクト） | CANoe（DBC読込） | DBCをマスターとし、ArXMLへインポート後はArXMLがマスターになる |
| J1939 | DBC拡張（J1939属性付き）または独自定義 | CANoe J1939 | PGN/SPN定義をDBC拡張属性で記述するのが一般的 |
| SOME/IP（Ethernet） | **ArXML**（AUTOSARエコシステム）またはFIBEX | CANoe（ArXML/FIBEX読込） | DBCのようなデファクト単一フォーマットは存在しない |

### SOME/IPにデファクト標準がない理由と現実的な方針

CANのDBCと異なり、SOME/IPの通信仕様フォーマットには業界統一のデファクトが存在しない。CANは「フレームのビット配列」という単純な構造のため1種類のフォーマット（DBC）で表現できるのに対し、SOME/IPは「サービス・メソッド・イベント・データ型」という複雑な概念を持つため、複数のフォーマットが競合している。

主な選択肢は以下の3つ。

| フォーマット | 特徴 | 採用場面 |
|---|---|---|
| **ArXML** | AUTOSARツール（DaVinci等）の出力。サービスID・メソッドID・データ型を一元管理 | AUTOSARエコシステム採用時（本プロジェクト推奨） |
| **FIBEX** | ASAMのXMLフォーマット。AUTOSARを使わない場合の中立的な選択肢 | AUTOSAR非採用で複数ツールと連携する場合 |
| **Franca IDL** | COVESAのテキストベースIDL。軽量・可読性高い | vsomeip等のOSSスタックを使うLinux系環境 |

**本プロジェクトの方針：** AUTOSARツールチェーンを前提とするため、**ArXMLをマスター仕様**とする。SOME/IPサービス定義（サービスID・メソッドID・イベントID・データ型）はArXML内のServiceInterface定義として記述し、CANoeでのテストもArXMLを読み込んで実施する。

## 1. 通信仕様書の全体構成

```
通信仕様書
├── ネットワーク構成
│   ├── CANバス構成（バスID・速度・接続ECU・バス用途）
│   └── Ethernetネットワーク構成（VLAN・IPアドレス体系・スイッチ有無）
├── CANメッセージリスト（バス別）
│   ├── メッセージ定義
│   ├── シグナル定義
│   └── CAN ID割り当てルール
├── J1939メッセージリスト（J1939バス使用時）
│   ├── PGN/SPN定義
│   └── J1939固有事項
├── Ethernetサービスリスト
│   ├── SOAPサービス定義（SOME/IP）
│   └── イベント定義
└── ネットワーク管理仕様
    ├── CANnm設定（ノードID・タイムアウト）
    └── バスオフ検出・回復動作
```

## 2. ネットワーク構成

### 2-1. CANバス構成

| バスID | バス名 | 通信速度 | プロトコル | 接続ECU | 用途 |
|---|---|---|---|---|---|
| CAN1 | ControlBus | 500kbps / CAN FD 2Mbps | AUTOSAR COM | VehicleCtrlECU, HydraulicECU | 制御系 |
| CAN2 | BodyBus | 250kbps | J1939 | BodyECU, EngineECU | エンジン・外部機器 |
| CAN3 | DiagBus | 500kbps | ISO 15765（UDS） | All ECU | 診断 |

**CAN ID割り当てルール：**
- J1939バス（CAN2）：29bit拡張ID固定（J1939規格に従う）
- 社内制御バス（CAN1）：11bit標準ID使用
- ID割り当て表は通信仕様担当者が管理し、重複・優先度競合がないことを確認すること

**CAN ID優先度について：**  
CAN IDの値が小さいほど送信優先度が高い（アービトレーション方式）。複数ECUが同時に送信しようとした場合、IDの小さいフレームが優先される。制御系の緊急コマンドは小さいIDを割り当て、テレメトリや診断は大きいIDを割り当てること。

### 2-2. Ethernetネットワーク構成

| 項目 | 設定値 |
|---|---|
| 物理層 | 100BASE-T1（車載Ethernet）またはGigabit Ethernet |
| IPアドレス体系 | 192.168.X.X（要確定） |
| VLAN | VLAN ID：制御系=10、HMI系=20、クラウド通信=30（要確定） |
| スイッチ | 有無・型番（要確定） |

**VLANを使う理由：**  
VLANはEthernetネットワークを論理的に分割する仕組みである。制御系（VLAN10）とHMI系（VLAN20）を分けることで、HMIの通信障害が制御系に波及することを防ぎ、ネットワークセキュリティの境界を設ける。

## 3. CAN通信仕様

### 3-1. メッセージリスト

| Message ID | メッセージ名 | DLC | 送信ECU | 送信周期 | バス |
|---|---|---|---|---|---|
| 0x100 | HydraulicCommand | 8 | VehicleCtrlECU | 10ms | CAN1 |
| 0x101 | HydraulicStatus | 8 | HydraulicECU | 10ms | CAN1 |
| 0x200 | OperatorInput | 8 | BodyECU | 20ms | CAN1 |
| 0x300 | SystemStatus | 4 | VehicleCtrlECU | 100ms | CAN1 |
| 0x7DF | DiagRequest | 8 | DiagTool | Event | CAN3 |

**各列の意味：**
- **Message ID**：CAN IDとも呼ぶ。CANバス上でフレームを一意に識別する番号。16進数表記（0x〜）が一般的
- **メッセージ名**：DBCファイル内でフレームを識別する名前。ツールがArXMLに変換する際のキーになる
- **DLC（Data Length Code）**：データ部のバイト数（0〜8。CAN FDは最大64バイト）。DBCの定義と実際の送信バイト数が一致しないと受信側でシグナルの解釈が壊れる
- **送信ECU**：このメッセージを送信（発行）するECU。「誰が送るか」が明確でないと実装責任が曖昧になる
- **送信周期**：フレームの送信間隔。「Event」は変化時のみ送信することを意味する。周期が短いほどバス負荷が上がる
- **バス**：どのCANバスに流れるか。複数バスが存在するため必ず明示すること

### 3-2. シグナル定義

**例：HydraulicCommand（0x100）**

バイトオーダーは**Intel（リトルエンディアン、@1）**に統一する。  
（Motorola＝ビッグエンディアンの場合は @0。DBCとテーブルで必ず一致させること）

| シグナル名 | 開始ビット | ビット長 | バイトオーダー | 型 | 係数 | オフセット | 最小値 | 最大値 | 単位 | 初期値 | タイムアウト | タイムアウト時フェールセーフ値 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| BoomLiftCmd | 0 | 16 | Intel | uint16 | 0.1 | -100.0 | -100.0 | 100.0 | % | 0 | 50ms | 0（安全値は機構設計確認要） |
| ArmLiftCmd | 16 | 16 | Intel | uint16 | 0.1 | -100.0 | -100.0 | 100.0 | % | 0 | 50ms | 0 |
| BucketCmd | 32 | 16 | Intel | uint16 | 0.1 | -100.0 | -100.0 | 100.0 | % | 0 | 50ms | 0 |
| ControlMode | 48 | 4 | Intel | uint8 | 1 | 0 | 0 | 15 | - | 0 | - | - |
| SafetyFlag | 52 | 2 | Intel | uint8 | 1 | 0 | 0 | 3 | - | 0 | - | - |
| Reserved | 54 | 10 | - | - | - | - | - | - | - | 0 | - | - |

**各列の意味（シグナル定義の読み方）：**

- **シグナル名**：個々のデータ項目の名前。DBCではSG_で始まる行として記述される。受信側SWCはこの名前でデータを参照する
- **開始ビット**：シグナルがCANフレームのデータ部の何ビット目から始まるか（0始まり）。Intelバイトオーダーの場合はLSB（最下位ビット）の位置を指定する
- **ビット長**：このシグナルが何ビットを占めるか。ビット長と型のビット数は一致させること（例：uint16なら16ビット）
- **バイトオーダー（Intel/Motorola）**：バイト列の格納順序。Intel（リトルエンディアン）はLSBが先、Motorola（ビッグエンディアン）はMSBが先。異なるECU間で誤認すると数値が完全に化ける。本プロジェクトはIntelに統一する
- **型**：データの型（uint8, uint16, int16 等）。符号付き（int）か符号なし（uint）かで受信側の解釈が変わる
- **係数（Factor）**：CANフレームに入っている生値（Raw値）に掛けて物理値に変換する倍率。例：係数0.1の場合、Raw値1000 → 物理値100.0（%）
- **オフセット（Offset）**：係数適用後に加算するバイアス値。物理値 = Raw値 × 係数 + オフセット。例：係数0.1・オフセット-100の場合、Raw値0 → 物理値-100.0（%）、Raw値1000 → 0.0（%）、Raw値2000 → 100.0（%）
- **最小値・最大値**：物理値の有効範囲。この範囲を外れた値は受信側で異常値として扱う
- **単位**：物理値の単位（%、degC、kPa、rpm 等）
- **初期値**：ECU起動時・通信開始前のデフォルト値。未定義のまま送信すると受信側が誤動作する可能性がある
- **タイムアウト**：このシグナルを含むメッセージが指定時間内に届かなかった場合にタイムアウトとみなす時間。受信タイムアウトはAUTOSARのCOM Deadlineモニタリング機能で検出する
- **タイムアウト時フェールセーフ値**：タイムアウト検出時に使用する代替値。制御シグナルの場合は必ず「安全な状態」を定義しておくこと

**値の定義（Enum型シグナル）**

```
ControlMode:
  0x0 = Manual
  0x1 = Assisted
  0x2 = Auto
  0xF = Invalid（受信側はこの値を無効として扱い、フェールセーフ動作に移行する）

SafetyFlag:
  0x0 = Normal
  0x1 = Warning
  0x2 = Fault
  0x3 = Reserved
```

**タイムアウト・無効値について：**  
CAN受信タイムアウト（メッセージが一定時間来ない）を検出した場合の動作を必ず定義すること。制御側が「値が来ていない」と判断する方法を定義しないと、委託先が独自に実装してしまうリスクがある。また、AUTOSARのCOM設定でDeadline Monitoringを有効にすれば、タイムアウト検出からフェールセーフ値への切り替えまでを自動で行える。手書きコードで独自実装させないよう、この設定を仕様書に明示すること。

### 3-3. DBC形式での記述（ツール入力用）

**プロジェクト初期はDBCを一次ソースとし、AUTOSARツールへインポート後はArXMLをmasterとして管理する。DBCはArXMLから逆生成で同期させること。**

```dbc
VERSION ""

NS_ :

BS_:

BU_: VehicleCtrlECU HydraulicECU BodyECU

BO_ 256 HydraulicCommand: 8 VehicleCtrlECU
 SG_ BoomLiftCmd : 0|16@1+ (0.1,-100) [-100|100] "%" HydraulicECU
 SG_ ArmLiftCmd : 16|16@1+ (0.1,-100) [-100|100] "%" HydraulicECU
 SG_ BucketCmd : 32|16@1+ (0.1,-100) [-100|100] "%" HydraulicECU
 SG_ ControlMode : 48|4@1+ (1,0) [0|15] "" HydraulicECU
 SG_ SafetyFlag : 52|2@1+ (1,0) [0|3] "" HydraulicECU

BO_ 257 HydraulicStatus: 8 HydraulicECU
 SG_ BoomActualPos : 0|16@1+ (0.1,-100) [-100|100] "%" VehicleCtrlECU
 SG_ ArmActualPos : 16|16@1+ (0.1,-100) [-100|100] "%" VehicleCtrlECU
 SG_ HydraulicTemp : 32|8@1+ (1,-40) [-40|215] "degC" VehicleCtrlECU
 SG_ HydraulicPressure : 40|16@1+ (0.1,0) [0|6553] "kPa" VehicleCtrlECU

BA_DEF_ BO_ "GenMsgCycleTime" INT 0 10000;
BA_ "GenMsgCycleTime" BO_ 256 10;
BA_ "GenMsgCycleTime" BO_ 257 10;
```

**DBC書式の読み方：**

```
SG_ BoomLiftCmd : 0|16@1+ (0.1,-100) [-100|100] "%" HydraulicECU
      │            │  │  │  │    │      │     │    │   └─ 受信ノード名
      │            │  │  │  │    │      │     │    └─ 単位
      │            │  │  │  │    │      └─────┘ 物理値の最小・最大
      │            │  │  │  └────┘ （係数, オフセット）
      │            │  │  └─ @1=Intel, @0=Motorola; +=符号なし, -=符号あり
      │            └──┘ 開始ビット|ビット長
      └─ シグナル名
```

**BA_（Attribute）とは：** DBCのメタデータ拡張機能。標準フォーマットにない情報（送信周期等）をキーバリュー形式で付加する。AUTOSARツールは `GenMsgCycleTime` 属性を読み込んでCOMモジュールの送信周期に自動設定する。

**注：** `@1` はIntelバイトオーダー（リトルエンディアン）、`@0` はMotorolaバイトオーダー（ビッグエンディアン）を示す。テーブルのバイトオーダー定義と必ず一致させること。

## 4. J1939通信仕様

### 4-1. 概要

建設機械では外部エンジンECU・アタッチメント機器との通信にJ1939が使われることが多い。J1939は29bit拡張IDを用い、PGN（Parameter Group Number）でメッセージ種別を、SPN（Suspect Parameter Number）でシグナルを識別する。

**なぜJ1939が建設機械で多いのか：**  
J1939はもともとトラック・農機・建設機械向けにSAE（米国自動車技術者協会）が策定した通信プロトコルである。エンジンECU・変速機ECU・ポンプ制御などの規格が国際標準として定められており、異なるメーカーのECU間でも共通のPGN/SPNでデータをやり取りできる。一方でAUTOSARは主に乗用車向けに発展してきたため、J1939の扱いはAUTOSAR上で追加モジュール（J1939Tp・J1939Nm等）が必要になる。

**PGN・SPNとは何か：**
- **PGN（Parameter Group Number）**：CANのメッセージIDに相当する概念。「どの種類のデータかを表すグループ番号」。例：PGN 0xF004（EEC1）はエンジン制御データグループを意味する
- **SPN（Suspect Parameter Number）**：CANのシグナルに相当する概念。「グループの中のどの項目か」を示す番号。SPN190はエンジン回転数、SPN91はアクセルペダル開度のように、SAE規格で番号と意味が紐付いている
- **SA（Source Address）**：送信ノードのアドレス（0x00〜0xFE）。CANのDBCにおける「送信ECU」に相当するが、J1939では起動時に動的に割り当てる（アドレスクレーム）

### 4-2. J1939メッセージリスト

| PGN | メッセージ名 | SA（送信ノードアドレス） | DA | 送信周期 | 用途 |
|---|---|---|---|---|---|
| 0xF004 (EEC1) | エンジン制御1 | EngineECU (0x00) | 全局 | 10ms | エンジン回転数・トルク |
| 0xFEF1 (CCVS) | 車速 | BodyECU (0x21) | 全局 | 100ms | 車速 |
| 0xFECA (DM1) | アクティブ診断コード | 各ECU | 全局 | Event + 1s | 故障コード通知 |

**DA（Destination Address）が「全局」の意味：**  
J1939では特定の相手にのみ送る「ピアツーピア」と、全ノードに向けて送る「ブロードキャスト」の2種類がある。DA=全局（0xFF）はブロードキャスト送信を意味し、バス上の全ノードがこのPGNを受信できる。

### 4-3. J1939 SPNリスト（例：EEC1）

| SPN | シグナル名 | 開始ビット | ビット長 | 係数 | オフセット | 単位 |
|---|---|---|---|---|---|---|
| 190 | EngineSpeed | 24 | 16 | 0.125 | 0 | rpm |
| 91 | AcceleratorPedalPosition | 8 | 8 | 0.4 | 0 | % |
| 92 | EnginePercentTorque | 16 | 8 | 1 | -125 | % |

**物理値の計算例（EngineSpeed）：**  
Raw値 = 0x1770（16進数）= 6000（10進数）  
物理値 = 6000 × 0.125 = 750 rpm

### 4-5. J1939固有の注意事項

- **アドレスクレーム：** J1939ネットワーク上の各ノードはSA（Source Address）を動的に取得するアドレスクレーム処理が必要。AUTOSARの場合、J1939Nm（J1939 Network Management）モジュールが担当する。アドレスクレームが失敗した場合の動作（別のSAへの切り替え、または通信停止）を仕様に明記すること
- **診断（DM1/DM2）：** 建設機械の外部診断ツールはJ1939 DM1（アクティブ診断コード）を前提とすることが多い。UDSのみでは対応できないケースがあるため、J1939診断が必要か事前に確認すること
- **ゲートウェイ処理：** エンジンECU等の外部サプライヤ製ECUとのJ1939通信は、ゲートウェイSWCでAUTOSARシグナルに変換して内部に取り込む構成にすること
- **Transport Protocol（J1939 TP）：** 8バイトを超えるデータを送る場合はJ1939-21で定義されたTPプロトコルが必要。AUTOSARではJ1939Tpモジュールが担当する。DM1/DM2等の可変長PGNに適用される

## 5. Ethernet（SOME/IP）通信仕様

### 5-1. サービスリスト

| Service ID | サービス名 | 提供ECU | AUTOSAR種別 | プロトコル | 用途 |
|---|---|---|---|---|---|
| 0x0100 | PersonDetectionService | LinuxECU | Adaptive | UDP Multicast | 人検知結果の配信 |
| 0x0101 | CloudTelemetryService | LinuxECU | Adaptive | TCP | クラウドへのデータ送信 |
| 0x0200 | HMIControlService | HmiECU | Adaptive | TCP | HMI制御コマンド |

**注：** SOME/IPはClassic AUTOSARとAdaptive AUTOSARで実装方法が異なる。Classic側でSOME/IPを受信する場合（SomeIpTransformer経由）の仕様も別途定義すること。

### 5-2. SOME/IP Service Discovery（SD）の仕組み

SOME/IPのSDは「サービスを提供するECUが他のECUにサービスを公告し、必要なECUが購読する」という手順を自動で行う仕組みである。CANではCAN IDが固定なので特別な発見処理は不要だが、Ethernet/IPの環境ではサービスのIPアドレス・ポート番号が起動時まで分からないため、このSDが必要になる。

```
SOME/IP-SD の通信フロー（例：人検知サービスの起動〜データ受信）

  LinuxECU（サービス提供側）          VehicleCtrlECU（サービス消費側）
       │                                       │
       │  ①[OfferService] ─────────────────►  │
       │   UDP Multicast 239.255.0.1:30490      │
       │   「PersonDetectionService(0x0100)を    │
       │    192.168.10.5:50100 で提供中です」   │
       │                                       │
       │  ◄──────── ②[FindService] ────────── │
       │   （任意。サービスを探したい場合）      │
       │                                       │
       │  ①が届いていれば FindService不要       │
       │                                       │
       │  ◄────── ③[SubscribeEventgroup] ──── │
       │   「EventGroup 0x0001 を購読します」   │
       │                                       │
       │  ④[SubscribeEventgroupAck] ─────────► │
       │   「購読を受理しました」                │
       │                                       │
       │  ⑤[Event: PersonDetected(0x8001)] ──► │
       │   UDP Multicast or Unicast             │
       │   ペイロード：DetectionStatus=1 等     │
       │                                       │
       │  ⑤を検知状態変化時 or 1000ms周期で繰り返し送信
       │                                       │
  ～～～～～～～～～～～～ 通常運転中 ～～～～～～～～～～～～～
       │                                       │
       │  ⑥[StopOfferService] ───────────────► │
       │   （LinuxECUシャットダウン時）         │
       │                                       │

  ポート：
    SD通信     → UDP 30490（SOME/IP-SD標準ポート。全ECUがこのポートを監視）
    イベントデータ → UDP 50100（サービスごとに割り当て）

  SDのタイムアウトパラメータ：
    InitialDelayMin / Max : 最初のOfferを送るまでの待ち時間（ランダム化して衝突を防ぐ）
    RepetitionsBaseDelay  : Offer送信の繰り返し間隔
    CyclicOfferDelay      : 定常状態でのOffer送信間隔（消費側のタイムアウト検出用）
    TTL（Time To Live）   : Offer/Subscribeの有効期限（秒）。切れると再ネゴシエーション
```

**SDの実装上の注意点：**
- SDパラメータ（InitialDelay・CyclicOfferDelay・TTL等）はAUTOSARのSDモジュールの設定に直接反映される。仕様書に数値を明記しないと委託先がデフォルト値で実装する
- 消費側はCyclicOfferDelayの2〜3倍をSD受信タイムアウトとして設定し、タイムアウト時の動作（エラーフラグ立て・フェールセーフ移行等）を定義すること
- VLANを使う構成ではSDのマルチキャストパケットがVLAN境界を越えないよう、スイッチのマルチキャストフィルタ設定が必要

## 6. ネットワーク管理（NM）仕様

### 6-1. CAN Network Management（CANnm）

| 設定項目 | 設定値 |
|---|---|
| ノードID | ECUごとに割り当て（0x01〜0xFE、要確定） |
| NMタイムアウト | XX ms（要確定） |
| バスオフ検出方式 | AUTOSARネイティブ（バスオフカウンタ方式） |
| バスオフ後の自動回復 | 最大3回、回復間隔：100ms |
| 3回回復後も復帰しない場合 | 上位レイヤー（システムマネージャ）へ通知しフェールセーフへ移行 |

**CANnmの役割：**  
CANnmはCANバス上の全ノードが協調して「バス全体を起こす・寝かせる」制御を行うためのプロトコルである。単純にCANフレームを送受信するだけでは、エンジン停止後にどのECUがいつスリープに入るかがECUごとにバラバラになり、他のECUがまだ通信中なのにスリープするECUが出てしまう。CANnmを使うことで全ノードが合意してから安全にスリープに移行できる。

**バスオフとは：**  
CANバスのエラーが連続して発生すると、CANコントローラがバスから自分を切断する「バスオフ」状態になる。バスオフになるとそのECUはCANフレームを送受信できなくなる。配線断線・終端抵抗不足・ノイズ等が原因。AUTOSARではバスオフ検出後に自動回復シーケンスを定義できる。

### 6-2. スリープ・ウェイクアップ制御

建設機械ではエンジン停止後のECUスリープと、キースイッチONによるウェイクアップ処理が必要。

```
スリープシーケンス：
  エンジン停止信号受信 → XX秒後にNMスリープ要求送出 → 全ECU応答確認 → スリープ

ウェイクアップシーケンス：
  キースイッチON（またはWake-on-CAN） → NMウェイクアップ → 全ECU起動確認 → 通常動作
```

**スリープシーケンスで全ECU応答確認が必要な理由：**  
一部のECUがまだデータの書き込み（例：EEPROM保存・不揮発メモリ更新）中にスリープ要求で電源を落とされると、データが壊れる可能性がある。CANnmの「PrepSleepInd」によりすべてのノードが「スリープ可能」と応答してから電源を制御する設計にすること。

## 7. 通信仕様書作成・管理のルール

### ツール管理

通信仕様をExcelのみで管理すると、ビット割り付けミス・DBC変換ミス・整合性崩壊が起きやすい。

**推奨管理方法：**
- プロジェクト初期：DBCを一次ソースとして作成し、ツールへインポート
- AUTOSARツール導入後：ArXMLをmasterとし、DBCは逆生成で同期
- Excelは閲覧・共有用の派生物として扱う

**ExcelとDBCの二重管理を避けるべき理由：**  
Excelで仕様を変更してDBCを手修正し、AUTOSARのArXMLと3つが並存する状態になると、どれが最新か分からなくなる。整合性確認の工数が増大し、変換ミスも発生しやすい。「ArXMLが唯一のマスター」というルールを委託先も含めて全員が守ることが重要。

### バージョン管理

- 通信仕様（DBC・ArXML）はgitで管理する
- メッセージ・シグナルの追加・変更はコミットログに理由を記載する
- 変更の承認者（自社担当者名）を決め、委託先への通知フローを確立する
- 量産に近づく時期（XX週前）以降の変更はCCB（変更管理委員会）の承認を要するルールを設ける

### バス負荷率の確認

変更のたびにバス負荷率を確認する。

```
バス負荷率 ≈ Σ（フレームビット長 × 送信頻度） ÷ バス帯域[bps]

目安：
  CAN（500kbps）：60%以下を維持
  CAN FD（2Mbps）：70%以下を目安
  Ethernet：物理帯域に余裕があっても SOME/IP SDの帯域消費・遅延に注意

確認ツール：Vector CANoeのBus Statistics機能が最も手軽

バス負荷率の計算例（CAN 500kbps, DLC=8バイト, 標準ID）：
  1フレームのビット長 ≈ 111ビット（オーバーヘッド含む）
  10ms周期メッセージの帯域 = 111 ÷ 0.010s = 11,100 bps
  500kbpsバスへの負荷 = 11,100 ÷ 500,000 × 100 ≒ 2.2%
  → 10msメッセージを約27本追加するとバス負荷60%に到達する計算。
     メッセージを増やすときは必ず集計すること。
```
