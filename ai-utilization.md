# 生成AI・AIエージェント活用ガイド

建設機械向けAUTOSARプラットフォーム開発・SDV化における生成AI／AIエージェントの活用方針をまとめる。

---

## 1. 生成AI・AIエージェントとは

| 区分 | 説明 | 代表例 |
|------|------|--------|
| LLM | 単発の質問応答・テキスト生成 | GPT-4o、Claude 3.7 Sonnet |
| コーディング支援 | エディタ補完・コード提案 | GitHub Copilot、Cursor |
| AIエージェント | ツール呼び出し・反復実行・自律タスク完遂 | Claude Code、Devin |

**コーディング支援の限界**：ファイル単位の補完は得意だが、リポジトリ横断の依存関係把握や複数ファイルの同時変更は苦手。変更の「意図」を理解せずに補完するため、AUTOSAR構成のような相互依存が多い環境では誤った提案を受け入れるリスクがある。

**AIエージェントが変える点**：ツール（ファイル読み書き・コマンド実行・Web検索）を自律的に呼び出し、複数ステップのタスクを人間の指示なしに完遂できる。例えばArXMLの差分確認 → 影響範囲の特定 → テスト仕様の更新 という一連の作業を単一の指示で実行できる。

## 2. AUTOSAR・車載ソフトウェア開発での活用領域

### 要求仕様作成支援

**課題**：要求仕様書の品質が低いと、委託先との認識齟齬・手戻りが発生する。「〜を考慮すること」「適切に対処する」といった曖昧表現は検証基準を持てない。

**AIでできること**

- 仕様書の曖昧表現を自動検出し、EARS記法（`The system shall…`形式）への変換案を提示
- 要求間の矛盾・抜け漏れをチェックするプロンプトをCI組み込み可能

**プロンプト例**：

```
以下の要求仕様を読んで、
①曖昧な表現（主語なし・定量基準なし・条件未定義）をリストアップ
②それぞれをEARS記法（"The <system> shall <function> <object>"形式）に書き直す
③相互に矛盾する要求があれば指摘する

--- 要求仕様 ---
[要求仕様文書のテキストをここに貼る]
```

**期待される出力例**：「要求 REQ-003『適切なタイムアウト処理を実施する』は定量基準がない。提案：The hydraulic control module shall transition to safe state within 100 ms when communication timeout is detected.」

---

### ArXML差分レビュー支援

**課題**：ArXMLはXML構造が深く、`git diff` の生出力を人間が読んで変更の意味を把握するのに時間がかかる。見落としも発生しやすい。

**AIでできること**

- `git diff` で取得したArXML差分をLLMに渡し、変更点を自然言語で説明
- 変更が元の要求仕様と整合しているかを問い合わせ形式で確認

**操作例**：

```bash
# 差分をテキストに書き出す
git diff HEAD~1 HEAD -- config/Ecu.arxml > arxml_diff.txt

# Claude Code に読ませる
cat arxml_diff.txt | claude "このArXML差分を読んで、
①変更されたBSWモジュールとパラメータを箇条書きで列挙
②変更によって影響を受けそうな他のモジュールを推定
③要求仕様REQ-012（下記）との整合性を評価してください
[REQ-012の内容]"
```

**期待される出力例**：「ComStack の ComMainFunctionTx の周期が10ms → 5msに変更されています。これはOsTaskの実行周期との整合が必要です。REQ-012で要求されている通信レイテンシ要件を満たすための変更と解釈できますが、OsTask側の変更が伴っていないため確認が必要です。」

---

### コードレビュー支援

**課題**：MISRA-C違反は静的解析ツールが検出できても、「なぜルールが存在するか」「この文脈での危険性」の説明に時間がかかる。レビュアーの力量に品質が左右される。

**AIでできること**

- MISRA-C違反箇所の意味説明と修正案の生成（最終判断は人間）
- コーディング規約チェックの前処理として活用し、レビュー工数を削減
- 機能安全観点（フェールセーフ・デッドロック・スタックオーバーフロー）でのコードコメント生成

**プロンプト例**：

```
以下のC言語コードをレビューしてください。
観点：①MISRA-C 2012 違反の可能性、②フェールセーフ設計上の問題、③境界条件・オーバーフローリスク
違反箇所はルール番号と理由を添えて指摘し、修正案のコードを提示してください。
最終承認は人間が行うため、確信が持てない箇所はその旨を明記してください。

--- コード ---
[コードをここに貼る]
```

### テストケース自動生成

**課題**：インタフェース仕様書から手動でテストケースを起こすのは工数がかかる。境界値・異常系の網羅が属人的になりやすい。

**AIでできること**

- インタフェース仕様書を入力にRobot Frameworkのテストケース初稿を生成
- 境界値・異常系の網羅漏れ指摘に有効

**プロンプト例**：

```
以下のAUTOSARポートインタフェース仕様を読んで、
Robot Framework形式のテストケースを生成してください。
必ず含めること：①正常系、②上下限境界値、③異常入力（NULL・範囲外・タイムアウト）
各テストケースには期待値とPass/Fail判定基準を記述してください。

--- インタフェース仕様 ---
[仕様書のテキストをここに貼る]
```

**期待される出力例**：Robot Framework のキーワード形式（`*** Test Cases ***` ブロック付き）でテストケース初稿が生成される。エンジニアはこの初稿を実際のHIL環境変数名に合わせて修正するだけでよい。

---

### ドキュメント生成

**課題**：設計書・議事録の作成は時間がかかる割に価値創造に直結しない。フォーマット合わせ・用語統一の作業が特に非効率。

**AIでできること**

- 設計書初稿の生成（テンプレートと実装コードを同時に渡す）
- 会議メモからアクションアイテム・決定事項・課題を構造化
- 英語/日本語の相互変換・用語統一（グロッサリとの突き合わせ）

**プロンプト例（議事録構造化）**：

```
以下の会議メモを読んで、次の形式で整理してください。
①決定事項（番号付き）、②アクションアイテム（担当者・期限付き）、③継続検討事項、④次回アジェンダ案

--- 会議メモ ---
[テキストをここに貼る]
```

## 3. 建設機械開発固有の活用例

建設機械はAUTOMOTIVE向けとは異なる固有の技術・環境条件を持つ。以下は特に効果が高い活用パターンを具体的なインプット・アウトプットとともに示す。

---

### 活用例1：J1939 PGN/SPN辞書の即時参照と変換スクリプト生成

**場面**：CAN通信仕様の確認・実装

**インプット**：「PGN 0xF004（Electronic Engine Controller 1）のSPN 190（Engine Speed）の信号定義、スケーリング、受信後のエンジン回転数計算式を教えてください。受信データからrpm値を計算するCコードも生成してください。」

**アウトプット**：J1939規格に基づくSPN定義（解像度0.125rpm/bit、オフセット0、範囲0〜8031.875rpm）と、8バイトのCANデータからエンジン回転数を取り出す計算コード。変換テーブルや単位変換の実装も含む。

**効果**：規格書を開かずに仕様確認 + 実装の初稿が同時に得られる。ただしJ1939の数値は出典確認が必須（後述の注意点参照）。

---

### 活用例2：DBCファイルとArXMLの整合性チェックスクリプト生成

**場面**：通信仕様バリデーション

**インプット**：CANデータベース（DBCファイル）とAutosar ComStackの設定（ArXML）を同時に渡し、「信号名・ビット長・バイトオーダーが両方で一致しているか確認するPythonスクリプトを生成して実行してください。不一致があればCSV形式で出力してください。」

**アウトプット**：DBCパーサ（python-can/cantools）とArXML解析を組み合わせたスクリプト。実行すると不整合一覧がCSVで出力される。

**効果**：手動照合では見落としが発生しやすい数百信号の一括チェックを自動化できる。CI組み込みで変更のたびに自動検証が可能。

---

### 活用例3：油圧制御コードのフェールセーフレビュー

**場面**：油圧アクチュエータ制御コードのレビュー

**インプット**：油圧バルブ制御のCコード（200〜400行程度）を渡し、「建設機械の安全要件（ISO 25119 AgPL c相当）の観点から、①センサ故障時のフェールセーフ処理が適切か、②スプール弁の急開閉による油圧サージへの対処があるか、③二重系センサの診断ロジックに論理的な誤りがないかをレビューしてください。」

**アウトプット**：各観点での指摘と修正案のコード。「センサ値がXms以上更新されない場合のデフォルト出力設定が見当たらない」「バルブ開度変化レートに上限がない」などの具体的指摘。

**留意点**：AIの出力はあくまでレビューコメントの初稿。安全要件適合の最終判断は機能安全エキスパートが行う。

---

### 活用例4：HARA・FMEA初稿生成

**場面**：機能安全文書の作成（ISO 25119 / ISO 26262）

**インプット**：「掘削アームの電動アクチュエータ制御システムについてHARAを作成してください。対象機能：アーム角度制御、アーム速度制限。ハザードのトリガー：センサ故障・通信途絶・制御ユニット電源瞬断。ISO 25119のAgPL評価も含めてください。」

**アウトプット**：ハザード一覧（ハザードID・ハザード説明・状況・重篤度・暴露頻度・制御可能性・AgPL）の表形式初稿。各ハザードに対する安全目標案も含む。

**効果**：ゼロから表を作る工数を大幅削減。エキスパートは内容の精査・加筆に集中できる。

---

### 活用例5：委託先へのレビューコメントの日英変換・整形

**場面**：海外委託先・海外サプライヤとのドキュメントやり取り

**インプット**：日本語で書いたレビューコメント（技術用語・AUTOSAR固有語を含む）を渡し、「AUTOSAR技術用語を英語の標準表記に統一しながら、英語のレビューコメント形式（QuestID・Severity・Comment・Proposed Resolution）に変換してください。」

**アウトプット**：AUTOSAR公式表記（`SwcInternalBehavior` など）を使った英語のコメント票。日本語の技術表現が正確に英訳されているかレビュー担当者が最終確認する。

**効果**：英語が得意でないエンジニアでも仕様レビューに参加しやすくなる。翻訳の品質を均質化できる。

---

### 活用例6：OTAキャンペーンのリスク評価支援

**場面**：フィールド機械へのファームウェア更新計画策定

**インプット**：「対象機体数500台、うち100台は山岳現場で電波品質が不安定。更新ファイルサイズ120MB。更新所要時間推定45分。以下のOTAキャンペーン計画のリスクポイントと軽減策を列挙してください。」

**アウトプット**：更新失敗時のロールバック機構の有無・電波途絶時の中断/再開設計・更新中の機能制限設計・稼働中機体への配信タイミングなど、見落としやすいリスク観点と対策案の一覧。

## 4. AIエージェントの実践的活用（Claude Code等）

### 基本的なエージェント活用パターン

- **コードベース調査**：`grep` や `find` を自律呼び出しし、特定シグナルの全参照箇所を即座に報告。例：「ENGINE_SPEED が参照されているファイルをすべて調べて、各ファイルでの使用目的をまとめてください」
- **ドキュメント自動更新**：コード変更に連動してREADMEや設計書の該当箇所を更新。例：「この関数のインタフェースが変わったので、doc/interface-spec.md の対応する節を更新してください」
- **CIパイプライン構築支援**：GitHub Actionsワークフローのひな形生成・デバッグ
- **ArXMLスクリプト自動生成**：「BSWモジュールXのタイムアウト値を一括変更したい」という指示からPythonスクリプトを生成・実行

### 並行エージェントによる高速化

複数のAIエージェントを同時並行で走らせることで、直列作業では数日かかる調査・文書化を数時間に短縮できる。

**活用例**：
- エージェントAが既存コードの依存関係を調査している間に、エージェントBが要求仕様書の曖昧表現を抽出する
- エージェントCがテストケース初稿を生成している間に、エージェントDが設計書の更新箇所を特定する

Claude Code では `--dangerously-skip-permissions` フラグを使わずに複数セッションを並行起動し、それぞれに独立したタスクを割り当てるパターンが実用的。

### 本リポジトリでの活用実績

本リポジトリの立ち上げ時に、以下の作業でClaude Codeを活用した。

| 作業 | 活用内容 | 効果 |
|------|---------|------|
| ディレクトリ構成の初期スキャフォールディング | 「AUTOSAR Classic＋Adaptive＋SDV化対応の構成にして」という指示で初稿を生成。既存プロジェクトの構成を参考にしながら反復的に修正 | ゼロから考える時間を削減し、議論のたたき台を即座に用意できた |
| ai-utilization.md の初稿生成 | 本ドキュメントの初稿をAIエージェントで生成後、担当者がレビュー・加筆 | ドキュメントのゼロベース作成が不要になり、「何を書くか」から「どう精度を上げるか」に集中できた |
| roadmap.md のフェーズ構成設計 | 他社AUTOSARロードマップの公開情報を参照させながら、建設機械固有の条件（ISO 25119、非常時安全要件等）を反映したフェーズ案を生成 | 複数のドキュメントを横断した整合性チェックをAIが担当し、人間のレビュー工数を削減 |
| MISRA-Cルール説明資料の下書き | 社内向け勉強会資料の初稿をAIが生成し、社内規格担当者が修正 | 規格書の読み込みと平易な説明への変換をAIが先行実施 |

## 5. 活用上の注意点・限界

### リスクと対処の一覧

| リスク | 対処 |
|--------|------|
| **機能安全の証跡にならない** | AI出力はドラフト。ISO 26262・ISO 25119の作業成果物はエキスパートが作成・承認する（詳細は後述） |
| **機密情報の漏洩** | 社内システムはオンプレLLMまたは契約上のデータ保護が担保された商用APIのみ使用 |
| **ハルシネーション** | 事実確認が必要な数値（PGN番号・規格条項等）は必ずソースを参照して検証 |
| **著作権** | 生成コードのライセンスを確認。MISRAルールセット等の再配布には注意 |
| **過信による手戻り** | AIの出力を「正しいもの」として採用してしまい、後工程でミスが発覚するリスク。必ずレビューステップを設ける |

### 機能安全証跡としてのAI出力の扱い（重要）

機能安全規格（ISO 26262・ISO 25119）では、安全要求の導出・HARA・FMEA・設計根拠などの「作業成果物（Work Product）」は、資格を持つ担当者が責任を持って作成・承認することが求められる。**AIが生成した内容をそのまま作業成果物として提出することはできない。**

具体的な運用ルールを以下に示す。

**AI出力を使ってよい場面**：
- HARA・FMEAのドラフト作成（エキスパートが内容を精査・加筆・承認する）
- レビューコメントのリストアップ（担当者が判断して採否を決める）
- テストケースの初稿（テスト担当者がカバレッジ・妥当性を確認して確定する）

**AI出力をそのまま使ってはいけない場面**：
- 安全目標（Safety Goal）の最終文書
- ASIL/AgPL判定の根拠（規格の正確な理解が必要）
- テスト合否判定の最終判断
- ソフトウェアコンポーネントの型式証明文書

**証跡管理上の実務ポイント**：
- AIを使って生成した場合でも、提出ドキュメントに「AI生成」と明記する必要はない（現時点では規格上の要件なし）が、内容の正確性・承認責任は人間の担当者が負う
- 社内のプロセス規定に「AIツール使用時の承認フロー」を明文化しておくことを推奨
- 将来的に機能安全の認証審査が入る場合、審査機関がAI活用の範囲について質問するケースが増えている。対応方針を事前に整理しておく

## 6. 今後の展望

- **自律化開発**：AIがテストを書き、CIを回し、失敗原因を分析して修正PRを出す。人間はアーキテクチャ判断と機能安全の承認に専念する形態へ
- **デジタルツイン×AI**：油圧・電気系シミュレーションにAIが参数最適化ループを組み込み、検証サイクルを短縮。フィールドデータとシミュレーションを継続的に照合することで、摩耗・故障の予兆検知に活用できる
- **自律施工への寄与**：建設機械の自律施工には高信頼な車載ソフトウェアが前提。SDV化で得たAUTOSARアーキテクチャがAI制御層（認識・判断・制御）の足場になる
- **組織的問い**：AIは「代替」ではなく「協業者」。「何をAIに任せ、何を人間が判断するか」の役割定義が開発組織の競争力を左右する。特に機能安全領域では、AIを使う前提でのプロセス設計が今後の標準になる可能性が高い
