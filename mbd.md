## 1. MBDとは

### 定義と目的

MBD（Model-Based Development / モデルベース開発）は、ソフトウェアの制御ロジックをグラフィカルなモデルで記述し、そのモデルから自動的にCコードを生成する開発手法である。

従来の手書きコード開発では「仕様書→コーディング→テスト」というウォーターフォール的な流れをたどるが、MBDでは「モデル（仕様と実装を兼ねる）→シミュレーション→コード生成→テスト」という流れになる。モデル自体が仕様書・設計書・実装の三役を兼ねるため、ドキュメントとコードの乖離が構造的に起きにくい。

### 従来の手書きコード開発との比較

| 観点 | 手書きコード | MBD |
|---|---|---|
| 仕様からの変換 | 人手で翻訳（バグ混入リスクあり） | モデルがそのまま動作仕様になる |
| 制御アルゴリズムの確認 | 実機またはテストコード | シミュレーションで早期確認 |
| リグレッションテスト | テストコードの別途作成が必要 | MIL/SILで自動化しやすい |
| トレーサビリティ | 手動で仕様書↔コードを紐づけ | ツールが自動管理（要確定） |
| コードサイズ | 最適化しやすい | 自動生成のため大きくなりがち |
| 開発者スキル | C言語能力が中心 | 制御工学＋ツール操作能力が必要 |
| ライセンスコスト | 低い（IDEのみ） | 高い（Simulink等の年間ライセンス） |

### V字モデルとMBDの関係

MBDはV字モデルの左辺（設計フェーズ）と右辺（検証フェーズ）の両方に関与する。

```
要求定義
  │
  ▼
システム設計 ─────────────────────────── システム検証（HIL）
  │                                            ▲
  ▼                                            │
ソフトウェア要求 ────────────────── SWテスト（SIL / PIL）
  │                                            ▲
  ▼                                            │
ソフトウェア設計（モデリング）──── ユニットテスト（MIL）
  │                                            ▲
  ▼                                            │
コーディング（自動コード生成）────────────────┘
```

- **MIL（Model-in-the-Loop）**：設計モデルのシミュレーション検証（左辺の設計段階）
- **SIL（Software-in-the-Loop）**：生成コードのPC上での検証
- **PIL（Processor-in-the-Loop）**：ターゲットプロセッサ上での検証
- **HIL（Hardware-in-the-Loop）**：実ECU＋環境シミュレータでの統合検証

### なぜ自動車業界でMBDが普及したか

1. **制御の複雑化**：パワートレイン制御（エンジン・トランスミッション）は制御パラメータが数千を超え、手書きコードでの管理が限界に達した
2. **機能安全対応**：ISO 26262が要求するトレーサビリティ・カバレッジ証跡をMBDは効率的に生成できる
3. **Tier1の先行投資**：ボッシュ・デンソー等がMBDを内製ツールとして早期に採用し、業界標準化が進んだ
4. **AUTOSAR対応**：Classic AUTOSARのSWC（ソフトウェアコンポーネント）の概念とSimulinkモデルの対応が取りやすく、コード生成からArXML出力まで一気通貫できる環境が整備された

## 2. MBDのツールエコシステム

### MATLAB/Simulink（MathWorks）

自動車・航空・産業機械分野で最も広く使われるMBDツール群。

#### Simulink

- ブロック線図をグラフィカルに組み合わせてシステムを記述する環境
- 車載での主な用途：
  - **制御ロジックモデル**：フィードバック制御、PID、状態遷移など
  - **プラントモデル**：エンジン、油圧、車体ダイナミクスなど制御対象のモデル化
  - **環境モデル**：路面、負荷、温度など外部環境のモデル化

#### Stateflow

Simulinkと統合された状態機械（ステートマシン）モデリング環境。モード管理、フォールトハンドリング、シーケンス制御の記述に適している。複雑な条件分岐をif-elseの羅列ではなく、状態遷移図として視覚的に表現できる。

#### Simulink Coder / Embedded Coder

- **Simulink Coder**：SimulinkモデルからC/C++コードを生成する基本ツール
- **Embedded Coder**：組み込みターゲット向けに最適化された生成コードを出力。Simulink Coderのスーパーセット
  - **MISRA-C対応**：Embedded CoderはMISRA-C:2012への準拠オプションを持つ（要設定、100%保証ではない点に注意）
  - **AUTOSARターゲット**：Classic AUTOSARのSWCとして直接コード生成が可能（AUTOSAR Blocksetと組み合わせて使用）
  - **汎用組み込みCターゲット**：AUTOSAR非対応の組み込みシステムへも対応

#### Simulink Test

MILおよびSILテストの自動化ツール。テストケースをモデルベースで記述し、シミュレーション結果とリファレンス信号を比較することで回帰テストを自動化できる。

#### Polyspace

生成Cコードの静的解析ツール。ランタイムエラー（配列の範囲外アクセス、ゼロ除算等）の検出と、MISRA-Cルール違反の自動チェックを行う。機能安全の証跡作成に活用される。

#### Model Advisor

Simulinkモデルに対してモデリングルールのチェックを自動実行するツール。MAAB（MathWorks Automotive Advisory Board）ガイドラインやAUTOSARモデリング規約への適合を確認できる。

#### AUTOSAR Blockset

SimulinkとClassic AUTOSARを統合するツールボックス。

- SimulinkモデルをAUTOSAR SWCとして記述するためのAPIとブロックを提供
- ArXMLの入出力（既存のArXMLをインポートしてモデルと紐づけ、またはモデルからArXMLを生成）
- RTE（Runtime Environment）のAPIをSimulinkブロックとして利用可能

#### 料金形態

ライセンスは年間サブスクリプション形式（要確定）。Simulink本体に加え、Embedded Coder、Simulink Test、Polyspace、AUTOSAR Blocksetはそれぞれ別ライセンスが必要。小規模チームでも数百万円/年規模になることが多く、ライセンスコストは導入判断の重要要素になる。

---

### MATLAB/Simulink以外の選択肢

#### SCADE Suite（Ansys）

- 航空（DO-178C）・鉄道（EN 50128）での認証実績が豊富
- 同期データフロー言語「SCADE」を使用しており、形式的な数学基盤を持つ
- コード生成器（KCG）がDO-330 TQL-1認証を取得しており、「生成コードの検証を省略できる」レベルの信頼性を持つ
- Simulinkより安全性・形式検証を重視した設計思想
- 自動車業界ではSimulinkほど普及していないが、ASIL-D相当の高安全要件で採用される

#### TargetLink（dSPACE）

- Simulinkと統合されたコード生成ツール（Simulinkに置き換わるのではなく、Simulinkの上位レイヤとして動作）
- Classic AUTOSARへの対応が強く、Tier1での採用が多い
- dSPACEのHILシステム（SCALEXIO等）との連携がシームレス
- 生成コードの品質と効率性がEmbedded Coderより高いとされる（要確定）

#### ASCET（ETAS/Bosch）

- ETASはボッシュグループのツール会社
- Classic AUTOSARのSWC開発に特化した設計環境
- ボッシュ系サプライヤや欧州Tier1での採用事例あり
- RhapsodyやSimulinkと組み合わせて使われることもある

#### IBM Rhapsody

- UML/SysMLベースのモデルベース開発環境
- 組み込みリアルタイムシステム向けのコード生成に対応
- 制御ロジックよりもシステムアーキテクチャ設計・状態機械モデリングに強み
- Simulinkとの役割分担：Rhapsodyでアーキテクチャ設計→Simulinkで制御アルゴリズム、という組み合わせも存在する

#### Modelica / OpenModelica

- 物理モデリングのための方程式ベース言語
- 機械・油圧・電気・熱など複数の物理ドメインを統合的に記述できる
- プラントモデル（制御対象の物理モデル）の記述に適しており、制御ロジックモデルとは役割が異なる
- OpenModelicaはオープンソース版（無料）
- SimulinkのSimscapeと競合する領域

## 3. MBDのワークフロー詳細

全体の流れを示す。

```
要求仕様
  │
  ▼
┌─────────────────────────────┐
│  モデリング（Simulink等）     │
│  ・制御ロジックモデル         │
│  ・プラントモデル             │
└─────────────┬───────────────┘
              │
              ▼
         MIL検証
    （ホストPC上のシミュレーション）
              │
              ▼
┌─────────────────────────────┐
│  自動コード生成               │
│  （Embedded Coder等）        │
└─────────────┬───────────────┘
              │
              ▼
         SIL検証
    （生成コードをPC上で実行）
              │
              ▼
         PIL検証
    （ターゲットプロセッサ上で実行）
              │
              ▼
         HIL検証
    （実ECU＋HWシミュレータ）
              │
              ▼
         実機評価
```

### MIL（Model-in-the-Loop）

**目的**：制御モデルの動作をホストPC上でシミュレーションし、制御アルゴリズムの妥当性を早期に確認する。

**実施内容**：
- 制御モデルとプラントモデルを閉ループ接続してシミュレーション
- ステップ応答・正弦波応答・フォールト注入等のシナリオを実行
- 制御目標（応答速度・定常偏差・オーバーシュート）を満たすかを確認

**プラントモデルの例（建設機械の場合）**：
- 油圧シリンダモデル（Simscape Fluids / Amesim）
- エンジントルクモデル
- 車体・アーム慣性モデル

**使用ツール**：Simulink、Stateflow、Simscape Fluids（油圧）、Simulink Test

**メリット**：実機がなくてもアルゴリズム検討が可能。シミュレーションは実時間より速く回せるため、長時間動作や異常シナリオのテストが容易。

---

### SIL（Software-in-the-Loop）

**目的**：Simulinkモデルから自動生成したCコードをホストPC上で実行し、MILの結果と比較することで「モデルとコードの等価性」を確認する。

**実施内容**：
- Embedded Coderでコードを生成
- Simulink TestのSILモードで生成コードを実行
- MILとSILの出力信号を比較（数値的一致を確認）

**注意点**：MILとSILで結果が異なる場合、固定小数点演算の丸め誤差、データ型の違い、浮動小数点例外などが原因になることが多い。

**使用ツール**：Embedded Coder、Simulink Test、Polyspace（生成コードの静的解析）

---

### PIL（Processor-in-the-Loop）

**目的**：実際のターゲットプロセッサ（ECU上のマイコン）で生成コードを実行し、タイミング・割り込み・アーキテクチャ固有の動作を確認する。

**実施内容**：
- ターゲットマイコンにコードをフラッシュし、PC側のSimulinkと接続
- タスク実行時間の計測（最悪実行時間の確認）
- SILとの出力比較

**SILとの違い**：SILはx86 PC上での実行のため、ターゲットマイコン固有の整数オーバーフロー動作、エンディアン差異、演算精度の差を確認できない。PILはこれらを実際のハードウェアで確認する。

**使用ツール**：Embedded Coder（PIL対応ターゲット設定）、JTAG/デバッガ

---

### HIL（Hardware-in-the-Loop）

**目的**：実際のECUハードウェアを使い、制御対象（プラント）をリアルタイムシミュレータで模擬して統合検証を行う。

**実施内容**：
- HILシミュレータ（dSPACE SCALEXIO、NI VeriStand等）に油圧・エンジン等のプラントモデルを実装
- ECUのI/O（CAN信号、センサ電圧、アクチュエータ電流等）をHILシステムと物理接続
- 通常運転・フォールト注入・境界値テスト等を自動実行

**建設機械での用途例**：
- 油圧ポンプ制御ECUのHIL検証（ポンプ・シリンダモデルをHILで模擬）
- エンジンコントロールユニットのHIL検証
- CAN/J1939通信を含むシステム統合テスト

**使用ツール**：dSPACE SCALEXIO、Vector CANoe、NI VeriStand

**HIL環境のコスト**：機材・ソフトウェアで数千万円規模になることが多い（要確定）。外部委託先が保有している場合はその設備を活用する契約を検討する。

## 4. AUTOSARとMBDの統合

### SimulinkモデルからClassic AUTOSAR SWCを生成するワークフロー

```
Simulinkモデル（制御アルゴリズム）
        │
        │  AUTOSAR Blockset
        │  ・RTEポート（Rポート/Pポート）をブロックで定義
        │  ・送受信データ要素・クライアント/サーバーインタフェースを設定
        ▼
   ArXML出力 ＋ 生成Cコード
        │
        ├─── ArXML → DaVinci Configurator Pro（Vector）
        │             またはSystemDesk（dSPACE）でBSWと統合
        │
        └─── 生成Cコード → RTE生成コードと結合してビルド
```

### AUTOSAR Blocksetの役割

- SimulinkモデルのポートをAUTOSAR SWCのP/Rポートとして定義
- データ型・スケーリングをAUTOSAR Internal Behavior仕様に合わせて設定
- モデルからArXMLを直接エクスポートし、設定ツール（DaVinci等）でインポート可能
- Runnable（実行エンティティ）の周期・イベントトリガをSimulinkの関数呼び出しと対応付け

### TargetLinkとDaVinci Configurator Proの連携

TargetLink（dSPACE）はdSPACEのSystemDeskと連携してAUTOSAR対応コード生成を行う。

```
Simulinkモデル
    │
    │ TargetLink
    ▼
生成Cコード（AUTOSAR SWC実装）＋ ArXML
    │
    ▼
SystemDesk / DaVinci Configurator Pro
（BSW設定・RTE生成・統合ビルド）
```

### 生成コードのRTEとの接続

- 生成コードは `Rte_Read_<Port>_<Element>()` / `Rte_Write_<Port>_<Element>()` 等のRTE APIを呼び出す
- RTE側のAPIはBSW設定ツールが生成するため、両者のインタフェース定義（ArXML）を一致させる必要がある
- Simulinkモデルのデータ型とAUTOSAR基本型（`uint8`, `sint16`等）のマッピングが重要

### 課題：モデルとArXMLの整合性管理

- モデル変更（ポート追加・データ型変更）のたびにArXMLを再エクスポートしてBSW設定ツールに再インポートする作業が発生する
- 複数モデル・複数SWCが存在するシステムでは、ArXMLの変更管理をCIパイプラインに組み込むことが望ましい（要確定）
- AUTOSAR BlocksetのバージョンとターゲットとするAUTOSAR Schemaのバージョン（R4.x等）の組み合わせに注意が必要

## 5. 機能安全（ISO 26262 / ISO 25119）とMBD

### MBDが機能安全に有利な理由

| 機能安全要求 | 手書きコード | MBD |
|---|---|---|
| 要求↔設計のトレーサビリティ | 手動で文書化 | ツールが自動生成 |
| 設計レビュー | コードレビュー（属人的） | モデルレビュー（視覚的） |
| テストカバレッジ証跡 | 手動でテスト作成 | MIL/SILで自動集計 |
| 回帰テスト | テストコード保守が必要 | モデル変更後の再実行が容易 |
| コーディングガイドライン準拠 | Lintツール等で確認 | Polyspace・Model Advisorで一括チェック |

### 開発ツール資格（ISO 26262 Part 8、Annex B）

MBDツールは「開発ツール」として、その誤作動がソフトウェアの安全性に影響する可能性があるため、ツール資格（Tool Qualification）が必要になる。

- **TCL（Tool Confidence Level）1〜3**：TCL3が最も厳しい要件
- MathWorks Embedded Coder・SCADE KCGはTCL3相当の認証取得を支援するドキュメント・テストスイートを提供
- ツールベンダーが提供する「Qualification Support Package」を活用し、プロジェクト固有の資格付けプロセスを実施する

ISO 25119（農業・建設機械向け機能安全規格）はISO 26262をベースにした規格であり、MBDとツール資格に関する考え方は基本的に同じアプローチが適用できる。

### モデルカバレッジとコードカバレッジ

| 観点 | モデルカバレッジ | コードカバレッジ |
|---|---|---|
| 対象 | Simulinkモデルのブロック・遷移 | 生成Cコードの命令・分岐 |
| 取得タイミング | MIL | SIL/PIL/実機テスト |
| 必要性 | モデルの設計上の抜けを検出 | 実装の実行経路を保証 |

機能安全規格ではモデルカバレッジとコードカバレッジの両方が要求される。MBDであってもコードカバレッジを省略できるわけではない点に注意。

### MBD証跡のセット

機能安全の審査・認証で必要になる代表的な証跡：

- **モデル（バージョン管理済み）**：設計の根拠と実装の一体成果物
- **MILテストケースと結果レポート**：要求に対する動作の検証証跡
- **SILカバレッジレポート**：生成コードのMC/DC（Modified Condition/Decision Coverage）等
- **生成コードとコード生成設定**：再現性のある証跡として保管
- **Model Advisor / Polyspaceレポート**：ルール準拠確認の証跡
- **ツール資格付けドキュメント**：使用ツールの資格付け証跡

## 6. 建設機械でのMBD活用可能性

### 自動車業界でのMBD普及状況

自動車業界ではパワートレイン制御（エンジン制御・トランスミッション制御）でのMBD採用が1990年代後半から始まり、現在はADAS・ボディ制御・シャシー制御まで広く普及している。量産規模が大きく（年間数十万台〜数百万台）、開発投資を回収しやすいことが普及の背景にある。

### 建設機械での適用可能性

#### 油圧システムのモデリング

建設機械の核心は油圧制御である。Simulinkと組み合わせて使えるツールとして以下がある。

- **Simscape Fluids**（MathWorks）：Simulink統合の油圧・流体モデリングライブラリ。油圧ポンプ・バルブ・シリンダをブロックで接続してプラントモデルを構築できる
- **Amesim**（Siemens）：油圧・機械・電気の複合物理モデリングツール。航空・建設機械分野での採用実績あり。Simulinkとの共同シミュレーションも可能

油圧システムのプラントモデルをMBDで精度よく作れれば、MIL段階での制御アルゴリズム評価が大幅に強化される。ただしプラントモデルの作成・検証には相応の工数と油圧専門知識が必要。

#### 建設機械のMBD適合度評価

| 観点 | 評価 | 備考 |
|---|---|---|
| ECU数 | 適（少ない） | 1〜数個のECUを段階的にMBD化しやすい |
| 制御の複雑度 | 適（高い） | 油圧制御は非線形・干渉が多く、MBDの恩恵を受けやすい |
| 生産台数 | 課題（少ない） | 年間数百〜数千台規模ではライセンスコスト回収が困難 |
| 機能安全要件 | 適 | ISO 25119への対応でMBDの証跡生成能力が活きる |
| 社内スキル | 課題 | 制御工学・ツール操作の両方ができる人材が必要 |
| 外部委託との相性 | 条件次第 | 委託先がMBD環境を保有している場合は活用しやすい |

#### ライセンスコストと開発規模のバランス

Simulink + Embedded Coder + Simulink Test + AUTOSAR Blocksetの構成で年間数百万〜一千万円規模のライセンスコストが想定される（要確定）。年間生産台数が少ない建設機械では、このコストを機種あたりの開発費で吸収できるかが鍵になる。

外部委託先がMBD環境を保有している場合、自社でのライセンス取得を最小限にとどめ、モデルレビューと要求管理に集中するアプローチが現実的。

#### 外部委託でMBDを採用した場合の自社側スキル要件

| スキル | 必要理由 |
|---|---|
| Simulinkモデルの読解・レビュー | 委託先の設計物を評価するため |
| MIL/SILテストの理解 | 検証結果の妥当性を判断するため |
| AUTOSARインタフェース設計 | SWCのポート・インタフェース要求を出すため |
| 機能安全の知識（ISO 25119） | 委託先から提出される証跡を評価するため |
| 油圧制御の制御工学知識 | モデルの制御アルゴリズムをレビューするため |

### 「MBDを使う」vs「手書きコード＋AUTOSAR」の現実的な比較

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   手書きコード＋Classic AUTOSAR                           │
│   ・コード生成はAutosar SWCの骨格のみ（Vector等のツール）  │
│   ・制御ロジックはCで手書き                              │
│   ・低ライセンスコスト                                   │
│   ・委託先に幅広い選択肢                                 │
│   ・MBD固有のMIL/SILが使えない                          │
│                                                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│   MBD（Simulink等）＋Classic AUTOSAR                    │
│   ・制御ロジックもモデルから自動生成                     │
│   ・MIL/SIL/HILが揃った検証環境                         │
│   ・高ライセンスコスト                                   │
│   ・委託先のMBDスキルが前提                              │
│   ・ISOトレーサビリティが強化される                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**現時点での推奨判断**：新規開発の複雑な制御機能（例：自動掘削制御・過負荷防止制御）についてMBDを試行し、ROIを見極めるアプローチが現実的。既存のシンプルな制御機能についてはMBD化のコストに見合わない可能性がある。

### まず試せる範囲

以下のように段階的に試すことで、MBD適用の可否を低リスクで評価できる。

1. **PoC（1〜2ヶ月）**：新規制御機能の1つをSimulinkでモデリング。実機コードは既存の手書きコードを使いつつ、モデルはシミュレーションのみで活用
2. **SIL導入（追加1〜2ヶ月）**：Embedded Coderでコード生成し、既存コードと動作比較。コード品質・サイズを評価
3. **HIL統合（別途半年〜）**：HIL環境を外部委託先と連携して構築し、統合検証プロセスを確立
4. **全面適用の判断**：上記PoC結果を基にコスト・品質・納期への影響を定量評価し、経営判断材料とする

## 7. MBDの課題と注意点

### モデルの可読性・保守性問題

Simulinkモデルはブロック数が増えると「モデルのスパゲッティ化」が起きやすい。複雑な制御ロジックでは、Cコードよりモデルの方が理解しにくい場合がある。

対策：
- モデリングガイドライン（MAABガイドライン等）を最初から適用する
- サブシステムへの適切な階層化
- モデルレビューを設計プロセスに組み込む

### モデルとコードの二重管理リスク

生成コードとモデルのどちらかが先行して変更されると、両者が乖離する。必ずモデルを唯一の設計源泉（Single Source of Truth）とし、コードを直接修正するワークフローを禁止するルールが必要。

### ライセンスコスト・ツール依存のリスク

- 特定ベンダー（MathWorks等）への強いロックインが発生する
- ツールのバージョンアップに伴うモデルのマイグレーションコストが発生する
- ライセンス費用の長期的な増加リスクを考慮した予算計画が必要

### モデル設計者のスキル要件

MBDエンジニアには「制御工学の知識」と「Simulinkの操作スキル」の両方が必要。どちらか一方だけでは設計品質が低下する。育成に1〜2年規模の期間が必要なことが多く、短期間での内製移行は困難。

### 生成コードの効率性

自動生成コードは手書きコードに比べてコードサイズが1.5〜3倍程度大きくなる傾向がある（要確定、設定と対象制御の複雑度による）。RAM/Flash容量が厳しいマイコンターゲットでは、自動生成コードのメモリフットプリントを事前に評価することが重要。

### 外部委託時のモデル資産の帰属

委託先がモデルを作成した場合、モデル（知的財産）の帰属を契約で明確にする必要がある。モデルを自社資産として受け取れない契約では、委託先変更時に再開発が必要になる。
